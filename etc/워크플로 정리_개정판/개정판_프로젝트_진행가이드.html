AI 협업 환경 이용 가이드

AccountInterlockHub 워크스페이스의 AI 협업 환경을 담당자 시점에서 정리한 안내서입니다. "어디서 무엇을 요청하면 · 누가 어떤 순서로 처리해 · 무엇이 만들어지는가"를 빠르게 찾기 위한 용도입니다.

이 환경의 핵심 — 담당자는 에이전트 프롬프트를 직접 입력하지 않습니다. Slack 에서 마스터 세션 ai-pm 과 대화하면, ai-pm 이 필요한 작업을 판단해 하위 서브에이전트를 자동으로 기동합니다.
이 문서 사용법
• 처음이라면 → 좌측 개요의 협업 환경 기본 구성부터 읽어 전체 그림을 잡으세요.
• 바로 시작하려면 → 담당자가 하는 일 (Slack) 절만 봐도 됩니다.
• 내부 동작이 궁금하면 → ② 사양 · ③ 생산 · ④ 검증 파이프라인의 에이전트 카드를 확인하세요.
• 상단 검색창에 이름·역할 키워드를 넣으면 해당 카드만 필터링됩니다.

0협업 환경 기본 구성

작업은 사람 담당자와 AI가 Slack에서 채팅으로 협업하고, 실제 작업은 서브에이전트가 수행하며, 그 기록은 Redmine 일감에 남는 구조입니다. 세 축의 역할이 다릅니다.

축무엇역할
Slack프로젝트 워크스페이스·채널담당자 ↔ ai-pm 의 실시간 대화 통로. 요청·질의·승인·보고가 오갑니다.
ai-pm마스터 세션(봇)채널 대화에서 할 일을 식별해 서브에이전트로 디스패치하고 결과를 다시 Slack 으로 알리는 얇은 오케스트레이터. 직접 작업은 하지 않습니다.
Redmine이슈 트래커작업 수행 기록의 정본. 명세·진행 노트·상태·결과가 일감에 쌓입니다(대화는 휘발성, 이슈는 영속).

협업 흐름 한 사이클

[담당자] ──(Slack 채널/DM 에 요청·멘션)──▶ [ai-pm 마스터 세션]
                                             │  ① 처리 대상 인식 → 스레드에 '접수' 피드백 게시
                                             │  ② 관련 Redmine 일감 확인/생성
                                             ▼
                                      [서브에이전트] ──작업 수행──▶ 산출물(docs/·apps/) + Redmine 일감 갱신
                                             │
                   (질의·승인 필요 시) ◀── 중간 보고 ── │
  [담당자] ◀──(ai-pm 이 Slack 으로 릴레이)─────────────┘
                                             │  ③ 완료 보고 → ai-pm 이 담당자·채널에 Slack 통지
                                             ▼
                                      (담당자 승인 시) 단계 브랜치를 main 에 병합

꼭 알아둘 원칙 5가지

  • 담당자는 Slack 으로만 지시합니다. 에이전트 이름·프롬프트를 외울 필요 없이 자연어로 요청하면 ai-pm 이 어느 단계·에이전트로 처리할지 판단합니다.
  • 4단계는 자동으로 이어지지 않습니다. directing → spec → build → qa 는 각각 독립 완결 프로세스이며, 다음 단계로 넘어가려면 담당자 승인이 필요합니다(수동 이행).
  • 진행 상황은 하나의 피드백 메시지로 보입니다. 요청 1건마다 ai-pm 이 접수 → 작업 시작 → 진행 중 → 종료로 같은 메시지를 갱신합니다.
  • 질의·승인은 ai-pm 이 릴레이합니다. 서브에이전트는 백그라운드라 Slack 에 직접 닿지 못하므로, 판단이 필요하면 멈추고 ai-pm 이 담당자에게 물어 답을 전달합니다.
  • 정본은 Redmine·저장소 파일입니다. Slack 대화는 통로일 뿐, 증적은 Redmine 일감과 저장소 산출물에 남습니다.

◎전체 워크플로

각 단계는 담당자의 Slack 요청으로 착수되고, ai-pm 이 그 단계 오케스트레이터를 서브에이전트로 디스패치하며, 오케스트레이터가 다시 여러 doer를 순서대로 호출합니다(directing 은 ai-pm 이 직접 수행). 단계 사이 화살표는 자동 연쇄가 아니며 담당자 승인으로만 이행합니다.

☰에이전트 전체 지도

"호출 주체"는 그 에이전트를 누가 부르는지입니다 — 담당자(Slack)는 사람이 대화로, ai-pm·오케스트레이터는 상위가 자동 호출합니다. 이름을 누르면 상세 카드로 이동합니다.

구분이름종류모델호출 주체한 줄 역할

계획·판정·분석 역할은 정확도를 위해 opus·effort max, 확정 사양 기반의 정형 작성·실행은 sonnet·effort xhigh 를 씁니다. 마스터 ai-pm 은 세션 구성 모델(inherit)·max 로 기동합니다.

▶담당자가 하는 일 (Slack 기반 협업)

담당자의 실제 조작은 Slack 채팅이 전부입니다. 진입 프롬프트를 외울 필요 없이 아래 네 가지 상황에서 자연어로 대화합니다.

1) 작업 요청·착수

해당 작업 채널에 요청을 공유하거나 ai-pm 을 멘션/DM 합니다. 어느 단계인지 몰라도 ai-pm 이 판단합니다.

@ai-pm  spec 단계 착수해줘. 방향 정의서 최신본 기준으로
계정 연동 도메인 사양을 정리해줘.
@ai-pm  로그인 화면 목업을 먼저 만들어서 확인하고 싶어.

2) 질의에 응답 (검토 협의)

서브에이전트가 모호·비가역 판단으로 멈추면 ai-pm 이 질의를 Slack 으로 전달합니다. 담당자는 스레드에 그대로 답하면 되고, ai-pm 이 답변을 일감 노트에 기록해 작업을 재디스패치(재개)합니다.

3) 승인 (단계 이행·목업 등)

단계 완료·목업 확정처럼 승인 게이트가 오면 ai-pm 이 승인을 요청합니다. 담당자가 승인하면 해당 단계 브랜치가 main 에 병합되고 다음 단계에 착수할 수 있습니다.

4) 진행 현황 확인·세션 관리

진행 현황은 Redmine 프로젝트(accountinterlockhub) 이슈 목록·상태로 직접 조회합니다. 세션을 새로 시작하려면 Slack 에 ai-pm 초기화 를 입력합니다.

directing 단계는 별도 서브에이전트 없이 ai-pm 이 담당자와 직접 대화로 진행합니다 — 방향 정리 후 방향 정의서·PRD·IA 로 산출됩니다.

★마스터 세션

모든 협업의 창구입니다. 담당자의 Slack 대화를 받아 작업을 서브에이전트로 디스패치합니다.

②사양 분석 파이프라인 (spec)

방향 정의서·PRD → 도메인별 상세 사양서로 전환합니다. spec 오케스트레이터가 아래 흐름 안에서 doer를 순서대로 호출합니다. 상세: 사양분석-파이프라인.md

spec 오케스트레이터
 ├ 1  (계획)         브랜치·작업 범위(IA 노드) 확정
 ├ 2  prd-reviewer   요구사항 정합성 게이트 (치명 0일 때만 진입)
 ├ 3  도메인 순차     policies → service → datas → functions → screens → process → qa
 ├ 4  spec-reviewer  도메인 간 교차검증
 ├ 5  mockup-builder 목업 → 담당자 승인(ai-pm 릴레이)
 └ 6  (정리)         사양 일감 동기화·완료 보고(build 착수 승인 요청)

③제품 생산 파이프라인 (build)

확정 사양서 → 제품 코드 + 배포 산출물로 전환합니다. 무게중심은 BE 로직이며 FE 는 승인된 목업을 청사진으로 삼습니다. 상세: 제품생산-파이프라인.md

build 오케스트레이터
 ├ 1  (계획)          브랜치·구현 항목을 Phase로 편성·담당 doer 지정
 ├ 2  Phase별 (순차)  2-A developer(작성) → 2-B code-reviewer(리뷰) → 2-C tester(검증)
 │                    → 세 책임 통과 시 Phase ✅ · 기능 일감 동기화
 ├ 3  build-installer 배포 산출물 빌드
 ├ 4  tester          런타임 검증 게이트 (배포 산출물 = 사용자 동일 환경)
 └ 5  (정리)          완료 보고(qa 착수 승인 요청)

④품질 검증 파이프라인 (qa)

배포 산출물 → 품질검증 보고(배포 가능 여부 평가)를 만듭니다. 배포 가능 판정은 담당자 승인 게이트입니다. 상세: 품질검증-파이프라인.md

qa 오케스트레이터
 ├ 0  (환경 구성)   배포 산출물 설치·기동 → 사용자 동일 환경·기준선 확보
 ├ 1  test-planner  검증 계획 — TC를 Phase로 배정(1 TC = 1 Phase)
 ├ 2  tester        Phase별 TC 실행 · 5종 판정 · 실행 즉시 검증 일감 갱신(Pass→해결)
 ├ 3  (결과 평가)   누락 대조 + 품질 게이트 · 담당자별 그룹 보고
 └ 4  (정리)        검증 일감 동기화 확인 · 완료 보고

◷활용 사례 · 실제 Slack 채널 타임라인

이 워크스페이스의 account-interlock-hub Slack 채널에서 실제로 오간 담당자 요청과 ai-pm 실행 결과입니다. directing → spec → build → qa 전 단계가 어떻게 순환했는지 시간순으로 보여줍니다.

* 세션 재기동·진단 등 운영 메시지는 생략하고 워크플로 이정표만 추렸습니다. 민감정보(자격·키)는 대화에 노출되지 않았습니다.

§공통 규약 및 주의사항

  • 단계는 수동 이행 — 각 단계는 독립 완결 프로세스이며 담당자 승인으로만 다음 단계에 착수합니다(자동 연쇄 없음).
  • 책임 분리 — 한 단계 안에서 작성·리뷰·검증을 서로 다른 주체가 맡습니다. 작성 주체가 자기 산출을 합격 판정·검증하지 않습니다.
  • 실 동작 검증 의무 — 정적 검증(컴파일·타입 체크)만으로 완료 처리하지 않고, 실제 실행해 저장→조회 왕복(round-trip)까지 확인합니다.
  • 회귀 상한 — 검증 결과가 기준 미달이면 앞 단계로 회귀해 재시도하되 최대 3회, 시도 횟수는 Redmine 일감 노트로 기록합니다.
  • 정본 분리 — 대화는 Slack(휘발성), 작업 기록은 Redmine 일감, 산출물은 저장소 파일(docs/·apps/).
  • 민감정보 보호 — 계정·키 등 원문(CLAUDE.local.md 변수)은 리포트·커밋·Slack 어디에도 노출하지 않습니다.
  • etc/ 는 사람 참고 전용 — 이 폴더를 포함한 etc/ 하위는 사람 참고용입니다. 정책·산출물의 단일 출처는 항상 etc/ 바깥(ai/·루트 config)이며, 본 가이드가 정본과 어긋나면 정본이 우선합니다.

📖용어 설명

ai-pm프로젝트 Slack 워크스페이스를 감시하며 작업을 서브에이전트로 디스패치하는 마스터 세션(봇). 직접 작업하지 않는 얇은 오케스트레이터.
오케스트레이터 / doer오케스트레이터는 한 단계의 흐름을 조율하는 서브에이전트(spec·build·qa), doer 는 그 흐름 안에서 실제 산출을 만드는 실행 단위.
단계(stage)업무를 나누는 4구간 — directing·spec·build·qa.
Phase한 단계·브랜치 안의 세부 실행 단위(= commit 1건). build 는 구현 항목 1건, qa 는 검증 TC 1건이 1 Phase.
Redmine 일감(이슈)작업 수행 기록의 정본. 트래커(사양·기능·검증·오류 등)와 상태(신규→진행→해결→완료, 보류·거절)로 관리.
PRD / 사양서(스펙)PRD 는 제품 요구사항 문서(docs/prd/), 사양서는 이를 개발 가능한 수준으로 상세화한 문서(docs/specs/).
IA정보 구조(Information Architecture). 제품 기능 단위까지의 좌표 체계(ia-code)로, 작업 범위·이력의 기준.
5종 판정검증 결과 등급 — 🟢 Pass / 🔵 Pass-Mock / 🟣 Pass-Static / 🔴 Fail / 🟠 Block.
round-trip(왕복 검증)저장한 값을 다시 조회했을 때 그대로 나오는지 실 환경에서 확인하는 검증.
질의·승인 릴레이서브에이전트가 판단이 필요할 때 멈추고, ai-pm 이 담당자에게 Slack 으로 물어 답을 전달하는 절차.
git-flow단계 착수=브랜치, Phase=commit, 담당자 승인=main 병합(no-ff)의 저장소 운용 규칙.
