# 사양 분석 파이프라인 (spec 단계)

**spec 오케스트레이터**가 자동으로 호출하는 하위 doer 9종의 실행 시점·입력·결과 정리다.
상위 문서: [AI 협업 환경 이용 가이드](AI%20협업%20환경%20이용%20가이드.md)

이 파이프라인은 **방향 정의서·PRD → 도메인별 상세 사양서**로 전환한다.
담당자는 doer를 직접 부르지 않는다 — ai-pm 이 spec 오케스트레이터를 디스패치하면, 오케스트레이터가 아래 흐름 안에서 doer를 순서대로 호출한다.

```
ai-pm ──디스패치──▶ spec 오케스트레이터
 ├ 1단계  (계획)            브랜치 생성·작업 범위(IA 노드) 확정
 ├ 2단계  prd-reviewer      요구사항 정합성 게이트 (치명 0일 때만 진입)
 ├ 3단계  도메인 정의 순차   prd-to-policies → service → datas → functions → screens → process → qa
 ├ 4단계  spec-reviewer     도메인 간 교차검증 (치명·오류 0까지 회귀)
 ├ 5단계  mockup-builder    목업 생성 → 담당자 승인(ai-pm 릴레이)
 └ 6단계  (정리)            사양 일감 상태 동기화·완료 보고(build 착수 승인 요청)
```

> 공통 입력 규약: 오케스트레이터가 각 doer에 **작업 범위(IA 노드)·참조 사양·관련 `사양` 일감**을 전달한다. 원본 정본은 `docs/specs/`, Redmine `사양` 일감은 그 사본·추적이다. 상세 흐름 정본은 [`stages/spec.md`](ai/strategies/stages/spec.md).

## 오케스트레이터

### spec (`ai/agents/spec.md`)
- **모델 / 호출 주체**: opus·max / ai-pm
- **실행 시점**: 담당자가 Slack 으로 spec 단계 착수를 요청하면 ai-pm 이 디스패치한다.
- **입력 예시(ai-pm 디스패치 맥락)**: `작업 범위: 계정 연동 도메인(IA 노드) / 관련 일감: 사양 그룹 일감 / 요청 채널·내용`
- **실행 결과**: 브랜치(`spec/<이슈번호>-<주제>`) 생성, 아래 doer들을 순차 호출해 `docs/specs/` 사양서를 산출하고, 정리 후 ai-pm 에 완료(build 착수 승인 요청 포함)를 보고한다. main 병합은 담당자 승인 후 별도 수행.
- 원본: [spec.md](ai/agents/spec.md)

## 검토 (2종)

### prd-reviewer (`ai/agents/workflow-prd-review/prd-reviewer.md`)
- **모델 / 호출 주체**: opus·max / spec (2단계 게이트)
- **실행 시점**: 도메인 사양을 만들기 전에 요구사항과 기존 사양서의 정합성을 먼저 평가한다. **치명 결함 0** 일 때만 3단계로 진입한다.
- **실행 결과**: 사양서 간 누락·충돌·중복이 위험도 등급으로 정리된다. 치명 결함이 있으면 오케스트레이터가 진입하지 않고 일감을 `보류` 로 바꾼 뒤 중간 보고로 종료한다(directing 회귀 여부는 담당자 응답으로 결정).
- 원본: [prd-reviewer.md](ai/agents/workflow-prd-review/prd-reviewer.md)

### spec-reviewer (`ai/agents/workflow-prd-to-spec/spec-reviewer.md`)
- **모델 / 호출 주체**: opus·max / spec (4단계 교차검증)
- **실행 시점**: 모든 도메인 정의가 끝난 뒤 전체 모드로 호출되어 도메인 간 정합성(깨진 참조·미반영)을 검증한다.
- **실행 결과**: 도메인 교차 정합성 평가 결과가 나오고, 치명·오류가 남으면 해당 도메인 doer로 회귀해 보완한다(최대 3회).
- 원본: [spec-reviewer.md](ai/agents/workflow-prd-to-spec/spec-reviewer.md)

## 도메인별 사양 정의 (7종)

3단계에서 오케스트레이터가 **정책 → 서비스 → 데이터 → 기능·API → 화면 → 프로세스 → 검증 TC** 순서로 1개씩 순차 호출한다. 뒤 도메인이 앞 도메인 산출물을 참조하므로 순서가 고정되어 있다. 각 doer는 산출물을 해당 경로에 쓰고 자기 `사양` 일감을 등록·갱신한다. (범위에 해당 도메인 변경이 없으면 생략)

### prd-to-policies (`ai/agents/workflow-prd-to-spec/prd-to-policies.md`)
- **모델 / 호출 주체**: opus·max / spec
- **실행 시점**: 도메인 정의의 첫 단계 — 전체에 적용될 정책을 먼저 정의한다.
- **실행 결과**: 접근 제어·데이터·비즈니스·보안 정책이 `docs/specs/policies/` 에 생성된다.

### prd-to-service (`ai/agents/workflow-prd-to-spec/prd-to-service.md`)
- **모델 / 호출 주체**: opus·max / spec
- **실행 시점**: 정책 정의 후 실행되어 사용자 시나리오 중심의 서비스 기능을 정의한다.
- **실행 결과**: 서비스 기능 정의서가 `docs/specs/services/` 에 생성된다.

### prd-to-datas (`ai/agents/workflow-prd-to-spec/prd-to-datas.md`)
- **모델 / 호출 주체**: opus·max / spec
- **실행 시점**: 서비스 정의 후 실행되어 저장 구조(DB 엔터티·데이터 모델)를 설계한다.
- **실행 결과**: 데이터·DB 설계 정의서가 `docs/specs/datas/` 에 생성된다.

### prd-to-functions (`ai/agents/workflow-prd-to-spec/prd-to-functions.md`)
- **모델 / 호출 주체**: opus·max / spec
- **실행 시점**: 데이터 정의 후 실행되어 재사용 로직과 API 인터페이스를 정의한다.
- **실행 결과**: 공통 기능·API 정의서가 `docs/specs/functions/` 에 생성된다.

### prd-to-screens (`ai/agents/workflow-prd-to-spec/prd-to-screens.md`)
- **모델 / 호출 주체**: opus·max / spec
- **실행 시점**: 기능 정의 후 실행되어 사용자 화면과 디자인 시스템을 정의한다.
- **실행 결과**: 화면 정의서·디자인 시스템이 `docs/specs/screens/` 에 생성된다.

### prd-to-process (`ai/agents/workflow-prd-to-spec/prd-to-process.md`)
- **모델 / 호출 주체**: opus·max / spec
- **실행 시점**: 화면 정의 후 실행되어 각 기능의 실행 흐름을 의사코드 수준으로 통합한다.
- **실행 결과**: 프로세스 정의서(FE/BE 의사코드 포함)가 `docs/specs/processes/` 에 생성된다.

### prd-to-qa (`ai/agents/workflow-prd-to-spec/prd-to-qa.md`)
- **모델 / 호출 주체**: opus·max / spec
- **실행 시점**: 도메인 정의의 마지막 — 기획서 품질을 검증하고 검증 TC를 설계한다.
- **실행 결과**: 기획서 검증 결과와 검증 TC 정의서가 `docs/specs/qa/` 에 생성되고, 사양 미해소 항목이 후속 qa 단계로 전파된다.

## 마무리 (1종)

### mockup-builder (`ai/agents/workflow-mockup/mockup-builder.md`)
- **모델 / 호출 주체**: sonnet·xhigh / spec (5단계 마무리)
- **실행 시점**: 교차검증 후 확정 화면 사양을 기반으로 경량 목업을 만들어 담당자 리뷰로 사양을 검증한다.
- **실행 결과**: 목업이 생성되고, 담당자 승인 요청이 **ai-pm 릴레이**로 전달된다(일감 `해결` + 중간 보고 종료 → 승인 기록된 재디스패치에서 재개). 리뷰 보완은 사양 정의로 회귀해 반영한다(목업 = 사양 검증 도구).
- 원본: [mockup-builder.md](ai/agents/workflow-mockup/mockup-builder.md)
