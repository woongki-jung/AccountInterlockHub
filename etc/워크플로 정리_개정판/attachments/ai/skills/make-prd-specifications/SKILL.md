## 사용 상황

- 요구사항(`docs/prd/requirements/`) 정의가 완료되었고 개발 사양(`docs/prd/devspec/`) 이 비어 있을 때 (**신규 모드**)
- 기존 작성된 개발 사양 문서의 갱신을 수행할 때 (**갱신 모드**) — 요구사항 변경에 따른 후속, 기술 스택 변경, 외부 API 추가/교체, 인프라 재설계 등
- 후속 `prd-to-datas` / `prd-to-functions` 등 사양 분석 에이전트가 사용할 입력 문서를 준비할 때

## 산출 문서

| 산출 문서 | 경로 | 역할 |
|---|---|---|
| DB 설계 기반 | `docs/prd/devspec/database.md` | 데이터 엔티티별 출처·민감도·규모·일관성 요건 (엔티티별 `D<code>` 절) |
| 외부 API | `docs/prd/devspec/external-apis.md` | 연동 대상 외부 시스템별 인증·SLA·데이터 형식·장애 대응 (API 별 `A<code>` 절) |
| 인프라 설계 | `docs/prd/devspec/infra.md` | 배포 환경·트래픽·가용성·관측성·보안 (영역별 `I<code>` 절) |
| 기술 스택·프로그램 구성 | `docs/prd/PRD.md` §프로그램 구성표 | 클라이언트 형태·프로그램별 기술 선택·검증 방법·도입 제약 |

개발사양 문서는 필요 항목만 작성하고 해당 없으면 생략한다. 저장 경로 전체 트리는 [`ai/strategies/doc-structure.md`](../../strategies/doc-structure.md) 참조.

## 대화 원칙 (모든 단계 공통)

- **추측 금지** — 담당자가 명시하지 않은 결정은 반드시 질문. 단, 한 번에 묻는 질문 수는 최대 3개.
- 모호함 발견 시 → 옵션 2~3개 제시 + 권장안 1개 명시 → 담당자 선택.
- 기술 결정은 **제약 조건(팀 자산·예산·규제) → 옵션 → 권장안** 순으로 진행. 권장안 단독 제시 금지.
- 각 산출 문서 작성 직후 담당자 검토 1회 → 수정 반영 → 다음 단계 진행 합의 확인.
- 담당자가 기술 비전공자인 경우 옵션을 "선택 결과로 무엇이 달라지는가" 의 비즈니스 영향(비용·시간·확장성) 으로 환산하여 설명.

## 실행 동작

### 1. 모드 식별 및 사전 준비

작업 시작 시 다음 두 항목으로 진입 가능 여부와 모드를 판단:
- **선행 조건** — `docs/prd/requirements/requirements.md` 존재 + 하위 4개 카테고리 작성됨. 없으면 [`ai/skills/make-prd-requirements/SKILL.md`](../make-prd-requirements/SKILL.md) 선행 권고 후 중단.
- **모드 추정** — `docs/prd/devspec/` 문서 존재 여부로 신규/갱신 분기. 담당자 확인 후 §1.1 또는 §1.2 진입.

#### 1.1 신규 모드 — 사전 자료 수집

요구사항 문서 일괄 로딩 + 담당자에게 다음을 묻고 자료 우선 분석.

- 연동 예정 외부 시스템 문서(URL·PDF·Postman 컬렉션) 보유 여부
- 기존 팀 보유 인프라·코드 자산 (재사용 후보)
- 회사·프로젝트 차원의 기술 표준·금지 항목 (라이선스·보안 정책)

**활용 외부 스킬**:
- 외부 API 문서 URL → `WebFetch` (Claude Code 내장) 로 가져와 분석.
- 외부 API 문서 PDF → [`ai/skills/opendataloader-pdf/SKILL.md`](../opendataloader-pdf/SKILL.md) (워크스페이스 등록) 로 마크다운 변환.
- 시장 동향·기술 비교 조사 → `WebSearch` (Claude Code 내장).
- Google Drive 공유 자료 → `Google Drive` MCP (Anthropic 공식).

#### 1.2 갱신 모드 — 변경 범위 식별

1. 기존 사양 일괄 로딩: `docs/prd/devspec/` 문서 전수 + `PRD.md` §프로그램 구성표.
2. 담당자에게 갱신 트리거 질문:
   - 요구사항 변경 후속 / 신규 외부 API 도입 / 기술 스택 교체 / 인프라 재설계 / 결함·운영 이슈 중 어디인가
   - 트리거 발생 사실 1~2줄 (예: "트래픽 10배 증가 예상", "벤더 A → B 교체")
3. 변경 범위 매트릭스(분류 / 영향 카테고리 / 요약) 작성 후 담당자 확인.
4. 영향 카테고리에 해당하는 §2~§5 단계만 실행. 연쇄 영향은 §9.3 매트릭스로 점검.

### 2. DB 설계 기반 정보 (→ `devspec/database.md`)

**대화 흐름**:

1. `requirements/ia/` 의 엔티티를 데이터 후보로 변환. 누락·중복·합치기 여부를 담당자에게 제시.
2. 각 엔티티별 다음 질문(3개씩 묶어 송신):
   - 출처: 신규 생성 / 외부 시스템 인입 / 사용자 직접 입력
   - 보관 기간: 영구 / N년 / 임시(세션) / 캐시
   - 민감도: 일반 / PII(개인정보) / 결제·금융 / 의료 등 — **규제 영향 반드시 묻기**
3. 추가 질문:
   - 사용 패턴: 읽기 위주 / 쓰기 빈번 / 분석 위주
   - 규모 예상: 일 신규 행 수 / 누적 행 수 (근거 있으면 함께, 추측이면 추측 명시)
   - 일관성: 강한 일관성 필요 / 결과적 일관성 허용
4. 엔티티별 절 `D<code>`(D1 / D2... 형태) 로 `database.md` 에 기록하고 문서 서두 목차를 갱신.

**수집 피드백**:
- 민감 데이터 식별 누락 없는지 (PII·결제·의료는 §4 인프라의 보관 위치·암호화 결정에 영향)
- 규모 예측이 명백히 비현실적이면 단계별 가설(MVP / 1년 / 3년) 로 분리 제안

### 3. 외부 API 정의 (→ `devspec/external-apis.md`)

**대화 흐름**:

1. `requirements/services/` 에서 외부 시스템 의존 가능한 기능 식별 후 담당자에게 연동 대상 목록 확인.
2. 각 외부 API 별 다음 질문:
   - 사용 목적 (어떤 기능에서 어떤 호출)
   - 인증 방식 (OAuth / API Key / JWT / mTLS / 비공개 IP)
   - 가용성·Rate Limit·비용 (벤더 공식 자료 기준)
3. 추가 질문:
   - 데이터 형식 (REST / GraphQL / SOAP / 파일 / WebSocket / Webhook)
   - 장애·오류 시 대응 (fallback / 캐시 / 큐 / 사용자 에러 표시)
4. 벤더 문서가 있으면 §1.1 의 외부 스킬로 가져와 endpoint·request/response 형식을 해당 API 절에 요약.
5. API 별 절 `A<code>`(A1 / A2... 형태) 로 `external-apis.md` 에 기록하고 문서 서두 목차를 갱신.

**수집 피드백**:
- 인증 자격증명의 보관 위치·교체 절차가 명확한지
- 장애 시 사용자 경험 정의가 있는지 (없으면 §4 인프라 단계로 미루지 말고 본 단계에서 결정)

### 4. 인프라 설계 (→ `devspec/infra.md`)

**대화 흐름**:

1. 배포 환경 질문 (3개 묶음):
   - 배포 위치: 클라우드(AWS/GCP/Azure/...) / 온프레미스 / 하이브리드
   - 트래픽 예상 규모: 동시 사용자·일 요청 수 (불명확하면 MVP/1년/3년 단계별 가설)
   - 가용성 SLA: 99% / 99.9% / 99.99% (높을수록 비용 비선형 증가 안내)
2. 데이터·규제 질문:
   - 서비스 지역: 글로벌 / 단일 국가 / 사내망
   - 데이터 보관 위치 (§2 의 민감 데이터와 정합 확인)
   - 백업·복구 정책 (RPO·RTO)
3. 운영 질문:
   - 모니터링·관측성 도구 선호 (자체 / 클라우드 매니지드 / 오픈소스)
   - 보안 정책 (네트워크 격리 / 암호화 / 접근 통제)
4. 위 답변을 바탕으로 인프라 아키텍처 초안을 mermaid 다이어그램으로 작성, 복잡 시 `claude.ai Figma` MCP (`mcp__claude_ai_Figma__generate_diagram`) 로 FigJam 다이어그램 생성.
5. 영역별 절 `I<code>`(I1: 배포 / I2: 데이터 보관 / I3: 모니터링 / I4: 보안 등) 로 `infra.md` 에 기록. 문서 서두에 목차 + 아키텍처 다이어그램 첨부.

**수집 피드백**:
- 트래픽·SLA 요구와 비용 현실성이 부합하는지 (벗어나면 권장안 재제시)
- 데이터 보관 위치와 §2 민감도가 정합하는지

### 5. 기술 스택 정의 (→ `PRD.md` §프로그램 구성표)

**대화 흐름**:

1. 제약 조건 먼저 확인 (3개 질문):
   - 팀의 기존 기술 자산 (현재 익숙한 언어·프레임워크·도구)
   - 도입 제한 (라이선스·보안·예산·내부 표준)
   - 학습 곡선 허용 범위 (새 기술 도입 자유도)
2. 클라이언트 형태 결정: Web / Mobile native / Hybrid / Desktop / CLI / 혼합
3. 각 계층별 옵션 + 권장안 제시 (담당자 선택):
   - Frontend: 프레임워크·빌드 도구·상태관리·스타일링
   - Backend: 언어·프레임워크·런타임
   - DB: §2 데이터 특성에 맞춰 RDBMS / NoSQL / 검색엔진 / 캐시 조합 권장
   - 인증: 자체 구현 / 외부 IdP (OAuth/OIDC/SAML)
   - 빌드·배포: CI·CD 도구·컨테이너·IaC
4. 각 계층 결정(T1: FE / T2: BE / T3: DB / T4: 인증 / T5: 빌드·배포 등)을 `PRD.md` §프로그램 구성표와 그 부속 서술로 기록하고 전체 스택 요약을 덧붙인다.

**범위 명시**: 본 스킬은 **계층별 기술 선택까지** 책임. 세부 라이브러리·버전·디렉터리 구조는 후속 spec 단계에 위임.

**수집 피드백**:
- 선택 스택이 §2 데이터 패턴·§4 인프라와 정합하는지 (예: 강한 일관성 요구 + NoSQL 단독 = 충돌)
- 팀 자산을 적극 활용하는 선택인지 (새 기술 비중이 과도하면 학습 비용 안내)

### 6. 개발사양 묶음 정리

§2~§5 완료 후 정리:
- 각 devspec 문서 서두에 한 줄 설명 + 절 목차
- 카테고리 간 의존 관계 1줄 요약 (예: "T3 DB 선택은 D1~Dn 의 일관성 요건에 따른다")
- 변경 이력 (최초 작성일 + 담당자)

### 7. PRD 통합 갱신

`PRD.md` 의 "개발 사양" 섹션에 `docs/prd/devspec/` 문서 목록 링크 추가. PRD 본문은 사양 정의 과정에서 발견된 범위 변경(예: 기술 제약으로 인한 기능 폐기)만 반영.

### 8. 후속 단계 안내

작성 완료 후 담당자에게 다음 안내:

- 본 산출물은 spec 오케스트레이터 [`ai/agents/spec.md`](../../agents/spec.md) 의 입력으로 사용된다 (`prd-to-datas` / `prd-to-functions` 등이 본 사양을 참조).
- 갱신 모드인 경우 §9.4 의 다운스트림 영향 안내를 반드시 함께 전달.
- 사양 분석 작업을 계획적으로 분해하려면 [`ai/skills/writing-plans/SKILL.md`](../writing-plans/SKILL.md) (워크스페이스 등록) 활용 가능.

### 9. 갱신 모드 세부 지침

§1.2 에서 갱신 모드로 분기한 경우 §2~§8 실행 시 본 절 규칙을 함께 적용.

#### 9.1 변경 분류 (수정 / 추가 / 폐기)

| 분류 | 처리 |
|---|---|
| 수정 | 기존 코드(D1·A1·I1·T1 등) 유지. 본문 갱신 + 변경 로그 항목 추가 |
| 추가 | 다음 비어있는 코드 부여. 신규 문서 생성 |
| 폐기 | 본문에 `status: deprecated` + 폐기 사유·일자·대체 항목 기재. 파일 삭제 금지 |

#### 9.2 코드(식별자) 유지 원칙

- 의미가 본질적으로 동일하면 코드 유지 (벤더 교체로 인한 API 변경 = 동일 의미면 수정, 다른 의미면 새 코드 + 기존 폐기).
- 코드 재활용 금지 (이력 추적 보존).

#### 9.3 연쇄 영향 규칙

| 1차 변경 카테고리 | 점검할 후속 카테고리 | 점검 포인트 |
|---|---|---|
| data | tech-stack / infrastructure | DB 종류·보관 위치가 데이터 특성 변화와 정합? |
| apis | infrastructure / tech-stack | 신규 외부 의존이 네트워크 격리·인증 방식·SDK 추가를 요구? |
| infrastructure | tech-stack / data | 배포 환경 변경이 런타임·DB 호환성에 영향? |
| tech-stack | data / infrastructure | 계층 변경이 DB·인프라 선택을 재검토하게 만드는가? |
| requirements(외부 입력) | 전체 카테고리 | 신규 기능·페르소나가 신규 엔티티·API·인프라 항목을 요구? |

연쇄 영향이 식별되면 §1.2 매트릭스에 카테고리를 추가하고 담당자 재확인.

#### 9.4 다운스트림 영향 평가

본 사양 변경은 이미 작성된 상세 사양(`docs/specs/`) 에 영향을 줄 수 있다. §8 안내 직전에 점검.

1. 변경된 사양 항목별로 `docs/specs/` 하위 직간접 참조 검색.
2. 영향 받는 사양 문서 목록을 작성하여 담당자에게 보고 (폐기·수정·신규 후보 분류).
3. 후속 작업 안내:
   - 사양 정합성 검증 → `spec-reviewer` 에이전트 ([`ai/agents/workflow-prd-to-spec/spec-reviewer.md`](../../agents/workflow-prd-to-spec/spec-reviewer.md))
   - 전체 재실행 → spec 오케스트레이터 ([`ai/agents/spec.md`](../../agents/spec.md))

#### 9.5 변경 이력 기록 의무

- 갱신된 각 문서 하단 "변경 로그" 에 `<날짜> | <변경 종류> | <변경 요약> | <담당자>` 항목 추가.
- 사이클 종료 시 §1.2 매트릭스 + §9.4 다운스트림 목록을 관련 Redmine 일감에 기록한다.
