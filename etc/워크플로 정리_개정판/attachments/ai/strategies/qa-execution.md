# 품질 검증 수행 전략

본 문서는 검증 활동의 **공통 거버넌스**와, 프로젝트 구성에 따라 **케이스별 하위 지침을 선택**하는 모델을 정의한다. 단계 흐름은 [`stages/qa.md`](stages/qa.md) 가 정본이고, 본 문서는 "어떻게 검증하는가"의 공통 기준과 케이스 선택을 담당한다. 케이스별 최적화 동작은 하위 지침(`qa-execution/`)에 위임한다.

적용 범위: qa 단계의 검증과 build 단계의 기능검증([`stages/build.md`](stages/build.md))이 본 전략의 케이스별 도구·절차를 따른다. 검증 대상 환경은 단계별로 다르다 — qa 단계와 build 의 최종 런타임 게이트는 **배포 산출물(사용자 동일 환경)** 을 쓰고, build 의 Phase별 기능검증은 그 시점의 실행 가능한 직접 빌드 산출물로 수행할 수 있다.

## 공통 원칙

- **사용자 동일 환경** — 워크스페이스 직접 기동이 아니라 배포 가능한 산출물을 검증 대상으로 한다([`stages/qa.md`](stages/qa.md) §환경 구성). build 의 Phase별 기능검증만 예외(§적용 범위).
- **실동작 재현** — TC 실행은 케이스별 **지정 검증 도구(MCP)** 로 실제 프로그램을 조작·관측해 수행한다(도구 매핑·구축 절차: [tools-setup.md](qa-execution/tools-setup.md)). 코드 정독·정적 추론만의 통과는 🟣 Pass-Static 잠정 판정으로 한정하고, 지정 도구 미구축·미연결로 실행할 수 없는 TC 는 🟠 Block 으로 등재해 환경 보완을 요청한다.
- **누락 금지** — 범위 내 검증 TC 를 전수 처리한다(5종 판정 또는 사유 명시 이월).
- **독립 재현** — 작성자의 자가 검증은 완료 근거가 아니다. 검증 주체는 작성 주체와 분리한다.
- **재현 가능성** — 실패 보고에 재현 절차·기대값·실제값·환경 정보를 포함한다.
- **사용자 협조 분기** — 자동화 한계(외부 인증 콜백·환경 의존 결함·외부 부작용 등)는 🟠 Block 으로 등재하고 수동 검증 절차를 명시한다.
- **5종 판정** — 🟢 Pass / 🔵 Pass-Mock / 🟣 Pass-Static / 🔴 Fail / 🟠 Block. 잠정 Pass(Mock·Static)는 실 Pass 와 분리해 품질 게이트 신뢰도 왜곡을 막는다.
- **증빙 보존** — 준비·실행·결과 증빙을 보존한다.

## 검증 범위 (사양 대조)

검증 TC 는 사양서를 기준으로 한다. 사양서와 어긋난 TC 는 결함이 아니라 TC 결함으로 분류한다(사양 동기화).

| 영역 | 기준 문서 | 확인 항목 |
|---|---|---|
| 기능 | `docs/specs/services/`·`functions/` | 사양 명시 동작·예외 처리 |
| 정책 | `docs/specs/policies/` | 업무 규칙·권한·무결성 |
| 화면 | `docs/specs/screens/` | 화면 정의·디자인 시스템 일치 |
| 데이터 | `docs/specs/datas/` | 모델·필드 제약 |
| 프로세스 | `docs/specs/processes/` | 단계 전이·롤백·동시성 |

## 케이스 선택 (프로젝트 구성 기반)

검증 방식은 프로그램 유형·기술스택에 따라 크게 달라지므로, **프로그램마다 케이스 하위 지침을 선택**해 적용한다.

- 선택 입력은 directing 산출물의 **프로그램 구성표**(프로그램 × 기술스택 × 검증 방법)다([`stages/directing.md`](stages/directing.md) §프로그램 구성표).
- qa 계획([`stages/qa.md`](stages/qa.md) §계획)에서 각 프로그램의 케이스를 확정하고, 실행은 해당 케이스 하위 지침의 환경·도구·절차를 따른다.
- 한 서비스에 여러 프로그램이 있으면 케이스가 혼재한다 — 프로그램별로 다른 케이스를 동시에 적용한다.

## 케이스 카탈로그

하위 지침은 `qa-execution/` 에 케이스별로 둔다. 기본 케이스(확장 가능):

| 케이스 | 적용 대상 | 하위 지침 |
|---|---|---|
| 웹·SPA UI | 브라우저 기반 웹/SPA 프런트 | [web-ui.md](qa-execution/web-ui.md) |
| 데스크톱 네이티브 UI | 네이티브 데스크톱 앱 | [desktop-ui.md](qa-execution/desktop-ui.md) |
| API·서버 | HTTP API·백엔드 서버 | [api-server.md](qa-execution/api-server.md) |
| CLI·배치 | 커맨드라인·배치·잡 | [cli-batch.md](qa-execution/cli-batch.md) |

- 신규 케이스(모바일 등)는 본 카탈로그에 행을 추가하고 하위 지침을 만든다.
- 케이스 하위 지침 공통 양식: §적용 조건(선택 트리거) / §환경 구성 / §검증 도구·절차 / §판정·증빙(본 문서 공통 원칙 준용). 인스턴스·제품 고유 값(도구 경로·포트·셀렉터 등)은 케이스 문서에 프로젝트별로 채운다.
- 케이스별 지정 도구(MCP)의 설치·등록·연결 검증·식별자 수집 절차는 [tools-setup.md](qa-execution/tools-setup.md) 가 정본이다 — 수행 시점·체크 기록은 [`project-bootstrap.md`](project-bootstrap.md) §6.

## 산출물·추적

- 검증 산출물(결과 보고·실패/차단 스냅샷·증빙) 구조는 [`stages/qa.md`](stages/qa.md) §산출물·§결과 보고를 따른다.
- 작업 추적·미러는 [`work-tracking.md`](work-tracking.md)(`검증` 트래커 — 검증 대상의 하위 이슈, 5종 판정에 따라 상태 동기화).
- 검증 실행 doer(test-planner·tester — [`agents.md`](agents.md) 카탈로그)는 케이스 무관 공통이며, 실행 시 해당 케이스 하위 지침을 로드해 환경·도구·절차를 따른다.
