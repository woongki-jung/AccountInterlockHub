# 제품 생산 파이프라인 (build 단계)

**build 오케스트레이터**가 자동으로 호출하는 하위 doer 5종의 실행 시점·입력·결과 정리다.
상위 문서: [AI 협업 환경 이용 가이드](AI%20협업%20환경%20이용%20가이드.md)

이 파이프라인은 **확정 사양서 → 제품 코드 + 배포 산출물**로 전환한다.
담당자는 doer를 직접 부르지 않는다 — ai-pm 이 build 오케스트레이터를 디스패치하면, 오케스트레이터가 구현 항목을 Phase로 편성해 아래 흐름을 실행한다.

```
ai-pm ──디스패치──▶ build 오케스트레이터
 ├ 1단계  (계획)             브랜치 생성·구현 항목을 Phase로 편성·담당 doer 지정
 ├ 2단계  Phase별 구현 (순차·한 Phase 통과 후 다음)
 │        ├ 2-A  backend-developer / frontend-developer   (코드 작성)
 │        ├ 2-B  code-reviewer                            (코드리뷰 PASS/FAIL → 보완 시 2-A 회귀)
 │        └ 2-C  tester                                   (기능검증 round-trip → Fail 시 2-A 회귀)
 │              → 세 책임 통과 시 Phase ✅ · `기능` 일감 동기화
 ├ 3단계  build-installer     배포 산출물(패키지·번들 등) 빌드
 ├ 4단계  tester              런타임 검증 게이트 (배포 산출물 = 사용자 동일 환경)
 └ 5단계  (정리)              배포 산출물 식별정보와 함께 완료 보고(qa 착수 승인 요청)
```

> 공통 입력 규약: 오케스트레이터가 각 doer에 **작업 범위(IA 노드)·확정 사양(`docs/specs/`)·승인된 목업·관련 `기능` 일감·검증 케이스**를 전달한다. 무게중심은 BE 로직이며 FE 는 승인된 목업을 청사진으로 삼는다. 상세 흐름 정본은 [`stages/build.md`](ai/strategies/stages/build.md).

## 오케스트레이터

### build (`ai/agents/build.md`)
- **모델 / 호출 주체**: opus·max / ai-pm
- **실행 시점**: 담당자가 Slack 으로 build 단계 착수를 승인·요청하면 ai-pm 이 디스패치한다(보통 spec 완료 승인 후).
- **입력 예시(ai-pm 디스패치 맥락)**: `작업 범위: 계정 연동 API(IA 노드) / 관련 일감: 기능 그룹 일감 / 확정 사양·승인 목업`
- **실행 결과**: 브랜치(`build/<이슈번호>-<주제>`) 생성, 구현 항목을 Phase로 편성해 아래 doer들을 호출하고, 배포 산출물 식별정보(버전·경로·기준 commit)와 함께 ai-pm 에 완료(qa 착수 승인 요청 포함)를 보고한다.
- 원본: [build.md](ai/agents/build.md)

## 코드 작성 (2종)

작성 주체가 자기 산출을 합격 판정·검증하지 않는다(책임 분리). 작성은 developer, 판정은 code-reviewer, 검증은 tester가 맡는다.

### backend-developer (`ai/agents/workflow-code-write/backend-developer.md`)
- **모델 / 호출 주체**: sonnet·xhigh / build (2-A)
- **실행 시점**: Phase의 백엔드 구현 담당으로 지정되면 코드를 작성한다. 외부 API 사양을 의무로 참조한다.
- **실행 결과**: `apps/backend/` (NestJS)에 비즈니스 로직·API·연동 코드가 구현된다.

### frontend-developer (`ai/agents/workflow-code-write/frontend-developer.md`)
- **모델 / 호출 주체**: sonnet·xhigh / build (2-A)
- **실행 시점**: Phase의 프런트엔드 구현 담당으로 지정되면 승인된 목업을 청사진 삼아 화면을 구현한다(목업 DOM 복사 아님, 사양·디자인 시스템 우선).
- **실행 결과**: `apps/frontend/` (React·Vite)에 화면·상태·흐름이 구현된다.

## 코드리뷰 (1종)

### code-reviewer (`ai/agents/workflow-code-write/code-reviewer.md`)
- **모델 / 호출 주체**: opus·max / build (2-B)
- **실행 시점**: 코드 작성 후 구현 코드를 판정한다(첫 줄 PASS/FAIL 마커).
- **실행 결과**: Critical/Important 보완이 있으면 2-A(작성)로 회귀시키고, 통과면 다음 책임(기능검증)으로 넘어간다. 회귀 사유·시도 횟수는 `기능` 일감 노트로 기록(최대 3회).

## 배포 산출물 (1종)

### build-installer (`ai/agents/workflow-publish/build-installer.md`)
- **모델 / 호출 주체**: sonnet·xhigh / build (3단계)
- **실행 시점**: 모든 Phase의 세 책임(작성·리뷰·검증) 통과를 확인한 뒤 실행된다.
- **실행 결과**: 개발사양이 정한 케이스별 형식(설치 패키지·번들·이미지 등)의 배포 산출물이 빌드된다. 이후 4단계에서 tester가 이 산출물로 런타임 게이트를 검증한다.
- 원본: [build-installer.md](ai/agents/workflow-publish/build-installer.md)

## 기능검증 (공유 doer)

### tester (`ai/agents/workflow-qa/tester.md`)
- **모델 / 호출 주체**: sonnet·xhigh / build (2-C 기능검증 · 4단계 런타임 게이트)
- **실행 시점**: ① Phase마다 구현 결과의 독립 재현·round-trip 을 검증(2-C), ② 모든 Phase 통과 후 배포 산출물로 최종 런타임 검증(4단계).
- **실행 결과**: 실 환경 round-trip 으로 판정한다. Fail·비대칭이면 2-A(작성)로 회귀한다. 잠정 Pass(🔵 Mock·🟣 Static)는 런타임 게이트 통과 근거가 아니다 — 실 Pass 로 해소한다. build 단계 기능검증의 Fail 은 `오류` 일감 없이 `기능` 일감 노트로 기록한다(build 내부 회귀).
- 원본: [tester.md](ai/agents/workflow-qa/tester.md) · qa 단계와 공유([품질 검증 파이프라인](품질검증-파이프라인.md) 참조)
