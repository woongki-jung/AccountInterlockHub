# 품질 검증 파이프라인 (qa 단계)

**qa 오케스트레이터**가 자동으로 호출하는 하위 doer 2종의 실행 시점·입력·결과 정리다.
상위 문서: [AI 협업 환경 이용 가이드](AI%20협업%20환경%20이용%20가이드.md)

이 파이프라인은 **배포 산출물 → 품질검증 보고(배포 가능 여부 평가)**를 만든다.
담당자는 doer를 직접 부르지 않는다 — ai-pm 이 qa 오케스트레이터를 디스패치하면, 오케스트레이터가 사용자 동일 환경을 구성한 뒤 아래 흐름을 실행한다.

```
ai-pm ──디스패치──▶ qa 오케스트레이터
 ├ 0단계  (환경 구성·선행)   배포 산출물 설치·기동 → 사용자 동일 환경·검증 기준선 확보
 ├ 1단계  test-planner       검증 계획 — TC를 빠짐없이 Phase로 배정(1 TC = 1 Phase)
 ├ 2단계  tester             Phase별 TC 실행 · 5종 판정 · 실행 즉시 `검증` 일감 갱신
 ├ 3단계  (결과 평가)        누락 대조(전체=수행+이월) + 품질 게이트 평가 · 담당자별 그룹 보고
 └ 4단계  (정리)             `검증` 일감 동기화 확인 · 완료 보고
```

> 공통 입력 규약: 오케스트레이터가 각 doer에 **대상 배포 산출물(식별정보)·검증 대상 사양·검증 TC·선택된 검증 케이스**를 전달한다. 워크스페이스 직접 기동이 아니라 **배포 산출물 설치·기동본**을 검증 대상으로 삼는다(사용자 동일 환경, Debug 미사용). 상세 흐름 정본은 [`stages/qa.md`](ai/strategies/stages/qa.md).

## 오케스트레이터

### qa (`ai/agents/qa.md`)
- **모델 / 호출 주체**: opus·max / ai-pm
- **실행 시점**: 담당자가 Slack 으로 qa 단계 착수를 승인·요청하면 ai-pm 이 디스패치한다(보통 build 완료 승인 후). 핫픽스 검증은 축약 범위(결함 TC + 회귀 우선순위)로 지정될 수 있다.
- **입력 예시(ai-pm 디스패치 맥락)**: `작업 범위: 전체 또는 IA 노드 / 대상 빌드: 배포 산출물(버전·경로·기준 commit) / 관련 검증·참조 사양 일감`
- **실행 결과**: 브랜치(`qa/<이슈번호>-<주제>`) 생성, 환경 구성 후 doer들을 호출해 5종 판정을 채우고, 품질 게이트 결과·후속 액션을 ai-pm 에 보고한다. **배포 가능 판정은 담당자 승인 게이트** — 자동으로 배포로 이어지지 않는다.
- 원본: [qa.md](ai/agents/qa.md)

## 검증 계획 (1종)

### test-planner (`ai/agents/workflow-qa/test-planner.md`)
- **모델 / 호출 주체**: opus·max / qa (1단계)
- **실행 시점**: 환경 구성 후 검증 실행에 앞서 계획을 수립한다.
- **실행 결과**: 작업 범위 내 검증 TC를 **빠짐없이 Phase로 배정**하고(1 TC = 1 Phase), 각 TC의 IA 노드·프로그램 케이스를 매핑하며 환경 의존성을 식별한다. 각 TC를 Redmine `검증` 일감으로 등록한다(검증 대상 일감의 하위 이슈, 참조 `사양` 일감 연관). 환경 한계로 수행 못 하는 TC는 사유·해소조건과 함께 이월로 명시한다(누락 금지).
- 원본: [test-planner.md](ai/agents/workflow-qa/test-planner.md)

## 검증 실행 (1종)

### tester (`ai/agents/workflow-qa/tester.md`)
- **모델 / 호출 주체**: sonnet·xhigh / qa (2단계) · build(2-C·런타임 게이트)와 공유
- **실행 시점**: Phase별로 검증 TC를 사용자 동일 환경에서 실행한다. 도구·셀렉터·환경은 선택된 검증 케이스(웹·SPA / 데스크톱 / API·서버 / CLI·배치) 지침에서 가져온다.
- **실행 결과**: 각 TC를 실 환경 round-trip 으로 5종 판정한다 — 🟢 Pass / 🔵 Pass-Mock / 🟣 Pass-Static / 🔴 Fail / 🟠 Block. **각 TC 실행 즉시 `검증` 일감 상태를 갱신**한다(Pass→`해결`, Fail→`진행`, Block→`보류`). 🔴 Fail 은 `오류` 일감을 생성해 해당 `검증`·`사양` 일감과 연관한다. 판정 근거·증빙은 일감 노트·첨부로 보존한다.
- **완료 경계**: 검증 실행 주체(오케스트레이터·tester)는 `검증` 일감을 **`해결`까지만** 둔다 — 최종 `완료`(닫힘)는 담당자 승인(배포 가능 판정) 후에만 이뤄진다.
- 원본: [tester.md](ai/agents/workflow-qa/tester.md)

## 판정 등급 (5종)

| 등급 | 기준 | 신뢰도 |
|---|---|:---:|
| 🟢 Pass | 실 환경(실 BE·DB·외부 시스템) 응답으로 요구 동작 확인 | 100% |
| 🔵 Pass-Mock | Mock·스텁·시드로 검증(실 응답 회귀 미수행) | ~70% |
| 🟣 Pass-Static | 코드 정합 trace 만으로 통과(동적 실행 없음) | ~50% |
| 🔴 Fail | 의도한 결과 미확인(증빙·해결 제안 포함) | — |
| 🟠 Block | 동작 불가(불가 사유·해결 제안 포함) | — |

> 잠정 Pass(🔵·🟣)는 실 Pass 와 분리해 품질 게이트 신뢰도 왜곡을 막고, 회귀 우선순위 후보로 등재한다. 판정 정책 정본은 [`qa-execution.md`](ai/strategies/qa-execution.md)·[`tester.md`](ai/agents/workflow-qa/tester.md).
