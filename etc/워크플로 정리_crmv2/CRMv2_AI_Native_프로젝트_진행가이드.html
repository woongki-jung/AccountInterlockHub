에이전트 기반 프로젝트 실행 가이드

CRMv2 워크스페이스의 모든 AI 에이전트(스킬 2 + 에이전트 24)를 "언제 실행되고 · 어떤 프롬프트를 넣고 · 무엇이 만들어지는가" 기준으로 정리한 매뉴얼입니다.

이 매뉴얼 사용법
• 직접 프롬프트를 입력하는 담당자라면 → 좌측 사용자 진입점만 보면 됩니다 (사람이 실행하는 지점은 5개뿐).
• 내부 동작이 궁금하면 → ① 사양 · ② 생산 · ③ 검증 파이프라인에서 하위 에이전트 카드를 확인하세요.
• 상단 검색창에 이름·기능 키워드를 넣으면 해당 카드만 필터링됩니다.

◎전체 워크플로

제품은 사양 분석 → 제품 생산 → 품질 검증 3개 파이프라인을 거칩니다. 각 파이프라인은 사용자가 넣는 하나의 진입 프롬프트로 시작되고, 그 안에서 하위 에이전트가 순서대로 자동 실행됩니다.

☰에이전트 전체 지도

"실행 주체"는 그 에이전트를 누가 호출하는지입니다 — 사용자는 사람이 직접, 나머지는 상위 에이전트가 자동 호출합니다. 이름을 누르면 상세 카드로 이동합니다.

파이프라인이름종류모델실행 주체한 줄 역할

계획·리뷰·판정은 정확도를 위해 opus, 반복적인 코드 작성·TC 실행은 sonnet을 사용합니다.

▶사용자 진입점 (직접 실행)

사람이 직접 프롬프트로 시작하는 지점은 이 5개뿐입니다. 나머지 하위 에이전트는 이 진입점이 자동으로 불러옵니다.

①사양 분석 파이프라인

start-spec의 3단계 흐름 안에서 실행되는 하위 12종. PRD → 상세 사양서로 전환합니다.

start-spec
 ├ 1단계  spec-sprint-planner   → 계획(ROADMAP·Phase) 수립
 ├ 2단계  spec-sprint-worker    → Phase마다: prd-reviewer(정합성) → prd-to-* 7종(순차 생성)
 └ 3단계  spec-sprint-closer    → 리포트·커밋·푸시
        (별도) spec-reviewer    → 완성 사양서 6축 교차 검증
공통 입력 규약 · prd-reviewer와 prd-to-* 7종은 모두 작업 Phase 문서 경로 하나를 전달받아 실행됩니다.

②제품 생산 파이프라인

start-builder의 4단계 흐름 안에서 실행되는 하위 7종. 사양서 → 제품 코드 → 인스톨러로 전환합니다.

start-builder
 ├ 1단계  build-sprint-planner  → 계획 수립
 ├ 2단계  build-sprint-worker   → Phase마다: developer(구현) → code-reviewer(PASS/FAIL) → tester(QC)
 ├ 3단계  build-installer       → Debug/Release 인스톨러 빌드
 └ 4단계  build-sprint-closer   → 리포트·커밋·푸시

⚙QA 사전 환경 구성 · 최초 1회 · QA 실행 전 필수

③ 품질 검증(/sprint-qa·start-qa)의 자동화 도구가 동작하려면 검증 PC마다 최초 1회 아래 도구·MCP 서버·셀렉터 카탈로그를 구성해야 합니다. 전체 절차·명령·트러블슈팅은 tc환경구축.md 전문을 단일 출처로 따릅니다.

사전 환경 구성 체크리스트

tc환경구축.md에 로컬 환경에서 직접 실행 기반 테스트 환경을 구축하기 위한 가이드가 작성되어 있습니다. tc환경구축.md 의 내용에 따라 테스트 환경을 구축해줘. 와 같이 프롬프팅하여 초기 구성을 진행하세요.

1) 런타임·도구 (§1 사전 준비)
  • ⬜ Python 3.10+ (pywinauto-mcp 구동)
  • ⬜ Node.js 18+ (playwright-mcp 구동)
  • ⬜ Microsoft Edge WebView2 Runtime (Evergreen 권장)
  • ⬜ .NET 6+ (대상 WinForms 앱 빌드·실행 시)
  • ⬜ Claude Code (npm install -g @anthropic-ai/claude-code) · Windows 네이티브 쉘 권장
  • ⬜ (권장) Inspect.exe(UIA 트리) · Spy++(Win32/VCL 클래스명) — 컨트롤 식별자 사전 확보
2) MCP 서버 설치 (§2)
  • ⬜ pywinauto-mcp — git clone + venv + pip install -e ".[all]", venv Python 절대경로 확보
  • ⬜ playwright-mcp — npm install -g @playwright/mcp
  • ⬜ WinForms 앱 WebView2 CDP 활성화 — 환경변수 --remote-debugging-port=9222 주입 후 http://localhost:9222/json 확인
3) MCP 서버 등록 — Claude Code (§3)
  • ⬜ claude mcp add pywinauto -s project -- <venv python> -m pywinauto_mcp
  • ⬜ claude mcp add playwright -s project -- npx -y @playwright/mcp@latest
  • ⬜ claude mcp list로 등록 확인 (팀 공유 시 .mcp.json 커밋 / WSL은 powershell.exe 브릿지)
4) 연결 검증 (§4)
  • ⬜ 세션에서 /mcp → pywinauto·playwright 모두 connected
  • ⬜ Inspect.exe로 WinForms 컨트롤 트리(AutomationId/Name) 확인
  • ⬜ Spy++로 Delphi VCL 클래스명(TEdit·TButton) 확인
  • ⬜ http://localhost:9222/json에 WebView 페이지 노출
  • ⬜ 세션에서 pywinauto 창 목록 / playwright CDP 페이지 정보 정상 반환
5) 초기 셀렉터 수집 (§5)
  • ⬜ 각 앱 수동 실행 후 셀렉터 정리(Spy++ · Inspect.exe · DevTools)
  • ⬜ docs/control-catalog.md·docs/app-registry.md로 정리하고 CLAUDE.md에서 @ 경로로 참조

③품질 검증 파이프라인

start-qa가 인스톨러 설치본을 대상으로 검증합니다. 마무리(closer)는 별도 에이전트 없이 start-qa가 직접 수행합니다.

▶ 이 파이프라인의 자동화 도구·셀렉터는 QA 사전 환경 구성(최초 1회)을 먼저 완료해야 동작합니다.
start-qa
 ├ 사전   인스톨러 빌드 + 자동 설치  (start-qa 직접 / build.ps1 -Mode {Test|Release})
 ├ 1단계  test-planner           → 테스트 계획(ROADMAP·Phase·TC 매핑)
 ├ 2단계  tester                 → Phase마다 1회 TC 실행·판정
 └ 3단계  (start-qa 직접)         → 결과 집계·리포트·커밋
사전 단계 — 인스톨러 빌드 + 자동 설치 · 지정 모드(Test 기본/Release)로 인스톨러를 빌드하고 InnoSetup 무인 옵션으로 자동 설치한 뒤 표준 경로(D:\YSR2000\YSRCRMv2\App\) exe 정합을 확인합니다. 개발 빌드가 아니라 실제 배포되는 설치본을 검증 대상으로 고정합니다.

마무리 (start-qa 직접 수행)

  • 결과 집계 · 5종 판정을 도메인·자동화 수준별로 집계, Critical·High는 🟢 Pass만 품질 게이트에 반영.
  • 리포트 · 담당 파트(기획·디자인·개발·QA)별 그룹화한 qa-report.md/qa-dev-report.md 생성 (report-template.md 기준).
  • 후속 스냅샷 · 마일스톤 전체 롤업 tc-followup.md(리프별 집계 + Critical 요약 + 변화 + 방향) 1회 생성.
  • 커밋·정리 · [start-qa] 접두사로 커밋·푸시하고, 검증용으로 띄운 프로세스(CRM·WebView2·Playwright)를 정리.
  • resolved 검증 모드 · 사용자가 명시 호출할 때만 docs/sprint-feedbacks/resolved/를 검증해 결과에 따라 completed/<날짜>/·failed/·blocked/로 이동.

🔁피드백 작성·처리 흐름 · docs/sprint-feedbacks/

docs/sprint-feedbacks/는 사용자·기획·QA가 남긴 피드백을 등록 → 조치 → 검증 → 종결까지 상태별 폴더로 관리하는 단일 저장소입니다. build 파이프라인을 실행하여 1차 처리(resolved)하고, qa 파이프라인을 실행하여 결과 상태 폴더(completed / failed / blocked)로 옮깁니다.

처리 흐름 (상태 전이)

피드백 하나는 폴더 위치가 곧 처리 상태입니다. 폴더를 옮기는 주체는 각 파이프라인의 마무리(closer)와 start-qa입니다.

[신규 피드백 등록]
   │
   ▼
 pending/     처리 대기 · build·spec 스프린트가 최상위 중요도로 소비
   │
   │  build·spec-sprint-closer 가 조치 완료분 이동
   ▼
 resolved/    조치 완료 · QA 검증 대기
   │
   │  start-qa · resolved 검증 모드 (사용자 명시 호출 시)
   ▼
   ├─ 🟢 Pass  → completed/<날짜>/    검증 통과 = 종결 (보관)
   ├─ 🔴 Fail  → failed/             결함 재발
   └─ 🟠 Block → blocked/            외부 API·DBA·기획 미확정·QA 시드 등

 failed·blocked → 사용자 수동 회수 → pending/   (자동 회수 없음)
사양(spec) 스프린트 예외 · 개발 작업이 필요 없는 순수 기획·문서 반영 항목은 spec-sprint-closer가 pending/ → completed/로 바로 종결합니다 (별도 QA 검증 불필요).

상태별 폴더

폴더상태 의미이동 주체다음 단계
pending/처리 대기 — 신규 피드백이 최초 등록되는 곳작성자 / 사용자 회수build·spec 스프린트가 최상위 중요도로 소비
resolved/스프린트 조치 완료 → QA 검증 대기build·spec closerstart-qa resolved 검증

§공통 규약 · 주의사항

  • 재실행 방지 · 각 진입점은 완료 시 [start-spec]·[start-builder]·[start-qa] 접두사로 커밋합니다. 마지막 커밋에 이 접두사가 있으면 웹훅 재트리거를 막기 위해 실행을 중단합니다.
  • 자동화 실행 모드 · start-builder·start-qa는 --dangerously-skip-permissions 자동화 모드 전용이며, 사용자 확인이 필요한 항목은 "수동 실행 필요 항목"으로 리포트에 남기고 멈추지 않습니다.
  • 정확도 우선 · 병렬 실행은 품질 저하 우려가 없을 때만 사용하고 기본은 순차 실행입니다.
  • 실 동작 검증 의무 · 정적 검증(컴파일·타입 체크)만으로 완료 처리하지 않고, 변경 기능을 실제 실행해 저장→조회 왕복(round-trip)까지 확인합니다.
  • 민감정보 보호 · 계정·복호화 키 등 원문(CLAUDE.local.md 변수)은 리포트·커밋에 포함하지 않습니다.
  • 문서 길이 제어 · 하나의 문서가 너무 길어지지 않도록 조정합니다 (300줄 제한 권장).
  • WIKI 형식 작성 · 최상위 문서에서 하위 문서로 점진적으로 세분화된 내용으로 작성합니다. 처음부터 문서 구조를 세분화하지 않고, 하나의 문서로 시작한 뒤 세부 내용을 추가하는 과정에서 문서 길이를 고려해 내용이 많은 섹션부터 하위 문서로 분기합니다.
