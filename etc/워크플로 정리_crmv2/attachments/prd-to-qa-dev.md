본 문서는 프로젝트의 결과 제품 및 서비스에 대한 품질을 보증하기 위해 필요한 검증 항목을 정의하기 위한 규칙을 정의한다.
아래 지침에 따라 제공된 요구사항 및 관련 문서를 기반으로 제품 검증 방법과 세부 조건에 대하여 서술한다.

용어 약어 — TC는 테스트 케이스(Test Case), SVC는 서비스 시나리오, FN은 공통 기능, SCR은 화면, ENT는 DB 엔터티, MDL은 데이터 모델, POL(=policy)은 정책 규칙, PROC는 기능구현 프로세스 정의서, BR은 분기 코드, EX는 예외 코드를 의미한다.

# 실행 시 전달 파라메터
- 작업 phase 문서 경로 (`sprints/spec/spec-<n>-<git유저명>/phase-<n>.md`)
- 유효한 값이 전달되지 않은 경우 경고 노출후 작업 중단.

# 작업 프로세스

## 1단계: 요구사항 분석

- 시작 시 전달받은 phase 문서의 내용을 확인하고, 분석 대상 문서를 확인한다. 분석 대상 문서와 각 문서가 참조하는 연결 문서의 내용을 자세히 분석한다.
- 분석된 내용에서 다음 항목을 추출한다
	- 서비스 기능(SVC)별 검증이 필요한 시나리오 목록
	- **PROC 별 분기·예외 흐름** 코드 목록 (BR-/EX-) — 각 BR/EX에 대응하는 TC 가 1개 이상 필요
	- 정책(policy) 규칙 중 자동 검증 가능한 항목과 수동 검증이 필요한 항목
	- API(FN)별 정상/예외 응답 검증 조건
	- 데이터 모델(MDL)·엔터티(ENT)의 정합성 검증 포인트
	- 화면(SCR)별 UI/UX 검증 항목 (레이아웃, 인터랙션, 상태 표시, 접근성)
	- 역할별 접근 제어 검증 시나리오
	- 실행 환경별 테스트 전제 조건 (프로그램 기동, DB 초기화 등)

## 2단계: 기획서 검증

1단계에서 분석한 PRD 및 스펙 문서를 기준으로 기획서의 품질을 검증하고, 결과를 **새 리포트 파일**로 생성한다.

### 2-1. 검증 항목

TC 작성 관점에서 아래 항목을 식별한다.

> **정본 체크리스트:** [qm-03-prd-requirements.md](../../../docs/prd/quality-manage/qm-03-prd-requirements.md) §1~§4 (SVC 노드 / 정책 / 시나리오 사전 요건 / 레거시·외부 시스템 연계).
> 본 단계에서는 정본 체크리스트의 모든 항목을 적용하며, 본 문서의 표는 신속 참조용 요약이다. 정본과 본 문서가 충돌하면 정본을 따른다.

**기본 검증 (기획서 품질)**

| 검증 관점 | 확인 내용 |
|----------|----------|
| TC 작성 불가 항목 | 기대 결과·완료 조건이 정의되지 않아 TC로 표현할 수 없는 요구사항 |
| 누락 항목 | 연결 문서에 언급되었으나 기획서 본문에 서술이 없는 기능·정책 |
| 모호한 표현 | "적절히", "충분히" 등 수치·조건 없이 기술된 요구사항 |
| 모순·충돌 | 다른 요구사항과 양립할 수 없는 내용 |
| 미완성 항목 | 핵심 흐름이 누락되거나 완성되지 않은 요구사항 |

**TC 작성 관점 심층 검증 — SVC 노드별** (qm-03-prd-requirements §1)

각 SVC 노드에 대해 아래 항목을 추가로 확인한다.

| 검증 항목 | 확인 내용 | 결함 등급 |
|----------|----------|----------|
| Preconditions 구성 가능 여부 | 계정 상태·데이터 존재 조건·환경 변수 등 사전 조건이 스펙에 명시되어 있는가 | 미명시 시 🟠 ERROR |
| Expected Result 측정 가능 여부 | "정상 동작" 등 모호한 결과 대신 UI 상태·DB 값·HTTP 응답 등 구체적 수치·조건이 있는가 | 모호 시 🟡 WARNING |
| Happy Path 시나리오 존재 여부 | 정상 입력 → 성공 결과의 주요 흐름이 기술되어 있는가 | 누락 시 🟠 ERROR |
| Negative 시나리오 존재 여부 | 잘못된 입력·비정상 상태에서의 시스템 반응(오류 메시지·차단 동작 등)이 명시되어 있는가 | 누락 시 🟡 WARNING |
| 경계값 수치 명시 여부 | 바이트·글자 수·횟수 등 경계 조건이 구체적 수치로 기술되어 있는가 | 미명시 시 🟡 WARNING |
| 조합 경계값 명시 여부 | 두 개 이상 정책·경계가 동시에 적용될 때 동작이 명시되어 있는가 (qm-03 §6.7) | 누락 시 🟡 WARNING |
| 권한/인증 분기 명시 여부 | AUTH-* 정책이 적용되는 노드에서 권한 미충족 시 동작이 명시되어 있는가 | 누락 시 🟠 ERROR |
| 상태전이 흐름 명시 여부 | A → B → 취소 → 원복 시 데이터 유실 여부 등 상태 변화가 기술되어 있는가 | 누락 시 🟡 WARNING |
| 시스템 예외 처리 명시 여부 | API 실패·타임아웃·네트워크 단절 시 시스템 반응이 기술되어 있는가 | 누락 시 🟡 WARNING |
| 하이브리드 컨텍스트 명시 여부 | WebView2(React) / Native(VCL·WinForms) 어느 영역에서 동작하는지 구분되어 있는가 | 누락 시 🟡 WARNING |
| 로그·추적 컬럼 명시 여부 | DB 로그 테이블·컬럼명, 마스킹 정책, 추적 식별자가 정의되어 있는가 | 누락 시 🟡 WARNING |
| 요구사항 ID 추적성 | 본 노드가 매핑되는 PRD 요구사항 ID·섹션이 명시되어 있는가 | 누락 시 🟡 WARNING |
| 테스트 데이터 준비 가능 여부 | TC 실행에 필요한 데이터(계정·DB seed 등)를 환경에서 구성할 수 있는가 | 불가 시 🟠 ERROR |
| 테스트 데이터 시드 경로 명시 여부 | 준비 수단(SQL seed / UI 입력 / API 호출 / 외부 파일) 중 어느 경로가 권장되는지 명시되어 있는가 | 누락 시 🟡 WARNING |
| 자동화 가능 여부 | UI 셀렉터(AutomationId·testId)·API 명세가 충분히 정의되어 있는가 | 미정의 시 🟡 WARNING |

**정책별 TC 작성 가능 여부** (qm-03-prd-requirements §2)

BIZ-* / AUTH-* / SEC-* / DATA-* 정책 각각에 대해 아래를 확인한다.

| 확인 내용 | 결함 등급 |
|----------|----------|
| 정책 적용 조건(트리거)이 수치·상태로 명시되어 있는가 | 미명시 시 🟠 ERROR |
| 정책 위반 시 시스템 반응(차단·경고·안내 문구 등)이 명시되어 있는가 | 미명시 시 🟠 ERROR |
| 정책 우선순위(P0/P1/P2/P3)가 부여되어 있는가 | 누락 시 🟡 WARNING |
| 두 정책 충돌 시 우선순위 처리가 기술되어 있는가 | 누락 시 🟡 WARNING |
| 정책이 적용되는 도메인(화면) 목록이 명시되어 있는가 | 누락 시 🟡 WARNING |
| P0 정책의 경계값(직전·경계·초과)이 수치로 명시되어 있는가 | 미명시 시 🟡 WARNING |
| 정책 미적용 화이트리스트(특수 계정·내부 발송 등)가 명시되어 있는가 | 누락 시 🟢 정보 부족 |

**시나리오 TC 사전 요건** (qm-03-prd-requirements §3 — 복합 흐름·사용자 여정 TC 대상)

| 확인 내용 | 결함 등급 |
|----------|----------|
| 페르소나·User Goal이 정의되어 있는가 | 누락 시 🟠 ERROR |
| 가로지르는 도메인 목록이 명시되어 있는가 | 누락 시 🟠 ERROR |
| 단계 간 데이터 일관성(1단계 산출물 → 2단계 입력)이 기술되어 있는가 | 누락 시 🟡 WARNING |
| 이탈/복귀 흐름(ESC·뒤로가기·세션 만료) 상태 보존 정책이 명시되어 있는가 | 누락 시 🟡 WARNING |
| 종료 조건·성공 판정 기준이 정의되어 있는가 | 누락 시 🟠 ERROR |
| 외부 결과 확인 수단(수신 폰·DB 조회·UI)이 명시되어 있는가 | 누락 시 🟡 WARNING |

**레거시·외부 시스템 연계 요건** (qm-03-prd-requirements §4 — 마이그레이션·외부 API·Agent TC 대상)

| 확인 내용 | 결함 등급 |
|----------|----------|
| 레거시 ↔ 신규 테이블·컬럼 매핑이 명세되어 있는가 | 미명시 시 🟠 ERROR |
| 변환 규칙(코드 매핑·문자열 변환 등)이 구체적으로 기술되어 있는가 | 미명시 시 🟠 ERROR |
| 외부 API(카카오·UBMS·Agent) 요청·응답 형식·에러 코드가 정의되어 있는가 | 미명시 시 🟠 ERROR |
| 실 검증 수단(테스트 발신번호·수신 폰·검증 채널) 확보 가능한가 | 불가 시 🟡 WARNING |
| 부분 실패·재실행 멱등성·롤백 정책이 명시되어 있는가 | 누락 시 🟡 WARNING |
| 외부 상태 동기화 시점·방법(카카오 검수 결과 등)이 명시되어 있는가 | 누락 시 🟡 WARNING |

### 2-2. 등급 분류

| 등급 | 기준 |
|------|------|
| 🔴 **CRITICAL** | TC 작성 자체가 불가능하거나 프로젝트 전체에 돌이킬 수 없는 영향을 주는 항목 |
| 🟠 **ERROR** | TC 작성 전 반드시 해소되어야 할 결함 (기대 결과 미정의, 핵심 흐름 누락 등) |
| 🟡 **WARNING** | 현 상태로 TC 작성은 가능하나 향후 재작업·품질 저하로 이어질 수 있는 항목 |
| 🟢 **OK** | TC 작성에 문제없는 명확한 요구사항 |

### 2-4. TC 작성 영향 반영

CRITICAL·ERROR로 분류된 항목의 해당 SVC 코드에 대해서는 4단계 TC 설계 시 해당 항목을 `Block` 예정으로 표기하고, TC 내 비고에 리포트 파일 경로와 항목 번호를 기재한다.

### 2-5. spec-qa.md "사양 미해소 항목" 카탈로그 등재

CRITICAL·ERROR 항목을 `spec-qa.md`의 **§사양 미해소 항목 (Block 예정)** 섹션에 카탈로그로 등재한다. 이 섹션은 `test-planner` → `tester` → `start-qa` 리포트로 자동 전파되어 [기획] 후속 대응 카탈로그로 누적된다.

**섹션 위치:** `spec-qa.md` 최상단 메타 섹션 직후, TC 매핑 표 직전 (없으면 신설)

## 3단계: 테스트 전략 수립

- 테스트 범위 및 접근 방법을 정의한다
	- **단위 테스트(UNIT)**: 공통 기능(FN) 단위의 로직 검증
	- **통합 테스트(INTG)**: API 엔드포인트 요청/응답, DB 연동, 외부 시스템 연동 검증
	- **E2E 테스트(E2E)**: 사용자 시나리오 기반 화면 흐름 전체 검증
	- **데이터 정합성 테스트(DATA)**: 엔터티 제약조건, 데이터 생명주기, 마이그레이션 검증
	- **정책 준수 테스트(POL)**: 접근 제어, 입력값 검증, 비즈니스 규칙 위반 시 처리 검증
	- **UI/UX 테스트(UI)**: 레이아웃, 반응형, 접근성, 디자인 시스템 준수 검증
- 테스트 우선순위를 결정한다
	- 핵심 비즈니스 시나리오 (Happy Path) → 정책 위반 시나리오 → 예외/에러 시나리오 → 경계값 시나리오

## 4단계: 테스트 환경 정의

- PRD 하위 문서의 실행 환경 정보를 기반으로 테스트 환경을 정의한다
	- 프로그램 실행 절차 (빌드, 기동 명령어, 환경변수 설정)
	- 데이터베이스 초기화 절차 (마이그레이션, 시드 데이터 투입)
	- 테스트 데이터 준비 방법 (픽스처, 팩토리 등)
	- 외부 시스템 모킹 전략 (목 서버, 스텁 등)
	- 테스트 실행 명령어 및 결과 확인 방법

## 5단계: 테스트 케이스 설계

- 각 테스트 유형별로 테스트 케이스를 설계한다
	- 테스트 케이스 코드 체계 (TC-nnn 형태)
	- 케이스별 전제 조건, 실행 단계, 기대 결과를 명확하게 서술
	- 검증 대상 스펙 코드(SVC, FN, ENT, MDL, SCR, policy, **PROC**) 매핑
	- 자동화 가능 여부 판단 (자동 / 수동 / 반자동)
	- **PROC 분기·예외 매핑 강제**: TC 가 검증하는 PROC 코드와 분기/예외 코드(BR-/EX-)를 명시한다.
	  PROC 의사코드의 각 BR-/EX- 코드에 대응하는 TC 가 1개 이상 존재해야 한다 (PROC ↔ TC 양방향 추적).
- 경계값 및 예외 케이스를 누락 없이 도출한다
	- 입력값 경계 (최소, 최대, 초과, 빈 값, null)
	- 동시성 시나리오 (동시 수정, 중복 요청)
	- 권한 경계 (허용/비허용 역할 전환)
- **TC 수량 기준을 반드시 충족한다** (정성적 판단 금지 — 아래 수치를 기준으로 삼는다)
	- 기능 복잡도별 최소 TC 수
		- 단순 조회/표시 기능: **최소 8개**
		- 입력·저장 기능 (CRUD): **최소 12개**
		- 복합 흐름 (다단계·채널 분기 포함): **최소 15개**
	- 정책(policy)별 최소 TC 수
		- P0 정책 (BIZ-*/AUTH-*/SEC-*): **최소 3개** (Positive 1 + Negative 1 + 경계값 또는 권한 케이스 1)
		- P1 정책: **최소 2개** (Positive 1 + Negative 1)
		- P2 이하 정책: **최소 1개**
		- AUTH-* 정책: 정책이 적용되는 **각 화면(SCR)/도메인마다** 독립적으로 1개 이상 추가
	- API(FN)별 최소 TC 수
		- 각 API 엔드포인트마다 **최소 3개**: 정상 응답(2xx) 1 + 클라이언트 오류(4xx) 1 + 서버 오류(5xx) 또는 타임아웃 1
- **아래 7개 유형을 균형 있게 도출한다** (도메인 전체 기준 분포 목표)

	| 유형 | 목표 비율 | 작성 기준 |
	|------|--------:|---------|
	| Positive | ~40% | 정상 입력·Happy Path 시나리오 |
	| Negative | ~30% | 잘못된 입력·비정상 상태의 시스템 반응 |
	| Boundary | ~15% | 정책 명시 경계값 -1 / 경계값 / 경계값 +1 |
	| 권한/인증 | ~5% | AUTH-* 정책 적용 화면의 권한 미충족 상태 |
	| 상태전이 | ~5% | A → B → 취소 → 원복 흐름 (데이터 유실 여부 포함) |
	| 시스템예외 | ~3% | API 타임아웃·네트워크 실패·백그라운드 프로세스 오류 |
	| 조합경계값 | ~2% | 두 개 이상의 정책이 동시에 적용될 때 우선순위·분류 기준 |

	> Boundary TC가 전체의 20%를 초과하는 경우 유형 편중으로 판단하고, Negative·상태전이·시스템예외 유형으로 재분배한다.

## 6단계: 검증 절차 정의

### 7-2: 개별 테스트 케이스 정의 문서 생성

테스트 유형 또는 검증 대상별로 `docs/specs/qa-dev/tc_<test-code>.md` 파일을 생성한다. 세부 테스트 케이스 정의는 다음 템플릿 구조를 따른다.

```markdown
# [테스트 대상명] 테스트 케이스 정의

## 개요
- **검증 목적**: 이 테스트가 검증하려는 품질 속성 또는 요구사항
- **관련 PRD 요구사항**: PRD 내 관련 섹션 및 원문 인용

---

## [TC 코드] [테스트 케이스명]

### 기본 정보
| 항목 | 내용 |
|------|------|
| 테스트명 | 테스트 케이스 이름 |
| 테스트 유형 | UNIT / INTG / E2E / DATA / POL / UI |
| 검증 대상 | 관련 스펙 코드 (SVC-xxx, FN-xxx, SCR-xxx, ENT-xxx 등) |
| **검증 PROC** | 본 TC 가 검증하는 PROC 코드 (1개 이상) |
| **검증 분기/예외** | 검증하는 BR-/EX- 코드 (PROC 의사코드 분기·예외 흐름 표 인용) |
| 우선순위 | 높음 / 보통 / 낮음 |
| 자동화 여부 | 자동 / 수동 / 반자동 |

> **PROC ↔ TC 양방향 추적**: 본 TC 가 검증하는 PROC 의 분기/예외 코드를 명시하면, PROC 정의서에서 역으로 어느 TC 가 자신의 분기/예외를 검증하는지 추적 가능하다.
> spec-qa.md 의 매핑 표는 SVC/FN/PROC 별로 검증 TC 목록을 제공해야 한다.

### 전제 조건
| 조건 | 상세 내용 | 준비 방법 |
|------|-----------|-----------|
| 프로그램 실행 | 서버가 기동된 상태 | PRD 하위 실행 절차 참조 |
| DB 상태 | 시드 데이터 투입 완료 | 마이그레이션 + 시드 스크립트 실행 |
| 테스트 데이터 | 특정 사용자 계정 존재 | 픽스처 또는 API 호출로 생성 |

### 테스트 단계
| 단계 | 실행 내용 | 입력 데이터 | 기대 결과 | 매핑 PROC 단계 |
|------|-----------|-------------|-----------|---------------|
| 1 | 구체적인 실행 동작 | 입력값 또는 조작 내용 | 예상되는 응답 또는 상태 변화 | PROC-xxx F1 |
| 2 | 다음 실행 동작 | 입력값 또는 조작 내용 | 예상되는 응답 또는 상태 변화 | PROC-xxx B2 |

> **매핑 PROC 단계** 칸은 본 TC 단계가 검증하는 PROC 의사코드 단계명(F1·B2 등)과 1:1 매핑한다.
> PROC 분기/예외(BR-/EX-)를 검증하는 단계에서는 해당 코드를 함께 명시한다.

### 데이터 검증 (해당 시)
| 검증 대상 | 엔터티(ENT) | 검증 쿼리 또는 조건 | 기대 결과 |
|-----------|-------------|---------------------|-----------|
| 데이터 생성 확인 | ENT-xxx | SELECT 조건 서술 | 레코드 1건 존재, 컬럼값 일치 |

### 화면 검증 (해당 시)
| 검증 항목 | 화면(SCR) | 검증 내용 | 기대 결과 |
|-----------|-----------|-----------|-----------|
| 레이아웃 | SCR-xxx | 요소 배치 확인 | 디자인 시스템 규격 일치 |
| 반응형 | SCR-xxx | 브레이크포인트별 확인 | 레이아웃 정상 전환 |

### 에러/예외 케이스
| 케이스 | 입력 조건 | 기대 동작 | 관련 정책 |
|--------|-----------|-----------|-----------|
| 필수값 누락 | 필수 필드 비워둠 | 유효성 에러 메시지 표시 | BIZ-xxx R-001 |
| 권한 없음 | 비허용 역할로 접근 | 403 반환 또는 접근 차단 | AUTH-xxx |

### 판정 기준
- **Pass**: 모든 테스트 단계의 기대 결과가 일치
- **Fail**: 하나 이상의 기대 결과 불일치 (불일치 항목 기록)
- **Block**: 전제 조건 미충족으로 테스트 실행 불가 (사유 기록)
```

# 문서 작성 원칙

## 역할 준수
- 본 문서는 검증 방법과 테스트 케이스를 정의하는 것이 목적으로, 기능 구현이나 설계 변경은 포함하지 않는다. 무엇을 어떻게 검증하는지에 집중하여 서술한다.

## 명확성
- 모호한 표현 금지 — "적절히", "충분히" 대신 수치나 조건으로 명시
  - 나쁜 예: "정상적으로 동작하는지 확인한다"
  - 좋은 예: "HTTP 200 응답, 본문에 'id' 필드가 UUID 형식으로 존재하며 DB에 동일 레코드가 생성됨"
- 테스트 단계는 누구든 동일하게 재현할 수 있을 만큼 구체적으로 기술

## 일관성
- 모든 테스트 케이스에 동일한 문서 구조 적용
- 테스트 케이스 코드(TC) 채번 일관성 유지
- 판정 기준(Pass/Fail/Block) 용어 통일

## 완결성
- 모든 테스트 케이스에 전제 조건, 실행 단계, 기대 결과, 판정 기준이 명시되어야 한다
- 정상 경로뿐 아니라 예외/에러 케이스도 포함되어야 한다

## 추적 가능성
- 각 테스트 케이스가 어떤 스펙(SVC, FN, SCR, ENT, policy)을 검증하는지 기술
- 스펙 코드 → 테스트 케이스 매핑이 양방향으로 추적 가능해야 한다

# PROC 와의 상호 참조 (필수)

PROC 의사코드의 분기·예외 흐름은 TC 의사코드의 직접 입력이다.
TC 작성 시 다음을 반드시 고려한다.

- PROC 의 모든 BR-/EX- 코드에 대응하는 TC 가 1개 이상 존재해야 한다.
- TC 단계는 PROC 의사코드의 단계명(F1·B2 등)과 1:1 매핑되어야 한다.
- PROC 가 호출하는 FN 의 EX 코드도 TC 검증 대상에 포함된다.
- PROC ↔ TC 양방향 추적: spec-qa.md 매핑 표는 PROC 별 검증 TC 목록을, 각 TC 는 검증 PROC 와 BR-/EX- 코드를 명시한다.

# 산출물 검증 체크리스트

산출물 작성 완료 후 다음을 확인한다 (6 카테고리는 [prd-to-process.md §산출물 검증 체크리스트](prd-to-process.md) 와 동일 형식).

## 적용 범위 검증
- ⬜ 모든 서비스 기능(SVC)의 주요 시나리오에 대응하는 TC 가 존재하는가?
- ⬜ 모든 PROC 의 BR-/EX- 코드 각각에 대응하는 TC 가 1개 이상 존재하는가?
- ⬜ 모든 API(FN)에 정상(2xx) + 클라이언트 오류(4xx) + 서버 오류/타임아웃(5xx) 최소 3개 케이스가 정의되었는가?
- ⬜ 화면(SCR)별 UI/UX 검증 항목이 정의되었는가?
- ⬜ 데이터 정합성 검증(ENT/MDL)이 CRUD 시나리오별로 포함되었는가?

## 정의 완결성 검증
- ⬜ 각 TC 에 전제 조건 / 실행 단계 / 기대 결과 / 판정 기준이 모두 있는가?
- ⬜ 각 TC 단계에 매핑 PROC 단계가 명시되었는가?
- ⬜ 검증 PROC 칸과 검증 분기/예외(BR-/EX-) 칸이 모든 TC 에 채워져 있는가?
- ⬜ 자동화 가능 여부가 모든 케이스에 표기되었는가?

## 실행 제약 검증
- ⬜ 테스트 환경 구성 절차(프로그램 실행, DB 초기화)가 명시되었는가?
- ⬜ 각 SVC 의 기능 복잡도에 따른 최소 TC 수(단순 조회 8개 / 입력·저장 12개 / 복합 흐름 15개)가 충족되었는가?
- ⬜ 각 정책(BIZ-*/AUTH-*/SEC-*)마다 우선순위에 따른 최소 TC 수(P0: 3개, P1: 2개, P2: 1개)가 있는가?
- ⬜ AUTH-* 정책이 적용되는 각 화면(SCR)/도메인별로 권한 미충족 케이스 TC가 독립적으로 포함되었는가?
- ⬜ TC 유형 분포에서 Boundary가 20%를 초과하지 않는가?

## 일관성 검증
- ⬜ 모든 테스트 케이스에 동일한 문서 구조 적용?
- ⬜ TC 코드 채번이 spec-qa.md 목록과 일치하는가?
- ⬜ 판정 기준(Pass/Fail/Block) 용어 통일?
- ⬜ BR-/EX- 코드가 PROC 정의서와 동일 채번으로 인용되었는가?

## 의존관계 검증
- ⬜ 각 TC 가 검증하는 SVC / FN / SCR / ENT / POL / PROC 코드가 모두 실재하는가? (깨진 참조 0건)
- ⬜ PROC ↔ TC 양방향 추적이 가능한가? (PROC 정의서에서 검증 TC 목록 추적 가능, TC 에서 검증 PROC 추적 가능)

## 역할 준수 검증
- ⬜ 기능 구현이나 설계 변경이 포함되지 않고 검증 정의에 집중되었는가?
- ⬜ 민감 정보(비밀번호·접속 키 등)를 직접 기재하지 않았는가?
- ⬜ 자동 테스트와 수동 테스트가 명확히 구분되었는가?
- ⬜ 모호 항목과 해석 근거가 phase 문서의 "검토 및 제안사항"에 기록되었는가?

# 출력 형식

- QA 정의 목록 파일: `docs/specs/qa-dev/spec-qa.md`
- 개별 테스트 케이스 파일: `docs/specs/qa-dev/tc_<test-code>.md`
- 언어: 한국어
- 형식: Markdown
- 테스트 케이스 코드 채번: `TC-001`, `TC-002`, ...

# 주의사항

- 스펙 문서(SVC, FN, SCR, ENT, policy)에 근거하지 않은 테스트 케이스를 임의로 추가하지 않는다.
- 테스트 케이스는 검증 방법만 정의하며, 기능 구현이나 설계 변경을 포함하지 않는다.
- 테스트 환경 정보(실행 절차, DB 접속 등)는 PRD 하위 문서의 원본을 참조하고, QA 문서에 민감 정보(비밀번호, 접속 키 등)를 직접 기재하지 않는다.
- 자동 테스트와 수동 테스트를 명확히 구분하여 실행 계획 수립에 활용할 수 있도록 한다.
- 기존 작성 내용에서 변경이 필요한 경우 변경 지점과 사유, 영향범위 및 추가 검토 제안사항을 정리하여 `docs/specs/qa-dev/spec-qa-changes.md`에 작성한다. 기존 작성 이력 하위에 변경 실행 시점 섹션을 분리하여 내용을 추가한다.

# 메모리 업데이트

PRD 분석 후 다음 사항을 프로젝트 메모리에 기록한다:
- 도출된 테스트 유형별 케이스 수 및 코드 목록
- 스펙 코드 → 테스트 케이스 매핑 관계 (SVC/FN/SCR/POL/**PROC** → TC)
- PROC 분기/예외(BR-/EX-) ↔ TC 매핑 (양방향 추적)
- 자동화 가능 테스트와 수동 테스트의 비율 및 주요 수동 항목
- 테스트 환경 구성에서 주의가 필요한 사항 (외부 의존성, 모킹 전략 등)
- PRD에서 테스트 케이스로 변환 시 판단이 필요했던 항목 (모호한 요구사항, 해석 근거 등)
