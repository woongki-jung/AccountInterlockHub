본 문서는 제품 요구사항을 정의하고 개발 프로세스를 진행하기 위한 세부 스펙을 확정하기 위환 sprint 수행 계획 수립 지침을 제공한다.
스프린트의 주요 수행 목표는 요구사항 분석 및 정합성 평가, 요구사항 기반 세부 사양 정의서 생성, 사양기반 목업 페이지 구현을 순차적으로 진행하는 것이다.
스프린트의 목표를 달성하기 위해 아래 순서에 따라 작업 스프린트 실행 계획을 수립하고 결과를 보고한다.

# 작업 프로세스

## 1단계: 요구사항 문서 확인
아래 사항에 주의하여 요구사항 문서를 확인한다.
- 현재 요구사항 문서
	- **대상 문서 경로**: 
		- **요구사항 문서**: `docs/prd/PRD.md` 및 해당 문서에서 참조하는 모든 하위 문서.
		- **이전 스프린트 피드백 문서**: `docs/sprint-feedbacks/pending/` 폴더 문서. 담당자 최종 피드백 항목으로 반드시 적용되도록 최상위 중요도로 적용하고, 적용이 불가한 경우 관련 내용을 보고한다.
	- 누락되는 문서가 없도록 주의한다.
	- 직접적인 참조가 명시되지 않은 경우, 임의로 관련 파일 여부를 판단하지 않는다.
	- 참조 여부는 항상 상위 문서 -> 하위 문서로 참조하는 경우만 고려한다. 다른 위치에서 검토 대상 문서를 참조하는 문서가 있는 지 찾아보지 않는다.
- GIT 변경 이력 확인
	- `sprints/spec/` 폴더 하위의 마지막 스프린트 기록의 commit 시점을 기준으로 작업 디렉토리와 비교하여 변경된 부분으로 분석 범위를 좁힌다.
	- 기존 sprint가 없으면 전체 요구사항을 분석 대상으로 한다.

## 2단계: 요구사항 분석 그룹 구성
다음 사항을 참고하여 수집된 대상 문서를 연관성 따라 분석 대상 그룹을 분류한다.
### `docs/prd/`의 1차 하위 폴더 구성
- `requirements`: 서비스 정책 및 기능 요구사항
- `design-system`: 디자인 시스템 지침 / 디자인 샘플 이미지 및 분석 md
- `specification`: 시스템 구성 지침
- `quality-manage`: 품질 관리 지침
- `feedbacks`: 사양정의서 생성 스프린트 결과에 대한 추가 피드백 및 조치 요약.
### 지난 sprint 실행 결과
지난 sprint 실행 결과를 확인하여 phase 구성 시 참고사항을 활용하도록 한다.
- `sprints/spec/spec-<n>-<git 유저명>/` 폴더 중 가장 최근 폴더의 ROADMAP 및 phase 실행 결과
### 분석 그룹 지침
- 분석 그룹 분류 시, 기능/비기능 중심으로 1차 분류 후, 기능 항목들을 화면/기능을 중심으로 2차 분류한다.
- 각 분석 그룹은 상호 연관성을 최소화 하여 동시에 분석작업을 수행할 수 있도록 구분한다.
- 이전에 분석이 완료된 항목이 있는 경우, 변경이 발생한 부분에 대하여 집중적으로 분석한다.

## 3단계: ROADMAP 작성

`sprints/spec/spec-<n>-<git 유저명>/ROADMAP.md` 파일에 전체 Phase의 진행 방향과 현황을 작성한다. 다음 구조를 따른다.

```markdown
# 프로젝트 로드맵

## 개요
- 프로젝트 목표 한 줄 요약
- 현재 진행 단계
- 전체 Phase 수 및 예상 일정

## 진행 상태 범례
- ✅ 완료
- 🔄 진행 중
- 📋 예정
- ⏸️ 보류

## 분석 그룹 → Phase 매핑

| 분석 그룹 | 대상 경로 | 매핑 Phase |
|-----------|-----------|------------|
| [그룹명] | `docs/prd/requirements/` | Phase 1 |
| [그룹명] | `docs/prd/design-system/` | Phase 2 |

> 📌 2단계에서 분류한 분석 그룹이 각 Phase에 어떻게 배정되었는지를 명시한다.

## Phase 현황

| Phase | 단계명 | 분석 대상 | 산출물 | 상태 | 완료 기준 | 비고 | 상세 문서 |
|-------|--------|-----------|--------|------|-----------|------|-----------|
| 1 | [단계명] | 대상 문서/그룹 | 산출물 경로 | ✅ | 정합성 검증 통과, Critical 0건 | | [상세](phase-1.md) |
| 2 | [단계명] | 대상 문서/그룹 | 산출물 경로 | 🔄 | 정합성 검증 통과, Critical 0건 | Phase 1 완료 후 착수 | [상세](phase-2.md) |

> 📌 Phase 간 의존관계 및 리스크 사항은 각 Phase 상세 문서에서 관리한다.

## 마일스톤
- 주요 분석 산출물 확정 시점 (예: PRD 분석 완료, 정합성 검증 통과, 스펙 확정)

## 후속 분석 대상
- 현재 분석 범위 밖 항목 및 추후 검토 예정 사항
```

## 4단계: phase 구성
각 Phase별로 `sprints/spec/spec-<n>-<git 유저명>/phase-<n>.md` 파일을 생성한다. 다음 구조를 따른다.

```markdown
# Phase N: [단계명]

## 개요
- **목표**: 이 단계에서 달성하는 분석 목표
- **분석 라운드**: Round N
- **상태**: 진행 상태
- **선행 Phase**: 의존하는 Phase 번호 (없으면 없음)

## 분석 대상 문서
| 분류 | 경로 | 문서명 | 용량 | 비고 |
|------|------|--------|------|------|
| 요구사항 | `docs/prd/requirements/` | 문서명 | 00KB | |
| 디자인 | `docs/prd/design-system/` | 문서명 | 00KB | |
| 이전 작업결과 | `sprints/spec/spec-<n>-<git 유저명>/spec-report.md` | | 00KB | |

## 산출물
| 산출물 | 출력 경로 | 담당 에이전트 | 상태 |
|--------|-----------|---------------|------|
| 정책 스펙 | `docs/specs/policies/` | prd-to-policies | 📋 예정 |
| 서비스 스펙 | `docs/specs/services/` | prd-to-service | 📋 예정 |

## 분석 작업 목록
- ⬜ **[분석 항목명]**: 분석 내용 설명
  - 대상 문서: `경로`
  - 분석 관점: 정합성 / 완전성 / 명확성
  - 참고 피드백: `docs/prd/feedbacks/관련파일.md`

## 완료 기준 (Exit Criteria)
- 분석 대상 문서 전수 검토 완료
- 정합성 검증 통과 (Critical 0건)
- 산출물 스펙 문서 생성 완료
- 교차 검증(문서 간 충돌·중복·불일치) 완료

## 요구사항 품질 검증
### 문서 정합성 검증
- ⬜ **[PRD ↔ 스펙 정합성]**: 요구사항 ID별 스펙 반영 여부 확인
  - 검증 방법: 요구사항 추적 매트릭스 대조
  - 통과 기준: 미반영 요구사항 0건

### 문서 간 교차 검증
- ⬜ **[스펙 간 일관성]**: 산출물 문서 간 용어·데이터 구조 일치 확인
  - 검증 방법: 엔터티명·필드명·용어 교차 대조
  - 통과 기준: 불일치 항목 0건

## 요구사항 분석 결과
### 분석 이슈 요약
| 수준 | 건수 | 주요 내용 |
|------|------|-----------|
| Critical/High | 0 | - |
| Medium | 0 | - |
| Low | 0 | - |

### 검증 결과
| 검증 항목 | 결과 | 위험도 | 비고 |
|-----------|------|--------|------|
| PRD ↔ 스펙 정합성 | ✅ / ⚠️ / ❌ | Critical / High / Medium / Low | 상세 내용 |

### 검토 및 제안사항

### 요구사항 모호·누락 발견사항

## 사양정의서 작업 결과
- 분석 결과로 정리된 상세 사양정의서의 목록과 개요를 작서성

|--------- | -------- | ---- | -------- |
| `경로` | 정책정의서 | 주요 서술내용 | 최초작성 또는 변경 개요 |
```

# 제약사항 및 원칙
- 모든 작업은 이전 sprint의 진행 사항을 참고하여 추가/변경에 대하여 빠짐없이 서술한다.
- 기존 작업 사항의 변경은 다른 방법으로 대체 불가능하거나 요구사항에 기존 요건에 대한 변경이 필요한 점임을 명시한 경우에만 진행한다.

# 메모리 업데이트
실행 완료 후 다음 사항을 프로젝트 메모리에 기록한다:
1. **로드맵 구성 결과** — 생성된 Phase 수, 각 Phase별 분석 그룹 배정 요약. 다음 실행 시 기존 로드맵과의 변경점을 파악하는 데 활용.
2. **분석 대상 문서 현황** — 수집된 전체 문서 수, 총 용량, 분석 그룹 분류 결과. PRD 문서가 추가/변경되었을 때 계획 재수립의 기준이 됨.
3. **계획 수립 중 발견된 문서 구조 이슈** — 참조 누락, 접근 불가 문서, 예상 외 대용량 파일 등. 반복 발생 시 PRD 관리 개선 피드백으로 활용.
4. **계획 수립 중단 시 중단 지점 및 사유** — 재실행 시 이어서 진행하거나 동일 문제를 회피하기 위한 정보.
