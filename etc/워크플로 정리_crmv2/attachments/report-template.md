# 테스트 SPRINT 리포트 템플릿

본 문서는 `ai/agents/start-qa.md`의 3-2단계에서 생성하는 결과 보고서 파일의 작성 형식을 정의한다.

## 작성 원칙

- 모든 실패·Block·경고·후속 액션은 **주관 담당자(기획 / 디자인 / 개발 / QA)** 기준으로 그룹화하여 작성한다 (CLAUDE.md "제안 및 보고" 정책).
- 실패 항목은 표 대신 **섹션 형식**으로 작성하며, 항목별로 재현 절차·기대/실제 동작·오류 로그·예상 원인·해결 방안을 모두 포함한다.
- 담당 매핑 가이드:
  - **기획**: TC 기대 동작이 PRD/스펙과 불일치하거나 정의가 모호한 경우, 정책 결정 필요 항목
  - **디자인**: 레이아웃·간격·색상·아이콘·타이포·인터랙션 피드백·a11y 등 시각/UX 결함
  - **개발**: 기능 결함·예외 처리 미흡·API 오류·성능 저하·셀렉터(AutomationId/testId) 미비
  - **QA**: 환경 미비·외부 의존성 장애·TC 자동화 스크립트 결함·Flaky·셀렉터 best_match 우회

```markdown

**종합 평가**: ✅ 완료 / ⚠️ 조건부 통과 / ❌ 실패 — Pass N / Fail N / Block N
**대상 빌드**: <commit SHA>
**실행 기간**: <YYYY-MM-DD HH:mm ~ YYYY-MM-DD HH:mm>
> 종합 판정 사유 한 줄 요약

## 품질 게이트 판정

- High TC Pass율: N% (기준: ≥ 90%)

## Phase별 실행 결과
| Phase | TC ID / 시나리오 | 도메인 | 유형 | 자동화 | 종합 판정 | 소요시간 | 주관 담당 | 비고 |

| 1 | LAYOUT_001 | LAYOUT | E2E | 자동 | 🟢 Pass | Ns | 개발 | - |

## TC 실패·차단 스냅샷 링크

> 본 스프린트 종료 시 `start-qa §3-4` 가 후속 처리용 스냅샷 파일을 생성한다.
> 스냅샷은 **리프별 결과 집계(+합계)와 Critical·중요 결함 요약** 중심이다. 개별 TC 전체는 각 leaf `qa-report.md` 가 SoT.

- **본 스프린트 스냅샷**: `sprints/qa/<YYYY-MM-DD>/tc-followup.md`
- **직전 스냅샷 대비 변화**:

| 변동 유형 | 건수 |
|---------|----:|
| 🆕 신규 Fail | N |
| 🆕 신규 Block | N |
| ✅ 해소 (직전 Fail/Block → 본 스프린트 부재 or Pass) | N |
| 🔁 결과 변경 (Fail ↔ Block) | N |

본 스프린트 합계: 🔴 Fail N건 / 🟠 Block N건. 개별 TC ID·기대/실제 결과는 각 leaf `qa-report.md` 참조 (tc-followup 은 Critical 요약 중심).
  ```
