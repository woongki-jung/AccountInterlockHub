본 문서는 프로젝트 수행을 위한 요구사항을 최적화하기 위한 프로세스를 정의한다.
아래 지침에 따라 PRD와 서비스 정책, 데이터 설계, 비즈니스 로직 구현, 인터페이스 정의, 화면 정의 사항들을 검토하고 상호간 누락 또는 충돌하는 내용이 있는 지 검토하고 보완해야 할 점을 정리한다.

# 실행 시 전달 파라메터
- 작업 phase 문서 경로 (`sprints/spec/spec-<n>-<git유저명>/phase-<n>.md`)
- 유효한 값이 전달되지 않은 경우 경고 노출후 작업 중단.

# 작업 프로세스
**시작 시 공지 노출** "요구관리 정합성 분석 스킬을 사용하여 평가를 시작합니다."

## 1단계: 요구사항 분석
시작 시 전달받은 phase 문서의 내용을 확인하고, 분석 대상 문서를 확인한다. 분석 대상 문서와 각 문서가 참조하는 연결 문서의 내용을 자세히 분석한다.

## 2단계: 요구사항 정합성 분석
### 정합성 분석 프로세스
1. **계획 수립**: 분석 대상 파일의 위치와 용량에 따라 적정 단위로 나누어 분석 계획을 수립한다.
2. **개별 분석**: 계획에 따라 각 문서를 별도의 세션으로 독립적으로 검토하여 적합도 판단 기준에 따라 문제 항목을 도출한다. 세션 분리 시, 분석의 목표와 분석 방법이 유지될 수 있도록 지침을 공유한다.
3. **교차 검증**: 문서 간 충돌·중복·불일치 여부를 확인한다.
4. **등급 분류**: 도출된 문제 항목에 평가 기준에 따라 위험도 등급을 부여한다.
### 요구사항 분석 기준
요구사항 내용을 확인함에 있어 다음 항목에 유의한다.
- **작성된 내용 그대로 해석한다**: 문장의 완결성이 부족해 정확한 해석을 위해 추가정보가 필요하더라도 내용 해석을 위해 임의의 해설을 추가하지 않는다.
- **연관성을 추측하지 않는다**: 파일명 또는 작성 내용이 유사하거나 동일한 내용에 대한 서술이라 하더라도 문서상 직접적인 연관성에 대한 서술이 없다면 임의로 관계성을 판단하지 않는다.
- **누락 없이 진행한다**: 대상 요구사항 문서는 누락되는 파일이 없도록 정확히 분석한다. 이미지, PDF 문서 등을 분석해야 하는 경우 인식률이 충분한 지에 대한 분석을 진행하고, 인식률이 낮다고 판단되는 경우 평가에 반영한다.
- **분석 범위를 준수한다.**: 지정된 분석 대상 문서 이외의 경로에 있는 문서를 포함하지 않는다. 외부 경로에 대한 직접 잠조가 있는 경우는 요구사항 자료 구성 오류로 처리한다.
- **분석 대상 범위를 조절한다.**: 분석 대상 데이터가 많아 context 사용량이 50%를 넘지 않도록 중간중간 분석 내용을 요약하며 진행한다.
### 요구사항 적합도 판단 기준
- 모호하게 정의된 요구사항
- 완성된 문장이 아닌 누락 내용이 있는 요구사항
- 계정정보, 개인정보, 각종 키값 등 민감정보를 포함하는 요구사항
- 다른 요구사항과 충돌하는 내용
- 내용은 다르지만 동일 기능을 서술하는 요구사항 간 중복
- 불법성이 있는 요구사항
- 데이터 목록 등 상황에 따라 과부하가 발생할 수 있는 요소에 제한 범위가 없는 경우
- 의도가 포함되지 않은 요구사항 변경
- 현재 기술 스택 또는 환경에서 구현이 불가능한 요구사항
- 검증 기준이나 완료 조건이 정의되지 않아 테스트가 불가능한 요구사항
### 평가 기준
적합도 판단 기준에 따라 문제의 수준을 분석하고 아래 등급으로 정의한다.

| 등급              | 기준                                                                                                  |
| --------------- | --------------------------------------------------------------------------------------------------- |
| 🔴 **CRITICAL** | 즉시 중단이 필요한 수준. 민감정보 노출, 불법적 요구사항, 서로 다른 핵심 요구사항 간 직접적 충돌처럼 그대로 진행 시 프로젝트 전체에 돌이킬 수 없는 손상을 줄 수 있는 항목 |
| 🟠 **ERROR**    | 구현 전 반드시 해소되어야 할 결함. 핵심 흐름의 누락, 완성되지 않은 요구사항, 핵심 기능의 정의 부재 등 구현이 불가능하거나 잘못된 결과물로 이어지는 항목            |
| 🟡 **WARNING**  | 개선이 권장되나 현 상태로도 진행 가능한 항목. 모호한 표현, 제한 범위 미지정, 의도가 불명확한 변경사항 등 향후 재작업이나 품질 저하로 이어질 수 있는 항목           |
| 🟢 **SUCCESS**  | 요구사항이 명확하게 완결되어 보완 없이 구현 가능한 항목                                                                     |

## 3단계: 요구사항 고도화 제안
이전 단계의 요구사항 분석 내용을 바탕으로 목표에 부합하는 범위에서 새로운 요구사항 제안을 결과보고서에 추가한다.
새로운 요구사항 제안은, 정책, 기능, 화면, 데이터 구조 등 범위를 가리지 않고 모든 내용에 대해 고려하고, 최대 3가지 까지만 제시한다.
새로운 요구사항 제안은 반드시 기존 내용을 참고하여 주제를 선정하되 기존에 언급되지 않은 내용으로 구성한다.
새로운 요구사항은 고객/서비스/사업성 관련 내용을 우선으로 고려한다.
제안은 다음 구조로 제시한다.
- 제안 주제: 제안하는 내용을 한마디로 요약
- 제안 상세: 적용 범위와 상세한 내용
- 제안 이유: 제안사항을 적용했을 때의 효과
- 적용 예시: 제안 사항을 요구사항에 적용하기 위한 샘플
- 추가 보완 사항: 제안 사항에 덧붙여 줄 수 있는 내용
