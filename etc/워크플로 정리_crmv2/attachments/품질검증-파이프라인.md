# 품질 검증 파이프라인 에이전트

`start-qa` 진입점이 자동으로 호출하는 하위 에이전트 2종의 실행 시점·입력·결과 정리다.
상위 문서: [프롬프트 동작 모음](프롬프트%20동작모음.md)

이 파이프라인은 **인스톨러 설치본 → 품질 리포트**로 이어진다.
`start-qa`는 실제 사용자 환경과 동일한 설치본을 대상으로 검증하는 것이 원칙이다.

```
start-qa
 ├ 사전   인스톨러 빌드 + 자동 설치   (start-qa가 직접 수행)
 │          apps/deploy/scripts/build.ps1 -Mode {Test|Release}
 │          → /VERYSILENT 자동 설치 → 표준 경로 exe 정합 확인
 ├ 1단계  test-planner            → 테스트 계획(ROADMAP·Phase·TC 매핑)
 ├ 2단계  tester                  → Phase마다 1회 TC 실행·판정
 └ 3단계  (start-qa 직접)          → 결과 집계·리포트·커밋 (별도 closer 없음)
```

> `start-qa`는 spec/build와 달리 **마무리(closer)를 별도 에이전트에 위임하지 않고** 진입점이 직접 결과 집계·리포트·푸시까지 수행한다.

## 최초 1회 · QA 사전 환경 구성

QA 자동화 도구(MCP·pywinauto·playwright)는 검증 PC마다 **최초 1회** 구성해야 한다.
전체 절차·명령·트러블슈팅은 [`tc환경구축.md`](../tc환경구축.md) 전문을 단일 출처로 따른다.

- 도구 매핑 · Delphi(VCL)=pywinauto `win32` / WinForms=pywinauto `uia` / WebView2 웹=playwright `CDP`.
- 구성 순서 · ① 런타임·도구 → ② MCP 설치 → ③ MCP 등록 → ④ 연결 검증 → ⑤ 초기 셀렉터 수집.

### 구성 체크리스트
- ⬜ **런타임·도구** · Python 3.10+ · Node 18+ · WebView2 Runtime · .NET 6+ · Claude Code · (권장) Inspect.exe·Spy++
- ⬜ **MCP 설치** · pywinauto-mcp(venv `pip install -e ".[all]"`) · playwright-mcp(`npm i -g @playwright/mcp`) · WebView2 CDP 9222 활성화
- ⬜ **MCP 등록** · `claude mcp add pywinauto/playwright -s project …`
- ⬜ **연결 검증** · `/mcp` 에서 두 서버 `connected` · `http://localhost:9222/json` WebView 노출
- ⬜ **초기 셀렉터** · `docs/control-catalog.md`·`docs/app-registry.md` 정리 후 `CLAUDE.md` `@` 참조

### 실행 안내
아래 프롬프트로 에이전트가 문서 전문에 따라 환경을 구성하도록 지시한다.

```
etc/workflow-guide/tc환경구축.md 문서 전문을 읽고, 정의된 QA 실행 사전 환경 구성 절차를 순서대로 수행해줘.
- 각 단계 완료 여부를 체크리스트로 보고 (이미 구성된 항목은 "확인됨")
- 수행 불가·수동 조치 필요 항목(관리자 권한·사내 보안 등)은 사유·대응 가이드와 함께 별도 보고
- 문서에 명시되지 않은 임의 변경은 하지 않는다
```

> TC 실행 규칙은 [`tc-guide.md`](../tc-guide.md) 참조.

## 사전 단계 — 인스톨러 빌드 + 자동 설치

- **수행 주체**: `start-qa` 본체 (계획 수립 직전 필수).
- **동작**: 지정 모드(Test 기본 / Release)로 인스톨러를 1종 빌드하고, InnoSetup 무인 옵션(`/VERYSILENT /SUPPRESSMSGBOXES /NORESTART /CLOSEAPPLICATIONS`)으로 자동 설치한 뒤 표준 경로(`D:\YSR2000\YSRCRMv2\App\`)의 exe 정합을 확인한다.
- **의미**: 워크스페이스의 개발 빌드가 아니라 **실제 배포되는 설치본**을 검증 대상으로 고정한다.
- **결과별 분기**: 빌드/설치 실패 시 계획 진입을 보류하고 곧바로 결과 기록 단계로 이행한다.

## 하위 에이전트 (2종)

### test-planner (`ai/agents/workflow-qa/test-planner.md`)
- **모델 / 호출 주체**: opus / start-qa (1단계)
- **실행 시점**: 사전 설치가 성공한 직후, TC 실행에 앞서 테스트 계획을 수립한다.
- **입력 예시**:
  ```
  테스트 계획을 수립해줘.
  범위: 발송 도메인 / TC 스펙: docs/specs/qa-dev/ · docs/specs/qa/
  참조 피드백: docs/sprint-feedbacks/pending/
  ```
- **내부 처리**: 사양·TC 색인 → 실행 환경(MCP·설치본·self-host URI) 확인 → ROADMAP 작성 → Phase당 단일 TC/시나리오로 Phase 문서 생성.
- **Phase 분할 규칙**: Phase ≤ 30 단일 진행 / 30~45 우선순위 분할(Critical·High 우선, 나머지 이월) / 45 초과 도메인 2분할.
- **실행 결과**: `sprints/qa/qa-<n>-<git유저명>/ROADMAP.md`·`phase-<n>.md`와 UI 트리 덤프(`dumps/native-tree.json`·`webview-tree.json`)가 생성되어 tester가 곧바로 실행할 수 있는 계획이 완성된다.
- 원본: [test-planner.md](../../../ai/agents/workflow-qa/test-planner.md)

### tester (`ai/agents/workflow-qa/tester.md`)
- **모델 / 호출 주체**: sonnet / start-qa (2단계, Phase마다 1회) · build-sprint-worker (제품 생산 시 QC)
- **실행 시점**: 계획된 Phase 단위로 실제 TC를 실행할 때, 또는 빌드 스프린트에서 구현 결과를 독립 재현 검증(QC)할 때 실행된다.
- **입력 예시**: `작업 phase 문서: sprints/qa/qa-500-정웅기/phase-3.md 를 실행해줘`
- **내부 처리**: 요구사항 분석 → 실행 가능 환경 확인 → `tmp/tc-<TC코드>/`에 자동화 스크립트 작성 → 실행·판정 → 결과 보고 → 임시 파일 정리.
- **판정 5종 체계**:

  | 등급 | 의미 | 신뢰도 |
  |------|------|:------:|
  | 🟢 Pass | 실 BE·DB·외부 시스템 응답으로 검증 완료 | 100% |
  | 🔵 Pass-Mock | UI·기능은 정상이나 Mock/시드 의존 | ~70% |
  | 🟣 Pass-Static | 사양↔소스 코드 정합 trace만, 동적 검증 없음 | ~50% |
  | 🔴 Fail | 기대 결과 미충족 (증빙·제안 첨부) | — |
  | 🟠 Block | 환경·의존성으로 수행 불가 (사유·해결안 필수) | — |

- **실행 결과**: Phase 문서 "검증 결과"가 5종 등급으로 채워지고, phase별 `qa-report.md`(집계·Fail/Block 목록·증빙)가 생성되며 임시 스크립트는 정리된다.
- **주의**: Fail·Block은 TC ID를 개별 1건 1행으로 명시하고, 자동화 도구는 `pywinauto-mcp`(uia·win32) / `Playwright MCP` 공식 표기만 사용한다.
- 원본: [tester.md](../../../ai/agents/workflow-qa/tester.md)

## 마무리 (start-qa 직접 수행)

- **결과 집계**: 각 Phase의 5종 판정을 도메인·자동화 수준별로 집계하고, Critical·High는 🟢 Pass만 품질 게이트에 반영한다.
- **리포트 작성**: 담당 파트(기획·디자인·개발·QA)별로 그룹화한 `qa-report.md`/`qa-dev-report.md`를 생성한다 (형식은 [report-template.md](../../../ai/agents/workflow-qa/report-template.md) 기준).
- **후속 스냅샷**: 마일스톤 전체를 롤업한 `sprints/qa/<날짜>/tc-followup.md`(리프별 집계 + Critical 요약 + 변화 + 방향)를 1회 생성한다.
- **커밋**: `[start-qa]` 접두사로 커밋·푸시하고, 마지막 스프린트 사후에는 검증용으로 띄운 프로세스(CRM·WebView2·Playwright 브라우저)를 정리한다.
- **resolved 피드백 검증 모드**: 사용자가 명시 호출한 경우에만 `docs/sprint-feedbacks/resolved/`를 검증해 결과에 따라 `completed/<날짜>/`·`failed/`·`blocked/`로 이동한다.
