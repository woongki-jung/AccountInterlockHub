본 문서는 프로젝트 구현 과정에서 공통으로 사용되어야 할 비즈니스 로직을 서술하기 위한 규칙을 정의한다.
아래 지침에 따라 제공된 요구사항 및 관련 문서를 기반으로 프로젝트 전반에서 참조될 공통 비즈니스 로직 정의를 수행한다.

용어 약어 — FN은 공통 기능 정의서 코드, MDL은 데이터 모델, ENT는 DB 엔터티, SVC는 서비스 시나리오, POL(=policy)은 정책 규칙, PROC는 기능구현 프로세스 정의서, DTO는 데이터 전송 객체(Data Transfer Object), HTTP status는 응답 상태 코드를 의미한다.

# 실행 시 전달 파라메터
- 작업 phase 문서 경로 (`sprints/spec/spec-<n>-<git유저명>/phase-<n>.md`)
- 유효한 값이 전달되지 않은 경우 경고 노출후 작업 중단.

# 작업 프로세스

## 1단계: 요구사항 분석

- 시작 시 전달받은 phase 문서의 내용을 확인하고, 분석 대상 문서를 확인한다.
- 분석된 내용에서 다음 항목을 추출한다.
	- 2개 이상의 서비스(SVC)에서 반복되는 로직 패턴.
	- 정책(policy) 규칙을 코드로 구현해야 하는 항목 (검증, 권한 확인, 데이터 처리 등).
	- 서비스 데이터 모델(MDL)과 DB 엔터티(ENT) 간 변환 로직.
	- 외부 시스템 연동에 필요한 공통 처리 (API 호출, 인증, 에러 처리 등).
	- 횡단 관심사 (로깅, 감사, 캐싱, 에러 핸들링 등).

## 2단계: 공통 기능 식별 및 분류

- 1단계에서 추출한 항목을 공통 기능 단위로 그룹화한다.
- 다음 기준으로 공통 기능을 분류한다.
	- **정책 구현(POL)**: 정책 규칙을 실행하는 로직 (권한 검증, 입력값 검증, 비즈니스 규칙 적용 등).
	- **데이터 처리(DAT)**: 데이터 변환, 매핑, 집계, 포맷팅 등.
	- **외부 연동(EXT)**: 외부 API 호출, 인증, 재시도, 에러 핸들링 등.
	- **횡단 관심사(CRS)**: 로깅, 감사 추적, 캐싱, 에러 처리, 알림 등.
- 공통 기능 코드 체계를 설계한다 (FN-nnn 형태).

## 3단계: 기능 상세 설계

- 각 공통 기능에 대해 다음을 정의한다.
	- 입력 파라미터 및 출력 (MDL 코드 매핑 우선).
	- 처리 흐름 — **의사코드 수준** (조건문·반복·예외 분기 명시).
	- 참조하는 정책 규칙과 해당 규칙의 구현 방식.
	- 참조하는 DB 엔터티(ENT) 및 데이터 모델(MDL).
	- 에러 케이스 및 에러 응답 정의 (HTTP status + EX 코드).
- 기능 간 의존관계를 파악한다.
	- 한 기능이 다른 공통 기능을 호출하는 경우 시그니처(인자·반환·예외) 명시.
	- 순환 의존이 발생하지 않도록 검증.

## 4단계: API 인터페이스 정의

- 서비스 기능(SVC)을 외부에 노출하는 API를 정의한다.
	- 엔드포인트 경로 및 HTTP 메서드.
	- 요청/응답 DTO (MDL 참조 + JSON 스키마 의사코드).
	- 인증/인가 요구사항 (AUTH/SEC 정책 참조).
	- 에러 응답 코드 카탈로그 (HTTP status + EX 코드 + 메시지 + 발생 조건).
	- 요청 제한 정책 (Rate Limiting 등 OPS 정책 참조).
- API 간 공통 규칙을 정의한다.
	- 공통 응답 포맷 (성공/실패 구조).
	- 페이지네이션, 정렬, 필터링 파라미터 규약.
	- 에러 코드 체계.
	- API 버전 관리 규칙.

## 5단계: 공통 기능 정의 문서 생성

### 5-1: 공통 기능 목록 문서 생성

`docs/specs/functions/spec-functions.md` 파일에 전체 공통 기능 정의서의 개요와 하위 문서 목록을 작성한다.

- 공통 기능 설계 원칙 및 기본 방향.
- 공통 기능 코드 체계 (FN-nnn).
- 기능 분류별 목록.
- API 인터페이스 목록 (API 카탈로그 표 — 엔드포인트 / 메서드 / 호출 PROC / 인증 / 응답 DTO).
- 기능 간 의존관계 요약 (그래프 또는 표).
- 정책(policy) → 공통 기능(FN) 매핑 요약.
- 이전 버전 대비 변경사항 — 세부 개정사항은 별도 문서(`docs/specs/functions/spec-functions-changes.md`)로 분리.

### 5-2: 개별 공통 기능 정의 문서 생성

각 공통 기능별로 `docs/specs/functions/function_<function-code>.md` 파일을 생성한다.
세부 공통 기능 정의는 다음 템플릿 구조를 따른다.

```markdown
# [기능명] 공통 기능 정의

## 개요
- **기능 목적**: 이 공통 기능이 해결하려는 문제 또는 제공하려는 가치
- **관련 PRD 요구사항**: PRD 내 관련 섹션 및 원문 인용

---

## [FN 코드] [기능명]

### 기본 정보
| 항목 | 내용 |
|------|------|
| 기능명 | 공통 기능 이름 |
| 분류 | POL / DAT / EXT / CRS 중 1개 |
| 사용 서비스 | 이 기능을 사용하는 SVC 코드 목록 |
| 호출 PROC | 본 FN 을 호출하는 PROC 코드 목록 |
| 연관 정책 | 이 기능이 구현하는 POL 규칙 ID 목록 |
| 참조 데이터 | 관련 ENT / MDL 코드 |

### 시그니처

```
function <FN 코드> (
  <param1>: <type1>,        // 설명·제약(필수·길이·범위)
  <param2>: <type2>,        // 선택 사항 표기 + 기본값
): <ReturnType>
  throws <ExceptionType1> { code: EX-xxx, http: 400 }
        | <ExceptionType2> { code: EX-yyy, http: 422 }
```

### 입력/출력 정의

| 구분 | 항목명 | 데이터 타입 | 필수 | 제약 | 설명 |
|------|--------|------------|------|------|------|
| 입력 | param_a | string | Y | NotBlank, MaxLength(20) | 파라미터 설명 |
| 입력 | param_b | int | N | Range(1..100), default 10 | 페이지 크기 |
| 출력 | result | MDL-xxx | - | - | 반환 데이터 모델 |

### 처리 흐름 (의사코드)

```
1. 입력 검증
   if (!isValid(param_a))             → throw EX-FN-001 (400)
   if (param_b > 100)                 → throw EX-FN-002 (422)

2. 정책 적용 — POL <그룹>-NNN-RR (validate)
   if (!hasPermission(user, "READ"))  → throw EX-AUTH-001 (401)

3. 사전 조회
   SELECT id, name, status FROM TBL_<ENT>
   WHERE clinic_id = :clinic_id AND status != 'DELETED'
   LIMIT :param_b;

4. 도메인 변환
   domain = ENT 행 → MDL-xxx (FE 노출 필드만 선별)

5. 응답 반환
   return domain
```

> 처리 흐름은 [prd-to-process](prd-to-process.md) §3-2 의 BE 측 의사코드 작성 원칙과 동일 깊이를 유지한다.
> "API 호출", "데이터 조회", "응답 처리" 같은 추상 표기는 사용하지 않는다.

### API 인터페이스 (해당하는 경우)

| 항목 | 내용 |
|------|------|
| 엔드포인트 | POST /api/<resource> |
| HTTP 메서드 | POST / GET / PUT / DELETE |
| 인증 요구 | Bearer 토큰 / API Key / Public |
| 요청 DTO | RequestDto-xxx (필드별 타입·제약 명시) |
| 응답 DTO (200) | ResponseDto-xxx |
| 응답 DTO (4xx/5xx) | ErrorResponseDto (code, message, details) |
| Rate Limiting | OPS-NNN-RR 적용 (예: 60 req/min) |

### 에러 처리 (에러 코드 카탈로그)

| HTTP status | EX 코드 | 발생 조건 | 사용자 메시지 | 개발자 노트 |
|-------------|---------|-----------|---------------|-------------|
| 400 | EX-FN-001 | 입력 형식 위반 | "입력값이 올바르지 않습니다." | 필드별 errors 배열 동봉 |
| 401 | EX-AUTH-001 | 토큰 누락·만료 | "로그인이 필요합니다." | POL AUTH-001 위반 |
| 422 | EX-FN-002 | 비즈니스 규칙 위반 | (규칙별 상이) | POL 규칙 ID 명시 |
| 500 | EX-FN-999 | 시스템 오류 | "잠시 후 다시 시도해주세요." | 감사 로그 필수 |

### 의존 기능

| FN 코드 | 호출 시점 | 동기/비동기 | 실패 시 처리 |
|---------|----------|------------|--------------|
| FN-xxx | 단계 N | 동기 | 호출 실패 시 EX-FN-yyy 전파 |

### 구현 가이드
- 개발자가 이 기능을 구현할 때 참고할 핵심 사항.
- 관련 기술 패턴 (특정 라이브러리·프레임워크 강제 금지).

### 변경 이력
| 일자 | 변경 내용 | 사유 | 영향 범위 |
|------|----------|------|----------|
| YYYY-MM-DD | 변경 요약 | 변경 사유 | 영향 PROC/SVC |
```

# 문서 작성 원칙

## 역할 준수
- 본 문서는 공통 비즈니스 로직과 API 인터페이스를 정의하는 것이 목적으로, UI 구성이나 DB 스키마 상세는 포함하지 않는다.
- PROC 가 호출하는 단위 로직(FN)은 본 문서에 정의되며, PROC 의사코드는 본 문서의 시그니처로 호출된다.

## 명확성
- 모호한 표현 금지 — "적절한 에러 처리를 한다" 대신 "인증 실패 시 HTTP 401을 반환하고, 응답 본문에 EX-AUTH-001 코드를 포함한다".
- 처리 흐름은 개발자가 코드로 바로 변환할 수 있을 만큼 의사코드 수준으로 작성.
- 시그니처(인자·반환·예외)를 모두 명시한다.

## 일관성
- 모든 공통 기능에 동일한 문서 구조 적용.
- 공통 기능 코드(FN) 채번 일관성 유지.
- 에러 코드 체계 통일 (EX-<도메인>-<순번>).

## 완결성
- 모든 기능에 입력/출력 / 시그니처 / 처리 흐름 의사코드 / 에러 처리 카탈로그가 명시되어야 한다.
- 참조하는 정책·서비스·데이터 코드가 빠짐없이 기술되어야 한다.

## 추적 가능성
- 각 기능이 PRD의 어떤 요구사항에서 비롯되었는지 기술.
- 기능과 연관된 POL / SVC / ENT / MDL / 호출 PROC 코드 기술.
- 기능 간 의존관계 기술.

# PROC 와의 상호 참조 (필수)

PROC 의사코드는 본 문서의 FN 코드를 호출한다.
FN 작성 시 다음을 반드시 고려한다.

- 한 FN 은 0~다수의 PROC 에서 호출될 수 있다.
- FN 시그니처(인자·반환·예외)는 PROC 의사코드의 호출 라인에서 그대로 사용 가능해야 한다.
  예: PROC 의사코드 `result = FN-xxx(param_a, param_b)` ↔ FN 시그니처 정합.
- FN 에서 throw 하는 EX 코드는 PROC 의 분기/예외 표에 등재되어야 한다.

# 산출물 검증 체크리스트

산출물 작성 완료 후 다음을 확인한다 (6 카테고리는 [prd-to-process.md §산출물 검증 체크리스트](prd-to-process.md) 와 동일 형식).

## 적용 범위 검증
- ⬜ 2개 이상 서비스에서 동일한 내용이 반복되는 로직이 공통 기능으로 식별되었는가?
- ⬜ 모든 정책 규칙에 대응하는 구현 기능(POL 분류 FN)이 매핑되었는가?
- ⬜ 모든 PROC 가 호출하는 FN 코드가 본 문서에 실재하는가?

## 정의 완결성 검증
- ⬜ 각 기능의 입력/출력이 MDL 코드 참조와 함께 정의되었는가?
- ⬜ 시그니처(인자·반환·예외)가 코드 호출 가능 수준으로 명시되었는가?
- ⬜ 처리 흐름이 의사코드 수준으로 작성되었는가? (추상 표기 0건)
- ⬜ 에러 처리 카탈로그가 HTTP status + EX 코드 + 발생 조건 + 메시지 + 개발자 노트로 작성되었는가?
- ⬜ API 인터페이스의 엔드포인트, 메서드, 요청/응답 DTO 가 명시되었는가?

## 실행 제약 검증
- ⬜ Rate Limiting 등 OPS 정책 적용이 필요한 API 에 적용 정책 코드가 인용되었는가?
- ⬜ 외부 연동(EXT) FN 의 타임아웃·재시도·실패 처리가 명시되었는가?

## 일관성 검증
- ⬜ 모든 공통 기능에 동일한 문서 구조 적용?
- ⬜ FN 코드 채번이 spec-functions.md 목록과 일치하는가?
- ⬜ EX 코드 체계가 통일되었는가? (`EX-<도메인>-<순번>`)

## 의존관계 검증
- ⬜ 기능 간 의존관계에 순환 의존이 없는가?
- ⬜ 호출 PROC 가 모두 실재하는가? (깨진 참조 0건)
- ⬜ 인용된 POL / MDL / ENT / SVC 코드가 모두 실재하는가? (깨진 참조 0건)

## 역할 준수 검증
- ⬜ UI 구성·표시 형식 등 화면 정의서에 속하는 내용이 포함되지 않았는가?
- ⬜ DB 스키마 상세(컬럼 타입·인덱스 등)는 ENT 정의서로 분리되었는가?
- ⬜ 특정 프레임워크·라이브러리를 강제하는 표현이 없는가?
- ⬜ 모호 항목과 해석 근거가 phase 문서의 "검토 및 제안사항"에 기록되었는가?

# 출력 형식

- 목록 파일: `docs/specs/functions/spec-functions.md`
- 개별 파일: `docs/specs/functions/function_<function-code>.md`

# 주의사항

- PRD, 정책 정의, 서비스 기능 정의, 데이터 정의에 근거하지 않은 기능을 임의로 추가하지 않는다.
- 특정 구현 기술(프레임워크, 라이브러리)을 강제하지 않고 기능의 요구사항만 정의한다.
- 공통 기능은 2개 이상의 서비스에서 재사용되는 로직만 대상으로 한다.
  단일 서비스 전용 로직은 해당 서비스 기능 정의(SVC)에 포함한다.
- 기존 작성 내용에서 변경이 필요한 경우 변경 지점과 사유, 영향범위 및 추가 검토 제안사항을 정리하여 `docs/specs/functions/spec-functions-changes.md`에 작성한다.

# 메모리 업데이트

PRD 분석 후 다음 사항을 프로젝트 메모리에 기록한다:
- 도출된 공통 기능 분류 체계 및 코드 목록.
- POL → FN 매핑 관계.
- FN ↔ PROC 호출 관계 (호출 빈도 상위 항목).
- 기능 간 주요 의존 관계.
- API 인터페이스의 공통 규칙 (응답 포맷, 에러 코드 체계 등).
- EX 코드 카탈로그 요약.
- PRD에서 공통 기능으로 변환 시 판단이 필요했던 항목.
