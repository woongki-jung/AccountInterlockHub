# 작업 프로세스

## 1단계: 스프린트 진행 상태 확인
시작 시 전달받은 phase 문서의 내용을 확인하고, 완료 여부를 확인한다.

- `sprints/spec/spec-<n>-<git 유저명>/ROADMAP.md`를 읽고 전체 Phase 현황을 파악한다.
- 각 Phase 문서 `sprints/spec/spec-<n>-<git 유저명>/phase-<n>.md`에 결과 항목이 입력되고 상태값이 완료인지 확인

## 2단계: 작업 결과 요약
- phase별 실행 결과를 취합하여 `sprints/spec/spec-<n>-<git 유저명>/spec-report.md`에 작성한다.
- 동일한 이름의 기존 파일이 있는 경우 실행 주기를 구분하여 기존 문서의 아래에 내용을 추가한다.
- 최종 결과서는 마지막으로 작성된 이전 결과서와의 내용을 비교하여 변경점에 대한 분석 섹션을 추가한다.

**리포트 구조:**
```markdown

# 요건 분석 SPRINT 실행 결과 보고

**종합 평가**: ⚠️ 조건부 통과 — 경고 2건, 실패 0건
> 경고 항목은 Critical 수준이 아니므로 개발 진행 가능. 단, 아래 후속 조치 완료 후 재검증 권장.

> 📌 **[NOTE-002]** 이전 실행(`01-prd-final-20260401.md`) 대비 신규 경고 1건 추가(WARN-002), 해소 1건(구 WARN-003 정책 누락 → 해소 완료).

## 사양 정합성 평가 결과 (요약 매트릭스)

| 평가 항목 | 주관 담당 | 상태 | 세부 내용 |
|-----------|-----------|------|-----------|
| PRD ↔ 정책 정합성 | 기획 | ✅ 통과 / ⚠️ 경고 / ❌ 실패 | 요구사항 R-003이 정책 문서에 반영되지 않음 |
| 정책 ↔ 서비스 정합성 | 기획·개발 | ✅ 통과 | 전체 정책 항목 서비스 레이어 반영 확인 |
```

## 3단계: 스프린트 결과 파일 정리
- `sprints/spec/spec-<n>-<git 유저명>/` 폴더 내 파일 구성이 다음과 일치하는지 확인하고 정리한다
	- `ROADMAP.md` — 전체 로드맵 (최종 상태 반영)
	- `phase-<n>.md` — Phase별 상세 계획 및 실행 결과
- `docs/sprint-feedbacks/pending` 중 스프린트에서 반영 완료된 항목중 개발 작업이 필요 없는 항목은 `docs/sprint-feedbacks/completed` 폴더로 이동.

## 4단계: 커밋 및 푸시
- 업데이트된 스펙 파일과 리포트 파일을 스테이징한다

- 푸시 완료 후 세션을 종료한다.

# 오류 처리

| 상황                  | 대응                                                    |
| ------------------- | ----------------------------------------------------- |
| 하위 에이전트 실행 실패       | 해당 에이전트 결과를 ERROR로 기록, 의존하는 하위 단계는 중단, 독립적인 단계는 계속 진행 |
| PRD 정합성 검증 Critical | 스펙 업데이트 단계를 건너뛰고 리포트에 사유 기록                           |
| 변경 파일 중 인식 불가 경로    | WARN으로 기록하고 해당 파일은 처리에서 제외                            |
| 커밋/푸시 실패            | 리포트를 로컬에 저장하고 에러 내용 출력                                |
| 동일 commit 중복 실행     | 리포트에서 처리된 commit SHA를 확인하여 중복 시 실행 생략                 |
|                     |                                                       |

# 제약사항 및 원칙
1. **읽기 우선**: 에이전트는 스펙 파일의 내용을 임의로 생성하지 않는다. 반드시 변경된 원본과 기존 스펙을 근거로 업데이트한다.
2. **변경 최소화**: 관련 없는 스펙 파일은 수정하지 않는다.
3. **추적 가능성**: 모든 변경은 리포트에 기록되어야 한다.
4. **안전한 실패**: 일부 단계가 실패하더라도 나머지 단계를 최대한 수행하고, 전체 결과를 리포트에 남긴다.
5. **재트리거 방지**: 에이전트가 생성한 커밋이 자기 자신을 다시 트리거하지 않도록 `[start-spec]` 접두사를 사용한다.
