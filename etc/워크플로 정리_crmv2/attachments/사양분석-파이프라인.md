# 사양 분석 파이프라인 에이전트

`start-spec` 진입점이 자동으로 호출하는 하위 에이전트 12종의 실행 시점·입력·결과 정리다.
상위 문서: [프롬프트 동작 모음](프롬프트%20동작모음.md)

이 파이프라인은 **PRD → 상세 사양서**로 전환한다.
하위 에이전트는 사용자가 직접 부르지 않고, `start-spec`의 3단계 흐름 안에서 순서대로 실행된다.

```
start-spec
 ├ 1단계  spec-sprint-planner   → 계획(ROADMAP·Phase) 수립
 ├ 2단계  spec-sprint-worker    → Phase마다:
 │          prd-reviewer  (정합성 평가)
 │          → prd-to-policies → service → datas → functions → screens → process → qa-dev (순차)
 └ 3단계  spec-sprint-closer    → 리포트·커밋·푸시
        (별도) spec-reviewer    → 완성 사양서 6축 교차 검증
```

> 공통 입력 규약: `prd-reviewer`와 `prd-to-*` 7종은 모두 **작업 Phase 문서 경로 하나**를 전달받아 실행된다.
> 예: `작업 phase 문서: sprints/spec/spec-3-정웅기/phase-1.md`

## 스프린트 관리 (3종)

### spec-sprint-planner (`ai/agents/sprint-manager/spec-sprint-planner.md`)
- **모델 / 호출 주체**: opus / start-spec (1단계)
- **실행 시점**: 사양 분석 스프린트가 시작될 때 가장 먼저 실행되어 계획을 세운다.
- **입력 예시**: `요구사항 분석 계획을 수립해줘. 대상: docs/prd/ 변경분, 이전 스프린트: sprints/spec/ 최신 폴더`
- **실행 결과**: 스프린트 폴더와 `ROADMAP.md`·`phase-<n>.md`가 생성되고, "sprint 준비 완료"를 반환한다.
- 원본: [spec-sprint-planner.md](../../../ai/agents/sprint-manager/spec-sprint-planner.md)

### spec-sprint-worker (`ai/agents/sprint-manager/spec-sprint-worker.md`)
- **모델 / 호출 주체**: opus / start-spec (2단계)
- **실행 시점**: 계획 수립 후 Phase마다 1회 실행되어 검토·사양 생성 에이전트를 오케스트레이션한다.
- **입력 예시**: `작업 phase 문서: sprints/spec/spec-3-정웅기/phase-1.md 를 실행해줘`
- **실행 결과**: 해당 Phase의 정합성 평가와 `docs/specs/` 사양서가 갱신되고 Phase 상태가 ✅로 바뀐다.
- 원본: [spec-sprint-worker.md](../../../ai/agents/sprint-manager/spec-sprint-worker.md)

### spec-sprint-closer (`ai/agents/sprint-manager/spec-sprint-closer.md`)
- **모델 / 호출 주체**: opus / start-spec (3단계)
- **실행 시점**: 모든 Phase 완료 후 마무리 단계에서 실행된다.
- **입력 예시**: `스프린트 결과를 정리하고 커밋해줘. 폴더: sprints/spec/spec-3-정웅기/`
- **실행 결과**: 담당 파트(기획·디자인·개발·QA)별로 그룹화된 `spec-report.md`가 작성되고 `[start-spec]` 접두사로 커밋·푸시된다.
- 원본: [spec-sprint-closer.md](../../../ai/agents/sprint-manager/spec-sprint-closer.md)

## 검토 (2종)

### prd-reviewer (`ai/agents/workflow-prd-review/prd-reviewer.md`)
- **모델 / 호출 주체**: opus / spec-sprint-worker (2단계 선행)
- **실행 시점**: 사양서를 생성하기 전에 요구사항과 기존 사양서의 정합성을 먼저 평가한다.
- **입력 예시**: `작업 phase 문서: sprints/spec/spec-3-정웅기/phase-1.md`
- **실행 결과**: 사양서 간 누락·충돌·중복이 위험도 등급으로 정리되고, 최대 3건의 요구사항 고도화 제안이 Phase 문서 "요구사항 분석 결과"에 기록된다 (별도 파일은 만들지 않음).
- 원본: [prd-reviewer.md](../../../ai/agents/workflow-prd-review/prd-reviewer.md)

### spec-reviewer (`ai/agents/workflow-prd-to-spec/spec-reviewer.md`)
- **모델 / 호출 주체**: opus / 사양 검증 시점 (완성 사양서 대상)
- **실행 시점**: 사양서가 완성된 뒤 PRD 충실도와 문서 간 정합성을 최종 평가할 때 실행된다.
- **입력 예시**: `docs/prd/PRD.md 와 docs/specs/ 전체를 대조해 정합성을 평가해줘`
- **실행 결과**: PROC(프로세스)를 중심으로 서비스·기능·데이터·화면·정책·TC 6축을 교차 검증한 평가 리포트 `spec-report-prd-review.md`가 생성된다 (사양 본문은 수정하지 않음).
- 원본: [spec-reviewer.md](../../../ai/agents/workflow-prd-to-spec/spec-reviewer.md)

## 사양서 생성 (7종)

7종은 `spec-sprint-worker`가 **정책 → 서비스 → 데이터 → 기능 → 화면 → 프로세스 → QA** 순서로 호출한다.
뒤 단계가 앞 단계 산출물을 참조하므로 순서가 고정되어 있다.
입력은 모두 동일하게 **Phase 문서 경로**다.

### prd-to-policies (`ai/agents/workflow-prd-to-spec/prd-to-policies.md`)
- **모델 / 호출 주체**: opus / spec-sprint-worker
- **실행 시점**: 사양서 생성의 첫 단계 — 전체에 적용될 정책을 먼저 정의한다.
- **실행 결과**: 접근 제어·데이터·비즈니스·보안 정책이 채번되어 `docs/specs/policies/spec-policies.md`와 `policy_<코드>.md`로 생성된다.

### prd-to-service (`ai/agents/workflow-prd-to-spec/prd-to-service.md`)
- **모델 / 호출 주체**: opus / spec-sprint-worker
- **실행 시점**: 정책 정의 후 실행되어 사용자 시나리오 중심의 서비스 기능을 정의한다.
- **실행 결과**: SVC 코드로 채번된 서비스 기능 정의서 `docs/specs/services/spec-services.md`와 `service_<코드>.md`가 생성된다.

### prd-to-datas (`ai/agents/workflow-prd-to-spec/prd-to-datas.md`)
- **모델 / 호출 주체**: opus / spec-sprint-worker
- **실행 시점**: 서비스 정의 후 실행되어 저장 구조를 설계한다.
- **실행 결과**: DB 엔터티(ENT)와 서비스 데이터 모델(MDL) 정의서 `docs/specs/datas/spec-datas.md`·`spec-models.md` 및 개별 문서가 생성된다.

### prd-to-functions (`ai/agents/workflow-prd-to-spec/prd-to-functions.md`)
- **모델 / 호출 주체**: opus / spec-sprint-worker
- **실행 시점**: 데이터 정의 후 실행되어 재사용 로직과 API를 정의한다.
- **실행 결과**: 공통 기능(FN)·API 인터페이스 정의서 `docs/specs/functions/spec-functions.md`와 `function_<코드>.md`가 생성된다.

### prd-to-screens (`ai/agents/workflow-prd-to-spec/prd-to-screens.md`)
- **모델 / 호출 주체**: opus / spec-sprint-worker
- **실행 시점**: 기능 정의 후 실행되어 사용자 화면과 디자인 시스템을 정의한다.
- **실행 결과**: 화면(SCR) 정의서 `docs/specs/screens/spec-screens.md`·`screen_<코드>.md`와 `design-system.md`가 생성된다.
- **담당 파트**: 디자인·개발

### prd-to-process (`ai/agents/workflow-prd-to-spec/prd-to-process.md`)
- **모델 / 호출 주체**: opus / spec-sprint-worker
- **실행 시점**: 화면 정의 후 실행되어 각 기능의 실행 흐름을 의사코드 수준으로 통합한다.
- **실행 결과**: PROC로 채번된 프로세스 정의서 `docs/specs/processes/spec-process.md`·`process_<코드>.md`가 생성되며, FE/BE 의사코드까지 상세화된다.

### prd-to-qa-dev (`ai/agents/workflow-prd-to-spec/prd-to-qa-dev.md`)
- **모델 / 호출 주체**: opus / spec-sprint-worker
- **실행 시점**: 마지막 단계로 실행되어 기획서 품질을 검증하고 테스트 케이스를 설계한다.
- **실행 결과**: 기획서 검증 리포트와 TC 채번된 `docs/specs/qa-dev/spec-qa.md`·`tc_<코드>.md`가 생성되고, 사양 미해소 항목이 후속 QA 파이프라인으로 전파된다.
- **담당 파트**: QA·기획

> 표기 유의: 원본 문서상 `prd-to-qa-dev`(파일명 `spec-qa.md`/`spec-qa-dev.md` 혼재)와 `prd-to-process`(경로 `process`/`processes` 혼재)에 파일명 표기 불일치가 있어, 실제 생성 파일명은 원본 확인이 필요하다.
