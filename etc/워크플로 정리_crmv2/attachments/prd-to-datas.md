본 문서는 프로젝트 구축에 필요한 데이터 설계 및 활용 모델을 설계하기 위한 규칙을 정의한다.
아래 지침에 따라 제공된 요구사항 및 정책정의를 기반으로 요구사항 구현을 위한 세부 데이터 모델링과 데이터 저장 및 제공을 최적화 할 수 있는 데이터베이스 설계를 수행한다.

용어 약어 — ENT는 DB 엔터티 정의서 코드, MDL은 데이터 모델(DTO·도메인 모델) 코드, SVC는 서비스 시나리오, FN은 공통 기능, PROC는 기능구현 프로세스 정의서, DDL은 데이터 정의 언어(Data Definition Language), CHECK 제약은 컬럼값 범위 제약, PK는 기본 키(Primary Key), FK는 외래 키(Foreign Key)를 의미한다.

# 실행 시 전달 파라메터
- 작업 phase 문서 경로 (`sprints/spec/spec-<n>-<git유저명>/phase-<n>.md`)
- 유효한 값이 전달되지 않은 경우 경고 노출후 작업 중단.

# 작업 프로세스

## 1단계: 요구사항 분석

- 시작 시 전달받은 phase 문서의 내용을 확인하고, 분석 대상 문서를 확인한다.
- 분석된 내용에서 다음 항목을 추출한다.
	- 서비스 기능에서 다루는 핵심 데이터 항목 (입력/출력/저장 대상).
	- 데이터 간 관계 (1:1, 1:N, N:M 등).
	- 데이터 생명주기 (생성, 수정, 삭제, 보관 시점 및 조건).
	- 정책 문서에서 데이터에 영향을 주는 제약사항 (보존 기간, 암호화, 접근 제어 등).
	- 외부 시스템과 주고받는 데이터 항목 및 형식.
	- 조회/검색 요구사항 (목록 조회, 필터 조건, 정렬 기준 등) — **인덱스 설계 입력**.
	- 데이터 정합성 규칙 (고유성, 필수값, 참조 무결성 등).

## 2단계: 데이터 모델 설계

- 1단계에서 추출한 데이터 항목을 논리적 엔터티로 그룹화한다.
- 정책 문서에서 엔터티에 적용되는 제약사항을 매핑한다.

## 3단계: 데이터베이스 설계

- 논리 모델을 물리 설계로 변환한다.
	- 테이블 구조 정의.
	- 컬럼별 **데이터 타입(타입·길이·precision)·NULL 허용·기본값·CHECK 제약** 명시.
	- 인덱스 설계 (조회 패턴 기반).
	- 소프트 삭제 / 하드 삭제 적용 기준 반영.
- 서비스 기능 문서의 입력/출력과 테이블 구조의 정합성을 검증한다.
- PROC 의사코드의 SQL 쿼리 대상 컬럼이 모두 ENT 정의에 실재하는지 사전 검증한다 (Phase 5 산출 전 사전 점검 권장).

## 4단계: 데이터 정의 문서 생성(DB 엔터티)

### 4-1: 데이터 정의 목록 문서 생성

`docs/specs/datas/spec-datas.md` 파일에 전체 데이터 정의서의 개요와 하위 문서 목록을 작성한다.

- 데이터 모델 설계 원칙 및 기본 방향.
- 엔터티 코드 체계 (ENT-nnn).
- 엔터티 분류 및 목록.
- 엔터티 관계도 (ER Diagram 또는 관계 요약).
- 엔터티 간 주요 의존관계 요약.
- 인덱스 전략: 성능을 위한 인덱스 설계 — 조회 패턴별 카디널리티 추정 포함.
- 이전 버전 대비 변경사항 — 세부 개정사항은 별도 문서(`docs/specs/datas/spec-datas-changes.md`)로 분리.

### 4-2: 개별 데이터 정의 문서 생성

각 엔터티별로 `docs/specs/datas/data_<entity-code>.md` 파일을 생성한다.

```markdown
# [엔터티명] 데이터 정의

## 개요
- **데이터 목적**: 이 엔터티가 관리하려는 데이터의 역할 및 필요성
- **관련 PRD 요구사항**: PRD 내 관련 섹션 및 원문 인용

---

## [ENT 코드] [엔터티명]

### 기본 정보
| 항목 | 내용 |
|------|------|
| 엔터티명 | 엔터티 이름 |
| 물리 테이블명 | 예: TBL_<도메인>_<엔터티> |
| 분류 | 마스터 / 트랜잭션 / 이력 / 코드 등 |
| 관련 서비스 | 이 데이터를 사용하는 SVC 코드 |
| 보존 정책 | 보존 기간 및 삭제 방식 (관련 정책 코드) |
| 개인정보 여부 | 해당 / 비해당 (해당 시 처리 기준 명시) |
| CRUD 수행 PROC | 본 ENT 에 변경을 가하는 PROC 코드 (CRUD 별로 명시) |

### 속성 정의

| 속성명 | 데이터 타입 | 길이/precision | NULL | 기본값 | CHECK 제약 | 키 | 설명 |
|--------|-----------|----------------|------|--------|-----------|----|------|
| id | UUID | - | NOT NULL | gen_random_uuid() | - | PK | 고유 식별자 |
| status | VARCHAR | 20 | NOT NULL | 'PENDING' | IN ('PENDING','ACTIVE','DELETED') | - | 상태 플래그 |
| amount | NUMERIC | 15,2 | NOT NULL | 0 | >= 0 | - | 금액 |
| created_at | TIMESTAMPTZ | - | NOT NULL | NOW() | - | - | 생성 시각 |
| updated_at | TIMESTAMPTZ | - | NOT NULL | NOW() | - | - | 수정 시각 |
| deleted_at | TIMESTAMPTZ | - | NULL | NULL | - | - | 소프트 삭제 시각 |

> **필수 명시 항목**: 데이터 타입 / 길이·precision / NULL 허용 / 기본값 / CHECK 제약 / 키(PK·FK·UK).
> 길이·precision 미적용 타입(BOOLEAN·TIMESTAMPTZ 등)은 `-` 로 표기.

### 관계 정의

| 대상 엔터티 | 관계 유형 | FK 컬럼 | ON DELETE | ON UPDATE | 설명 |
|-------------|-----------|---------|-----------|-----------|------|
| ENT-xxx | 1:N | entity_id | RESTRICT / CASCADE | RESTRICT / CASCADE | 관계 설명 |

### 인덱스 정의

| 인덱스명 | 대상 컬럼 | 유형 | 카디널리티 추정 | 조회 패턴 (인용 PROC) |
|----------|-----------|------|-----------------|----------------------|
| idx_<table>_<col> | column_a, column_b | UNIQUE / BTREE / GIN | 높음 / 중간 / 낮음 | PROC-xxx B2 사전 조회 (WHERE column_a=?, ORDER BY column_b DESC) |

> **카디널리티 추정**: 컬럼 고유 값 분포에 따른 추정 — 높음(near-unique) / 중간(수십~수백 그룹) / 낮음(소수 코드값).
> **조회 패턴**: 본 인덱스를 사용하는 PROC 의사코드의 SQL 쿼리(WHERE / ORDER BY / JOIN) 인용. 인덱스 사용 PROC 가 0건이면 인덱스 신설 사유 재검토.

### 데이터 생명주기
- **생성 조건**: 어떤 PROC 의 어느 단계에서 INSERT 되는지 (PROC 코드 + 단계명).
- **수정 조건**: 어떤 항목이 어떤 조건에서 UPDATE 되는지 (PROC 코드 + 단계명 + 변경 컬럼).
- **삭제/보관 조건**: 소프트 삭제 / 하드 삭제 / 아카이브 기준 (PROC 코드 + 단계명).

### 연관 정책 (policy)

| 정책 코드 | 적용 컬럼/제약 | 적용 위치 (DB 무결성 / 응용 검증 / 마스킹) |
|-----------|----------------|------------------------------------------|
| <그룹>-NNN-RR | 적용 대상 | 적용 위치 |

### 구현 가이드
- 개발자가 이 엔터티를 구현할 때 참고할 핵심 사항 (DDL 변환·인덱스 빌드·마이그레이션 순서).
- 관련 기술 패턴 (특정 ORM·DBMS 강제 금지).

### 변경 이력
| 일자 | 변경 내용 (DDL 변경 사유 + 영향 컬럼·인덱스) | 사유 | 영향 범위 (PROC/MDL/마이그레이션) |
|------|----------------------------------------------|------|----------------------------------|
| YYYY-MM-DD | 변경 요약 | 변경 사유 | 영향 PROC/MDL/마이그레이션 |
```

## 5단계: 서비스별 데이터 모델 설계

- 서비스 기능 정의서(SVC)를 기준으로 각 서비스가 다루는 데이터 모델을 설계한다.
- 다음 기준으로 데이터 모델을 분류한다.
	- **공통 모델(COM)**: 2개 이상의 서비스에서 공유.
	- **서비스 모델(SVC)**: 특정 서비스 내부에서만 사용.
	- **외부 연동 모델(EXT)**: 외부 시스템과의 연동에 사용.
- 데이터 모델 코드 체계를 설계한다 (MDL-nnn 형태).

## 6단계: 서비스별 데이터 모델 정의서 생성

### 6-1: 서비스 데이터 모델 목록 문서 생성

`docs/specs/datas/spec-models.md` 파일에 전체 서비스 데이터 모델의 개요와 하위 문서 목록을 작성한다.

### 6-2: 개별 서비스 데이터 모델 정의 문서 생성

각 서비스별로 `docs/specs/datas/model_<model-code>.md` 파일을 생성한다.

```markdown
# [모델명] 데이터 모델 정의

## 개요
- **모델 목적**: 이 데이터 모델이 사용되는 맥락 및 필요성
- **관련 서비스**: 이 모델을 사용하는 SVC 코드

---

## [MDL 코드] [모델명]

### 기본 정보
| 항목 | 내용 |
|------|------|
| 모델명 | 데이터 모델 이름 |
| 분류 | 공통(COM) / 서비스(SVC) / 외부 연동(EXT) |
| 사용 서비스 | 이 모델을 사용하는 SVC 코드 목록 |
| 매핑 엔터티 | 관련 ENT 코드 |
| 사용 PROC | 본 MDL 을 입력/출력으로 사용하는 PROC 코드 목록 |
| 용도 | 요청 모델 / 응답 모델 / 도메인 모델 / 외부 DTO 등 |

### 속성 정의

| 속성명 | 데이터 타입 | 필수 | 기본값 | 유효성 규칙 | 마스킹 규칙 | 설명 |
|--------|-----------|------|--------|-------------|-------------|------|
| id | string(UUID) | Y | - | UUID 형식 | - | 고유 식별자 |
| phone | string | Y | - | `/^010\d{8}$/` | 응답 시 `010-****-1234` | 휴대폰 번호 |
| name | string | Y | - | NotBlank, MaxLength(20) | - | 이름 |

### 엔터티 매핑 (PROC 데이터 변환 흐름과 정합)

| 모델 속성 | 엔터티(ENT) | 엔터티 속성 | 변환 지점 | 변환 규칙 |
|-----------|-------------|-------------|-----------|-----------|
| id | ENT-001 | id | 도메인→ENT / ENT→도메인 | 직접 매핑 |
| phone | ENT-001 | phone_no | FE→요청 / 도메인→ENT | 하이픈 제거 후 저장 |
| phone | ENT-001 | phone_no | ENT→도메인 / 도메인→응답 | 마스킹 후 응답 (POL DATA-NNN) |

> **변환 지점** 칸은 [prd-to-process](prd-to-process.md) §3-2 의 데이터 변환 흐름 6 지점(FE→요청 / 요청→도메인 / 도메인→ENT / ENT→도메인 / 도메인→응답 / 응답→FE) 중 적용되는 지점을 명시한다.
> PROC 의 데이터 변환 흐름 표와 본 표는 1:1 정합해야 한다.

### 사용처

| SVC 코드 | 기능 | 용도 (요청/응답/도메인) | 사용 PROC | 비고 |
|----------|------|------------------------|----------|------|
| SVC-001 | 기능명 | 응답 | PROC-xxx | 목록 조회 시 사용 |

### 구현 가이드
- 개발자가 이 모델을 구현할 때 참고할 핵심 사항 (타입 정의 / 직렬화 / 마스킹 처리 위치).

### 변경 이력
| 일자 | 변경 내용 | 사유 | 영향 범위 |
|------|----------|------|----------|
| YYYY-MM-DD | 변경 요약 | 변경 사유 | 영향 PROC/SVC/ENT |
```

# 문서 작성 원칙

## 역할 준수
- 본 문서는 데이터 구조와 관계를 정의하는 것이 목적으로, 비즈니스 로직이나 UI 구성은 포함하지 않는다.
- ENT 는 DB 영속화 구조, MDL 은 응용 계층의 데이터 모델(DTO·도메인 모델). 양자의 책임 범위를 분리한다.

## 명확성
- 모호한 표현 금지 — "충분한 길이의 문자열 타입을 사용한다" 대신 "VARCHAR(255), NOT NULL".
- DB 엔터티의 속성 정의는 개발자가 DDL 로 바로 변환할 수 있을 만큼 구체적으로 기술 (타입·길이·NULL·기본값·CHECK).
- 서비스 데이터 모델의 속성 정의는 개발자가 타입 정의(interface/type)로 바로 변환할 수 있을 만큼 구체적으로 기술.

## 일관성
- 모든 엔터티 및 데이터 모델에 동일한 문서 구조 적용.
- 엔터티 코드(ENT) 및 데이터 모델 코드(MDL) 채번 일관성 유지.
- 속성명 명명 규칙 통일 (DB 컬럼은 snake_case, MDL 속성은 도메인 컨벤션 — 통상 camelCase).

## 완결성
- 모든 ENT 속성에 데이터 타입 / 길이·precision / NULL / 기본값 / CHECK / 키 6 항목이 명시되어야 한다.
- 모든 MDL 속성에 데이터 타입 / 필수 / 유효성 규칙 / 마스킹 규칙(해당 시) 이 명시되어야 한다.
- 엔터티 간 관계가 양방향으로 추적 가능해야 한다 (ON DELETE / ON UPDATE 포함).
- 서비스 데이터 모델과 DB 엔터티 간 매핑이 PROC 데이터 변환 흐름 6 지점과 정합해야 한다.

## 데이터 설계 원칙
- **정규화**: 3NF를 기본으로, 성능상 필요한 경우 의도적 비정규화 명시.
- **무결성**: FK 제약, NOT NULL, UNIQUE, CHECK 등 DB 레벨 무결성 적극 활용.
- **확장성**: 향후 컬럼 추가에 유연하도록 설계 (JSON/JSONB 컬럼 활용 등 고려).
- **감사 추적**: 중요 엔티티는 `created_at`, `updated_at`, 생성자/수정자 ID 포함.
- **소프트 삭제**: 복구 가능성이 있는 데이터는 `deleted_at` 컬럼으로 논리 삭제.
- **인덱스 신설 기준**: 본 ENT 의 컬럼을 조건절·정렬·조인에 사용하는 PROC 가 1개 이상 실재해야 한다.
  사용 PROC 0건인 인덱스는 신설 금지 (메모리 비용 낭비).

## 추적 가능성
- 각 엔터티가 PRD의 어떤 요구사항에서 비롯되었는지 기술.
- 엔터티와 연관된 SVC / POL 코드 기술.
- CRUD 수행 PROC 코드 인용 (어느 PROC 가 본 ENT 를 INSERT/UPDATE/DELETE/SELECT 하는지).
- 각 데이터 모델이 어떤 SVC / PROC 에서 사용되는지 기술.

# PROC 와의 상호 참조 (필수)

PROC 의사코드의 SQL 쿼리 대상 컬럼·인덱스가 본 문서에 실재해야 한다.
ENT/MDL 작성 시 다음을 반드시 고려한다.

- ENT 의 모든 컬럼은 1개 이상의 PROC 의사코드(또는 BE 의사코드)에서 SELECT/INSERT/UPDATE/DELETE 대상이어야 한다.
- 인덱스는 1개 이상의 PROC 의사코드 SQL 의 WHERE/ORDER BY/JOIN 에서 사용되어야 한다.
- MDL 의 엔터티 매핑 표는 PROC 의 데이터 변환 흐름 표(6 지점)와 1:1 정합해야 한다.
- ENT 변경(컬럼 추가·타입 변경·인덱스 신설) 발생 시 영향 PROC 코드를 변경 이력에 명시한다.

# 산출물 검증 체크리스트

산출물 작성 완료 후 다음을 확인한다 (6 카테고리는 [prd-to-process.md §산출물 검증 체크리스트](prd-to-process.md) 와 동일 형식).

## 적용 범위 검증
- ⬜ PRD 및 SVC 정의의 모든 데이터 항목이 하나 이상의 ENT 에 반영되었는가?
- ⬜ 모든 PROC 가 사용하는 ENT 컬럼이 본 문서에 실재하는가? (PROC SQL 쿼리 대상 1:1 일치)
- ⬜ MDL 이 모든 SVC 의 입력/출력을 포괄하는가?
- ⬜ 공통 데이터 모델(COM)이 식별되고 분리되었는가?

## 정의 완결성 검증
- ⬜ 모든 ENT 속성에 데이터 타입 / 길이·precision / NULL / 기본값 / CHECK / 키 6 항목이 명시되었는가?
- ⬜ 모든 ENT 관계에 카디널리티 / FK 컬럼 / ON DELETE / ON UPDATE 가 명시되었는가?
- ⬜ 모든 인덱스에 카디널리티 추정 / 조회 패턴(인용 PROC) 이 작성되었는가?
- ⬜ 데이터 생명주기(생성/수정/삭제 조건) 가 모든 ENT 에 PROC 코드 + 단계명 인용으로 작성되었는가?
- ⬜ 모든 MDL 속성에 데이터 타입 / 필수 / 유효성 규칙 / 마스킹 규칙(해당 시) 이 명시되었는가?
- ⬜ MDL 엔터티 매핑 표에 변환 지점(6 지점 중 해당) 이 명시되었는가?

## 실행 제약 검증
- ⬜ 개인정보 해당 ENT 에 보존/삭제/암호화 정책이 매핑되었는가?
- ⬜ 인덱스 신설 시 사용 PROC 가 0건인 항목이 없는가?
- ⬜ 소프트 삭제 적용 ENT 에 `deleted_at` 컬럼이 일관되게 적용되었는가?

## 일관성 검증
- ⬜ 모든 엔터티 및 데이터 모델에 동일한 문서 구조 적용?
- ⬜ ENT/MDL 코드 채번이 spec-datas.md / spec-models.md 목록과 일치하는가?
- ⬜ 속성명 명명 규칙(DB snake_case / MDL camelCase) 이 통일되었는가?

## 의존관계 검증
- ⬜ 엔터티 간 관계(FK, 카디널리티)가 양방향으로 정의되었는가?
- ⬜ MDL ↔ ENT 매핑이 모두 정의되었는가? (깨진 참조 0건)
- ⬜ CRUD 수행 PROC 코드 / 사용 PROC 코드가 모두 실재하는가?

## 역할 준수 검증
- ⬜ 비즈니스 로직(처리 흐름·검증 흐름)이 포함되지 않고 데이터 정의에 집중되었는가?
- ⬜ UI 구성·표시 형식이 포함되지 않고 SCR/MDL 마스킹 규칙으로 위임되었는가?
- ⬜ 특정 ORM·DBMS 를 강제하는 표현이 없는가?
- ⬜ 모호 항목과 해석 근거가 phase 문서의 "검토 및 제안사항"에 기록되었는가?

# 출력 형식

- DB 엔터티 목록 파일: `docs/specs/datas/spec-datas.md`
- DB 엔터티 개별 파일: `docs/specs/datas/data_<entity-code>.md`
- 서비스 데이터 모델 목록 파일: `docs/specs/datas/spec-models.md`
- 서비스 데이터 모델 개별 파일: `docs/specs/datas/model_<model-code>.md`
- 언어: 한국어
- 형식: Markdown
- 엔터티 코드 채번: `ENT-001`, `ENT-002`, ...
- 데이터 모델 코드 채번: `MDL-001`, `MDL-002`, ...

# 주의사항

- PRD 및 서비스 기능 정의에 명시되지 않은 엔터티나 데이터 모델을 임의로 추가하지 않는다.
- 특정 구현 기술(DBMS, ORM 등)을 강제하지 않고 데이터의 요구사항만 정의한다.
- 개인정보를 다루는 엔터티는 관련 법규(개인정보보호법 등)를 명시한다.
- ENT 컬럼 추가·삭제·타입 변경은 영향 PROC 가 식별되어야 진행 가능하다.
- 기존 작성 내용에서 변경이 필요한 경우 변경 지점과 사유, 영향범위 및 추가 검토 제안사항을 정리하여 `docs/specs/datas/spec-datas-changes.md` 또는 `docs/specs/datas/spec-models-changes.md` 에 작성한다.

# 메모리 업데이트

PRD 분석 후 다음 사항을 프로젝트 메모리에 기록한다:
- 도출된 엔터티 분류 체계 및 코드 목록 — 컬럼·인덱스·제약 변경 사항도 함께 등재.
- 도출된 서비스 데이터 모델 분류 체계 및 코드 목록.
- 엔터티 간 주요 의존 관계 (FK 그래프).
- 공통 데이터 모델(COM) 목록 및 공유 서비스 범위.
- 인덱스 카탈로그 (인덱스명·대상 컬럼·카디널리티·사용 PROC).
- ENT ↔ PROC CRUD 매핑 (어느 PROC 가 어느 ENT 를 변경하는지).
- 개인정보 해당 ENT 존재 여부 및 관련 법규 요구사항.
- PRD에서 데이터 모델로 변환 시 판단이 필요했던 항목.
