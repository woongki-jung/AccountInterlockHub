# 프롬프트 동작 모음 — CRMv2 에이전트 실행 가이드

이 문서는 현재 워크스페이스(CRMv2)에 정의된 **모든 AI 에이전트**의 실행 시점, 입력 프롬프트 예시, 실행 결과를 한곳에 정리한 안내서다.
"어떤 상황에서 어떤 프롬프트를 넣으면 무엇이 만들어지는가"를 빠르게 찾기 위한 용도다.

- 에이전트 정의 원본: [`ai/agents/`](../../../ai/agents/) / 스킬 정의 원본: [`ai/skills/`](../../../ai/skills/)
- 에이전트 구성 요약: [`ai/strategies/agents.md`](../../../ai/strategies/agents.md)
- 전체 업무 흐름 설계: [`워크플로우구성 협의자료/workflow정의.md`](../워크플로우구성%20협의자료/workflow정의.md)

## 1. 전체 워크플로 한눈에 보기

제품은 **사양 분석 → 제품 생산 → 품질 검증**의 3개 파이프라인을 거쳐 만들어진다.
각 파이프라인은 사용자가 넣는 하나의 진입 프롬프트로 시작되고, 그 안에서 여러 하위 에이전트가 순서대로 자동 실행된다.

```
[담당자] PRD 작성 (docs/prd/)
      │  (작업 브랜치 push → n8n 웹훅 자동 실행)
      ▼
① 사양 분석      start-spec
      │   ├ spec-sprint-planner  (계획)
      │   ├ spec-sprint-worker   (실행: prd-reviewer → prd-to-* 7종)
      │   └ spec-sprint-closer   (마무리·커밋)
      ▼   → docs/specs/ (정책·서비스·데이터·기능·화면·프로세스·TC)
② 제품 생산      /sprint-build → start-builder
      │   ├ build-sprint-planner (계획)
      │   ├ build-sprint-worker  (실행: frontend/backend-developer → code-reviewer → tester로 QC)
      │   ├ build-installer      (인스톨러 빌드)
      │   └ build-sprint-closer  (마무리·커밋)
      ▼   → apps/ 제품 코드 + 인스톨러
③ 품질 검증      /sprint-qa → start-qa
          ├ (사전) 인스톨러 빌드 + 자동 설치
          ├ test-planner         (테스트 계획)
          └ tester               (Phase별 TC 실행)
          → sprints/qa/ 품질 리포트
```

## 3. 사용자가 직접 실행하는 프롬프트 (진입점)

여기 5개가 사람이 직접 프롬프트로 시작하는 지점이다.
나머지 하위 에이전트는 이 진입점이 자동으로 불러오므로 직접 실행할 필요가 없다.

### 3-1. `/sprint-build` (스킬) — 제품 코드 생산 시작

- **실행 시점**: 마일스톤에서 지정한 영역의 코드 구현을 시작할 때.
- **핵심 동작**: 범위가 여러 leaf(개별 도메인 셀)면 **1 leaf = 1 스프린트**로 자동 분할해 순차 실행한다 (분할이 3개 이상이면 먼저 계획을 보고하고 진행 방식을 확인한다).
- **입력 프롬프트 예시**:
  ```
  /sprint-build
  목표: AF 템플릿 관리 화면의 카테고리 CRUD를 제품 코드로 구현
  범위: sprints/milestone 의 AFB 카테고리 관리 leaf
  ```
  (목표·범위를 비우고 실행하면 되묻는다.)
- **실행 결과**: leaf마다 `start-builder`가 완주하며 제품 코드·인스톨러·결과 리포트가 생성되고, 각 스프린트가 커밋·푸시된 뒤 다음 leaf로 넘어간다.

### 3-2. `/sprint-qa` (스킬) — 품질 검증 시작

- **실행 시점**: 마일스톤에서 지정한 영역의 품질 검수를 시작할 때.
- **핵심 동작**: N개 leaf → N개 스프린트로 분할하되 **반드시 직렬(포어그라운드)**로 실행한다 (검증 도구가 같은 포트·브라우저를 공유하므로 병렬 금지).
- **입력 프롬프트 예시**:
  ```
  /sprint-qa
  목표: 메시지 발송 도메인 릴리스 품질 검증
  범위: 발송 관련 leaf 전체
  모드: Test        (미지정 시 Test 기본)
  ```
- **실행 결과**: leaf마다 `start-qa`가 실행되어 TC를 검증하고, 품질 리포트(`qa-report.md`)와 실패·차단 요약(`tc-followup.md`)이 생성·커밋된다.

### 3-3. `start-spec` — 요구사항 분석 및 사양서 작성

- **실행 시점**: PRD(`docs/prd/`)가 갱신되어 작업 브랜치에 push될 때 n8n 웹훅이 자동 실행하거나, 사양을 새로 정비할 때 수동 실행한다.
- **입력 프롬프트 예시**:
  ```
  start-spec 에이전트로 요구사항 분석 스프린트를 실행해줘.
  대상: docs/prd/ 최근 변경분 (requirements·specification 폴더)
  ```
- **실행 결과**: `docs/specs/` 하위에 정책·서비스·데이터·기능·화면·프로세스·TC 사양서가 신규/갱신되고, 정합성 평가 리포트(`spec-report.md`)와 함께 `[start-spec]` 접두사로 커밋된다.
- 내부 3단계(planner → worker → closer)는 [사양 분석 파이프라인](사양분석-파이프라인.md) 참조.

### 3-4. `start-builder` — 제품 코드 생산

- **실행 시점**: `/sprint-build` 스킬이 leaf별로 호출하거나, 사양 push 웹훅·수동으로 실행한다 (자동화 모드 `--dangerously-skip-permissions` 전용).
- **입력 프롬프트 예시**:
  ```
  start-builder 에이전트로 빌드 스프린트를 실행해줘.
  범위: AFB 카테고리 관리 leaf (화면·모달·훅 + BE 컨트롤러 포함)
  ```
- **실행 결과**: `apps/` 하위에 제품 코드가 구현되고, Debug/Release 인스톨러가 빌드되며, `build-report.md`와 함께 `[start-builder]` 접두사로 커밋된다.
- 내부 4단계(planner → worker → installer → closer)는 [제품 생산 파이프라인](제품생산-파이프라인.md) 참조.

### 3-5. `start-qa` — 품질 검증

- **실행 시점**: `/sprint-qa` 스킬이 leaf별로 호출하거나 수동으로 실행한다 (자동화 모드 전용).
- **핵심 동작**: 계획 수립 직전에 **인스톨러 빌드 + 자동 설치**를 반드시 수행해, 실제 사용자 환경과 동일한 설치본을 검증 대상으로 삼는다.
- **입력 프롬프트 예시**:
  ```
  start-qa 에이전트로 품질검증 스프린트를 실행해줘.
  범위: 발송 도메인 / 모드: Test
  ```
  해소된 피드백만 재검증하려면 아래처럼 모드를 명시한다.
  ```
  start-qa 로 resolved 피드백 검증을 실행해줘.
  ```
- **실행 결과**: TC별로 🟢/🔵/🟣/🔴/🟠 5종 판정이 채워지고, `qa-report.md`·`tc-followup.md`가 생성되며 `[start-qa]` 접두사로 커밋된다.
- 내부 단계(test-planner → tester)는 [품질 검증 파이프라인](품질검증-파이프라인.md) 참조.

## 4. 하위 에이전트 상세

진입점이 자동 호출하는 하위 에이전트의 **실행 시점 / 입력(전달 파라미터) 예시 / 실행 결과**는 파이프라인별 문서로 분리했다.

- 📄 [사양 분석 파이프라인 에이전트](사양분석-파이프라인.md) — planner·worker·closer, prd-reviewer, prd-to-* 7종, spec-reviewer (12종)
- 📄 [제품 생산 파이프라인 에이전트](제품생산-파이프라인.md) — planner·worker·closer, backend·frontend developer, code-reviewer, build-installer (7종)
- 📄 [품질 검증 파이프라인 에이전트](품질검증-파이프라인.md) — test-planner, tester (2종)

## 5. 공통 규약 및 주의사항

- **재실행 방지**: 각 진입점은 완료 시 `[start-spec]`·`[start-builder]`·`[start-qa]` 접두사로 커밋한다. 마지막 커밋에 이 접두사가 있으면 웹훅 재트리거를 막기 위해 실행을 중단한다.
- **자동화 실행 모드**: `start-builder`·`start-qa`는 `--dangerously-skip-permissions` 자동화 모드 전용이며, 사용자 확인이 필요한 항목은 "수동 실행 필요 항목"으로 리포트에 남기고 멈추지 않는다.
- **정확도 우선**: 병렬 실행은 결과 품질 저하 우려가 없을 때만 사용하고, 기본은 순차 실행이다.
- **실 동작 검증 의무**: 정적 검증(컴파일·타입 체크)만으로 완료 처리하지 않고, 변경된 기능을 실제 실행해 저장→조회 왕복(round-trip)까지 확인한다.
- **민감정보 보호**: 계정·복호화 키 등 원문([`CLAUDE.local.md`](../../../CLAUDE.local.md) 변수)은 리포트·커밋에 포함하지 않는다.

## 6. 용어 설명

- **PRD**: Product Requirements Document, 제품 요구사항 문서 (`docs/prd/`).
- **스펙(사양서)**: PRD를 개발 가능한 수준으로 상세화한 문서 (`docs/specs/`).
- **마일스톤 / leaf**: 개발 범위 묶음(마일스톤)과 그 안의 최소 작업 단위(leaf, 페이지·모달·도메인 하나).
- **스프린트**: 하나의 작업 단위를 계획→실행→마무리까지 완주하는 실행 회차 (`sprints/`).
- **ROADMAP / Phase**: 스프린트 전체 계획(ROADMAP)과 그 안의 세부 실행 단위(Phase).
- **TC**: Test Case, 테스트 케이스.
- **round-trip(왕복 검증)**: 저장한 값이 다시 조회했을 때 그대로 나오는지 확인하는 검증.
- **인스톨러 / 설치본**: 실제 사용자가 설치하는 배포 패키지와 그 설치 결과물.
- **웹훅(n8n)**: git push를 감지해 해당 에이전트를 자동 실행하는 자동화 서비스.
