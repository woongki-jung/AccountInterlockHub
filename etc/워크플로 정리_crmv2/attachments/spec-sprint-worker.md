본 문서는 ROADMAP을 기반으로 작업 항목을 확인하고 요구사항을 사양서로 전환하기 위한 위한 프로세스를 정의한다.
아래 지침에 따라 ROADMAP을 기반으로 수행해야 할 phase를 선정하고 단계별 작업을 수행한다.

# 작업 프로세스

## 1단계: 작업 범위 확인
시작 시 전달받은 phase 문서의 내용을 확인하고, 분석 대상 문서를 확인한다.
실행해야 할 phase가 목업 생산이라면 4단계를 수행한다.

- `sprints/spec/spec-<n>-<git 유저명>/ROADMAP.md`를 읽고 전체 Phase 현황을 파악한다.
- 현재 진행해야 할 Phase를 확인하고 상태를 변경한다. (🔄 진행 중 또는 다음 📋 예정 Phase).
- 해당 Phase의 상세 문서 `sprints/spec/spec-<n>-<git 유저명>/phase-<n>.md`를 읽고 작업 목록을 파악한다.
- 분석 대상 문서를 확인하고 내용을 자세히 분석한다.

## 2단계: 요구사항 정합성 평가 수행
1단계에서 파악한 Phase 정보에 따라 요구사항 분석 에이전트를 실행한다.
- **실행 에이전트**: `ai/agents/workflow-prd-review/prd-reviewer.md`
- **에이전트 전달 파라메터**
	- 작업 phase 문서 경로 (`sprints/spec/spec-<n>-<git 유저명>/phase-<n>.md`)
- **기대 결과**: phase 문서의 "요구사항 분석 결과" 섹션에 실행 결과 업데이트

## 3단계: 사양정의서 작성
순차적으로 에이전트를 수행하여 상세 스펙 문서를 갱신한다.
- **실행 에이전트**
	1. `prd-to-policies`
	2. `prd-to-service`
	3. `prd-to-datas`
	4. `prd-to-functions`
	5. `prd-to-screens`
	6. `prd-to-process`
	7. `prd-to-qa-dev`
- **에이전트 전달 파라메터**
	- 작업 phase 문서 경로 (`sprints/spec/spec-<n>-<git 유저명>/phase-<n>.md`)
- **기대 결과**
	- phase 문서의 "사양정의서 작업 결과" 목록 업데이트
	- `docs/specs/` 폴더 하위 작성된 사양정의서

## 4단계: 외부 API 사양 검증

Phase 작업 범위에 외부 API 사양 문서가 포함된 경우, 문서에 기술된 요청/응답 형식이 실제 서버와 일치하는지 검증한다.

- 외부 API 스펙 문서에 정의된 각 엔드포인트에 대해:
	1. ⚠️ **데이터를 변경하지 않는 조회성 API만 실제 요청한다. 스펙 문서에서 각 엔드포인트의 용도(조회/생성/수정/삭제)를 판단하고, 데이터를 변경하는 API는 실제 서버에 요청하지 않는다** — 문서/Swagger 기반으로만 검증한다
	2. 검증 대상 엔드포인트에 대해, 스펙에 명시된 개발 서버 base URL로 실제 HTTP 요청을 보낸다 (Bash의 `curl` 사용)
	3. 스펙의 Method(POST/GET), Request Body, 필수 파라미터를 그대로 사용한다
	4. 실제 응답의 상태 코드, 응답 구조, 필드명을 스펙 문서와 대조한다
	5. 불일치가 발견되면 실제 서버 응답을 근거로 스펙 문서를 수정한다
- 외부 서버 접속 불가 시 해당 엔드포인트를 Block으로 기록하고 수동 확인 필요 항목으로 분류한다
- Swagger 등 API 문서가 있으면 교차 검증에 활용한다

## 5단계: phase 완료 처리
작업이 완료된 phase 문서의 진행상태를 "✅ 완료"로 수정한다.

# 오류 처리

| 상황                  | 대응                                                    |
| ------------------- | ----------------------------------------------------- |
| 하위 에이전트 실행 실패       | 해당 에이전트 결과를 ERROR로 기록, 의존하는 하위 단계는 중단, 독립적인 단계는 계속 진행 |
| PRD 정합성 검증 Critical | 스펙 업데이트 단계를 건너뛰고 리포트에 사유 기록                           |
| 변경 파일 중 인식 불가 경로    | WARN으로 기록하고 해당 파일은 처리에서 제외                            |

# 제약사항 및 원칙
1. **읽기 우선**: 에이전트는 스펙 파일의 내용을 임의로 생성하지 않는다. 반드시 변경된 원본과 기존 스펙을 근거로 업데이트한다.
2. **변경 최소화**: 관련 없는 스펙 파일은 수정하지 않는다.
3. **추적 가능성**: 모든 변경은 리포트에 기록되며, 커밋 메시지로 추적 가능해야 한다.
4. **안전한 실패**: 일부 단계가 실패하더라도 나머지 단계를 최대한 수행하고, 전체 결과를 리포트에 남긴다.

# 메모리 업데이트
실행 완료 후 다음 사항을 프로젝트 메모리에 기록한다:

- 실행 중단 시 중단 지점 및 사유
- 반복적으로 발생하는 오류 패턴
