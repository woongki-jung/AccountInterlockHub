# 제품 생산 파이프라인 에이전트

`start-builder` 진입점이 자동으로 호출하는 하위 에이전트 7종의 실행 시점·입력·결과 정리다.
상위 문서: [프롬프트 동작 모음](프롬프트%20동작모음.md)

이 파이프라인은 **사양서 → 제품 코드 → 인스톨러**로 전환한다.
하위 에이전트는 사용자가 직접 부르지 않고, `start-builder`의 4단계 흐름 안에서 실행된다.

```
start-builder
 ├ 1단계  build-sprint-planner  → 계획(ROADMAP·Phase) 수립
 ├ 2단계  build-sprint-worker   → Phase마다:
 │          frontend/backend-developer  (구현)
 │          → code-reviewer  (PASS/FAIL 리뷰)
 │          → tester        (독립 재현 QC)
 ├ 3단계  build-installer       → Debug/Release 인스톨러 빌드
 └ 4단계  build-sprint-closer   → 리포트·커밋·푸시
```

## 스프린트 관리 (3종)

### build-sprint-planner (`ai/agents/sprint-manager/build-sprint-planner.md`)
- **모델 / 호출 주체**: opus / start-builder (1단계)
- **실행 시점**: 빌드 스프린트가 시작될 때 가장 먼저 실행되어 구현 계획을 세운다.
- **입력 예시**: `구현 계획을 수립해줘. 대상 사양: docs/specs/ (AFB 카테고리 관리), 대상 프로그램: apps/CRMv2.Node.App.UI + apps/CRMv2.Net.App`
- **실행 결과**: 스프린트 폴더와 `ROADMAP.md`·`phase-<n>.md`가 생성되고(1 Phase = 1 기능 단위), "sprint 준비 완료"를 반환한다.
- **특징**: BE 변경 시 [`docs/prd/specification/api/`](../../../docs/prd/specification/) 외부 API 사양을 의무 참조해 "DDL 부재"만으로 구현을 미루지 않는다.
- 원본: [build-sprint-planner.md](../../../ai/agents/sprint-manager/build-sprint-planner.md)

### build-sprint-worker (`ai/agents/sprint-manager/build-sprint-worker.md`)
- **모델 / 호출 주체**: opus / start-builder (2단계)
- **실행 시점**: 계획 수립 후 Phase마다 실행되어 구현→리뷰→검증을 오케스트레이션한다 (직접 코딩하지 않고 developer에 위임).
- **입력 예시**: `작업 phase 문서: sprints/build/2026-07-09/build-460-정웅기/phase-2.md 를 실행해줘`
- **하위 developer 분기 규칙**:

  | 대상 프로그램 | 호출 에이전트 |
  |-------------|-------------|
  | `apps/CRMv2.Node.App.UI` (React) | web-frontend-developer |
  | `apps/CRMv2.Net.App` / `CRMv2.Net.Agent` / `YSRCRMv2.NET.Migrator` (C#) | backend-developer |

- **실행 결과**: 항목별로 developer→code-reviewer→tester가 돌고, Phase 문서 검증 결과가 채워진 뒤 Phase 상태가 갱신된다. closer 진입 전 실제 UI/API 왕복 검증까지 수행한다.
- 원본: [build-sprint-worker.md](../../../ai/agents/sprint-manager/build-sprint-worker.md)

### build-sprint-closer (`ai/agents/sprint-manager/build-sprint-closer.md`)
- **모델 / 호출 주체**: opus / start-builder (4단계)
- **실행 시점**: 모든 Phase와 인스톨러 빌드 후 마무리 단계에서 실행된다.
- **입력 예시**: `스프린트 결과를 정리하고 커밋해줘. 폴더: sprints/build/2026-07-09/build-460-정웅기/`
- **실행 결과**: `build-report.md`가 작성되고 ROADMAP·UI 진척 매트릭스가 갱신되며, `[start-builder]` 접두사로 커밋·푸시된다.
- 원본: [build-sprint-closer.md](../../../ai/agents/sprint-manager/build-sprint-closer.md)

## 코드 작성 (3종)

developer 2종은 `build-sprint-worker`가 작업 항목 단위로 호출하며, 아래 정보를 전달받는다.

- 입력(전달 파라미터) 예시:
  ```
  phase 문서: sprints/build/2026-07-09/build-460-정웅기/phase-2.md
  작업 항목: 카테고리 트리 렌더링 (SCR-AFB-001, SVC-012)
  대상 프로그램: apps/CRMv2.Node.App.UI
  선행 결과: BE /api/template/category/list 계약 확정됨
  ```

### backend-developer (`ai/agents/workflow-code-write/backend-developer.md`)
- **모델 / 호출 주체**: sonnet / build-sprint-worker
- **실행 시점**: 대상이 C# 프로그램(App·Agent·Migrator)인 작업 항목에서 실행된다.
- **내부 5단계**: 요구사항 분석 → 변경 확인 → 코드 수정 → code-reviewer 리뷰 → 완료 보고.
- **실행 결과**: OWIN 로컬 REST API·외부 UTEngine API 대리 호출·Npgsql/EMR DAO 등이 빌드 성공·커밋 상태로 구현되고, Phase 문서에 검증 결과가 기록된다.
- 원본: [backend-developer.md](../../../ai/agents/workflow-code-write/backend-developer.md)

### web-frontend-developer (`ai/agents/workflow-code-write/frontend-developer.md`)
- **모델 / 호출 주체**: sonnet / build-sprint-worker (frontmatter 이름은 `web-frontend-developer`)
- **실행 시점**: 대상이 `apps/CRMv2.Node.App.UI`(React 웹 UI)인 작업 항목에서 실행된다.
- **내부 5단계**: 요구사항 분석 → 퍼블리싱/앱 소스 비교 → 코드 수정 → code-reviewer 리뷰 → 완료 보고.
- **실행 결과**: 퍼블리싱 소스가 실 API 연동·디자인 토큰 적용·정책 반영된 React 제품 코드로 전환되어 커밋되고, Phase 문서에 검증 결과가 기록된다.
- **주의**: 외부 API 직접 호출 금지 — BE(`/api/*`) 경유(`apiPost`/`apiGet`)만 사용한다.
- 원본: [frontend-developer.md](../../../ai/agents/workflow-code-write/frontend-developer.md)

### code-reviewer (`ai/agents/workflow-code-write/code-reviewer.md`)
- **모델 / 호출 주체**: opus / 각 developer (내부 4단계)
- **실행 시점**: developer가 코드 수정을 마친 직후, 완료 보고 전에 호출된다.
- **입력(전달 파라미터) 예시**:
  ```
  구현 파일 목록(diff): src/features/template/CategoryTree.tsx 외 3건
  관련 스펙 코드: SCR-AFB-001, SVC-012
  조치한 피드백: gcsn1091-0625-01 (카테고리 순서 보존)
  ```
- **판정 방식**: Critical 1건 이상 → **FAIL**, 이월 정책 없는 Important → FAIL, Suggestion만/위반 0 → **PASS**.
- **실행 결과**: 첫 줄에 `PASS`/`FAIL` 마커, FAIL이면 위치·위반 패턴·사유·수정 가이드 4항목을 담은 이슈 목록을 반환한다. developer는 이 결과로 다음 단계 진행 여부를 자동 분기한다.
- 원본: [code-reviewer.md](../../../ai/agents/workflow-code-write/code-reviewer.md)

## 배포 (1종)

### build-installer (`ai/agents/workflow-publish/build-installer.md`)
- **모델 / 호출 주체**: sonnet / start-builder (3단계)
- **실행 시점**: 코드 생성이 끝난 뒤 인스톨러를 만들 때 실행된다.
- **입력 예시**: `코드 빌드 완료. 인스톨러를 빌드해줘 (Debug + Release)`
- **실행 결과**: `build.bat BOTH`로 Debug/Release 코드를 빌드하고, 인스톨러 4종(App/Agent × Debug/Release)을 생성한 뒤 정형 보고(✅/⚠️/❌)를 반환한다.
- **주의**: 본 에이전트 정의는 `build.bat` 기반이며, `start-qa`의 사전 환경 준비는 별도로 [`apps/deploy/scripts/build.ps1 -Mode {Test|Release}`](../../../apps/deploy/) 경로를 사용한다 (두 경로가 병존).
- 원본: [build-installer.md](../../../ai/agents/workflow-publish/build-installer.md)

> tester는 이 파이프라인에서도 `build-sprint-worker`의 QC(독립 재현 검증)로 호출된다. 상세는 [품질 검증 파이프라인](품질검증-파이프라인.md) 참조.
