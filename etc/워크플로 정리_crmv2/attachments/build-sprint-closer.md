본 문서는 ROADMAP을 기반으로 실행된 빌드 작업 결과를 확인하고 마무리하기 위한 프로세스를 정의한다.

# 작업 프로세스

## 1단계: 스프린트 진행 상태 확인

시작 시 전달받은 스프린트 정보를 확인하고, 완료 여부를 판정한다.

- `sprints/build/<날짜>/build-<n>-<git 유저명>/ROADMAP.md`를 읽고 전체 Phase 현황을 파악한다
- 각 Phase 문서 `sprints/build/<날짜>/build-<n>-<git 유저명>/phase-<n>.md`에서 다음을 확인한다
	- 작업 목록의 완료 현황 (✅/⬜ 상태)
	- 검증 결과 테이블의 Pass/Fail/Block 현황
	- 검토 및 제안사항 섹션의 내용 (수동 실행 항목, 리팩토링 제안, 스펙 발견사항 등)
- ROADMAP의 프로그램 구성 테이블에서 대상 프로그램 목록을 확인한다

## 2단계: 작업 결과 요약

각 Phase의 결과를 종합 분석하고 리포트를 작성한다.

### 결과 분석
- Phase별 작업 완료율 (완료 항목 / 전체 항목)
- Phase별 검증 통과율 (Pass / 전체 검증 항목)
- 프로그램별 구현 현황 (프로그램별 완료/미완료 항목 수)
- 미완료 또는 검증 실패 항목의 원인 및 영향 범위
- 수동 실행 필요 항목 취합 (DB 스키마 변경 등)
- 리팩토링 제안사항 취합
- 스펙 문서 관련 발견사항 취합

### 리포트 작성
`sprints/build/<날짜>/build-<n>-<git 유저명>/build-report.md` 파일을 생성하여 전체 실행 결과를 작성한다.
- 동일한 이름의 기존 파일이 있는 경우 실행 주기를 구분하여 기존 문서의 아래에 내용을 추가한다.
- 최종 결과서는 마지막으로 작성된 이전 결과서와의 내용을 비교하여 변경점에 대한 분석 섹션을 추가한다.

**리포트 구조:**
```markdown

# 빌드 SPRINT 실행 결과 보고

**종합 평가**: ✅ 완료 / ⚠️ 조건부 통과 / ❌ 실패 — 경고 N건, 실패 N건
> 종합 판정 사유 한 줄 요약

## 경고 및 참고사항

> ⚠️ **[WARN-001]** `대상 파일 또는 프로그램` — 경고 내용 설명
> - 영향 범위: 영향받는 프로그램, 서비스, 화면 등
> - 권장 조치: 해결 방법

> 📌 **[NOTE-001]** 참고 사항 설명
> - 관련 정보 또는 이전 실행 대비 변경점

## Phase별 실행 결과

| Phase | 단계명 | 대상 프로그램 | 작업 완료율 | 검증 통과율 | 상태 | 비고 |
|-------|--------|-------------|------------|------------|------|------|
| 1 | [단계명] | [프로그램명] | 5/5 (100%) | 3/3 (100%) | ✅ | - |
| 2 | [단계명] | [프로그램명] | 3/4 (75%) | 2/3 (67%) | ⚠️ | Fail 1건 |

## 프로그램별 구현 현황

| 프로그램 | 경로 | 구현 항목 | 완료 | 미완료 | 검증 Pass | 검증 Fail | 비고 |
|----------|------|----------|------|--------|----------|----------|------|
| [프로그램명] | `apps/[프로그램명]/` | N건 | N건 | N건 | N건 | N건 | - |

## 검증 결과 종합

| 검증 유형 | 전체 | Pass | Fail | Block | 비고 |
|-----------|------|------|------|-------|------|
| 자동 검증 | N | N | N | N | - |
| 수동 검증 | N | N | N | N | - |
| 데이터 정합성 | N | N | N | N | - |

### 검증 실패 항목 상세
| Phase | 검증 항목 | TC 코드 | 대상 프로그램 | 실패 원인 | 영향 범위 |
|-------|-----------|---------|-------------|----------|----------|
| N | 항목명 | TC-xxx | [프로그램명] | 원인 설명 | 영향 범위 |

## 수동 실행 필요 항목

- ⬜ **[Phase N]** 항목 설명 `[프로그램명]`
  - 실행 방법: 구체적 절차
  - 관련 파일: 마이그레이션 SQL 경로 등

## 후속 대응 체크리스트

- ⬜ **[WARN-xxx 대응]** 대응 항목 설명
- ⬜ **[Fail 항목 대응]** 실패 항목 수정 및 재검증
- ⬜ 수동 실행 항목 처리 후 관련 검증 재수행
- ⬜ 재검증 결과 확인 후 다음 Phase 착수 판단

## 제안사항 종합

### 리팩토링 제안
- Phase별 코드 리뷰에서 도출된 개선 항목

### 스펙 문서 개선 필요 항목
- 구현 중 발견된 스펙 문서의 모호하거나 누락된 부분

### 기술적 제약사항
- 스펙과 다르게 구현한 항목 및 사유

### 이전 결과 대비 변경점
- 이전 리포트 대비 신규/해소된 경고 및 이슈 변동 내역

```

### ROADMAP 최종 업데이트
- `sprints/build/<날짜>/build-<n>-<git 유저명>/ROADMAP.md`의 Phase 현황 테이블을 최신 상태로 업데이트한다
	- 모든 검증이 통과한 Phase의 상태를 ✅로 확정
	- 검증 미통과 항목이 남아있는 Phase는 🔄로 유지
- 리포트 문서 경로를 ROADMAP.md에 참조로 추가한다

## 3단계: 스프린트 결과 파일 정리 및 진척 매트릭스 갱신

### 스프린트 폴더 정리
- `sprints/build/<날짜>/build-<n>-<git 유저명>/` 폴더 내 파일 구성이 다음과 일치하는지 확인하고 정리한다
	- `ROADMAP.md` — 전체 로드맵 (최종 상태 반영)
	- `phase-<n>.md` — Phase별 상세 계획 및 실행 결과
- `docs/sprint-feedbacks/pending` 중 스프린트에서 반영 완료된 항목은 `docs/sprint-feedbacks/resolved` 폴더로 이동 (sprint-qa 검증 대기 상태)

### UI 스프린트 진척 매트릭스 갱신 (UI 작업 수반 시 필수)

UI 작업(`apps/CRMv2.Node.App.UI/` 변경)을 수반한 스프린트는 [`ai/strategies/sprint-ui-dev.md`](../../ai/strategies/sprint-ui-dev.md) §5 페이지 진척 매트릭스 및 §7 디자인 시스템 진척 표를 ROADMAP 결과와 정합되도록 반드시 갱신한다.

**갱신 대상 판정 기준**
- ROADMAP에서 ✅로 확정된 G1/G2 페이지 셀(L5 페이지 셸·L6 모달·L7 인터랙션·L9 API wire 등) 존재 시 → §5 해당 row 진척 갱신 필수
- L1~L4 디자인 시스템 자산(토큰/프리미티브/아이콘/도메인 컴포넌트/레이아웃) 변경 시 → §7 해당 row 진척 갱신 필수
- UI 작업이 없는 스프린트(백엔드 전용·문서 전용)는 본 단계를 생략한다

**갱신 항목 체크리스트**
- ⬜ §5 매트릭스의 해당 row 진척 컬럼 (⬜ → 🟡 또는 🟡 → ✅)
- ⬜ §5 footer 진척률 분자/분모 및 백분율 재계산 (G1 N/12 + G2 M/1)
- ⬜ §5 footer "직전 스프린트 반영" 라인에 본 스프린트 항목 추가
- ⬜ §5 footer "잔여" 라인에서 본 스프린트로 해소된 항목 제거 또는 갱신
- ⬜ §7 표 — L1~L4 변경 시 해당 row 진척 컬럼 갱신
- ⬜ 회귀 발생 시 §7 ✅ → 🟡 되돌림 (회귀 페이지 본문 보존)

**정합성 검증 (필수)**
- ROADMAP의 ✅ 페이지 셀과 §5의 ✅ row가 1:1 일치하는지 대조한다
- 불일치 발견 시 매트릭스를 ROADMAP 결과에 맞춰 즉시 갱신한다
- 매트릭스 갱신 사실은 리포트 "후속 대응 체크리스트" 또는 "이전 결과 대비 변경점" 섹션에 한 줄 등재한다 (감사 추적성)

## 4단계: 커밋 및 푸시

**스테이징 직전 체크리스트 (필수)**
- ⬜ ROADMAP 모든 Phase 상태(✅/🔄)가 검증 결과와 일치한다
- ⬜ ROADMAP에 리포트 경로가 실 경로로 등재되어 있다 (placeholder 잔존 0건)
- ⬜ UI 작업 수반 스프린트인 경우, `ai/strategies/sprint-ui-dev.md` §5/§7 매트릭스가 ROADMAP과 정합 (3단계 정합성 검증 통과)
- ⬜ 본 스프린트 범위 외 unstaged 변경은 보존(스테이징 제외)

체크리스트 미충족 항목 발견 시 누락 보정을 먼저 수행한 후 스테이징을 진행한다.

- 업데이트된 ROADMAP, Phase 문서, 리포트 파일을 스테이징한다
- 위 체크리스트의 §5/§7 매트릭스 갱신이 있었던 경우 `ai/strategies/sprint-ui-dev.md`도 함께 스테이징한다
- 커밋 메시지는 리포트 개요를 기반으로 작성한다

**커밋 메시지 형식:**
```
[start-builder] <상태>: <한 줄 요약>

구현: <구현_항목_수>건 (완료 <n>건, 미완료 <n>건)
검증: Pass <n>건, Fail <n>건, Block <n>건
대상 프로그램: <프로그램 목록>

📄 리포트: sprints/build/<날짜>/build-<n>-<git 유저명>/build-report.md
```

예시:
```
[start-builder] SUCCESS: Phase 1-2 기반 구축 및 사용자 인증 구현 완료

구현: 12건 (완료 12건, 미완료 0건)
검증: Pass 8건, Fail 0건, Block 0건
대상 프로그램: api-server, web-client

📄 리포트: sprints/build/<날짜>/build-<n>-<git 유저명>/build-report.md
```

- 에이전트 실행 중 일부 단계가 생략·실패한 경우 상태를 `FAILED`로 표기하고, 한 줄 요약에 중단된 지점을 명시한다
  예: `[start-builder] FAILED: Phase 2 검증 단계 오류로 중단`
- 작업 브랜치에 푸시한다
- **주의**: 커밋 메시지에 `[start-builder]` 접두사를 포함하여 웹훅 재트리거를 방지한다
- 푸시 완료 후 세션을 종료한다

# 오류 처리

| 상황 | 대응 |
| --- | --- |
| Phase 문서 읽기 실패 (경로 불일치, 누락) | WARN으로 기록하고 해당 Phase는 리포트에서 제외, 사용자에게 보고 |
| 리포트 작성 실패 | 리포트를 로컬에 저장하고 에러 내용을 출력 |
| ROADMAP 업데이트 실패 | 리포트 작성까지는 정상 수행, 업데이트 실패 사유를 리포트에 기록 |
| 커밋/푸시 실패 | 리포트를 로컬에 저장하고 에러 내용 출력 |
| 이전 리포트 비교 불가 (이전 파일 없음) | 변경점 분석 섹션을 "최초 실행" 으로 기록하고 계속 진행 |

# 제약사항 및 원칙

1. **읽기 전용**: Phase 상세 문서의 작업 목록이나 구현 코드를 수정하지 않는다. 상태 업데이트와 결과 기록만 수행한다.
2. **변경 최소화**: ROADMAP 상태 업데이트와 리포트 생성 외의 파일 변경을 수행하지 않는다.
3. **추적 가능성**: 모든 결과는 리포트에 기록되며, 커밋 메시지로 추적 가능해야 한다.
4. **안전한 실패**: 일부 단계가 실패하더라도 나머지 단계를 최대한 수행하고, 전체 결과를 리포트에 남긴다.
5. **재트리거 방지**: 에이전트가 생성한 커밋이 자기 자신을 다시 트리거하지 않도록 `[start-builder]` 접두사를 사용한다.

# 주의사항

- 스펙 문서나 구현 코드를 변경하지 않는다. 상태 업데이트와 결과 기록만 수행한다.
- 리포트에 민감 정보(비밀번호, API 키 등)를 포함하지 않는다.
- 검증 미통과 항목이 있는 Phase를 완료(✅)로 확정하지 않는다.

# 메모리 업데이트

작업 완료 후 다음 사항을 프로젝트 메모리에 기록한다:
- 완료된 Phase 목록 및 완료 일자 (프로그램별 구분)
- 미해결 항목 (검증 실패, 수동 실행 미완료 등)
- 리팩토링 제안사항 중 다음 Phase에 반영이 필요한 항목
- 스펙 문서 수정이 필요한 항목
- 반복적으로 발생하는 검증 실패 패턴
