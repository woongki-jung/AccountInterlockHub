본 문서는 PRD와 스펙 문서를 분석하여 제품화하기 위한 프로세스를 정의한다.
아래 지침에 따라 제공된 요구사항과 스펙 문서를 활용하여 프로젝트를 제품화하기 위한 단계별 제품 개발 프로세스를 생성한다.

# 작업 프로세스

## 1단계: 사양 문서 확인

아래 사항에 주의하여 사양 문서와 프로젝트 현황을 확인한다.

- 현재 사양 문서
	- **PRD 문서**: `docs/prd/PRD.md` 및 해당 문서에서 참조하는 모든 하위 문서
	- **스펙 문서**: `docs/specs/` 하위의 각 사양정의서
		- `docs/specs/policies/` — 접근 제어, 표시 규칙, 비즈니스 규칙 정책
		- `docs/specs/services/` — 서비스 기능(SVC) 및 사용 시나리오 정의
		- `docs/specs/datas/` — 엔터티(ENT) 구조, 서비스 데이터 모델(MDL) 정의
		- `docs/specs/functions/` — 공통 기능(FN) 및 API 인터페이스 정의
		- `docs/specs/screens/` — 화면(SCR) 구성 및 디자인 시스템 정의
		- `docs/specs/processes/` — 기능구현 프로세스 정의
		- `docs/specs/qa-dev/` — 테스트 케이스(TC) 및 검증 기준 정의 (AI 주도 산출물)
	- **이전 스프린트 피드백 문서**: `docs/sprint-feedbacks/pending/` 폴더 문서. 담당자 최종 피드백 항목으로 반드시 적용되도록 최상위 중요도로 적용하고, 적용이 불가한 경우 관련 내용을 보고한다.
	- 누락되는 문서가 없도록 주의한다
	- 직접적인 참조가 명시되지 않은 경우, 임의로 관련 파일 여부를 판단하지 않는다
	- 참조 여부는 항상 상위 문서 → 하위 문서로 참조하는 경우만 고려한다
- 퍼블리싱 코드 확인
	- UI 프로젝트의 경우, `pub/CRMV2.Node.App.UI` 폴더에 요구사항과 매칭되는 코드를 검색한 후, 개발 코드에 적용할 범위를 확인한다.
- 프로그램 구성 확인
	- `apps/` 하위 프로그램 목록과 각 프로그램의 기술 스택을 파악한다
	- 프로그램 간 공유 모듈 및 의존관계를 확인한다
	- 각 프로그램의 빌드·실행·테스트 환경 구성을 파악한다
- GIT 변경 이력 확인
	- `sprints/build/` 하위의 마지막 빌드 스프린트 결과의 commit 시점을 기준으로 변경된 사양을 파악하여 분석 범위를 좁힌다
		- git 이력상의 변경점 분석은 소스 영역으로 제한한다.
	- 기존 빌드 스프린트가 없는 최초 작업인 경우, git 이력확인은 생략한다.
- 이전 빌드 스프린트 확인
	- `sprints/build/<날짜>/build-<n>-<git 유저명>/` 폴더들 중 sprint 번호 `<n>` 가 가장 큰 폴더의 ROADMAP 및 phase 실행 결과를 확인한다 (날짜 폴더가 여러 개여도 sprint 번호가 단조 증가하므로 번호 기준 비교).
	- 기존 구현 진행 상황, 미완료 항목, 발견된 이슈를 파악한다
- 스프린트 피드백 확인
	- `docs/sprint-feedbacks/` 하위 문서를 확인하여 이전 스프린트에서 발생한 실제 문제와 피드백을 파악한다
	- Phase 계획 수립 시 동일한 유형의 문제가 재발하지 않도록 주의사항으로 반영한다

## 2단계: 구현 대상 분석 및 그룹 구성

1단계에서 수집한 정보를 기반으로 구현 대상을 분석하고 그룹을 구성한다.

### 구현 대상 분석

분석된 내용에서 다음 항목을 추출한다:
- 구현해야 할 서비스 기능(SVC) 전체 목록 및 우선순위
- 서비스 기능 간 의존관계 (어떤 SVC가 다른 SVC의 선행 조건인지)
- 공통 기능(FN) 중 여러 서비스에 선행되어야 하는 기반 기능
- 데이터베이스 엔터티(ENT) 간 의존관계 및 마이그레이션 순서
- 화면(SCR) 간 이동 경로에서 도출되는 구현 순서
- 외부 시스템 연동 등 기술적 불확실성이 높은 항목
- 기존 로드맵 대비 변경이 필요한 사항

### 프로그램별 구현 대상 분류

- 스펙 코드(SVC, FN, SCR, ENT, MDL)를 `apps/` 하위 프로그램 단위로 매핑한다
- 여러 프로그램에 걸치는 공통 모듈(인증, 로깅, 설정 등)을 식별한다
- 프로그램별 기술 스택에 따른 구현 방식 차이를 고려한다

### 구현 그룹 구성 지침

- 기능/비기능 중심으로 1차 분류 후, 프로그램/서비스 단위로 2차 분류한다
- 각 구현 그룹은 상호 의존성을 최소화하여 독립적으로 구현·검증 가능하도록 구분한다
- 이전에 구현이 완료된 항목이 있는 경우, 변경이 발생한 부분에 대하여 집중적으로 계획한다
- 각 Phase는 독립적으로 배포 및 검증 가능한 단위로 구성한다
	- 하나의 phase는 하나의 기능 구현만을 포함한다. 사용자 화면 또는 앱 단위가 아닌 아닌 코드 상의 기능으로, 세분화한다. (ex. 홈 화면 구현 (X) -> 홈 화면 - 예약 목록 구현)
	- phase 구성 순서는 빌드및 실행 가능 상태 확보 > 공통기능 (레이아웃/공용컴포넌트) > 세부기능 순으로 배치한다.
- 검증 계획을 Phase 구성 시 함께 정의한다
	- 단위/통합 테스트: 공통 기능(FN) 및 API 검증
	- 데이터 정합성 테스트: 엔터티(ENT) 제약조건, CRUD 정합성 검증
	- E2E 테스트: 서비스 시나리오(SVC) 기반 화면(SCR) 흐름 검증
	- 정책 준수 테스트: 접근 제어, 입력값 검증 등 정책 규칙 위반 시나리오 검증
- 검증 미통과 시 해당 Phase를 완료로 전환하지 않는다

### 우선순위 결정 기준

1. **MoSCoW 기법 적용**: Must Have → Should Have → Could Have → Won't Have
2. 기술적 의존성: 하위 레이어부터 상위 레이어 순서 (공통 모듈 → 백엔드 → 프론트엔드)
3. 비즈니스 가치: 사용자에게 가장 빠른 가치를 제공하는 순서
4. 리스크: 불확실성이 높은 기술은 초기에 검증

## 3단계: ROADMAP 작성

`sprints/build/<날짜>/build-<n>-<git 유저명>/ROADMAP.md` 파일에 전체 Phase의 진행 방향과 현황을 작성한다. 다음 구조를 따른다.

```markdown
# 프로젝트 로드맵

## 개요
- 프로젝트 목표 한 줄 요약
- 현재 진행 단계
- 전체 Phase 수 및 예상 일정

## 진행 상태 범례
- ✅ 완료
- 🔄 진행 중
- 📋 예정
- ⏸️ 보류

## 프로그램 구성

| 프로그램 | 경로 | 기술 스택 | 역할 |
|----------|------|-----------|------|
| [프로그램명] | `apps/[프로그램명]/` | 기술 스택 | 역할 설명 |

## 구현 그룹 → Phase 매핑

| 구현 그룹 | 대상 프로그램 | 관련 스펙 코드 | 매핑 Phase |
|-----------|-------------|---------------|------------|
| [그룹명] | [프로그램명] | SVC-xxx, FN-xxx | Phase 1 |

> 📌 2단계에서 분류한 구현 그룹이 각 Phase에 어떻게 배정되었는지를 명시한다.

## Phase 현황

| Phase | 단계명 | 대상 프로그램 | 스프린트 | 상태 | 핵심 목표 | 상세 문서 |
|-------|--------|-------------|----------|------|-----------|-----------|
| 1 | [단계명] | [프로그램명] | Sprint 1 | ✅ | 핵심 목표 요약 | [상세](phase-1.md) |
| 2 | [단계명] | [프로그램명] | Sprint 2 | 🔄 | 핵심 목표 요약 | [상세](phase-2.md) |

> 📌 Phase 간 의존관계 및 리스크 사항은 각 Phase 상세 문서에서 관리한다.

## 마일스톤
- 주요 릴리스 포인트

## 리스크 및 완화 전략
- 예상 이슈와 대응 방안

## 기술 부채 관리
- 기술 부채 항목 및 각 Phase에의 배분 계획

## 후속 개발 대상
- 현재 구현 범위 밖 항목 및 추후 검토 예정 사항
```

## 4단계: Phase 구성

각 Phase별로 `sprints/build/<날짜>/build-<n>-<git 유저명>/phase-<n>.md` 파일을 생성한다. 다음 구조를 따른다.

```markdown
# Phase N: [단계명]

## 개요
- **목표**: 이 단계에서 달성하는 핵심 가치
- **대상 프로그램**: 구현 대상 프로그램 목록
- **스프린트**: Sprint N
- **상태**: 진행 상태
- **선행 Phase**: 의존하는 Phase 번호 (없으면 없음)

## 관련 스펙 코드
| 유형 | 코드 | 설명 | 대상 프로그램 |
|------|------|------|-------------|
| 서비스 | SVC-xxx | 서비스 기능명 | [프로그램명] |
| 공통 기능 | FN-xxx | 공통 기능명 | [프로그램명] |
| 화면 | SCR-xxx | 화면명 | [프로그램명] |
| 엔터티 | ENT-xxx | 엔터티명 | 공통 |
| 데이터 모델 | MDL-xxx | 모델명 | [프로그램명] |

## 작업 목록
- ⬜ **[기능명]** (SVC-xxx, FN-xxx) `[프로그램명]`: 상세 설명
  - 세부 태스크 1
  - 세부 태스크 2

## 완료 기준 (Definition of Done)
- 기준 1
- 기준 2

## 검증 계획
### 자동 검증
- ⬜ **[테스트 유형]**: 검증 대상 (관련 TC 코드) `[프로그램명]`
  - 실행 명령어 또는 실행 방법
  - 통과 기준

### 수동 검증
- ⬜ **[검증 항목]**: 검증 내용 (관련 TC 코드)
  - 검증 절차
  - 판정 기준

### 데이터 정합성 검증
- ⬜ **[검증 항목]**: 대상 엔터티 (관련 TC 코드)
  - 검증 쿼리 또는 확인 방법
  - 기대 결과

## 검증 결과
| 검증 항목 | TC 코드 | 대상 프로그램 | 결과 | 비고 |
|-----------|---------|-------------|------|------|
| 항목명 | TC-xxx | [프로그램명] | Pass / Fail / Block | 비고 |

## 검토 및 제안사항

### 코드 리뷰 요약
| 수준 | 건수 | 주요 내용 |
|------|------|-----------|
| Critical/High | 0 | - |
| Medium | 0 | - |
| Low | 0 | - |

### 스펙 관련 발견사항
- 구현 중 발견된 스펙 문서의 모호하거나 누락된 부분

### 리팩토링 제안
- 코드 리뷰에서 도출된 개선 항목 (현재 Phase 범위 외)

### 기술적 제약사항
- 스펙과 다르게 구현한 항목 및 사유

### 수동 실행 필요 항목
- 사용자가 직접 수행해야 할 항목 (DB 스키마 변경 등)
```

# 문서 작성 원칙

## 역할 준수
- 본 문서는 제품 개발 로드맵을 정의하는 것이 목적으로, 스펙 상세(기능 정의, 데이터 설계 등)는 포함하지 않는다. 구현 순서·범위·일정에 집중하여 서술한다.

## 명확성
- 모호한 표현 금지 — "적절히", "충분히" 대신 수치나 조건으로 명시
  - 나쁜 예: "성능을 최적화한다"
  - 좋은 예: "Lighthouse 성능 점수 90+ 달성"
- 각 태스크는 개발자가 즉시 시작할 수 있도록 구체적으로 기술

## 일관성
- 모든 Phase에 동일한 문서 구조(목표, 작업 목록, 완료 기준, 검증 시나리오) 적용
- 작업 항목에 관련 스펙 코드(SVC, FN, SCR, ENT, TC 등)와 대상 프로그램을 명시

## 완결성
- 모든 스펙 문서의 항목이 하나 이상의 Phase에 배정되어야 한다
- 각 Phase에 완료 기준(Definition of Done)과 검증 시나리오가 포함되어야 한다
- `apps/` 하위 모든 프로그램이 구현 그룹에 포함되어야 한다
- FE - BE 연동이 필요한 경우, 반드시 한 스프린트에서 양쪽 작업을 모두 진행하여 기능 구현을 완결하도록 구성해야 한다.

## 추적 가능성
- 각 작업 항목이 어떤 스펙 코드에서 비롯되었는지 기술
- 스펙 코드 → Phase/스프린트 매핑이 양방향으로 추적 가능해야 한다
- 작업 항목이 어떤 프로그램에 속하는지 명시

# 산출물 검증 체크리스트

ROADMAP 작성 완료 후 다음을 확인한다.
- ⬜ PRD 및 스펙 문서의 모든 핵심 기능이 로드맵에 반영되었는가?
- ⬜ 각 Phase의 의존성이 올바르게 설정되었는가?
- ⬜ MVP 범위가 명확하게 정의되었는가?
- ⬜ 각 태스크가 실제로 실행 가능한 수준으로 구체적인가?
- ⬜ `apps/` 하위 프로그램 구성이 정확히 반영되었는가?
- ⬜ 프로그램 간 공유 모듈 및 의존관계가 식별되었는가?
- ⬜ 완료 기준(Definition of Done)이 측정 가능한가?
- ⬜ 각 Phase에 테스트 케이스(TC) 기반 검증 시나리오가 포함되었는가?
- ⬜ 각 Phase의 검증 계획이 자동/수동/데이터 정합성으로 구분되어 있는가?
- ⬜ 검증 항목이 해당 Phase에서 구현된 스펙 코드를 빠짐없이 포괄하는가?
- ⬜ 검증 미통과 시 Phase 완료 불가 원칙이 반영되었는가?
- ⬜ 작업 항목에 관련 스펙 코드와 대상 프로그램이 명시되었는가?

# 출력 형식

- 스프린트 폴더: `sprints/build/<날짜>/build-<n>-<git 유저명>/`
- 전체 로드맵: `sprints/build/<날짜>/build-<n>-<git 유저명>/ROADMAP.md`
- Phase별 상세: `sprints/build/<날짜>/build-<n>-<git 유저명>/phase-<n>.md`
- 폴더 경로 명명 규칙:
	- `<날짜>`: 스프린트 시작일을 `YYYY-MM-DD` 형식으로 표기 (예: `2026-05-20`).
		- 같은 날 여러 sprint 가 시작될 수 있으므로 날짜 폴더 하위에 다수의 `build-<n>-<git 유저명>` 폴더가 공존할 수 있다.
	- `<n>`: 워크스페이스 전체에서 단조 증가하는 빌드 스프린트 번호. 가장 최근 sprint 번호 + 1.
	- `<git 유저명>`: 작업자의 git 사용자명 (`git config user.name` 결과).
- 언어: 한국어
- 형식: Markdown
- 개발자가 바로 작업을 시작할 수 있을 만큼 상세하게 작성

# 제약사항 및 원칙

- 모든 작업은 이전 빌드 스프린트의 진행 사항을 참고하여 추가/변경에 대하여 빠짐없이 서술한다.
- 기존 작업 사항의 변경은 다른 방법으로 대체 불가능하거나 요구사항에 기존 요건에 대한 변경이 필요한 점임을 명시한 경우에만 진행한다.

# 주의사항

- PRD 및 스펙 문서에 명시되지 않은 기능을 임의로 추가하지 않는다.
- 특정 구현 기술(프레임워크, 라이브러리)을 강제하지 않고 스펙 문서의 요구사항을 기반으로 계획한다.
- 기술적 실현 가능성을 항상 고려한다.
- 완벽한 계획보다 적응 가능한 계획을 우선시한다 (Agile 원칙).
- 기존 작성 내용에서 변경이 필요한 경우 변경 지점과 사유, 영향범위를 ROADMAP.md 내에 명시한다.

# 메모리 업데이트

실행 완료 후 다음 사항을 프로젝트 메모리에 기록한다:
1. **로드맵 구성 결과** — 생성된 Phase 수, 각 Phase별 구현 그룹 배정 요약, 대상 프로그램 구성. 다음 실행 시 기존 로드맵과의 변경점을 파악하는 데 활용.
2. **프로그램 구성 현황** — `apps/` 하위 프로그램 목록, 기술 스택, 프로그램 간 의존관계. 프로그램 추가/변경 시 계획 재수립의 기준이 됨.
3. **계획 수립 중 발견된 이슈** — 스펙 모호, 프로그램 간 충돌, 기술적 제약 등. 반복 발생 시 스펙 개선 피드백으로 활용.
4. **계획 수립 중단 시 중단 지점 및 사유** — 재실행 시 이어서 진행하거나 동일 문제를 회피하기 위한 정보.
