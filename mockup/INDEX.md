# 목업 인덱스 — AccountInterlockHub

확정 화면 사양([`../docs/specs/screens/`](../docs/specs/screens/))을 반영한 **UI 중심 경량 목업**(버려지는 static HTML)이다. 실코드·비즈니스 로직·실 API 없음. 화면 외형·흐름·상태(정상/Loading/Empty/Error)만 시각화한다. 데이터는 모두 명백한 더미값이다.

> **2026-07-11 `#214` 개정판**: 핵심 연동 플로우를 **단일 암호화 JSON(encX/encY)·허브 복호화·수신처 서버-서버 전달·연동 추적 키(X 내부 필드)**로 재정의한 화면 사양(`docs/specs/screens/` 전체)을 반영해 목업을 갱신했다. 사용자 흐름을 **발송처 링크 진입(접근 주소=발송처 판별)→생년월일 입력→동의→승인→(허브 복호화·전달)→완료/실패**로 바꾸고, SCR-005 에 **생년월일 입력 필드**와 복호화 실패 재입력·링크 오류 상태를 신설했다. 관리자 폼(SCR-003)·상세(SCR-004)에서 **전달 파라미터 정의·사용자 키값 exactly-one 지정(`#33`)·서비스 A 진입 주소를 폐기**했다. SCR 코드·파일명은 유지하고 의미만 갱신했다.
>
> **2026-07-18 `#408` 개정판(사용자 페이지 디자인 개선)**: 화면·플로우 변경 없이 SCR-005·006 의 **시각 디자인만** 개선했다 — 근거는 [`design-system.md`](../docs/specs/screens/design-system.md) §사용자 표면 확장 신설(`-u-` 접두 색·타이포·간격·모션 토큰, `Card(user)`·`Button(user)`·`TextField(user/birthdate)`·`ConsentItem(user)`·`Badge(user)`·`Banner(user)`·`Modal(user/terms)`·`ResultIcon(user)` 사용자 변형, 콘텐츠 상한·오버플로 규칙). **플로우·유효성 규칙·트리거 PROC·고정 문구 원문·민감값 노출 규칙은 불변**이다. 개선 전(main 원본) 화면은 `SCR-005-before.html`·`SCR-006-before.html` 로 보존했다 — 이 두 파일은 **담당자 승인 게이트의 비교 참고 자료이며 승인 후 폐기 예정**이다(정본 아님). 상세는 §개선 전(비교용) 사본 및 §사양 보완 후보(`#408` 라운드) 참고.

## 열람 방법

- **진입점은 [`index.html`](index.html) 허브 페이지**다. 브라우저로 `index.html` 를 열면 6개 화면을 관리자/사용자 흐름으로 묶어 카드로 보여주며, 카드를 눌러 각 화면으로 이동한다(로컬 파일, 서버 불필요).
- 모든 화면(허브 포함) 상단에는 **공통 내비게이션 바**(어두운 바)가 있어, 어느 화면에서든 다른 화면·허브로 자유롭게 오갈 수 있다. 현재 보고 있는 화면은 파란색으로 표시된다.
- 내비게이션 바 아래의 **검은색 "상태 시연" 바**는 목업 전용 도구다(제품 UI 아님). 버튼으로 정상/Loading/Empty/Error 등 상태를 전환해 확인한다.
- 사용자 화면(SCR-005·006, 각 `-before` 사본 포함)에는 상태 시연 바 위에 **버전 비교 바**가 추가로 있어 개선 후 ↔ 개선 전을 원클릭으로 오갈 수 있다(`#408`).
- 화면 전환은 실제 링크(`<a href>`)로도 연결되어 목록→상세→편집, 동의→결과 등 제품 흐름을 클릭으로 그대로 따라갈 수 있다.

> **`index.html` ↔ 본 문서(`INDEX.md`)**: `index.html` 는 클릭으로 오가는 열람 진입 허브이고, 본 `INDEX.md` 는 화면↔SCR↔IA↔상태 매핑·흐름·디자인 근거를 담은 추적 문서다(병존). 상세 매핑은 아래 표를 참고한다.

## 화면 흐름 요약

**관리자 흐름**

1. SCR-001 로그인 성공 → SCR-002 목록.
2. SCR-002 목록 "발송처 접근 주소 등록" → SCR-003(신규), 행(구성명) 클릭 → SCR-004 상세.
3. SCR-004 상세 "편집" → SCR-003(편집), 저장 성공 → SCR-004 복귀.
4. SCR-004 상세 "삭제" 확정 → SCR-002 목록 복귀.
5. 모든 관리자 화면에서 세션 만료 → SCR-001(`?expired=1`) 재인증 유도(SCR-001 "세션만료 안내" 상태로 시연).

**사용자 흐름 (`#214` 개정)**

1. 발송처 링크(`/interlock/entry/:accessAddressId?encX=…&encY=…`)로 SCR-005 이용 동의에 **직접 진입** — 접속한 접근 주소(고유 ID)가 발송처 판별값이다. 선행 화면 없음(구 요청 키값(UUID) 발급 경로는 `#214` 로 폐기).
2. SCR-005 상단에는 관리자 구성의 **동의 대상 설명 문구(consentNotice, 선택·`#215` 추가)**가 노출된다. **생년월일 입력** + 필수 동의 체크 후 승인 제출 → 허브 복호화·수신처 B 전달(목업은 `SCR-006.html?result=…` 쿼리로 결과 상태를 전달해 시연).
3. 결과에 따라 SCR-006 이동: 완료(success)·거부(reject)·**링크 오류(linkerr, `#214` 신설)**. 단 **복호화 실패**(생년월일 인라인 에러)와 **전달 실패**(`#215`: "동의 처리에 실패했습니다. 다시 시도해주세요." 배너 + 재승인)는 **SCR-005 에 머물러** 재시도를 유도한다(화면 이동 없음 — 목업은 별도 "상태 시연" 버튼으로 시연).
4. SCR-006 은 결과 유형만 표시하고 신규 화면 이동을 강제하지 않는다(Fallback 은 안내 문구만 제공하고 버튼·링크는 두지 않는다 — 유효한 `accessAddressId` 가 없어 재진입 CTA 를 구성할 수 없음, §이동 경로. **`#408`: 개선 전 목업엔 "동의 화면으로 이동" 버튼이 있었으나 사양 규정에 맞춰 개선 후에서 제거**).

## 반영한 디자인

[`../docs/specs/screens/design-system.md`](../docs/specs/screens/design-system.md) 토큰을 하드코딩 반영: primary #2563EB / danger #DC2626 / success #16A34A / info #0891B2 / text #111827, 간격 4·8·12·16·24·32·48px, 라운딩 4·8·12px, 타이포 display28/h1 22/h2 18/body15/label14/caption13. 관리자 셸=헤더(AdminNav)+1120px 컨테이너, 사용자 셸=중앙 카드 480px. SCR-005 본인확인 섹션은 디자인 시스템의 `TextField(birthdate)` 변형(6자리 숫자·`inputMode=numeric`·YYMMDD 플레이스홀더)을 반영했다. SCR-006 거부 상태는 design-system §상태 표현 규칙의 "거부(info 중립)" 정의에 맞춰 `info` 배지(청록)를 사용한다(구 버전은 회색 neutral 오표기 — 본 개정에서 수정).

**SCR-005·006 개선 후(`#408`)**: 위 전역 토큰 대신 §사용자 표면 확장의 `-u-` 접두 토큰을 하드코딩 반영했다 — 배경 `color-u-canvas` #EEF2F7, 보조텍스트 `color-u-text-muted` #5F6773, 경계 `color-u-border`/`-strong` #7D8698/#4B5563, 구분선 `color-u-divider` #DDE3EA, 행 상태 `color-u-row-hover`/`color-u-row-selected`, 결과 색 3종 쌍(success #E7F6EC·#166534·#15803D / info #E6F6FB·#155E75·#0E7490 / danger #FEF2F2·#B91C1C·#DC2626 / neutral #F1F5F9·#334155·#64748B), 타이포(`text-u-title` 24/32·`text-u-lead` 15/24·`text-u-section` 13/18·`text-u-item` 15/22·`text-u-value` 17/24·`text-u-caption` 13/18), 간격(`space-u-shell-y` 48/24·`space-u-card` 28/20·`space-u-section` 28·`space-u-stack` 12), 라운딩(`radius-u-card` 16·`radius-u-control` 10·`radius-u-pill` 999), 모션(`motion-u-fast` 120ms·`motion-u-base` 200ms·`motion-u-slow` 320ms + `ease-u-standard`/`ease-u-decelerate`, `prefers-reduced-motion: reduce` 대응 — 애니메이션·전환 지속시간 0 처리). `ResultIcon(user)`(원형64px·글리프28px·선굵기2.5px)은 형태 서술(체크/가로막대/느낌표/물음표)에 맞춰 인라인 SVG 로 새로 그렸다. `SCR-005-before.html`·`SCR-006-before.html`(개선 전)은 전역 토큰 하드코딩 방식을 그대로 유지한다 — 두 세대의 시각 차이 자체가 비교 대상이다.

encX·encY·생년월일 등 민감값은 design-system §마스킹·노출 제어(SEC-005-06·SEC-006-06)에 따라 어떤 화면·더미 데이터에도 값으로 노출하지 않는다 — 생년월일 입력 필드는 항상 빈 값(placeholder만 `YYMMDD`)으로 두고, SCR-005 mock-bar 태그·SCR-006 경로 표기에도 실제 encX·encY 값은 등장하지 않는다. 개선 후 SCR-005 의 "필수 동의 자동 체크(시연)" 편의 버튼도 체크박스만 조작할 뿐 생년월일 필드는 채우지 않는다(활성화까지 보려면 담당자가 직접 6자리를 입력).
