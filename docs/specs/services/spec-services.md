# 서비스 기능 정의서 — AccountInterlockHub

본 문서는 AccountInterlockHub 전체 서비스 기능 정의의 개요와 하위 문서 목록을 정의한다. spec 단계 산출물(도메인 2순위)이며, 방향 근거는 [`../../prd/PRD.md`](../../prd/PRD.md), 정보 구조는 [`../../prd/ia/IA.md`](../../prd/ia/IA.md), 선행 정책은 [`../policies/spec-policies.md`](../policies/spec-policies.md)다. 후행 도메인(데이터·프로세스·화면·QA)이 SVC 코드로 본 문서군을 인용한다. 추적 일감: `accountinterlockhub#217`.

> **2026-07-11 `#214` 개정**: 핵심 연동 플로우를 **단일 암호화 JSON(encX/encY)·허브 복호화·수신처 서버-서버 전달·연동 추적 키(전달 데이터 X 내부 필드) 기반 추적**으로 재정의했다. 이용 주체를 4종(발송처·수신처·사용자·연동 관리자)으로 재편하고, 구 '전달 파라미터 + 사용자 키값 exactly-one 지정(`#33`)'·'허브 발급 요청 키값(UUID)'을 폐기했다. SVC 코드는 유지하되 의미를 갱신했다. 후행 도메인(datas/functions/screens/processes/qa)은 v0.1.0 상태이며 PROC/MDL 등 후행 코드는 예약 채번(실재 확인은 교차검증 시점).

## 사용자 정의

| 역할 | 설명 | 접근 범위·맥락 |
|------|------|----------------|
| 연동 관리자(허브 운영자) | 허브 관리자 화면에서 발송처별 접근 주소(고유 ID)·수신처(B) 전달 주소·동의 항목을 등록·편집·조회·활성/삭제 | IP 허용 목록 + 로그인 인증 통과 시 관리자 경로 접근. 처리 상태·회원 키·복호화 원문에는 접근하지 않음 |
| 발송처(서비스 A 운용사) | 연동 요청 주체·외부 시스템 사용자. 전달 데이터 X 를 생성·이중 암호화(encX/encY)해 접근 주소에 붙여 사용자를 유도하고, 연동 추적 키로 처리상태·완료 확인을 조회 | 접근 주소 고유 ID 로 발송처 식별. 서버 대면 API(처리상태·완료 확인) 사전 공유 자격 보유(주체 분리) |
| 수신처(서비스 B 운용사) | 연동 대상·외부 시스템 사용자·완료 통지 주체. 허브가 서버-서버로 전달한 복호화 원문 X 의 회원 키로 회원 여부를 판단하고, 완료를 콜백으로 통지 | 허브 서버-서버 전달 수신. 완료 콜백 API 사전 공유 자격 보유(주체 분리) |
| 사용자(서비스 A의 고객) | 최종 사용자·동의/본인확인 주체. 발송처 링크로 접근해 생년월일을 입력하고, 동의 항목 확인 후 동의/거부·승인을 결정 | 발송처 링크 진입(접근 주소=발송처 판별). 생년월일은 복호화 요소이며 인증·세션 미발급(AUTH-004) |
| 상태 보관 배치 | 사람이 호출하지 않는 스케줄 시스템 액터 | 단일 App Service 내 일 배치. 개인정보·복호화 원문 미취급 |

### 역할 관계

1. 연동 관리자가 등록한 발송처 접근 주소 구성이 사용자 접근·동의 화면·수신처 전달의 전제가 된다(구성 → 실행 방향 의존).
2. 발송처가 X 를 이중 암호화해 접근 주소에 붙여 사용자를 유도하고, 사용자가 생년월일 입력·동의로 승인하면, 허브가 복호화해 수신처 B 로 서버-서버 중개한다.
3. 허브는 복호화 원문·회원 키·생년월일·암호값을 저장하지 않고, X 내부 **연동 추적 키**(불투명 문자열)만 추출해 처리상태·연동이력의 키로 저장한다(무저장·불투명 키 원칙).
4. 관리자 경로와 사용자·API·배치 경로는 인증·접근 방식이 분리되고, 서비스 대면 API 인증 자격도 주체별(발송처/수신처)로 분리된다(SEC-003-03).
5. 발송처·수신처는 공유된 연동 추적 키(X 내부 필드)로 각각 처리상태·완료 확인 조회(발송처)와 완료 콜백 통지(수신처)를 수행한다(연동이력 근거).

## 서비스 코드 체계

- 코드 형식: `SVC-<순번 3자리>` (예: `SVC-001`). 하나의 서비스 시나리오 단위.
- 기능 번호: 서비스 내 `F-<순번>` (문서 로컬).
- 분기 코드: `BR-<번호>` — PROC 대역(1xx 관리자·2xx 사용자·3xx API·4xx 배치)과 정합. PROC 분기 흐름과 동일 코드 공유.
- 예외 코드: `EX-<그룹>-<순번>` — 정책 정의서의 위반 처리 코드를 그대로 인용. PROC 예외 흐름과 동일 코드 공유.
- 각 SVC 는 1개 이상의 PROC 로 구현된다(SVC ↔ PROC 1:N). PROC 코드는 후행 도메인이 채번하며 본 문서는 예약 코드로 인용한다.

## 서비스 시나리오 요약

1. **관리자 진입·구성(SVC-003 → SVC-001 → SVC-002)**: IP·인증 통과 후 발송처 접근 주소(고유 ID)를 생성하고 수신처 B 전달 주소·동의 항목을 등록·편집하고 조회·활성/삭제로 운영한다.
2. **사용자 접근·승인(SVC-004 → SVC-005)**: 발송처 링크(접근 주소 + encX·encY)로 진입해 발송처를 판별하고, 생년월일 입력·동의 항목 확인·필수 동의 체크 후 승인한다. 승인 시 허브가 복호화(생년월일→encY→키→encX→X)해 연동 추적 키를 확보하고 연동이력 1건을 생성한 뒤 X 를 수신처 B 로 서버-서버 전달하고 처리 상태 1건을 저장한다. 거부·복호화 실패는 추적 키가 없어 상태·이력을 남기지 않는다(감사만).
3. **완료 통지·확인(SVC-009 → SVC-008)**: 수신처 B 가 X 에서 얻은 연동 추적 키로 완료를 통지하면 연동이력에 수신 여부·수신 일시를 기록하고(처리상태 불변경), 발송처가 동일 추적 키의 최신 이력으로 처리완료 여부를 확인한다.
4. **조회·정리(SVC-006, SVC-007)**: 발송처가 연동 추적 키로 처리·결과 확인 상태를 조회하고, 일 배치가 처리상태·연동이력의 보관 기간 경과분을 삭제한다(90/180일 fallback).

- 사용자 거부·필수 동의 미충족 승인 차단·세션 만료·배치 실패는 EX 코드가 아니다 — 거부·미충족은 200 정상 종료·정상 흐름 차단, 세션 만료는 재인증 유도, 배치 실패는 다음 주기 재시도로 처리한다. 완료 콜백 재통지(완료 이력만 존재)도 오류가 아니라 멱등 성공으로 처리한다(EXC-BIZ-10).
- 결번: EX-BIZ-003(미사용)·EX-BIZ-007(지정 파라미터 값 누락 — `#214` 폐기)은 재사용하지 않는다(신규 채번은 EX-BIZ-009부터). EX-SEC-006/007·EX-BIZ-008 은 `#214` 신규 채번이다.

## 담당자 확정 대기 항목

정책 정의서 §담당자 확정 대기·§spec 확정 기본값과 동일 근거의 확정 기본안이다(승인 전 잠정).

- **Q1 관리자 인증**: IP 제한 + 로그인 인증 병행(SVC-003). 잠금 임계치·세션 유휴·비밀번호 복잡도 수치는 기본안.
- **연동 추적 키**: 발송처가 전달 데이터 X 내부에 넣는 불투명 문자열이다(허브 미발급·미해석). 허브는 비어있지 않음·최대 길이만 검증하고 처리상태·연동이력·서비스 대면 API 의 단일 키로 사용한다(DATA-002, SVC-005·006·008·009).
- **보관 상한**: 처리상태·연동이력 모두 결과 확인/수신 후 90일과 생성/요청 후 180일 중 먼저 도래·미확인/미수신 180일 절대 상한(SVC-007, DATA-004/006 spec 확정값). 통계 목적 장기 보관 여부는 담당자 확정 대기.
- **암호 규약**: AES-256-CBC + PKCS#7·키 32B/IV 16B `_` 정규화·encY 평문=encX 키 문자열·Base64URL(SEC-006, spec 확정값). encX 전달 형식(GET 파라미터 길이 한계) 시 접근 URL 구성은 화면·build 확정.
- **실패·거부 UX**: 복호화 실패=생년월일 재입력 재시도(SVC-005·BR-204), 전달 실패=재시도 안내+처리상태 실패(SVC-005·EX-BIZ-004), 거부·필수 미충족=취소·복호화 미수행 최소 기록(SVC-004, BIZ-002·BIZ-003 spec 확정값).
- **완료 판정·콜백 특정**: 연동 추적 키 단독·연동 요청 일시 최신 1건(SVC-008·SVC-009, BIZ-004-09/10 기본안).
- **보안 강화(후속 보완)**: 링크 만료(TTL)·1회성 사용·발송처 서명 검증은 초기 범위 제외의 후속 보완 항목이다(SEC-008, 범위 밖).
- **(폐기)** `#33` 사용자 키값 파라미터 exactly-one 지정 체계·허브 발급 요청 키값(UUID)은 `#214` 로 폐기됐다 — 입력이 단일 암호화 JSON 으로 바뀌고 추적 키가 X 내부 필드로 대체됐다.
