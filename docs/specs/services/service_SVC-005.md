# 연동 실행·서비스 B 전달 기능 정의

## 개요

- **서비스 목적**: 사용자 승인이 완료된 연동 요청에 대해 허브가 **사용자 생년월일로 복호화**(encY→복원 키→encX→전달 데이터 X)를 수행하고, X 에서 연동 추적 키를 확보해 **연동이력 1건을 생성**한 뒤, 복호화 원문 X 를 접근 주소 구성의 **수신처(B) 전달 주소로 서버-서버 전달**하고 처리 상태 4항목을 1건 저장한다. 성공 시 사용자에게 원문을 포함하지 않는 완료 페이지를 노출한다. 복호화 원문·암호값·생년월일은 메모리에서만 다루고 전달 후 폐기한다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §수행 범위 4 "연동 실행(허브 복호화·전달): 승인 시 허브가 생년월일로 encY→키→encX→X 복호화 → X 를 접근 주소에 설정된 수신처(B) 주소로 전달 → 성공 시 연동 완료 페이지" · §수행 범위 7 "상태 보관: 추적 키별 최소 상태 저장" / IA [`../../prd/ia/IA.md`](../../prd/ia/IA.md) `USR-02`·`BAT-01`·`BAT-03`.

> **2026-07-11 `#214` 개정**: 회원 키 수신·전달을 **허브 복호화(encX/encY)·서버-서버 전달**로 재정의하고, 연동이력 생성을 **복호화 성공 이후**로 이동해 본 서비스에 편입했다(구 SVC-004 진입 시 생성 폐기). 전달 페이로드는 복호화 원문 X 이며, 추적 키는 X 내부 필드다.

### 사용자 정의

- **발송처(서비스 A 운용사)**: 연동 요청의 원 주체. 전달 데이터 X 를 생성·이중 암호화(encX/encY)하고 값 신뢰성을 책임진다(SEC-002). 허브는 X 내용의 업무 유효성을 판단하지 않는다.
- **수신처(서비스 B 운용사)**: 서버-서버로 전달받은 복호화 원문 X 의 회원 키로 회원 여부를 판단하고 서비스를 제공한다. 완료는 콜백(SVC-009)으로 통지한다.

### 관련 서비스

- **SVC-004 이용 동의**: 승인 완료가 실행의 전제(선행). 승인 후 본 서비스가 복호화·전달을 개시한다.
- **SVC-001 발송처 접근 주소 구성**: 수신처 B 전달 주소·동의 항목의 소스(선행).
- **SVC-006 처리상태 확인 API**: 본 서비스가 저장한 처리 상태를 추적 키 기준 조회 대상으로 함(후행).
- **SVC-007 상태 보관 배치**: 저장된 처리 상태·연동이력의 보관·삭제 대상(후행).
- **SVC-009 완료 콜백 API**: 수신처 B 가 X 에서 얻은 추적 키로 완료를 통지할 대상 연동이력을 본 서비스가 생성한다(후행).
- **SVC-008 연동 완료 확인 API**: 본 서비스가 생성한 연동이력이 완료 판정 근거가 된다(후행).

### 예외 사항

- 복호화 원문 X·생년월일·암호값·복원 키는 요청 처리 컨텍스트 밖으로 넘기지 않고 전달 완료 즉시 참조를 해제한다 — 어떤 테이블·파일·로그에도 남기지 않는다(DATA-001-04·SEC-006-06). 저장은 연동 추적 키(불투명)와 처리상태 4항목·연동이력 상한으로 한정한다.
- 전달 실패(EX-BIZ-004)여도 복호화 성공 이후이므로 연동이력(F-004)은 유지하고 처리 상태 1건을 반드시 남긴다(EXC-BIZ-06·EXC-BIZ-11).
- 회원 키는 복호화 원문 X 의 일부로 수신처 전달 페이로드에만 실리고 저장 모델에서 원천 배제한다. 값 변형·정규화·업무 검증을 수행하지 않는다(SEC-002).

### 구현 가이드

- 복호화 함수의 입력은 생년월일·encX·encY 뿐이며, 키·IV 정규화(32B·IV 16B·`_` 우패딩)와 encY→encX→X 복호화 순서를 결정론적으로 구현한다(SEC-006-02/04, 라이브러리 강제 없음). 출력 X 는 수신처 전달 페이로드로만 사용하고 저장하지 않는다.
- 복호화 성공 후 X 에서 연동 추적 키를 추출해 연동이력을 먼저 생성하고(BIZ-004-07), 전달 결과를 처리상태 4항목에 반영한다(DATA-003). 연동이력 기록(PROC-403)과 처리상태 저장(PROC-401)은 분리된 흐름으로 두되 추적 키로 연결한다(DATA-005-08).
- 수신처 전달은 허브 서버에서 수신처 B 로 직접 서버-서버 POST 로 호출하고, 사용자에게는 전달 결과만 반영한 완료 페이지를 반환한다 — 원문·회원 키를 완료 페이지 DTO·URL 에 싣지 않는다(SEC-007).
- 복호화 실패(EX-SEC-006)와 암호 파라미터 형식 오류(EX-SEC-007)·X 데이터 오류(EX-BIZ-008)는 사용자 재시도 가능 여부가 달라 구분해 처리한다(재입력 가능 vs 발송처 오류 — EXC-BIZ-13).

### 담당자 확정 대기

- 수신처 B 전달 실패 재시도 최대 횟수(2회)는 확정 기본안이다(EXC-BIZ-05).
- 암호 규약 세부(AES-256-CBC·PKCS#7·키 32B/IV 16B 정규화·Base64URL)는 오케스트레이터 spec 확정값이며(SEC-006), encX 전달 형식(GET 파라미터 길이 한계) 시 접근 URL 구성 방식은 화면·build 확정 대상이다(EXC-SEC-08).
- 정적 IV 파생·생년월일 저강도의 전수대입 취약은 알려진 제약으로 초기 범위에서 수용하며(SEC-008), 링크 TTL·1회성·발송처 서명 검증은 후속 보완 항목이다.
