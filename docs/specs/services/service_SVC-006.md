# 처리상태 확인 API 기능 정의

## 개요

- **서비스 목적**: 서비스 A가 요청 키값을 기준으로 연동 요청의 처리·결과 확인 상태를 조회하는 서버 대면 API를 제공한다. 응답에는 처리 상태 4항목만 담고 회원 키·개인정보는 포함하지 않으며, 최초 조회 성공 시 결과 확인 여부를 갱신한다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §수행 범위 "처리상태 확인 API: 서비스 A가 요청 키값 기준으로 처리·결과 확인 상태를 조회" / IA [`../../prd/ia/IA.md`](../../prd/ia/IA.md) `API-01`.

| 우선순위 | 높음 |

### 서비스 기능

| 기능 번호 | 기능 내용 | 입력 (MDL 코드) | 출력 (MDL 코드) | 비고 |
|-----------|-----------|----------------|----------------|------|
| F-001 | API 인증(사전 공유 API 키 또는 서명 헤더 검증) | 인증 자격 | 통과/거부 | SEC-003 |
| F-002 | 요청 제한 검사(분당 60회) | 요청 주체 | 통과/거부 | OPS-001 |
| F-003 | 요청 키값 형식 검증(UUID 형식) | MDL-202 후보(요청 키값) | 검증 결과 | DATA-002-04 |
| F-004 | 요청 키값으로 처리 상태 조회 | MDL-202 후보 | MDL-301 후보(처리 상태) | 미존재 시 404 |
| F-005 | 조회 성공 시 결과 확인 여부·결과 확인 일시 갱신 | 조회 결과 | 갱신된 상태 | DATA-003-03 |
| F-006 | 응답 DTO 변환(상태 4항목만, 마스킹) | 조회 결과 | MDL-302 후보(조회 응답) | SEC-005-02 |

> 조회 응답(MDL-302 후보)에는 회원 키·개인정보 필드를 두지 않는다.

### 사용자 정의

- **서비스 A**: 조회 주체. 진입 응답으로 받은 요청 키값으로 처리·결과 확인 상태를 조회한다. 사전 공유된 인증 수단을 보유한다.

### 관련 서비스

- **SVC-004 이용 동의**: 조회 키(요청 키값)의 발급원(선행).
- **SVC-005 연동 실행**: 조회 대상 처리 상태의 생성원(선행).
- **SVC-007 상태 보관 배치**: 삭제된 요청 키값 조회는 404로 응답(정합).

### 예외 사항

- 형식은 맞으나 미존재(만료 삭제 포함)인 요청 키값 조회는 404 EX-DATA-003으로 응답한다(EXC-DATA-02·EXC-DATA-04).
- 인증 실패 감사 로그에서 요청 키값은 마스킹한다(SEC-003-02·SEC-005-01).

### 구현 가이드

- 조회 응답 DTO에 회원 키 필드를 두지 않고, 마스킹은 응답 DTO 변환·로그 포맷터 계층에서 일괄 적용한다.
- 결과 확인 여부 갱신은 최초 조회 성공 시 1회 수행하도록 멱등하게 설계한다.

### 담당자 확정 대기

- 서비스 대면 API 인증 수단 구체안(API 키/서명 알고리즘, SEC-003)과 요청 제한 임계치(분당 60회, OPS-001)는 확정 기본안이다.
