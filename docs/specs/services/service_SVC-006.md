# 처리상태 확인 API 기능 정의

## 개요

- **서비스 목적**: 발송처가 **연동 추적 키**(전달 데이터 X 내부 필드값)를 기준으로 연동 요청의 처리·결과 확인 상태를 조회하는 서버 대면 API 를 제공한다. 응답에는 처리 상태 4항목만 담고 회원 키·복호화 원문·개인정보는 포함하지 않으며, 최초 조회 성공 시 결과 확인 여부를 갱신한다. 추적 키만으로 신뢰하지 않고 사전 공유 시크릿 인증을 요구한다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §수행 범위 5 "처리상태 확인 API: 발송처가 연동 추적 키 기준으로 처리·결과 확인 상태를 조회" / IA [`../../prd/ia/IA.md`](../../prd/ia/IA.md) `API-01`.

> **2026-07-11 `#214` 개정**: 조회 키를 **허브 발급 요청 키값(UUID)에서 연동 추적 키(X 내부 필드·불투명 문자열·허브 미발급)로 전환**했다. 형식 검증은 UUID 형식이 아니라 비어있지 않음·최대 길이다(DATA-002-07).

### 사용자 정의

- **발송처(서비스 A 운용사)**: 조회 주체. 전달 데이터 X 안에 자신이 넣은 연동 추적 키로 처리·결과 확인 상태를 조회한다. 사전 공유된 발송처 인증 자격을 보유한다.

### 관련 서비스

- **SVC-005 연동 실행**: 복호화 후 추적 키로 저장한 처리 상태의 생성원(선행).
- **SVC-007 상태 보관 배치**: 삭제된 추적 키 조회는 404로 응답(정합).
- **SVC-008 연동 완료 확인 API**: 추적 의미 구별 — API-01의 "처리 성공"은 허브→수신처 B 전달 성공, API-02의 "처리완료"는 수신처 B 의 완료 콜백 수신이다(BIZ-004-11). 두 추적은 연동 추적 키로 연결된다.

### 예외 사항

- 형식은 맞으나 미존재(보관 만료 삭제 포함)인 연동 추적 키 조회는 404 EX-DATA-003으로 응답한다(EXC-DATA-02·EXC-DATA-04).
- 인증 실패·오류 로그에서 연동 추적 키는 앞 2·뒤 2자만 노출하고 마스킹한다(SEC-005-04). 추적 키의 개인정보 비포함 구성은 발송처 책임이며 허브는 값을 해석하지 않는다(DATA-002-06·EXC-DATA-12).
- 동일 연동 추적 키가 재사용된 경우 처리 일시(processed_at) 기준 최신 1건을 반환한다(tiebreaker, EXC-BIZ-12 준용·SVC-008/009 정합).

### 구현 가이드

- 조회 응답 DTO 에 회원 키·복호화 원문 필드를 두지 않고, 마스킹은 응답 DTO 변환·로그 포맷터 계층에서 일괄 적용한다.
- 연동 추적 키는 특정 형식(UUID 등)을 강제하지 않고 비어있음·최대 길이만 검증한다(발송처 구성 자유, DATA-002-07). 결과 확인 여부 갱신은 최초 조회 성공 시 1회 수행하도록 멱등하게 설계한다(DATA-003-06).

### 담당자 확정 대기

- 서비스 대면 API 인증 수단 구체안(API 키/서명 알고리즘, SEC-003)과 요청 제한 임계치(분당 60회, OPS-001)는 확정 기본안이다.
