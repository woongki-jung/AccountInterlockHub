# 이용 동의 기능 정의

## 개요

- **서비스 목적**: 서비스 A에서 진입한 최종 사용자에게 해당 연동 구성의 동의 항목을 노출하고 동의/거부를 처리한다. 진입 시 허브가 불투명 요청 키값을 발급해 처리 추적의 기준으로 삼고, 회원 키는 저장하지 않는다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §수행 범위 "사용자 — 이용 동의 페이지: 구성된 동의 항목 노출, 동의/거부 처리" / IA [`../../prd/ia/IA.md`](../../prd/ia/IA.md) `USR-01`.

---

## SVC-004 이용 동의

### 기본 정보

| 항목 | 내용 |
|------|------|
| 기능명 | 이용 동의(진입·표시·동의/거부) |
| 분류 | 사용자 연동 |
| 주 사용자 | 최종 사용자(서비스 A 회원) / 서비스 A(진입 주체) |
| 우선순위 | 높음 |
| 선행 기능 | SVC-001(연동 구성 등록) |
| 매핑 PROC | PROC-201(진입·표시), PROC-202(동의/거부 처리) |
| 관련 IA 항목 | USR-01 |

### 서비스 기능

| 기능 번호 | 기능 내용 | 입력 (MDL 코드) | 출력 (MDL 코드) | 비고 |
|-----------|-----------|----------------|----------------|------|
| F-001 | 서비스 A 진입 수신(회원 키·전달 파라미터·구성 참조) | MDL-201 후보(진입 요청) | 진입 컨텍스트 | 회원 키 무저장(메모리 경유) |
| F-002 | 요청 키값(불투명 UUID v4) 자체 발급 | 진입 컨텍스트 | MDL-202 후보(요청 키값) | DATA-002, 진입 응답으로 서비스 A 반환 |
| F-003 | 동의 화면 데이터 구성(구성 동의 항목만 노출) | 구성 코드 | 동의 항목 목록 | BIZ-002-01(구성 외 노출 금지) |
| F-004 | 사용자 동의/거부 결과 수신·검증 | MDL-203 후보(동의 결과) | 검증 결과 | 구성 매칭 근거로 검증 |
| F-005 | 거부 시 종료 안내 + 처리 상태 성공 여부=실패(미전달) 기록 | 거부 결과 | 처리 상태 반영 | BIZ-002-03, 오류 아님 |

> 요청 키값은 진입 응답으로 서비스 A에 반환되어 이후 처리상태 조회(SVC-006)의 조회 키가 된다.

### 사용 시나리오

#### Happy Path

| 단계 | 단계명 | 주체 | 행동/처리 | 입력 | 출력 | 매핑 PROC 단계 |
|------|--------|------|----------|------|------|---------------|
| 1 | 진입 | 서비스 A | 회원 키·파라미터로 사용자 진입, 요청 제한·입력 검증 | MDL-201 후보 | 진입 컨텍스트 | PROC-201 · 진입 요청 제한 검사 / 입력 DTO 재검증 |
| 2 | 키 발급 | 시스템 | 요청 키값(UUID v4) 발급 | 진입 컨텍스트 | MDL-202 후보 | PROC-201 · 진입 처리·요청 키값 발급 |
| 3 | 화면 구성 | 시스템 | 연동 구성의 동의 항목만 조회·렌더 | 구성 코드 | 동의 항목 목록 | PROC-201 · 동의 화면 데이터 구성 |
| 4 | 결정 | 사용자 | 동의 또는 거부 선택 | 사용자 입력 | MDL-203 후보 | PROC-202 · 동의 결과 검증·분기 |
| 5 | 결과 검증 | 시스템 | 동의 결과를 구성 매칭 근거로 검증·분기 | MDL-203 후보 | 검증 결과 | PROC-202 · 동의 결과 검증·분기 |
| 6 | 연동 트리거 | 시스템 | 동의 시 연동 실행(SVC-005)으로 전달 | 검증 결과 | 실행 요청 | PROC-202 · 동의 결과 반영 |

#### 분기 흐름

| 코드 | 발생 조건 | 분기 처리 | 결과 |
|------|----------|----------|------|
| BR-201 | 사용자가 동의 / 거부 | 동의는 연동 실행(SVC-005)로 진행, 거부는 전달 없이 종료 | 동의→전달 / 거부→종료(상태=실패·미전달) |

#### 예외 흐름

| 코드 | 발생 조건 | 처리 방식 | 사용자 응답 |
|------|----------|----------|-------------|
| EX-OPS-001 | 진입 요청 분당 60회 초과 | 요청 거부, 감사 로그 | 429 잠시 후 다시 시도해주세요. |
| EX-SEC-004 | 진입 파라미터 형식 위반·유효하지 않은 구성 참조 | 진입 거부 | 400 요청이 올바르지 않습니다. |
| EX-SEC-005 | 요청 본문 1MB 초과 | 요청 거부 | 413 요청이 너무 큽니다. |

> 사용자 거부는 오류(EX)가 아니라 200 정상 종료로 처리한다(EXC-BIZ-03). 처리 상태에는 성공 여부=실패(미전달)로 1건 반영한다(EXC-DATA-03).

### 사용자 정의

- **최종 사용자(서비스 A 회원)**: 동의 주체. 자신에게 노출된 동의 항목을 확인하고 동의/거부를 스스로 결정한다.
- **서비스 A**: 진입 주체. 회원 키·전달 파라미터를 넘기고 응답으로 요청 키값을 수신한다.

### 관련 서비스

- **SVC-001 연동 구성 등록·편집**: 동의 항목·구성 정보의 소스(선행).
- **SVC-005 연동 실행**: 동의 완료 시 전달 트리거(후행).
- **SVC-006 처리상태 확인 API**: 발급된 요청 키값을 조회 키로 사용(후행).

### 예외 사항

- 회원 키는 진입~전달 목적의 메모리 처리로만 경유하고 어떤 저장소에도 남기지 않는다(DATA-001-01).
- 동의 화면은 해당 구성에 설정된 동의 항목만 노출하며 구성 외 항목 노출을 금지한다(BIZ-002-01).

### 구현 가이드

- 요청 키값은 표준 UUID v4로 발급하고, 예측 가능한 시퀀스·타임스탬프 노출식 식별자를 쓰지 않는다.
- 동의 결과는 서버가 구성 매칭 근거와 함께 검증한다(화면 값 단독 신뢰 금지).

### 담당자 확정 대기

- Q2: 요청 키값을 허브 발급 불투명 UUID로 두는 안(DATA-002)은 확정 기본안이다.
- Q3: 개인식별 동의 증빙 원장은 MVP 제외, 동의/거부 결과만 처리 상태에 반영하는 안(BIZ-002-04)은 확정 기본안이다.
