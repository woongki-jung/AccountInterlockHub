# 인증/인가 정책 정의 (AUTH)

## 개요

- **정책 목적**: 관리자 경로의 로그인 인증·세션·계정 잠금 규칙을 정의하여 IP 접근 제한(SEC-001)과 함께 관리자 화면을 이중으로 보호하고, 사용자 접근 페이지의 **생년월일 본인확인**이 인증·인가 수단이 아니라 복호화 요소임을 규정해 그 처리 방향을 확정한다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §미결·확인 필요 "IP 제한 외 로그인 인증(계정·비밀번호 등)을 둘지는 spec 단계에서 확정" · §서비스 이용 대상 3(사용자 — 생년월일 입력) / [`../../prd/devspec/infra.md`](../../prd/devspec/infra.md) §접근 제어.
- **담당자 확정 대기 (Q1)**: 로그인 인증 도입 여부는 담당자 확정 대기다. 본 문서는 "IP 제한 + 로그인 인증 병행"을 확정 기본안으로 정의한다. 계정 발급 절차·비밀번호 복잡도·세션 유휴시간·잠금 임계치의 수치는 기본안이며 확정 시 리비전한다.
- **생년월일 본인확인 성격 (spec 확정)**: 사용자 생년월일은 연동 실행의 **복호화 요소**이며 인증·세션을 발급하지 않는다(AUTH-004). 암호 규약·미기록은 SEC-006, 실패 시 재입력 흐름은 BIZ-003 이 담당한다.

### 기본 정보

| 항목 | 내용 |
|------|------|
| 정책명 | 관리자 로그인 인증 |
| 분류 | AUTH |
| 적용 범위 | 관리자 웹·API 경로(연동 구성 등록·편집·조회) 접근 |
| 우선순위 | 높음 |
| 관련 정책 | SEC-001(IP 제한), AUTH-002, AUTH-003, OPS-002 |
| 적용 PROC | PROC-103 |
| 관련 IA 항목 | ADM-03 |

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |
|---------|-----------|-----------|-----------|----------------------------|
| AUTH-001-01 | 관리자 기능은 계정 ID·비밀번호 검증을 통과한 세션에서만 수행한다. 미인증 요청은 거부한다. | 모든 관리자 경로 요청 | validate | 401 EX-AUTH-001 |
| AUTH-001-02 | 관리자 비밀번호는 8자 이상, 영문 대문자·소문자·숫자·특수문자를 각 1자 이상 포함한다. | 비밀번호 설정·변경 시 | validate | 422 EX-AUTH-004 |
| AUTH-001-03 | 비밀번호는 평문 저장·평문 로그를 금지하고 단방향 해시(솔트 포함)로 저장한다. | 자격 저장 시 | transform | 감사 로그 기록 + 내부 차단(사용자 미노출) |

### 예외 사항

- **EXC-AUTH-01**: dev/local 환경에서 IP 제한이 비활성일 때에도 로그인 인증은 유지한다(우회 금지).
- 계정 발급·초기 비밀번호 전달 절차는 MVP 범위에서 운영 수동 절차로 두며 별도 화면을 강제하지 않는다(담당자 확정 대기).

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 (PROC 의사코드 단계명) | 적용 위치 |
|---------|----------|-------------------------------|-----------|
| AUTH-001-01 | PROC-103 | 진입 인증 가드 | 검증 |
| AUTH-001-02 | PROC-103 | 비밀번호 설정·변경 검증 | 검증 |
| AUTH-001-03 | PROC-103 | 자격 영속화 | 가공 |

### 구현 가이드

- 인증은 세션 기반으로 처리하고, 자격 저장은 검증된 단방향 해시 알고리즘을 사용한다(특정 라이브러리 강제하지 않음).
- 관리자 경로 진입 가드는 SEC-001(IP 확인) 이후 AUTH-001(인증 확인) 순으로 배치한다.

| AUTH-002-02 | 세션 식별자는 추측 불가한 난수로 발급하고 HttpOnly·Secure 속성으로 전달한다. | 세션 생성 시 | transform | 내부 차단(사용자 미노출) |

### 예외 사항

- **EXC-AUTH-02**: 세션 만료(EX-AUTH-002)는 오류가 아닌 정상 재인증 유도다. 화면은 로그인 페이지로 안내한다.

| 규칙 ID | 적용 PROC | 적용 단계 | 적용 위치 |
|---------|----------|-----------|-----------|
| AUTH-002-01 | PROC-103 | 세션 유효성 검사 | 검증 |
| AUTH-002-02 | PROC-103 | 세션 발급 | 가공 |
| AUTH-002-03 | PROC-103 | 커밋 후 감사 로그 | 감사 |

### 구현 가이드

- 세션 저장소는 단일 App Service 구성에 맞춰 애플리케이션 세션으로 두되, 스케일아웃 시 공유 저장소 전환 여지를 남긴다(구체안은 build 단계).

## AUTH-004 사용자 생년월일 본인확인 (비인증 복호화 요소)

### 기본 정보

| 항목 | 내용 |
|------|------|
| 정책명 | 사용자 생년월일 본인확인 (비인증 복호화 요소) |
| 분류 | AUTH |
| 적용 범위 | 사용자 접근 페이지의 생년월일 입력·연동 실행 복호화 진입 |
| 우선순위 | 높음 |
| 관련 정책 | SEC-006(암호 규약), BIZ-003(복호화·상태 전이), DATA-001(미저장), OPS-002(감사) |
| 적용 PROC | PROC-201, PROC-203 |
| 관련 IA 항목 | USR-01, USR-02 |

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |
|---------|-----------|-----------|-----------|----------------------------|
| AUTH-004-01 | 사용자 생년월일(yyMMdd)은 연동 실행의 복호화 입력값이며 인증·인가·세션 발급 수단이 아니다. 생년월일로 관리자·서비스 권한을 부여하지 않고, 검증 결과는 복호화 성공/실패로만 귀결한다(SEC-006). | 사용자 접근·연동 실행 시 | validate | (성격 규정 — 위반 EX 없음, 오용 시 감사 로그) |
| AUTH-004-02 | 생년월일 불일치로 복호화가 실패하면 계정·링크를 잠그지 않고 재입력 재시도를 허용한다(MVP 하드 잠금 없음). 복호화 실패 자체는 SEC-006 이 400 EX-SEC-006 으로 처리하고, 사용자는 생년월일 재입력으로 재시도한다. | 복호화 실패 시 | validate | 400 EX-SEC-006(재입력 유도) |
| AUTH-004-03 | 생년월일 입력·복호화 시도는 성공·실패 여부만 감사 로그에 남기고, 생년월일 원문·암호값·복호화 원문은 로그·저장 어디에도 기록하지 않는다(SEC-005-06·DATA-001). | 복호화 시도 시 | audit | 감사 로그 기록(원문 미포함) |

### 예외 사항

- **EXC-AUTH-04**: MVP 하드 잠금 없음(AUTH-004-02)은 알려진 보안 제약(SEC-008 — 생년월일 저강도)과 함께 수용된 초기 범위 결정이다. 오프라인 전수대입 방어를 위한 링크 TTL·1회성·시도 임계 잠금은 후속 보완 항목으로 SEC-008 이 관장한다.
- **EXC-AUTH-05**: 반복 재시도의 남용 억제는 요청 제한(OPS-001, 사용자 진입 분당 상한)으로 1차 방어하며, 이는 계정 잠금과 구별된다(요청 빈도 제한이지 본인확인 잠금이 아님).

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 (PROC 의사코드 단계명) | 적용 위치 |
|---------|----------|-------------------------------|-----------|
| AUTH-004-01 | PROC-201, PROC-203 | 생년월일 입력 수신·복호화 진입 | 검증 |
| AUTH-004-02 | PROC-203 | 복호화 실패 분기(재입력 유도) | 검증 |
| AUTH-004-03 | PROC-203 | 복호화 시도 후 감사 로그 | 감사 |

### 구현 가이드

- 생년월일은 복호화 함수의 입력값으로만 전달하고 세션·토큰·권한 컨텍스트로 승격하지 않는다. 재시도는 동일 접근 주소·암호값(encX·encY)에 대해 생년월일만 다시 받아 복호화를 재수행하는 형태로 구성한다(암호값 재수신 불요).
- 감사 로그에는 접근 주소 고유 ID·시각·복호화 성공 여부만 남기고 생년월일·암호값·복호화 원문은 제외한다(SEC-005-06). 세부 재입력 화면 흐름은 화면 정의서가, 복호화 규약은 SEC-006 이 확정한다.
