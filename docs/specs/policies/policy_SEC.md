# 보안 정책 정의 (SEC)

## 개요

- **정책 목적**: 관리자 경로 IP 접근 제한, 회원 키 신뢰 위임, 서비스 대면 API 인증, 공통 입력 검증·주입 방어, 민감값 마스킹을 확정한다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §시스템 제약사항(관리자 IP 제한·신뢰 위임) / [`../../prd/devspec/infra.md`](../../prd/devspec/infra.md) §접근 제어 / [`../../prd/devspec/external-apis.md`](../../prd/devspec/external-apis.md) 접점 신뢰 위임.
- **담당자 확정 대기**: 관리자 IP 제한과 로그인 인증 병행(SEC-001·Q1), 서비스 대면 API 인증 수단 구체안(SEC-003)은 확정 기본안이다.

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |

| SEC-001-02 | 허용 IP 목록은 운영 구성값으로 관리한다(코드 하드코딩 금지). 목록 변경은 감사 로그에 기록한다. | IP 목록 변경 시 | audit | 감사 로그 기록 |

### 예외 사항

- **EXC-SEC-01**: dev/local 환경은 IP 제한을 비활성할 수 있으나(운영 구성), 로그인 인증(AUTH-001)은 유지한다.

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 | 적용 위치 |
|---------|----------|-----------|-----------|
| SEC-001-01 | PROC-104 | 진입 IP 가드 | 검증 |
| SEC-001-02 | PROC-104 | 구성 변경 후 감사 로그 | 감사 |
| SEC-001-03 | PROC-104 | 차단 후 감사 로그 | 감사 |

### 구현 가이드

- IP 가드는 관리자 라우트 진입점(NestJS 미들웨어/가드 계층)에 두고 AUTH-001 인증 가드보다 앞단에 배치한다. 프록시 환경의 원 출발지 IP 판별 방식은 build 단계 확정.

| 관련 정책 | DATA-001, BIZ-003 |

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |
|---------|-----------|-----------|-----------|----------------------------|
| SEC-002-01 | 허브는 회원 키 값의 신뢰성(암·복호화·해시·위변조)을 판단·검증하지 않고 그대로 서비스 B 로 중개한다. 신뢰성 책임은 서비스 A 에 있다. | 회원 키 전달 시 | transform | 감사 로그 기록 |
| SEC-002-02 | 회원 키에 대해 값 변형·정규화·복호화를 수행하지 않는다(전달 목적 최소 처리만). | 회원 키 처리 시 | transform | 감사 로그 기록 |

### 예외 사항

- **EXC-SEC-02**: 신뢰 위임은 SEC-004 의 형식·크기·주입 방어(전송 안전성) 검증을 면제하지 않는다. 값의 의미적 신뢰성만 위임하며, 전송 계층 검증은 수행한다.

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 | 적용 위치 |
|---------|----------|-----------|-----------|
| SEC-002-01 | PROC-203 | 서비스 B 전달 페이로드 구성 | 가공 |
| SEC-002-02 | PROC-203 | 전달 페이로드 구성 | 가공 |

### 구현 가이드

- 회원 키는 원본 그대로 전달 페이로드에 실어 보내고 저장하지 않는다(DATA-001). 값 해석 로직을 두지 않는다.

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |
|---------|-----------|-----------|-----------|----------------------------|
| SEC-003-01 | 서비스 대면 API 는 사전 공유된 인증 수단(API 키 또는 서명 헤더)을 검증해 통과한 요청에만 응답한다. | API 진입 시 | validate | 401 EX-SEC-003 |
| SEC-003-02 | 인증 실패·서명 불일치를 감사 로그에 기록한다(요청 키값은 마스킹). | 인증 실패 시 | audit | 감사 로그 기록 |

### 예외 사항

- **EXC-SEC-03**: 인증 수단 구체안(API 키/서명 알고리즘)은 담당자 확정 대기다. 확정 시 본 정책을 리비전한다.

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 | 적용 위치 |
|---------|----------|-----------|-----------|
| SEC-003-01 | PROC-301 | API 진입 인증 검증 | 검증 |
| SEC-003-02 | PROC-301 | 인증 실패 감사 로그 | 감사 |

### 구현 가이드

- 인증 수단은 전송 계층 보안(HTTPS) 위에 두고, 자격 값은 로그에 남기지 않는다.

|---------|-----------|-----------|-----------|----------------------------|

| SEC-004-02 | SQL·스크립트 주입을 방지한다 — 파라미터 바인딩·출력 이스케이프를 적용하고 원시 쿼리 문자열 결합을 금지한다. | DB 접근·출력 생성 시 | transform | 400 EX-SEC-004 |

### 예외 사항

- **EXC-SEC-04**: 본문 크기 상한(1MB)은 기본안이며 담당자 확정 대기다.

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 | 적용 위치 |
|---------|----------|-----------|-----------|
| SEC-004-01 | PROC-101, PROC-201, PROC-301 | 입력 DTO 재검증 | 검증 |
| SEC-004-02 | PROC-101 | DB 접근·출력 생성 | 가공 |
| SEC-004-03 | PROC-201, PROC-301 | 요청 수신 | 검증 |

### 구현 가이드

- 입력 검증은 DTO 스키마 검증 계층에서 일괄 처리하고, DB 접근은 파라미터 바인딩만 사용한다.

## SEC-005 민감값 마스킹

### 기본 정보

|------|------|
| 정책명 | 민감값 마스킹 |
| 분류 | SEC |
| 적용 범위 | 로그·감사·오류 응답·조회 응답 |
| 우선순위 | 높음 |
| 관련 정책 | DATA-001, DATA-003, OPS-002 |
| 적용 PROC | PROC-102, PROC-301 |
| 관련 IA 항목 | 공통 |

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |
|---------|-----------|-----------|-----------|----------------------------|
| SEC-005-01 | 로그·감사·오류 응답에 회원 키·인증 자격을 남길 경우 앞 2·뒤 2자만 노출하고 나머지를 마스킹한다. | 로그·응답 생성 시 | mask | 감사 로그 기록 |
| SEC-005-02 | 처리상태 조회 응답에는 상태 4항목만 포함하고 회원 키·개인정보를 포함하지 않는다. | 조회 응답 생성 시 | mask | 감사 로그 기록 + 내부 차단 |

### 예외 사항

- **EXC-SEC-05**: 관리자 연동 구성 조회 응답의 서비스 A/B 주소는 마스킹 대상이 아니다(설정 데이터, 개인정보 아님).

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 | 적용 위치 |
|---------|----------|-----------|-----------|
| SEC-005-01 | PROC-301 | 응답·로그 생성 | 마스킹 |
| SEC-005-02 | PROC-102, PROC-301 | 응답 DTO 변환 | 마스킹 |

### 구현 가이드

- 마스킹은 응답 DTO 변환·로그 포맷터 계층에서 일괄 적용한다. 조회 응답 DTO 에 회원 키 필드를 두지 않는다.
