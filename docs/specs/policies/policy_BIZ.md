# 비즈니스 규칙 정책 정의 (BIZ)

## 개요

- **정책 목적**: 발송처 접근 주소 구성 등록의 입력 검증·고유성, 사용자 동의 처리, 승인 후 허브 복호화·수신처(B) 전달과 연동 실행의 상태 전이, 연동이력의 기록·완료 판정을 확정한다. 관리자 구성이 사용자 페이지·연동 실행의 전제이므로 BIZ-001 을 최우선에 둔다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §수행 범위(발송처 접근 주소 구성·사용자 접근·동의·승인·연동 실행 복호화·전달·완료 확인·완료 콜백·연동이력) / [`../../prd/devspec/external-apis.md`](../../prd/devspec/external-apis.md) 발송처→허브→수신처 접점·완료 확인(§4)·완료 콜백(§5).
- **spec 확정 기본값**: 승인 게이팅(필수 연동 동의 미체크 시 승인 차단·약관 동의 선택), 실패·거부 UX(복호화 실패 재입력·전달 실패 재시도·거부 최소 기록), 전달 재시도 2회는 오케스트레이터 spec 확정값이다([`spec-policies.md`](spec-policies.md) §spec 확정 기본값).
- **담당자 확정 대기 (Q3)**: 개인식별 동의 증빙 원장은 MVP 제외, 동의/거부 결과만 반영하는 안(BIZ-002-04)은 확정 기본안이다.
- **담당자 확정 대기 (약관 컨텐츠)**: 동의 항목별 약관 본문('약관 컨텐츠')을 '선택' 입력으로 두는 안(BIZ-001-06)은 기본안이다 — 약관 미설정 항목은 사용자 동의 화면 [상세] 버튼을 노출하지 않는다(BIZ-002-05). '필수' 전환은 담당자 확정 대기(EXC-BIZ-07).

### 예외 사항

- **EXC-BIZ-02**: 편집 시 자기 자신과의 고유 ID 충돌은 고유성 위반으로 보지 않는다.
- **EXC-BIZ-07 (담당자 확인)**: 약관 컨텐츠(BIZ-001-06)를 '선택'으로 두는 것은 기본안이다 — 약관 미설정 항목은 사용자 동의 화면에서 [상세] 버튼을 노출하지 않는다(BIZ-002-05). '필수'로 전환할지는 담당자 확정 대기다. 약관 컨텐츠는 항목 정의 메타(설정 데이터)로 무저장 원칙과 무관하다.
- **EXC-BIZ-14**: 입력이 단일 암호화 JSON(encX·encY)이므로 접근 주소 구성에는 전달 파라미터 정의를 두지 않는다 — 수신처에 넘길 회원 키와 연동 추적 키는 발송처가 전달 데이터 X 안에 담아 전달하며(SEC-006), 허브는 복호화 후 추적 키만 추출·저장한다(DATA-002).

### 구현 가이드

- URL·필수값 검증은 서버단에서 재수행한다(화면 검증에 의존하지 않음). 고유 ID 고유성은 저장 직전 조회로 확인한다. 접근 주소 고유 ID 는 관리자가 직접 입력하며 자동 생성하지 않는다 — 중복 값이면 등록을 실패시킨다(BIZ-001-10).
- 접근 주소 고유 ID 는 발송처 식별자이자 사용자 진입 경로의 일부다 — 접근 URL 은 `허브 접근 주소(고유 ID) + encX·encY 파라미터`로 구성되며, 발송처키·암호값은 구성에 저장하지 않는다(DATA-001·SEC-002).
- 약관 컨텐츠(BIZ-001-06)는 선택 입력이라 크기 상한(1MB)만 적용하고, 동의 항목의 본문 속성으로 부모 구성과 함께 영속화한다.
- **동의 대상 설명 문구(consent_notice)**는 접근 주소 구성 단위의 자유 텍스트 선택 입력이다 — 미입력(NULL)을 허용하고 크기 상한(1MB) 내로만 제한하며, 부모 구성과 함께 영속화한다. 사용자 동의 화면 상단 노출은 BIZ-002-08 이 담당한다.

### 예외 사항

- **EXC-BIZ-03**: 동의/거부는 오류가 아닌 정상 업무 결과다. 거부는 EX 코드가 아니라 취소·종료 안내로 처리한다.
- **EXC-BIZ-04 (Q3)**: 동의 증빙 원장 미저장은 MVP 확정 기본안이며 담당자 확정 대기다. 증빙 요건이 확정되면 DATA 정책과 함께 리비전한다.
- **EXC-BIZ-08**: 약관 상세 모달의 [동의] 버튼은 해당 항목을 체크(동의) 처리하고, [닫기] 버튼은 모달만 닫고 체크 상태를 바꾸지 않는다. 모달 상호작용은 동의/거부 제출(PROC-202) 전의 클라이언트 상태 조작이며 별도 서버 호출을 발생시키지 않는다.

### 구현 가이드

- 동의 항목 목록은 접근 주소 구성에서 조회해 렌더링하며, 서버는 승인 시 필수 동의 충족 여부를 구성 매칭으로 재검증한다(화면 검증에 의존하지 않음).
- 약관 컨텐츠는 동의 항목 조회 응답(PROC-201)에 포함해 내려보내고, [상세] 버튼·약관 모달은 화면이 클라이언트에서 표시한다. 약관 미설정 항목은 [상세] 버튼을 렌더하지 않는다.
- 동의 대상 설명 문구(consent_notice)는 동의 항목 조회 응답(PROC-201)에 구성 단위 값으로 함께 내려보내고, 화면 상단(제목 아래)에 노출한다(BIZ-002-08). 미설정 시 해당 영역을 렌더하지 않는다.

### 예외 사항

- **EXC-BIZ-05**: 재시도 최대 횟수(2회)는 spec 확정 기본값이며 담당자 조정 가능하다.
- **EXC-BIZ-06**: 복호화 성공 이후의 전달 실패(EX-BIZ-004)는 처리 상태 1건을 반드시 남긴다(DATA-003-05 정합). 복호화 이전 거부·복호화 실패는 추적 키가 없어 키 기반 처리상태를 남기지 않고 감사 로그(결과 코드)로만 기록한다(EXC-DATA-03).
- **EXC-BIZ-13**: 복호화 실패(생년월일 불일치)와 X 데이터 오류(추적 키 필드 누락)는 구별한다 — 전자는 사용자 정정 가능(생년월일 재입력, 400 EX-SEC-006), 후자는 발송처 오류로 재입력 불가(400 EX-BIZ-008). 복호화 원문 X 의 수신처 전달은 서버-서버로만 수행한다(SEC-007 — 브라우저 미경유).

- **EXC-BIZ-11**: 복호화 성공 후 전달 실패 건의 연동이력도 생성(BIZ-004-07)을 유지한다 — 이력을 삭제하지 않고 완료 콜백 미수신 상태로 남겨 DATA-006-05 기산(연동 요청 일시)으로 정리한다. 복호화 이전 거부·복호화 실패는 추적 키가 없어 연동이력을 생성하지 않는다.
- **EXC-BIZ-12**: 완료 판정 식별 단위를 연동 추적 키 단독으로 두는 것(BIZ-004-09·10)은 확정 기본안이다 — 추적 키는 발송처·수신처가 공유하는 요청별 조회 기준이다. 발송처가 추적 키를 요청별 고유로 구성하면 대상이 유일하며, 동일 추적 키 재사용에 대비해 연동 요청 일시 최신 1건을 tiebreaker 로 둔다.
