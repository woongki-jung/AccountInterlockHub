# 비즈니스 규칙 정책 정의 (BIZ)

## 개요

- **정책 목적**: 연동 구성 등록의 입력 검증·고유성, 사용자 동의 처리, 서비스 B 전달과 연동 실행의 상태 전이, 연동이력의 기록·완료 판정을 확정한다. 관리자 구성이 사용자 페이지·연동 실행의 전제이므로 BIZ-001 을 최우선에 둔다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §수행 범위(연동 구성 관리·이용 동의·연동 실행·완료 확인·완료 콜백·연동이력) / [`../../prd/devspec/external-apis.md`](../../prd/devspec/external-apis.md) 서비스 A→허브→서비스 B 접점·완료 확인(§4)·완료 콜백(§5).
- **담당자 확정 대기 (Q3)**: 개인식별 동의 증빙 원장은 MVP 제외, 동의/거부 결과만 처리 상태에 반영하는 안(BIZ-002)은 확정 기본안이다.
- **담당자 확정 대기 (약관 컨텐츠)**: 동의 항목별 전체 약관 본문('약관 컨텐츠')을 '선택' 입력으로 두는 안(BIZ-001-06)은 기본안이다 — 약관 미설정 항목은 사용자 동의 화면 [상세] 버튼을 노출하지 않는다(BIZ-002-05). '필수' 전환·항목당 약관 1건 가정은 담당자 확정 대기(EXC-BIZ-07).
- **담당자 확정 (사용자 키값 필수화 — `accountinterlockhub#33`, 2026-07-08)**: 사용자 키값 파라미터 지정은 **필수**다 — 연동 구성은 '사용자 키값 속성(isUserKey)'을 가진 전달 파라미터를 **정확히 1개**(exactly-one) 반드시 포함해야 하며, 지정이 0개(미지정)이거나 2개 이상(다중)이면 등록·편집 저장을 차단한다(BIZ-001-07·EXC-BIZ-09). 이 확정으로 '미지정 구성'은 정상 경로에서 생성되지 않으므로, BIZ-004-05 등 '미지정' 전제 규칙·분기는 레거시·비정상 데이터에 대비한 **방어적 잔존**으로만 유지한다. 완료 판정 스코프(연동 구성 식별자 + 사용자 키값 복합, 최신 건 — BIZ-004-03·04)는 확정 기본안이다.

### 예외 사항

- **EXC-BIZ-01**: BIZ-001-05 는 무저장 원칙(DATA-001) 상 지양 대상이나, devspec 데이터 원칙에 따라 유효성 차단은 하지 않고 경고만 남긴다.
- **EXC-BIZ-02**: 편집 시 자기 자신과의 식별자 충돌은 고유성 위반으로 보지 않는다.
- **EXC-BIZ-07 (담당자 확인)**: 약관 컨텐츠(BIZ-001-06)를 '선택'으로 두는 것은 기본안이다 — 약관 미설정 항목은 사용자 동의 화면에서 [상세] 버튼을 노출하지 않는다(BIZ-002-05). '필수'로 전환할지는 담당자 확정 대기다. 약관 컨텐츠는 항목 정의 메타(설정 데이터)로 무저장 원칙과 무관하다.
- **EXC-BIZ-09 (담당자 확정 — `accountinterlockhub#33`, 2026-07-08)**: 사용자 키값 파라미터 지정(BIZ-001-07)은 **필수(exactly-one)**로 확정됐다(선택 입력 잔재 제거). 등록·편집 저장은 지정이 정확히 1개일 때만 허용하며, 0개·2개 이상은 422 EX-BIZ-001 로 차단한다. 이 확정으로 '미지정 구성'은 정상 경로로 생성되지 않으므로, 미지정 전제의 BIZ-004-05·완료 확인/콜백 대상 제외 분기는 방어적 잔존으로만 유지한다(BLK-13 해소).

### 구현 가이드

- URL·필수값 검증은 서버단에서 재수행한다(화면 검증에 의존하지 않음). 고유성은 저장 직전 조회로 확인한다.
- 약관 컨텐츠(BIZ-001-06)는 선택 입력이라 필수·형식 차단 없이 크기 상한(1MB)만 적용하고, 동의 항목(ENT-002)의 본문 속성으로 부모 구성과 함께 영속화한다.
- 사용자 키값 파라미터 지정(BIZ-001-07)은 전달 파라미터 정의에 대한 참조로 저장한다. 지정이 필수(정확히 1개)이므로 정상 등록된 구성은 항상 지정을 가지며 연동이력 기록 대상이 된다(BIZ-004) — '미지정' 분기는 방어적 잔존이다.

### 예외 사항

- **EXC-BIZ-03**: 동의/거부는 오류가 아닌 정상 업무 결과다. 거부는 EX 코드가 아니라 처리 상태(실패)와 종료 안내로 처리한다.
- **EXC-BIZ-04 (Q3)**: 동의 증빙 원장 미저장은 MVP 확정 기본안이며 담당자 확정 대기다. 증빙 요건이 확정되면 DATA 정책과 함께 리비전한다.
- **EXC-BIZ-08**: 약관 상세 모달의 [동의] 버튼은 해당 동의 항목을 체크(동의) 처리하며, [닫기] 버튼은 모달만 닫고 항목 체크 상태를 바꾸지 않는다. 모달 상호작용 자체는 동의/거부 제출(PROC-202) 전의 클라이언트 상태 조작이며 별도 서버 호출을 발생시키지 않는다.

### 예외 사항

- **EXC-BIZ-10**: 스코프 내 미수신 이력이 없고 이미 완료된 이력만 있는 재통지 콜백은 오류가 아니다 — 상태 변경 없이 성공 응답(멱등)하고 감사 로그를 남긴다.
- **EXC-BIZ-11**: 동의 거부·전달 실패 건의 연동이력도 생성 시점(진입) 기록을 유지한다 — 이력을 삭제하지 않고 완료 콜백 미수신 상태로 남겨 DATA-006-02 기산(연동 요청 일시)으로 정리한다.
- **EXC-BIZ-12 (담당자 확인)**: 완료 판정의 식별 단위를 {연동 구성 식별자 + 사용자 키값} 복합 스코프·최신 건으로 두는 것(BIZ-004-03·04)은 확정 기본안이다 — 동일 사용자 키값이 복수 구성·복수 요청에 걸칠 수 있어 사용자 키값 단독으로는 판정 대상이 유일하지 않기 때문이다. 담당자 이견 시 리비전한다.

### 구현 가이드

- 완료 콜백 특정(BIZ-004-03)과 완료 판정(BIZ-004-04)은 동일 스코프 정의({연동 구성 식별자 + 사용자 키값})를 공유한다.
- 연동이력 기록은 처리상태 저장(PROC-401)과 분리된 저장 흐름(PROC-403)으로 두되 요청 키값으로 연결한다(DATA-005-04).
- 콜백·완료 확인 요청이 구성 식별자를 얻는 방식(서비스 B 전달 페이로드에 포함해 회신 등) 상세는 서비스 정의·PROC 에서 확정한다.
