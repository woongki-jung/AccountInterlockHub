# 운영 정책 정의 (OPS)

## 개요

- **정책 목적**: 서비스 대면 API·사용자 진입의 요청 제한, 감사 로그 대상·항목·보존, 보관 배치(처리상태·연동이력)의 실행 규칙(스케줄·멱등·실패 처리)을 확정한다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §수행 범위(상태 보관·정리) / [`../../prd/devspec/database.md`](../../prd/devspec/database.md) §보관 정책(일 배치) / [`../../prd/devspec/infra.md`](../../prd/devspec/infra.md) 로깅·모니터링(build 상세).
- **담당자 확정 대기**: 요청 제한 임계치(OPS-001)와 감사 로그 보존 기간(OPS-002)은 확정 기본안 수치다.

## OPS-001 요청 제한 (Rate Limiting)

### 기본 정보

| 항목 | 내용 |
|------|------|
| 정책명 | 요청 제한 (Rate Limiting) |
| 분류 | OPS |
| 적용 범위 | 서비스 대면 API(API-01·API-02·API-03), 사용자 진입(USR-01) |
| 우선순위 | 보통 |
| 관련 정책 | SEC-003, SEC-001 |
| 적용 PROC | PROC-201, PROC-301, PROC-302, PROC-303 |
| 관련 IA 항목 | API-01, API-02, API-03, USR-01 |

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |
|---------|-----------|-----------|-----------|----------------------------|
| OPS-001-01 | 서비스 대면 API·사용자 진입은 출발지(또는 인증 주체) 기준 분당 60회를 초과하는 요청을 거부한다. | 요청 진입 시 | validate | 429 EX-OPS-001 |
| OPS-001-02 | 요청 제한 초과 이벤트를 감사 로그에 기록한다. | 제한 초과 시 | audit | 감사 로그 기록 |

### 예외 사항

- **EXC-OPS-01**: 분당 60회 임계치는 기본안이며 담당자 확정 대기다. 관리자 경로는 IP 제한(SEC-001)으로 1차 보호되어 본 제한의 우선 대상이 아니다.

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 | 적용 위치 |
|---------|----------|-----------|-----------|
| OPS-001-01 | PROC-201, PROC-301, PROC-302, PROC-303 | 진입 요청 제한 검사 | 검증 |
| OPS-001-02 | PROC-201, PROC-301, PROC-302, PROC-303 | 제한 초과 감사 로그 | 감사 |

### 구현 가이드

- 요청 제한 카운터는 진입점 미들웨어에서 처리하고, 단일 App Service 기준으로 구성하되 스케일아웃 시 공유 카운터 전환 여지를 둔다(build 상세).

## OPS-002 감사 로그

### 기본 정보

| 항목 | 내용 |
|------|------|
| 정책명 | 감사 로그 |
| 분류 | OPS |
| 적용 범위 | 관리자 인증·연동 구성 변경·IP 차단·API 인증·연동 실행·완료 콜백·배치 |
| 우선순위 | 높음 |
| 관련 정책 | AUTH-003, SEC-001, SEC-003, BIZ-003, BIZ-004, DATA-001, SEC-005 |
| 적용 PROC | PROC-101, PROC-103, PROC-104, PROC-105, PROC-106, PROC-203, PROC-301, PROC-302, PROC-303, PROC-402 |
| 관련 IA 항목 | 공통 |

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |
|---------|-----------|-----------|-----------|----------------------------|
| OPS-002-01 | 감사 로그 기록 대상 — 관리자 로그인/로그아웃/실패/잠금, 연동 구성 등록·수정·삭제, IP 차단, API 인증 실패, 연동 전달 실패, 완료 콜백 수신(대상 미특정·멱등 재통지 포함), 배치 실행. | 해당 이벤트 발생 시 | audit | 감사 로그 기록 |
| OPS-002-02 | 감사 로그 항목은 시각·행위자(관리자 ID 또는 서비스 식별)·행위·대상·결과로 구성하고, 회원 키·개인정보는 포함하지 않는다(SEC-005 마스킹). | 감사 기록 생성 시 | audit | 감사 로그 기록 |
| OPS-002-03 | 감사 로그는 최소 1년간 보존한다. | 로그 보존 관리 시 | audit | 내부 처리(사용자 미노출) |

### 예외 사항

- **EXC-OPS-02**: 감사 로그 보존 기간(1년)은 기본안이며 담당자 확정 대기다. 감사 로그는 처리 상태(DATA-004)와 달리 무저장 원칙 대상이 아니며 개인정보를 포함하지 않는다.

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 | 적용 위치 |
|---------|----------|-----------|-----------|
| OPS-002-01 | PROC-101, PROC-103, PROC-104, PROC-105, PROC-106, PROC-203, PROC-301, PROC-302, PROC-303, PROC-402 | 커밋 후 감사 로그 | 감사 |
| OPS-002-02 | PROC-101, PROC-103, PROC-105, PROC-106, PROC-203, PROC-301, PROC-302, PROC-303 | 감사 로그 생성 | 감사 |
| OPS-002-03 | PROC-402 | 로그 보존 관리 | 감사 |

### 구현 가이드

- 감사 로그는 애플리케이션 로그와 분리 가능한 채널로 남기고, 기록 전 SEC-005 마스킹을 적용한다.

## OPS-003 보관 배치 실행

### 기본 정보

| 항목 | 내용 |
|------|------|
| 정책명 | 보관 배치 실행 |
| 분류 | OPS |
| 적용 범위 | 처리 상태·연동이력 보관·삭제 배치의 실행·멱등·실패 처리 |
| 우선순위 | 보통 |
| 관련 정책 | DATA-004, DATA-006, OPS-002 |
| 적용 PROC | PROC-402 |
| 관련 IA 항목 | BAT-02 |

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |
|---------|-----------|-----------|-----------|----------------------------|
| OPS-003-01 | 보관 배치는 일 1회 스케줄로 실행한다(애플리케이션 스케줄 작업). | 스케줄 도래 시 | audit | 감사 로그 기록 |
| OPS-003-02 | 배치는 멱등하게 동작한다 — 재실행 시 중복 삭제·부작용이 없어야 한다. | 배치 실행 시 | transform | 내부 처리(사용자 미노출) |
| OPS-003-03 | 배치 실행 결과(대상 건수·삭제 건수·소요)를 감사 로그에 기록하고, 실패 시 다음 주기에 재시도한다. | 배치 종료 시 | audit | 감사 로그 기록 |

### 예외 사항

- **EXC-OPS-03**: 삭제 대상 선정 기준은 처리상태 DATA-004(완료/미완료 3개월)·연동이력 DATA-006(콜백 수신/미수신 3개월)을 따른다. 본 정책은 실행 방식만 정의한다.

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 | 적용 위치 |
|---------|----------|-----------|-----------|
| OPS-003-01 | PROC-402 | 배치 스케줄 기동 | 감사 |
| OPS-003-02 | PROC-402 | 삭제 실행(멱등) | 가공 |
| OPS-003-03 | PROC-402 | 배치 종료 감사 로그 | 감사 |

### 구현 가이드

- 배치는 단일 App Service 내 스케줄 작업으로 구성하고, 삭제 조건은 DATA-004(처리상태)·DATA-006(연동이력) 규칙을 한 배치 실행 흐름에서 청크 DELETE + 청크 단위 커밋으로 처리한다. 중단·재실행 시 커밋된 청크는 유지되고 잔여분만 삭제되어 멱등(OPS-003-02)이 성립한다 — 청크 크기·실행 상세는 PROC-402·ENT-004 참조.
- 배치 실행 결과 감사(OPS-003-03)의 대상 건수·삭제 건수는 처리상태·연동이력을 각각 집계해 기록한다.
