# 보안 정책 정의 — 암호화 연동 (SEC-006~008)

본 문서는 보안 정책([`policy_SEC.md`](policy_SEC.md))의 하위 문서로, 암호화 연동 플로우에 특화된 세 정책 — 이중 암호화·복호화 규약(SEC-006), 복호화 원문 서버-서버 전달(SEC-007), 알려진 보안 제약·후속 보완(SEC-008) — 을 정의한다. 코드 체계·용어·우선순위는 [`spec-policies.md`](spec-policies.md) 를, 나머지 보안 정책(SEC-001~005)은 [`policy_SEC.md`](policy_SEC.md) 를 따른다.

- **spec 확정 기본값**: 암호 규약(AES-256-CBC·PKCS#7·키 32B 정규화·IV 파생·encY 평문·Base64URL — SEC-006)과 서버-서버 전송(SEC-007)은 오케스트레이터 spec 확정값이다([`spec-policies.md`](spec-policies.md) §spec 확정 기본값).
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §시스템 제약사항(암호화 복호화 규약·보안 강도 한계) / [`../../prd/devspec/external-apis.md`](../../prd/devspec/external-apis.md) §암호화 연동 규약.

## SEC-006 이중 암호화·복호화 규약

### 기본 정보

| 항목 | 내용 |
|------|------|
| 정책명 | 이중 암호화·복호화 규약 |
| 분류 | SEC |
| 적용 범위 | 발송처 이중 암호값(encX·encY) 수신·허브 복호화 |
| 우선순위 | 높음 |
| 관련 정책 | DATA-001, SEC-002, SEC-005, SEC-007, SEC-008, BIZ-003, AUTH-004 |
| 적용 PROC | PROC-203 |
| 관련 IA 항목 | USR-02, 공통 |

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |
|---------|-----------|-----------|-----------|----------------------------|
| SEC-006-01 | 암호 알고리즘은 AES-256-CBC + PKCS#7 패딩을 encX·encY 공통으로 사용한다. | 암·복호화 시 | transform | 감사 로그 기록 |
| SEC-006-02 | 키 32B 정규화 — 키 문자열 UTF-8 바이트가 32 초과 시 앞 32B 절단, 미만 시 `_`(0x5F) 우측 패딩. IV 16B — 해당 키 문자열 UTF-8 앞 16B 절단·부족분 `_` 우패딩. | 키·IV 유도 시 | transform | 감사 로그 기록 |
| SEC-006-03 | 이중 구조 — encX = AES(X, 키=발송처키+생년월일), encY = AES(encX 키 문자열, 키=생년월일). encY 평문 = encX 키 문자열(발송처키+생년월일 정규화 전 원문). 전달 인코딩은 Base64URL(접근 주소 URL 파라미터). | 규약 적용 시 | transform | 감사 로그 기록 |
| SEC-006-04 | 복호화(허브) 순서 — 사용자 생년월일로 encY 복호화 → encX 키 문자열 복원 → encX 복호화 → 전달 데이터 X. 복호화는 연동 실행 시점 메모리에서만 수행하고 전달 후 즉시 폐기한다(DATA-001). | 승인 후 복호화 시 | transform | 감사 로그 기록 |
| SEC-006-05 | 암호 파라미터(encX·encY) 누락·Base64URL 형식 오류는 복호화 미수행·거부한다. 복호화 실패(패딩·키 불일치=잘못된 생년월일)는 재입력 유도로 처리한다(AUTH-004-02). | 복호화 진입·실행 시 | validate | 형식 오류 400 EX-SEC-007 / 복호화 실패 400 EX-SEC-006 |
| SEC-006-06 | encX·encY·생년월일·복호화 원문·발송처키·복원 키를 저장·로깅하지 않는다(SEC-005-06·DATA-001 준거). | 복호화 처리 시 | mask | 감사 로그 기록(성공 여부만) |

### 예외 사항

- **EXC-SEC-06**: 정적 IV 파생(IV 를 키 문자열에서 결정론적으로 유도) + 생년월일 저강도는 알려진 약점이다 — 오프라인 전수대입 취약은 SEC-008 이 관장하며 초기 범위에서 수용한다.
- **EXC-SEC-08**: encX 전달 형식(GET 파라미터 길이 한계)은 발송처·허브 합의 사항으로, 길이 초과 시 접근 URL 구성 방식(경로·본문 대체 등)은 서비스·화면 정의서·build 에서 확정한다. 규약(알고리즘·키·IV)은 본 정책이 고정한다.

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 | 적용 위치 |
|---------|----------|-----------|-----------|
| SEC-006-01 | PROC-203 | 복호화 알고리즘 적용 | 가공 |
| SEC-006-02 | PROC-203 | 키·IV 유도 | 가공 |
| SEC-006-03 | PROC-203 | 규약 해석(encX·encY) | 가공 |
| SEC-006-04 | PROC-203 | 복호화 실행 | 가공 |
| SEC-006-05 | PROC-203 | 암호 파라미터 검증·복호화 실패 분기 | 검증 |
| SEC-006-06 | PROC-203 | 복호화 로그(원문·키 미기록) | 마스킹 |

### 구현 가이드

- 키 문자열은 정규화 전 원문으로 encY 평문에 담기며, 정규화(32B·IV 16B)는 암·복호화 직전에 적용한다. 복호화 실패(EX-SEC-006)와 형식 오류(EX-SEC-007)는 사용자 재시도 가능 여부가 달라 구분해 처리한다(재입력 가능 vs 발송처 오류).
- 검증 도구가 규약을 재현할 수 있도록, 키·IV 정규화와 encY→encX→X 복호화 순서를 결정론적으로 구현한다(라이브러리 강제 없음).

## SEC-007 복호화 원문 서버-서버 전달

### 기본 정보

| 항목 | 내용 |
|------|------|
| 정책명 | 복호화 원문 서버-서버 전달 |
| 분류 | SEC |
| 적용 범위 | 허브→수신처 B 전달, 사용자 완료 페이지 노출 |
| 우선순위 | 높음 |
| 관련 정책 | SEC-006, DATA-001, BIZ-003 |
| 적용 PROC | PROC-203 |
| 관련 IA 항목 | USR-02 |

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |
|---------|-----------|-----------|-----------|----------------------------|
| SEC-007-01 | 복호화 원문 X 는 사용자 브라우저를 경유하지 않는다 — 허브→수신처 B 는 서버-서버 POST 로 전달하고, 리다이렉트·URL 파라미터로 X 를 노출하지 않는다(기밀성 핵심가치 보호). | 수신처 전달 시 | transform | 감사 로그 기록 + 내부 차단 |
| SEC-007-02 | 사용자에게 노출되는 완료 페이지는 복호화 원문 X·회원 키·전달 상세를 포함하지 않고 성공/실패 결과만 안내한다. | 완료 페이지 생성 시 | mask | 감사 로그 기록 |

### 예외 사항

- **EXC-SEC-09**: 수신처 B 전달 프로토콜 세부(요청 필드·응답 규약)는 서비스·PROC 정의가 확정한다. 본 정책은 "브라우저 미경유·서버-서버" 원칙을 고정한다.

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 | 적용 위치 |
|---------|----------|-----------|-----------|
| SEC-007-01 | PROC-203 | 수신처 전달(서버-서버) | 가공 |
| SEC-007-02 | PROC-203 | 완료 페이지 응답 구성 | 마스킹 |

### 구현 가이드

- 전달은 허브 서버에서 수신처 B 로 직접 호출하고, 사용자에게는 전달 결과만 반영한 완료 페이지를 반환한다. 원문·회원 키를 완료 페이지 DTO·URL 에 싣지 않는다.

|------|------|

| 우선순위 | 보통 |

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |
|---------|-----------|-----------|-----------|----------------------------|
| SEC-008-01 | 알려진 제약을 사양에 명시하고 수용한다 — encX·encY 가 URL 파라미터로 노출되고 생년월일(yyMMdd) 경우의 수가 작아, 링크·암호값을 확보한 자가 오프라인 전수대입으로 원문을 복원할 수 있다. 이상 징후는 복호화 시도·전달 감사(OPS-002)로 관측한다. | 상시 | audit | (알려진 제약 명시 — EX 없음) |
| SEC-008-02 | 후속 보완 항목(이번 범위 제외)을 백로그로 관리한다 — 링크 만료(TTL)·1회성 사용·발송처 서명 검증. 도입 확정 시 관련 정책(SEC-003·SEC-006·AUTH-004)을 리비전한다. | 후속 검토 시 | audit | (후속 항목 — EX 없음) |

### 예외 사항

- **EXC-SEC-07**: 강화 항목 도입 전까지 반복 시도·남용 억제는 요청 제한(OPS-001, 사용자 진입 분당 상한)과 감사 로그(OPS-002)로 1차 대응한다. 이는 본인확인 잠금이 아니라 요청 빈도 제한이다(AUTH-004-02·EXC-AUTH-05).

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 | 적용 위치 |
|---------|----------|-----------|-----------|
| SEC-008-01 | PROC-203 | 복호화·전달 감사(제약 관측) | 감사 |
| SEC-008-02 | (후속) | 후속 보완 백로그 | 감사 |

### 구현 가이드

- 본 정책은 알려진 약점을 은폐하지 않고 명시하는 것이 목적이다. 후속 보완 도입 시 링크 발급(TTL·1회성)·발송처 서명 검증을 SEC-003·SEC-006 규약에 얹는 방향으로 확장한다(초기 범위 미포함).
