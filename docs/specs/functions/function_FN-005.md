# 공통 입력 검증·주입 방어 공통 기능 정의

## 개요

- **기능 목적**: 모든 외부 입력(관리자 폼·진입 파라미터·API 파라미터)을 서버단에서 길이·형식·허용 문자로 재검증하고, SQL·스크립트 주입을 방지하며, 요청 본문 크기 상한을 강제한다. 화면 검증에 의존하지 않는 서버 공통 방어선이다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §시스템 제약사항(무저장·신뢰 위임 전제의 전송 안전성) / 정책 SEC-004.

## FN-005 공통 입력 검증·주입 방어

### 기본 정보

| 항목 | 내용 |
|------|------|
| 기능명 | 공통 입력 검증·주입 방어 |
| 분류 | POL |
| 사용 서비스 | SVC-001, SVC-002, SVC-004, SVC-006 |
| 호출 PROC | PROC-101, PROC-102, PROC-201, PROC-301 |
| 연관 정책 | [SEC-004](../policies/policy_SEC.md#sec-004-입력-검증주입-방어)(01·02·03) |
| 참조 데이터 | 대상 요청 DTO(MDL-101·201·202·203 등), 검증 스키마 |
| 관련 IA 항목 | 공통 |

### 시그니처

```
function FN-005_validateInput<T> (
  raw: unknown,           // 원시 요청 본문/파라미터
  schema: Schema<T>,      // 대상 DTO 스키마(길이·형식·허용 문자)
  rawSize: number,        // 요청 본문 바이트 크기
): T                      // 검증·정규화된 DTO
  throws BodyTooLargeError { code: EX-SEC-005, http: 413 }
        | InvalidInputError { code: EX-SEC-004, http: 400 }
```

### 입력/출력 정의

| 구분 | 항목명 | 데이터 타입 | 필수 | 제약 | 설명 |
|------|--------|------------|------|------|------|
| 입력 | raw | unknown | Y | - | 검증 전 원시 입력 |
| 입력 | schema | Schema | Y | DTO별 정의 | 길이·형식·허용 문자 규칙 |
| 입력 | rawSize | number | Y | ≤ 1MB(기본안) | 본문 크기 |
| 출력 | T(DTO) | 대상 MDL | - | 정규화 | 검증 통과 DTO |

### 처리 흐름 (의사코드)

```
1. 본문 크기 검사 — POL SEC-004-03 (validate)
   if (rawSize > 1MB)                 → throw BodyTooLargeError (413, EX-SEC-005)

2. 스키마 검증 — POL SEC-004-01 (validate)
   errors = schema.validate(raw)      // 길이·형식·허용 문자·필수
   if (errors.length > 0)
        → throw InvalidInputError (400, EX-SEC-004, details: errors)  // 필드별 오류 배열

3. 주입 방어 정규화 — POL SEC-004-02 (transform)
   dto = schema.coerce(raw)           // 타입 정규화
   assert(noRawQueryConcat)           // DB 접근은 파라미터 바인딩만(호출부 계약)
   escapeOnRender(dto)                // 출력 이스케이프 대상 표시

4. 반환
   return dto
```

> DB 접근 계층은 파라미터 바인딩만 사용하고 원시 쿼리 문자열 결합을 금지한다(SEC-004-02). 본 FN 은 검증·정규화를 담당하고, 바인딩 강제는 데이터 접근 계층 규약으로 관철한다.

### API 인터페이스

해당 없음 — 각 API 진입점(PROC-101·201·301)의 입력 검증 단계에서 호출되는 공통 유틸리티다. 독립 엔드포인트가 아니다.

### 에러 처리 (에러 코드 카탈로그)

| HTTP status | EX 코드 | 발생 조건 | 사용자 메시지 | 개발자 노트 |
|-------------|---------|-----------|---------------|-------------|
| 400 | EX-SEC-004 | 길이·형식·허용 문자 위반·주입 패턴 | "입력 형식이 올바르지 않습니다." | details 에 필드별 오류 배열 |
| 413 | EX-SEC-005 | 본문 1MB 초과 | "요청이 너무 큽니다." | SEC-004-03(기본안 1MB) |
| 500 | EX-FN-999 | 스키마 로드·검증 오류 | "잠시 후 다시 시도해주세요." | - |

### 의존 기능

없음(leaf). 신뢰 위임(SEC-002)의 회원 키도 전송 안전성 검증(형식·크기·주입)은 면제되지 않으므로(EXC-SEC-02) 본 FN 을 거친다.

### 구현 가이드

- 입력 검증은 DTO 스키마 검증 계층에서 일괄 처리한다(특정 검증 라이브러리 강제 없음). 본문 크기 상한(1MB)은 기본안이며 확정 시 SEC-004 를 리비전한다.
- 회원 키(memberKey)는 값 의미는 검증하지 않되(신뢰 위임) 크기·형식·주입 방어는 수행한다(EXC-SEC-02).
