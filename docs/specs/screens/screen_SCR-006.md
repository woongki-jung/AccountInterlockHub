# 동의 결과 화면 정의

## 개요

- **화면 목적**: 이용 동의(SCR-005) 승인/거부 제출 이후 처리 결과를 최종 사용자에게 안내한다. 연동 완료(허브 복호화·수신처 B 전달 성공)·거부 종료·전달 실패·**링크 오류**의 결과를 명확한 문구로 표시한다. 신규 API·PROC 를 트리거하지 않고 앞선 승인 처리(PROC-202/내부 PROC-203) 결과를 표시한다. 연동 추적 키·회원 키·개인정보를 노출하지 않는다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §수행 범위 4 "연동 실행(허브 복호화·전달): … 성공 시 사용자에게 연동 완료 페이지 노출" 결과 안내. 서비스 근거 [SVC-004](../services/service_SVC-004.md)·[SVC-005](../services/service_SVC-005.md).
- **디자인**: [`design-system.md`](design-system.md) 사용자 셸(중앙 카드, 최대 폭 480px)·`Card`·`Badge`·`Banner` 인용.

> **2026-07-11 `#214` 개정**: 허브 발급 요청 키값 경로(`/consent/:requestKey/result`)를 폐기하고, 결과를 SCR-005 승인 제출 응답의 클라이언트 상태로 전달받는다(민감값 미포함 경로 `/interlock/result`). 결과 유형에 **링크 오류(암호 파라미터 형식·추적 키 필드 오류 — 발송처 문의, 재입력 불가)**를 추가했다. 완료 페이지는 복호화 원문·회원 키·추적 키를 포함하지 않는다(SEC-007-02).

## SCR-006 동의 결과

### 기본 정보

| 항목 | 내용 |
|------|------|
| 화면명 | 동의 결과 |
| 화면 경로 | `/interlock/result` (SCR-005 제출 결과 컨텍스트) |
| 접근 권한 | Public(발송처 진입 흐름) + SCR-005 제출 결과 컨텍스트(클라이언트 상태). 관리자 인증 경로와 분리 |
| 관련 서비스 | [SVC-004](../services/service_SVC-004.md) 이용 동의 · [SVC-005](../services/service_SVC-005.md) 연동 실행 |

| 관련 IA 항목 | USR-02 |

### 레이아웃 구성

- 헤더 없는 단일 중앙 `Card`(최대 폭 480px, 상하 여백 48px).
- `Card` 내부: 결과 상태 아이콘·`Badge` → 결과 제목(`text-h1`) → 결과 설명 문구 → (해당 시)안내 `Banner` → 안내 마무리 문구.
- 결과에 따른 색: 연동 완료=success, 거부 종료=info(중립), 전달 실패=warning/error, 링크 오류=error.
- 반응형: mobile 좌우 여백 16px 전체 폭.

### 데이터 표시

| 표시 항목 | 데이터 모델(MDL) | 속성 | 표시 형식 | 마스킹 규칙 |
|-----------|-----------------|------|-----------|-------------|
| 처리 결과 유형 | (PROC-202/203 응답 상태) | 연동완료 / 거부 / 전달실패 / 링크오류 | `Badge` + 제목 | 마스킹 없음(개인정보·추적 키 미포함) |
| 결과 안내 문구 | (승인 응답 결과 코드 파생) | 성공 여부·거부·오류 유형 | 텍스트 | 마스킹 없음 |

> 결과 화면은 회원 키·연동 추적 키·개인정보·복호화 원문·처리 상태 값을 표시하지 않는다(무저장·최소 노출, DATA-001·SEC-007-02). 처리 상태 4항목은 발송처 대면 API(API-01)에서 조회하며 사용자 화면에는 결과 유형만 안내한다.

### 사용자 인터랙션

| 인터랙션 | 대상 요소 | 동작 | 호출 API(FN) | 트리거 PROC | 결과 |
|----------|-----------|------|-------------|-------------|------|
| 페이지 mount | (자동) | SCR-005 제출 결과 상태 렌더 | (없음, 클라이언트 상태) | (트리거 없음) | 결과 유형별 안내 표시 |
| 클릭 | 닫기/발송처 복귀(있을 시) | 진입 서비스로 복귀 안내 | (없음) | (네비게이션, PROC 없음) | 발송처 흐름 복귀 |

> 본 화면은 신규 서버 호출을 하지 않는다. 결과는 SCR-005 의 승인/거부 제출(PROC-202/내부 PROC-203) 응답을 네비게이션 상태로 전달받아 표시한다.

### 화면 상태 전이

| 상태 | 진입 조건 | PROC 단계 매핑 | 표시 내용 |
|------|----------|---------------|-----------|
| Loaded(연동 완료) | 승인→복호화→수신처 전달 성공(200) | PROC-202 승인 결과 반영 / PROC-203 전달 응답 처리(성공) | success `Badge` + "연동이 완료되었습니다." |
| Loaded(거부 종료) | 거부(200 정상 종료) | PROC-202 거부 처리·최소 감사 | info `Badge` + "연동이 취소되었습니다." |
| Loaded(전달 실패) | 502 EX-BIZ-004(재시도 후 실패) | PROC-203 전달 응답 처리(실패 확정) | error `Banner` + "연동 처리 중 오류가 발생했습니다. 잠시 후 다시 시도해 주세요." |
| Loaded(링크 오류) | 400 EX-SEC-007(암호 파라미터 형식) / 400 EX-BIZ-008(추적 키 필드 누락·X 파싱 실패) | PROC-203 암호 파라미터·추적 키 검증 실패(종료) | error `Badge` + "연동 링크가 올바르지 않습니다. 발송처에 문의해 주세요."(재입력 불가) |
| Fallback(직접 진입) | 결과 상태 없음(새로고침·직접 URL) | (해당 없음) | 안내 "표시할 결과가 없습니다. 발송처에서 다시 진입해 주세요." |

> 연동 완료·전달 실패는 복호화 성공 이후라 처리 상태 1건·연동이력 1건이 서버에 기록된다(EXC-BIZ-06·EXC-BIZ-11). 거부·복호화 실패·링크 오류는 추적 키가 없어 처리상태·연동이력을 남기지 않고 감사 로그(결과 코드)로만 기록된다(BIZ-002-07·EXC-DATA-03). 화면은 결과 유형만 안내하고 상태 항목 값·추적 키를 노출하지 않는다. 복호화 실패(EX-SEC-006)는 SCR-005 에서 생년월일 재입력으로 처리되므로 본 화면으로 넘어오지 않는다.

### 입력 폼 정의

해당 없음 — 결과 안내 전용 화면으로 입력 폼이 없다.

### 구현 가이드

- 결과는 SCR-005 제출 응답에서 전달받은 클라이언트 상태(결과 유형)로 렌더한다(추가 서버 호출 없음). 새로고침·직접 진입 시 상태가 없으면 Fallback 안내를 표시한다. 결과 경로(`/interlock/result`)에는 encX·encY·생년월일·추적 키 등 민감값을 싣지 않는다(SEC-005-06·SEC-007-02).
- 회원 키·연동 추적 키·처리 상태 값을 화면에 노출하지 않고 결과 유형만 사용자 친화 문구로 안내한다. 상태 색은 텍스트 라벨(완료/취소/오류)과 병기한다(디자인 시스템 §접근성).
- 링크 오류(EX-SEC-007·EX-BIZ-008)는 발송처 링크·데이터 구성 문제로 사용자가 정정할 수 없으므로 재입력·재시도 유도 대신 발송처 문의를 안내한다. 전달 실패(EX-BIZ-004)는 일시적 오류일 수 있어 "잠시 후 다시 시도" 안내와 함께 재진입(SCR-005) 경로를 열어 둔다(build 에서 재시도 액션 제공 여부 확정).
