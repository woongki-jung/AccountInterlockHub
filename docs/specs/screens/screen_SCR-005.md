# 사용자 이용 동의 화면 정의

## 개요

- **화면 목적**: 발송처 링크로 진입한 최종 사용자에게 **본인확인용 생년월일**을 입력받고, 진입한 접근 주소(발송처 판별)에 설정된 동의 항목만 노출해 동의/거부를 스스로 결정하게 한다. 승인 시 허브가 생년월일로 복호화해 수신처 B 로 서버-서버 전달하는 연동 실행(SVC-005)을 트리거한다. 생년월일·암호값(encX·encY)은 화면에 저장·표시·로깅하지 않는다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §수행 범위 3 "사용자 — 접근·동의·승인 페이지: 발송처 링크로 진입, 생년월일 입력, 구성된 동의 항목 노출·동의 체크 후 연동 승인". 서비스 근거 [SVC-004](../services/service_SVC-004.md).
- **디자인**: [`design-system.md`](design-system.md) §사용자 표면 확장 인용 — 사용자 셸(중앙 카드, 최대 폭 480px)·`Card(user)`·`TextField(user/birthdate)`·`Checkbox(user)`·`ConsentItem(user)`·`Button(user/primary·secondary·ghost-link)`·`Banner(user)`·`Modal(user/terms)` 및 사용자 전용 색·타이포·간격·모션 토큰(`-u-`).

> **2026-07-11 `#214` 개정**: 입력이 단일 암호화 JSON(encX·encY)으로 바뀌어 **허브 발급 요청 키값(UUID) 경로(`/consent/:requestKey`)를 폐기**하고, 발송처 링크로 진입한 **접근 주소(고유 ID) 자체를 발송처 판별값**으로 삼는다. **생년월일 입력 필드를 신설**(본인확인·복호화 요소, AUTH-004)하고, 승인 제출은 접근 컨텍스트(encX·encY·생년월일)를 함께 담아 `/api/interlock/approve` 로 전송한다. 복호화·수신처 전달은 승인 이후 허브가 수행한다(SVC-005·PROC-203).
>
> **2026-07-12 `#215` 개정(목업 리뷰 보완)**: **동의 대상 설명 문구(consentNotice)**를 제목 아래 상단에 노출하도록 추가했다(BIZ-002-08, 미설정 시 영역 미렌더). 복호화 실패(EX-SEC-006) 인라인 문구를 "사용자 정보가 일치하지 않습니다."로 고정해 FE 형식 안내와 구분했다. **전달 실패(EX-BIZ-004)를 SCR-006 이동에서 본 화면 잔류로 이관**했다 — error `Banner`("동의 처리에 실패했습니다. 다시 시도해주세요.") + 승인 재제출(재승인) 허용.
>
> **2026-07-18 `#408` 개정(사용자 페이지 시각 개선)**: 시각 폴리시만 개선했다 — 레이아웃 구성의 여백·크기·색·모션을 사용자 표면 토큰 기준 수치로 확정하고, 본 화면의 **고정 문구 정본 표**를 추가했으며, 복호화 실패 후 생년월일 필드 상태를 **값 유지 + 포커스 이동 + 전체 선택**으로 확정했다. 약관 [상세] 버튼을 `Button(user/ghost-link)`(가시 높이 32px·히트 영역 44x44px)로 바꿔 터치 타깃 기준 충돌을 해소하고, 긴 콘텐츠(설명 문구·동의 항목)의 오버플로 규칙을 항목 수와 무관한 상시 상한으로 명시했다. **기능·플로우·유효성 규칙·트리거 PROC·민감값 노출 규칙·고정 문구 자체는 불변**이다.
>
> **2026-07-18 `#408` 목업 리뷰 보완**: 목업 제작 중 드러난 모호·이중 서술 3건을 사양에 반영했다 — (1) 필수/선택 표시를 **텍스트 태그("필수"/"선택")로 단일화**하고 별표(`*`) 표기를 폐지했다(생년월일 라벨 포함, [`design-system.md`](design-system.md) `ConsentItem(user)` 정의와 이중 서술 해소 — 색·기호가 아닌 텍스트로 의미를 전달하는 §접근성 원칙과 정합). (2) 승인·거부 액션의 **데스크톱 정렬 방향**(≥640px 가로 배치·우측 정렬, 좌→우 승인→거부, 간격 12px)을 명시했다. (3) 섹션 구분선이 들어가는 **위치를 2곳으로 확정**했다(상단 블록↔본인확인, 본인확인↔동의 항목 — 액션 앞에는 두지 않음). 배정된 타이포·색 토큰 값과 고정 문구는 불변이다.

### 레이아웃 구성

- 헤더 없는 단일 중앙 `Card(user)`(최대 폭 480px, 상하 여백 `space-u-shell-y` 48px/모바일 24px, 내부 여백 `space-u-card` 28px/모바일 20px, `radius-u-card` 16px, `shadow-u-card`, 상단 4px `color-primary` 액센트 라인). 페이지 배경 `color-u-canvas`. 모바일 진입 비중 고려 단일 컬럼.
- `Card` 내부 섹션 순서(PRD 흐름: 생년월일 입력 → 동의 항목). 섹션 간 간격 `space-u-section` 28px, 섹션 내 요소 간 `space-u-stack` 12px:
	1. 안내 제목(`text-u-title` 24px/32px, 예 "서비스 연동 동의") → **동의 대상 설명 문구**(구성에 설정된 consentNotice 를 `text-u-lead` 로 그대로 노출, 영역 최대 높이 144px(6줄)·초과 시 내부 세로 스크롤, 미설정 시 영역 자체 미렌더, BIZ-002-08).
	2. **본인확인 섹션**: 섹션 라벨 "본인확인"(`text-u-section`) + 생년월일 `TextField(user/birthdate)`(라벨 "생년월일(YYMMDD)" + 라벨 우측 **"필수" 텍스트 태그**(라벨과 간격 4px, 규격은 `ConsentItem(user)` 의 필수/선택 텍스트 태그와 동일 — 13px/18px·700·`color-text`, **별표(`*`) 미사용**), 높이 48px, 6자리 숫자 입력, `inputMode=numeric`) + 안내 캡션("연동 값 복원을 위한 본인확인입니다", `text-u-caption`).
	3. **동의 항목 섹션**: 섹션 라벨 "동의 항목"(`text-u-section`) + 동의 항목 목록(`ConsentItem(user)` 행 목록, 행 간격 8px·행 여백 12px 14px, 각 항목 라벨(`text-u-item`)·설명(`text-u-caption`)·**필수/선택 텍스트 태그** 표시, 약관 컨텐츠가 있으면 라벨 우측에 [상세] `Button(user/ghost-link)`). 필수/선택 텍스트 태그는 항목 라벨 우측(간격 8px)에 **"필수" 또는 "선택" 을 텍스트로 표기**하며 `text-u-caption` 크기 13px/18px — 필수 700·`color-text`, 선택 400·`color-u-text-muted` 로 둔다(**별표(`*`)·색·기호 단독 표기 금지** — 정본은 [`design-system.md`](design-system.md) `ConsentItem(user)`). 목록 컨테이너는 항목 수와 무관하게 최대 높이 360px(모바일 320px) + 세로 스크롤(내용이 더 낮으면 스크롤 없음).
	4. 액션: 승인(동의) `Button(user/primary)` · 거부 `Button(user/secondary)` — 높이 48px, 버튼 간 간격 12px. 데스크톱(≥640px)은 한 줄 가로 배치 + **카드 우측 정렬**(`justify-content: flex-end`)이고 좌→우 배치 순서는 승인(동의) → 거부다(본문은 좌측 정렬이므로 액션만 우측에 모아 읽기 흐름의 종료 지점에 둔다). 모바일(<640px)은 같은 순서로 세로 스택(각 폭 100%·간격 12px, §반응형).
- **섹션 구분선**: 카드 본문에 `color-u-divider` 1px 가로 구분선을 **정확히 2곳**에 둔다 — (1) 상단 블록(안내 제목·동의 대상 설명 문구·카드 상단 `Banner(user)`) ↔ 본인확인 섹션 사이, (2) 본인확인 섹션 ↔ 동의 항목 섹션 사이. 동의 항목 섹션 ↔ 액션 사이에는 두지 않는다(액션은 섹션이 아니라 카드 하단 액션 영역이며 `space-u-section` 28px 간격으로만 분리한다). 구분선 상하 여백은 각 14px 로 `space-u-section` 28px 를 이등분해 배치한다. 구분선은 장식이다 — 지워도 제목·섹션 라벨로 구조가 전달된다.
- [상세] 클릭 시 약관 상세 콘텐츠 `Modal(user/terms)`(제목=항목 라벨, 스크롤 본문=약관 컨텐츠, 하단 [동의]`Button(user/primary)`·[닫기]`Button(user/secondary)`)을 연다. [동의]=해당 항목 체크(동의) 후 모달 닫기, [닫기]=모달만 닫기(체크 불변).
- 반응형: mobile(<640px) 좌우 여백 16px 전체 폭, 버튼 세로 스택(폭 100%). 약관 모달은 mobile 에서 뷰포트에 맞춰 축소하고 본문 영역만 스크롤한다(폭·최대 높이 수치 정본은 [`design-system.md`](design-system.md) §컴포넌트 사용자 변형).
- 마이크로 인터랙션: 카드·인라인 에러·`Banner(user)` 등장은 `motion-u-base`·`ease-u-decelerate`(불투명도 0→1 + 8px 상승), 버튼·체크박스·항목 행 hover 색 전환은 `motion-u-fast`·`ease-u-standard`. `prefers-reduced-motion: reduce` 이면 이동 없이 최종 상태를 즉시 렌더한다.

> 동의 대상 설명 문구(consentNotice)는 진입한 접근 주소 구성에 설정된 값을 카드 상단(제목 아래)에 노출한다(BIZ-002-08, 미설정 시 생략). 동의 화면은 진입한 접근 주소 구성에 설정된 동의 항목만 노출한다(BIZ-002-01, 구성 외 항목 노출 금지). 회원 키·연동 추적 키는 화면에 존재하지 않는다(복호화 이전·무저장, DATA-001). encX·encY 는 URL 파라미터로 수신하되 화면에 렌더하지 않으며, 승인 제출 시에만 요청 본문에 실어 서버로 보낸다(URL→본문 이동, 재노출 금지). 생년월일은 입력 필드 값으로만 메모리에 두고 제출 후 화면에 남기지 않는다.

### 구현 가이드

- 회원 키·개인정보·연동 추적 키를 화면에 표시하지 않는다(복호화 이전 단계라 존재하지 않음). encX·encY 는 URL 파라미터로 읽어 컴포넌트 상태(메모리)로만 보유하고 화면에 렌더하지 않으며, 승인 제출 시 요청 **본문**에 실어 보낸다(URL 재기록·로컬 저장 금지). 생년월일은 입력 필드 상태로만 두고 어떤 로그·URL·화면에도 노출하지 않는다(AUTH-004-03·SEC-005-06).
- 접근 주소 고유 ID(경로 `:accessAddressId`)가 곧 발송처 판별값이다 — FE 는 접속한 접근 주소로 발송처를 판별하고 별도 발송처 선택 UI 를 두지 않는다(BIZ-001-11). 접근 URL 의 정확한 파라미터 배치(접근 주소 경로/쿼리, encX·encY 쿼리 vs 본문, GET 파라미터 길이 한계 대응)는 build 확정 대상이다(EXC-SEC-08).
- 약관 컨텐츠가 있는 항목에만 [상세] 버튼을 렌더한다(BIZ-002-05). 약관 모달은 포커스 트랩·ESC·배경 클릭 닫기를 적용하고 본문 영역만 스크롤한다(디자인 시스템 `Modal(user/terms)` — 전역 `Modal` 콘텐츠 변형 규칙 승계). 모달 [동의]는 해당 항목을 체크한 뒤 닫고, [닫기]는 체크 상태를 바꾸지 않고 닫는다 — 두 동작 모두 서버 호출이 아니라 승인/거부 제출(PROC-202) 전의 클라이언트 상태 조작이다(EXC-BIZ-08).
- 복호화 실패(EX-SEC-006) 응답 시 화면을 이탈하지 않고 생년월일 필드에 인라인 에러("사용자 정보가 일치하지 않습니다.")를 노출한 뒤, **입력값을 유지한 채 해당 필드로 포커스를 옮기고 값 전체를 선택 상태로 둔다** — 값 삭제 없이 이어서 숫자를 누르면 전체가 대체되므로 재입력 마찰이 가장 적다(하드 잠금 없음, AUTH-004-02). 여기서 유지하는 값은 **사용자가 방금 직접 입력한 값**이며, 민감값 규칙의 "표시(에코) 금지"는 서버 응답에 담긴 값을 화면에 되비추지 말라는 뜻이므로 서로 상충하지 않는다(서버 응답은 생년월일을 포함하지 않는다). 사용자가 값을 수정하기 시작하면 인라인 에러를 즉시 해제하고 캡션을 FE 형식 안내로 되돌린다. 이 서버 판정 문구는 FE 형식 위반 안내("생년월일 6자리(YYMMDD)를 정확히 입력해 주세요.")와 구분해 같은 캡션 영역에서 텍스트만 교체한다(형식 위반=blur·입력 시 FE, 복호화 실패=제출 응답 시 서버). 재제출은 동일 접근 주소·encX·encY 에 대해 생년월일만 갱신해 다시 POST 한다(암호값 재수신 불요). 남용 억제는 요청 제한(OPS-001)으로 서버가 1차 방어한다.
- 전달 실패(EX-BIZ-004) 응답 시에도 별도 결과 화면(SCR-006)으로 이동하지 않고 본 화면에 전달 실패 `Banner(user)`(danger, "동의 처리에 실패했습니다. 다시 시도해주세요.")를 노출하고 승인 재제출(재승인)을 허용한다. 복호화 성공 이후라 처리 상태·연동이력 1건은 이미 저장돼 있으며, 재제출은 새 승인 제출(PROC-202)로 처리된다. 링크 오류(EX-SEC-007·EX-BIZ-008)만 SCR-006 로 종료 안내한다(재입력 불가, EXC-BIZ-13).
- 생년월일 형식 미충족 또는 필수 동의 미체크 시 승인 버튼을 비활성으로 유지한다. 비활성 사유는 색·커서만이 아니라 텍스트(형식 안내·"필수 동의 항목에 동의해 주세요.")로도 안내한다. 거부는 오류가 아닌 정상 종료로, 거부 제출 후 SCR-006 에서 취소 안내를 표시한다(EXC-BIZ-03). 처리 완료·거부 후 접근 컨텍스트(encX·encY·생년월일)는 서버 메모리에서 폐기된다(DATA-001·SEC-006-06).
- 시각 규격(여백·크기·색·모션)은 [`design-system.md`](design-system.md) §사용자 표면 확장을 그대로 따른다 — 임의 색 신설·전역 토큰 값 변경 금지. 모든 상호작용 요소는 **히트 영역 44x44px 이상**을 확보하고([상세] 버튼은 가시 높이 32px + 투명 여백으로 확장), 포커스 링 2px·offset 2px 를 상시 노출한다. 상태·오류는 색 단독으로 전달하지 않고 텍스트를 병기한다.
- 콘텐츠 길이 방어: 동의 대상 설명 문구(최대 1000자)는 영역 최대 높이 144px(6줄) + 내부 세로 스크롤으로 흡수한다. 동의 항목 목록은 **항목 수·설명 길이와 무관하게 상시** 최대 높이 360px(모바일 320px) + 세로 스크롤을 적용해, 항목이 적어도 설명이 길면 스크롤이 흡수하도록 한다(내용이 상한보다 낮으면 스크롤이 생기지 않는다). 항목 라벨(최대 200자)·설명(최대 1000자)은 **말줄임 없이 전량 표시**한다(고지 정보 은닉 금지). 스크롤 컨테이너에는 `tabindex="0"`·`role="group"`·`aria-label` 을 부여해 키보드만으로도 스크롤할 수 있게 한다.
