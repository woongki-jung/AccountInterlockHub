# 사용자 이용 동의 화면 정의

## 개요

- **화면 목적**: 서비스 A에서 진입한 최종 사용자에게 해당 연동 구성에 설정된 동의 항목만 노출하고, 동의/거부를 스스로 결정하게 한다. 동의 시 서비스 B 연동 실행으로 진행하고, 거부 시 전달 없이 종료한다. 회원 키는 화면에 표시하지 않고 저장하지 않는다.
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §수행 범위 "사용자 — 이용 동의 페이지: 구성된 동의 항목 노출, 동의/거부 처리". 서비스 근거 [SVC-004](../services/service_SVC-004.md).
- **디자인**: [`design-system.md`](design-system.md) 사용자 셸(중앙 카드, 최대 폭 480px)·`Card`·`Checkbox`·`Button`·`Banner`·`Modal`(약관 상세 콘텐츠 변형) 인용.

---

## SCR-005 사용자 이용 동의

### 기본 정보

| 항목 | 내용 |
|------|------|
| 화면명 | 사용자 이용 동의 |
| 화면 경로 | `/consent/:requestKey` |
| 접근 권한 | Public(서비스 A 진입 흐름) + 요청 키값 진입 컨텍스트 유효([DATA-002](../policies/policy_DATA.md#data-002-요청-키값-발급불투명성)). 관리자 인증 경로와 분리 |
| 관련 서비스 | [SVC-004](../services/service_SVC-004.md) 이용 동의 |
| 트리거 PROC | PROC-201(진입·동의 화면 데이터 구성) · PROC-202(동의/거부 처리) |
| 선행 화면 | 없음(서비스 A `/interlock/entry` 진입 → 요청 키값 발급 후 본 화면으로 유입) |
| 관련 IA 항목 | USR-01 |

### 레이아웃 구성

- 헤더 없는 단일 중앙 `Card`(최대 폭 480px, 상하 여백 48px). 모바일 진입 비중 고려 단일 컬럼.
- `Card` 내부: 안내 제목(`text-h1`, 예 "서비스 연동 동의") → 연동 대상 안내 문구 → 동의 항목 목록(`Checkbox` 목록, 각 항목 라벨·설명·필수 표시, 약관 컨텐츠가 있으면 라벨 우측에 [상세] `Button(ghost/sm)`) → 동의 `Button(primary)` · 거부 `Button(secondary)`.
- [상세] 클릭 시 약관 상세 콘텐츠 `Modal`(제목=항목 라벨, 스크롤 본문=약관 컨텐츠, 하단 [동의]`Button(primary)`·[닫기]`Button(secondary)`)을 연다. [동의]=해당 항목 체크(동의) 후 모달 닫기, [닫기]=모달만 닫기(체크 불변).
- 반응형: mobile 좌우 여백 16px 전체 폭, 버튼 세로 스택. 약관 모달은 mobile 에서 전체 폭에 가깝게 표시하고 본문 영역만 스크롤한다.

### 데이터 표시

| 표시 항목 | 데이터 모델(MDL) | 속성 | 표시 형식 | 마스킹 규칙 |
|-----------|-----------------|------|-----------|-------------|
| 동의 항목 라벨 | (구성 동의 항목, [MDL-101](../datas/model_admin.md).consentItems) | label | 텍스트(체크박스 라벨) | 마스킹 없음(구성 메타) |
| 동의 항목 설명 | MDL-101.consentItems | description | 보조 텍스트(`text-caption`) | 마스킹 없음 |
| 약관 컨텐츠 | MDL-101.consentItems | termsContent | [상세] 버튼→약관 모달 본문(값 있을 때만) | 마스킹 없음(설정 메타) |
| 필수 여부 | MDL-101.consentItems | required | 필수 표식(*) | 마스킹 없음 |
| 요청 키값 | [MDL-202](../datas/model_user.md) | requestKey | (URL 경로, 화면 비표시) | 인증 실패 로그 시 마스킹(SEC-003-02) |

> 동의 화면은 해당 연동 구성에 설정된 동의 항목만 노출한다(BIZ-002-01, 구성 외 항목 노출 금지). 회원 키는 화면에 표시하지 않으며(무저장·미노출, DATA-001), 요청 키값은 경로 컨텍스트로만 사용한다.

### 사용자 인터랙션

| 인터랙션 | 대상 요소 | 동작 | 호출 API(FN) | 트리거 PROC | 결과 |
|----------|-----------|------|-------------|-------------|------|
| 페이지 mount | (자동) | 진입 컨텍스트의 동의 항목 조회·표시(약관 컨텐츠 포함) | GET /api/consent/:requestKey([FN-005](../functions/function_FN-005.md)·FN-007·[FN-008](../functions/function_FN-008.md)) | PROC-201 | 동의 항목 렌더링 / Error |
| 입력 | 동의 항목 `Checkbox` | 필수 항목 체크 상태 갱신 | (없음, 클라이언트) | (트리거 없음) | 동의 버튼 활성/비활성 |
| 클릭 | 항목 [상세] 버튼 | 약관 상세 `Modal` 열기(약관 본문 표시) | (없음, 클라이언트) | (트리거 없음) | 약관 모달 노출(BIZ-002-05) |
| 클릭 | 약관 모달 [동의] | 해당 항목 체크(동의) 후 모달 닫기 | (없음, 클라이언트) | (트리거 없음) | 항목 체크·동의 버튼 활성 재계산(EXC-BIZ-08) |
| 클릭 | 약관 모달 [닫기] | 모달만 닫기(체크 불변) | (없음, 클라이언트) | (트리거 없음) | 모달 닫힘 |
| 클릭 | 동의 버튼 | 동의 결과 제출(AGREE) | POST /api/consent/:requestKey([FN-008](../functions/function_FN-008.md)·FN-012·FN-009·FN-013) | PROC-202 | SCR-006(결과) 이동 |
| 클릭 | 거부 버튼 | 거부 결과 제출(REJECT) | POST /api/consent/:requestKey(FN-008·FN-009·FN-013) | PROC-202 | SCR-006(거부 종료 안내) 이동 |

### 화면 상태 전이

| 상태 | 진입 조건 | PROC 단계 매핑 | 표시 내용 |
|------|----------|---------------|-----------|
| Initial | 페이지 mount | (트리거 전) | 카드 Skeleton |
| Loading(항목 로드) | 항목 요청 대기 | PROC-201 동의 화면 데이터 구성 진입 직후 | Skeleton 유지 |
| Loaded | 200 응답 | PROC-201 동의 화면 데이터 구성 | 동의 항목·버튼 렌더링 |
| Submitting | 동의/거부 제출 대기 | PROC-202 동의 결과 검증·분기 진입 직후 | 버튼 Spinner + `disabled` |

| Error(키/컨텍스트) | 400 EX-DATA-002 | PROC-201/202 진입 컨텍스트 조회 실패 | `Banner`(error) "요청이 올바르지 않습니다."(만료·불일치) |
| Error(요청 제한) | 429 EX-OPS-001 | PROC-201 진입 요청 제한 검사 | `Banner` "잠시 후 다시 시도해주세요." |

### 입력 폼 정의

| 필드명 | 입력 유형 | 데이터 모델(MDL) | 필수 | 유효성 규칙 (FE 검증 의사코드) | 에러 메시지 |
|--------|-----------|-----------------|------|--------------------------------|-------------|
| 동의 항목 체크 | checkbox 목록 | [MDL-203](../datas/model_user.md).decision(파생) | 필수 항목 한정 | `consentItems.filter(i => i.required).every(i => checked[i.order])` | "필수 동의 항목에 동의해주세요." |
| 결정 값 | (AGREE/REJECT 버튼) | MDL-203.decision | Y | `['AGREE','REJECT'].includes(decision)` | (버튼 선택으로 보장) |

> 동의 결과(decision)는 화면 값 단독으로 신뢰하지 않고 서버가 구성 매칭 근거(configCode)로 검증한다([FN-008](../functions/function_FN-008.md)). requestKey 형식(UUID v4)은 서버가 재검증(FN-007, EX-DATA-002)한다.

### 구현 가이드

- 회원 키·개인정보를 화면에 표시하지 않는다. 요청 키값은 경로 컨텍스트로만 사용하고 로컬 저장하지 않는다.
- 약관 컨텐츠가 있는 항목에만 [상세] 버튼을 렌더한다(BIZ-002-05). 약관 모달은 포커스 트랩·ESC·배경 클릭 닫기를 적용하고 본문 영역만 스크롤한다(디자인 시스템 `Modal` 콘텐츠 변형). 모달 [동의]는 해당 항목을 체크한 뒤 닫고, [닫기]는 체크 상태를 바꾸지 않고 닫는다 — 두 동작 모두 서버 호출이 아니라 동의/거부 제출(PROC-202) 전의 클라이언트 상태 조작이다.
- 필수 동의 항목 미체크 시 동의 버튼을 비활성으로 유지한다. 거부는 오류가 아닌 정상 종료로, 거부 클릭 후 SCR-006 에서 종료 안내를 표시한다(EXC-BIZ-03). 처리 완료 후 진입 컨텍스트(회원 키 포함)는 서버 메모리에서 폐기된다.
