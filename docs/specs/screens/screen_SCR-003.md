# 연동 구성 등록·편집 폼 화면 정의

## 개요

- **화면 목적**: 연동 관리자가 서비스 A↔서비스 B 연동 1건을 코드 개발 없이 화면 구성만으로 등록·편집한다. 서비스 A 호출 주소·전달 파라미터 정의·사용자 동의 항목·서비스 B 전달 주소를 입력하며, 동의 항목·파라미터는 동적 반복 행으로 여러 개 입력한다. 전달 파라미터 중 하나를 **사용자 키값 파라미터로 명시 지정**해야 한다(필수·구성당 정확히 1개 — 지정한 파라미터가 연동이력·완료 확인/콜백의 사용자 키값 근거가 된다).
- **관련 PRD 요구사항**: [`../../prd/PRD.md`](../../prd/PRD.md) §수행 범위 "관리자 — 연동 구성 관리: 서비스 A의 호출 주소·전달 파라미터, 사용자 노출 동의 항목, 완료 시 서비스 B로 전달할 주소값 설정. 전달 파라미터 중 **사용자 키값 파라미터를 명시적으로 지정**할 수 있어야 한다"(`accountinterlockhub#33` 요구 추가). 서비스 근거 [SVC-001](../services/service_SVC-001.md), 검증 [FN-006](../functions/function_FN-006.md)(BIZ-001-07).
- **디자인**: [`design-system.md`](design-system.md) 관리자 셸·`TextField`·`Select`·`RepeatableRows`·`Toast` 인용.

### 레이아웃 구성

- 관리자 셸 + 본문 컨테이너(최대 폭 1120px). 상단 제목("연동 구성 등록" / "연동 구성 편집") + 하단 저장·취소 `Button`.
- 섹션 순서: (1) 기본 정보(구성 코드·구성명·활성 여부) → (2) 서비스 A 진입(호출 주소) → (3) 전달 파라미터 정의(`RepeatableRows`) → (4) 사용자 동의 항목(`RepeatableRows`, 각 행 라벨·설명·약관 컨텐츠·필수) → (5) 서비스 B 전달(전달 주소·HTTP 메서드).
- 동의 항목 각 반복 행은 라벨·설명·필수 입력에 더해 '약관 컨텐츠'(전체 약관 본문) 여러 줄 입력(textarea)을 둔다. 선택 입력이며(BIZ-001-06), 입력하면 사용자 동의 화면(SCR-005)에서 해당 항목에 [상세] 버튼이 노출된다(BIZ-002-05).
- 전달 파라미터 각 반복 행은 파라미터명·원천 키명·서비스 B 전달 여부에 더해 '사용자 키값' 단일 선택(라디오)을 둔다. 라디오는 전달 파라미터 반복 행 전체를 하나의 그룹으로 묶어 **정확히 1개 행만 선택**되며(구성당 정확히 1개 필수, BIZ-001-07), **반드시 1개를 선택해야 한다(none 불가)**. 지정은 **필수**이며(EXC-BIZ-09), 지정한 파라미터의 진입 값이 연동이력·완료 확인·완료 콜백의 사용자 키값으로 사용된다(BIZ-004). 어떤 행도 선택되지 않은 상태로 저장을 시도하면 검증 오류로 저장이 차단된다(서버 FN-006 step4 EX-BIZ-001 의 1차 방어). 라디오는 실재하는 파라미터 행에 귀속되므로 "지정 대상은 이미 입력된 파라미터 중 하나"가 구조적으로 보장된다.
- 반응형: desktop 2컬럼(라벨-입력), mobile 1컬럼. 반복 행은 mobile 에서 필드 세로 스택. 전달 파라미터 행의 '사용자 키값' 라디오는 행 헤더(또는 파라미터명 옆)에 배치해 어느 행이 지정됐는지 한눈에 보이게 한다.

| 사용자 키값 파라미터 지정(행) | radio(전 행 단일 선택 그룹) | MDL-101.parameters[].isUserKey | Y | `parameters.filter(p => p.isUserKey).length === 1`(정확히 1개 필수 — 라디오 그룹이 ≤1 을 구조적 보장, 미선택(0개)은 저장 차단, BIZ-001-07) · 지정 행은 실재 파라미터에 귀속(라디오 구조상 보장) | "사용자 키값 파라미터를 1개 지정해주세요."(미선택 시 저장 차단. 다중·미실재 지정은 라디오 구조상 발생 불가) |
| 전달 파라미터 목록 | RepeatableRows | MDL-101.parameters | Y | `parameters.length >= 1` (BIZ-001-01) | "전달 파라미터를 1개 이상 정의해주세요." |

> 모든 규칙은 서버단에서 재검증([FN-005](../functions/function_FN-005.md)·[FN-006](../functions/function_FN-006.md))되며 FE 검증은 1차 방어다. 고유성(BIZ-001-03)·개인정보 파라미터 경고(BIZ-001-05)는 서버 전담이며 FE 는 사전 판단하지 않는다.
> 사용자 키값 지정(isUserKey)은 서버 검증 FN-006 step 4(BIZ-001-07)와 정합한다 — 서버는 `parameters.filter(p => p.isUserKey).length !== 1`(0개 미지정 또는 2개 이상 다중) 또는 지정 파라미터 미실재를 422 EX-BIZ-001 로 거부한다. FE 는 라디오 단일 선택으로 다중 지정을, 라디오의 행 귀속으로 미실재 지정을 구조적으로 차단하고, **저장 제출 전 정확히 1개 선택 여부를 검증**해 미선택(0개) 저장을 차단한다(1차 방어 — 정상 흐름에서 이 422 미발생). 지정 행 삭제로 지정이 풀리면 저장을 막고 다른 행 재지정을 유도해 "지정 유지 상태의 파라미터 삭제 제출"·"미지정 저장"(서버 422)을 사전 차단한다. 지정 파라미터의 진입 값 누락은 등록·편집이 아닌 진입 시점(FN-016 EX-BIZ-007) 검증이므로 본 폼의 책임 밖이다.

### 구현 가이드

- 반복 행 추가·삭제·순서(order)는 클라이언트 상태로 관리하고 제출 시 배열로 직렬화한다. `order` 는 화면 순서로 부여한다(display_order 오름차순 정합).
- 개인정보 파라미터 경고는 저장을 차단하지 않는다(BIZ-001-05). FE 는 안내만 노출하고 서버가 감사 기록한다. 구성 코드는 편집 시 변경을 막아 고유성·참조 안정성을 지킨다(값 변경 필요 시 신규 등록 권장).
- 사용자 키값 지정은 전달 파라미터 반복 행 전체를 단일 라디오 그룹으로 묶어 상태(선택된 행 인덱스)를 관리한다. 지정은 필수라 **정확히 1개를 선택해야 저장할 수 있고**, 미선택(0개) 상태에서는 저장 버튼을 비활성화하거나 제출 시 검증 오류로 차단한다. 제출 시 선택 행만 `isUserKey=true`, 나머지는 `false` 로 직렬화한다(서버는 `user_key_param_id`=지정 행의 저장 id 로 참조 저장 — 값 복제 없음, MDL-101 매핑). 편집 진입 프리필 시 `user_key_param_id` 와 매칭되는 파라미터 행의 라디오를 선택 상태로 복원한다(MDL-101 ENT→도메인 매핑).
- 지정된 파라미터 행을 삭제하면 지정이 풀리므로(미선택), 저장을 막고 다른 파라미터 행 재지정을 유도한다 — 삭제된 파라미터를 참조하거나 미지정으로 제출되지 않게 한다(서버 RESTRICT·EX-BIZ-001 사전 방어, [FN-006](../functions/function_FN-006.md)). 지정 대상을 다른 파라미터로 바꾸려면 그 행의 라디오를 선택하면 되고(라디오 단일 선택), 지정을 0개로 되돌리는 조작은 제공하지 않는다(필수).
