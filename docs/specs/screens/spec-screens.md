# 화면 정의서 — AccountInterlockHub

본 문서는 AccountInterlockHub 전체 화면 정의의 개요와 하위 문서 목록을 정의한다. spec 단계 산출물(도메인 5순위)이며, 방향 근거는 [`../../prd/PRD.md`](../../prd/PRD.md), 정보 구조는 [`../../prd/ia/IA.md`](../../prd/ia/IA.md), 선행 도메인은 정책 [`../policies/spec-policies.md`](../policies/spec-policies.md)·서비스 [`../services/spec-services.md`](../services/spec-services.md)·데이터 [`../datas/spec-models.md`](../datas/spec-models.md)·기능 [`../functions/spec-functions.md`](../functions/spec-functions.md)다. 화면의 사용자 인터랙션은 후행 도메인(프로세스 PROC)의 진입 트리거가 되며, PROC 코드로 양방향 추적한다. 추적 일감: `accountinterlockhub#220`.

> **2026-07-11 `#214` 개정**: 핵심 연동 플로우를 **단일 암호화 JSON(encX/encY)·허브 복호화·수신처 서버-서버 전달·연동 추적 키(X 내부 필드)**로 재정의한 선행 도메인 개정을 화면에 반영했다. 사용자 흐름을 **발송처 링크 진입(접근 주소=발송처 판별)→생년월일 입력→동의→승인→(허브 복호화·전달)→완료/실패**로 개정하고, SCR-005 에 **생년월일 입력 필드**와 복호화 실패 재입력·링크 오류 상태를 신설했다. 관리자 폼(SCR-003)에서 **전달 파라미터·사용자 키값 exactly-one 지정(`#33`)을 폐기**했다. SCR 코드는 유지하고 의미를 갱신했다. 후행 도메인(processes/qa)은 v0.1.0 상태이며 트리거 PROC 는 예약 채번(실재 확인은 교차검증 시점)이다.

> **2026-07-12 `#215` 개정(목업 리뷰 보완)**: (1) 동의 대상 설명 문구(consentNotice·선택)를 SCR-003 입력·SCR-004 표시·SCR-005 상단 노출로 추가, (2) 복호화 실패 문구를 "사용자 정보가 일치하지 않습니다."로 고정(FE 형식 안내와 구분), (3) 전달 실패(EX-BIZ-004)를 SCR-006 대신 SCR-005 재시도 알림으로 이관, (4) 거부 배지 info(청록) 확정(변경 없음), (5) 접근 주소 고유 ID 관리자 직접 입력·중복 시 실패 확정.

## 대상 범위·스택

- **스택**: React(TypeScript) SPA + NestJS 정적 서빙. 관리자 웹·사용자 웹은 단일 App Service 로 배포되며 동일 디자인 시스템을 공유한다([`design-system.md`](design-system.md)).
- **화면 있는 IA**: 관리자(ADM-01·02·03)·사용자 연동(USR-01·02). 서비스 대면 API(API-01·02·03)와 배치(BAT-01·02)는 **화면이 없다**(서버 대면·백그라운드).
- **화면 없는 IA 처리**: API-01·02·03(처리상태·완료 확인·완료 콜백)은 발송처·수신처 대면 API, BAT-01·02(상태 저장·보관 배치)는 스케줄 백그라운드로, 사용자 접점 화면이 존재하지 않는다.

## 네비게이션 흐름 요약

관리자 흐름은 다음 순서로 이동한다.

1. SCR-001 로그인 성공 → SCR-002 목록.
2. SCR-002 목록에서 "발송처 접근 주소 등록" → SCR-003(신규), 행 클릭 → SCR-004 상세.
3. SCR-004 상세에서 "편집" → SCR-003(편집), 저장 성공 → SCR-004 로 복귀.
4. SCR-004 상세에서 "삭제" 확정 → SCR-002 목록으로 복귀.
5. 모든 관리자 화면에서 세션 만료(EX-AUTH-002) → SCR-001 로 재인증 유도.

사용자 흐름은 다음 순서로 이동한다.

1. 발송처 링크(`/interlock/entry/:accessAddressId?encX=…&encY=…`)로 SCR-005 이용 동의에 직접 진입 — 접속한 접근 주소(고유 ID)가 발송처 판별값이다(PROC-201).
2. SCR-005 에서 생년월일 입력·필수 동의 체크 후 승인 제출(PROC-202→내부 PROC-203) → 허브 복호화·수신처 B 전달.
3. 결과에 따라 SCR-006 로 이동: 완료·거부(취소)·링크 오류(발송처 문의). 단 복호화 실패(EX-SEC-006)·전달 실패(EX-BIZ-004)는 SCR-005 에 머무르며 각각 생년월일 재입력·승인 재제출(재승인)로 재시도한다(`#215`).
4. SCR-006 은 결과 유형만 표시하고 신규 화면 이동을 강제하지 않는다(Fallback 은 재진입 경로 안내).

## 데이터·마스킹 요약

- 관리자 화면(SCR-002·003·004)은 설정 데이터(MDL-101·102)만 표시하며 회원 키·처리 상태·연동 추적 키를 포함하지 않는다. 접근 주소 고유 ID·수신처 B 주소는 마스킹 대상이 아니다(설정 데이터, EXC-SEC-05).
- 접근 주소 구성(SCR-003·004)은 접근 주소 고유 ID(발송처 식별자)·수신처 B 전달 주소·전달 방식·동의 대상 설명 문구(선택)·동의 항목으로 구성한다. `#214` 로 **전달 파라미터 정의·사용자 키값 파라미터 exactly-one 지정(`#33`)·서비스 A 진입 주소를 폐기**했다 — 회원 키·연동 추적 키는 발송처가 전달 데이터 X 안에 담고 허브가 복호화 후 추출하므로(EXC-BIZ-14·DATA-002), 접근 주소 구성에 두지 않는다. 접근 주소 고유 ID 는 **관리자 직접 입력**·불변이며 등록 시 중복 검사로 유일성을 강제한다(BIZ-001-10·11). `#215` 로 동의 대상 설명 문구(consentNotice·선택)를 추가했다 — SCR-003 폼 입력·SCR-004 표시·SCR-005 상단 노출.
- 사용자 화면(SCR-005·006)은 회원 키·연동 추적 키·처리 상태 값을 표시하지 않는다(DATA-001 무저장·최소 노출). 동의 화면은 진입한 접근 주소 구성 소속 동의 항목만 노출한다(BIZ-002-01).
- **민감값 미노출·미로깅(SEC-005-06·SEC-006-06)**: SCR-005 는 encX·encY 를 URL 파라미터로 받되 화면에 렌더하지 않고 승인 제출 시 요청 본문으로만 보낸다. 생년월일은 입력 필드 값으로만 메모리에 두고 화면·로그·URL 에 남기지 않는다. 완료 페이지(SCR-006)는 복호화 원문·회원 키·추적 키를 포함하지 않는다(SEC-007-02).

## 담당자 확정 대기·보류 항목

선행 도메인의 확정 기본안을 승계하며, 화면 도메인에서 추가로 도출한 해석 항목을 포함한다(승인 전 잠정).

- **Q1 관리자 인증**: IP 제한 + 로그인 화면(SCR-001) 병행은 확정 기본안. 로그인 화면 도입 자체가 이 결정에 종속된다.
- **[신규 해석] 발송처 링크 URL 형태(EXC-SEC-08)**: SCR-005 진입 URL 을 `허브 접근 주소(고유 ID) + encX·encY 파라미터`로 정의하고, 브라우저 라우트를 `/interlock/entry/:accessAddressId?encX=…&encY=…`(접근 주소=경로 판별값)로 기본안화했다. 접근 주소의 경로/쿼리 배치, encX·encY 의 쿼리 vs 본문 전달, GET 파라미터 길이 한계 대응(경로·본문 대체)은 build 확정 대상이다. FE 는 접속한 접근 주소로 발송처를 판별한다(별도 발송처 선택 UI 없음).
- **[신규 해석] 생년월일 입력**: SCR-005 에 생년월일(yyMMdd 6자리) 입력 필드를 신설했다. FE 는 형식(6자리 숫자·월/일 범위)만 검증하고 정오는 서버 복호화 성공/실패로 귀결한다(AUTH-004-01). 마스킹 여부(입력 중 표시 방식)는 6자리 본인확인 특성상 필드 내 표시를 기본안으로 두되, 로그·URL·제출 후 화면 노출은 금지한다(build 상세 확정).
- **[신규 해석] 복호화 실패 재입력 UX(BR-204·AUTH-004-02)**: 복호화 실패(EX-SEC-006)는 SCR-005 에 머무르며 생년월일 재입력·재제출을 허용한다(하드 잠금 없음). 암호 파라미터 형식 오류(EX-SEC-007)·추적 키 필드 누락(EX-BIZ-008)은 SCR-006 링크 오류로 종료 안내한다(재입력 불가, EXC-BIZ-13).
- **[확정(`#215`)] SCR-006 결과 전달·전달 실패 UX**: 결과를 SCR-005 승인 제출 응답의 클라이언트 상태로 전달(추가 API 없음)하고 경로에 민감값을 싣지 않는다(`/interlock/result`). 새로고침·직접 진입 시 Fallback 안내. **전달 실패(EX-BIZ-004)는 SCR-006 로 이동하지 않고 SCR-005 에서 "동의 처리에 실패했습니다. 다시 시도해주세요." 알림 + 승인 재제출(재승인)로 처리**한다(SCR-006 은 완료·거부·링크 오류·Fallback 만 표시). Fallback 재진입 액션 제공 여부만 build 확정 대상이다.
- **[확정(`#215`)] 접근 주소 고유 ID 입력 방식**: 관리자가 등록 시 **직접 입력**하고(자동 생성 아님) 등록 시 중복 검사로 유일성을 강제한다(중복 값이면 409 EX-BIZ-002·BIZ-001-10). 편집 시 읽기전용·불변(BIZ-001-11). 허용 문자·형식 세부 규칙만 build 확정.
- **[신규 해석] 목록 페이지네이션·정렬 UI**: MVP 는 활성 필터·생성일 정렬 기본. 페이지네이션 규약은 build·[FN-015](../functions/function_FN-015.md) 확정.
- **활성/비활성 전환**: SVC-002 파생 운영 기능. 목록·상세 양쪽에서 전환 가능하도록 화면 배치했으며 기본 활성 여부·상태 모델은 확정 대기.
- **(폐기)** 구 요청 키값(UUID) 진입 경로(`/consent/:requestKey`)·사용자 키값 파라미터 지정(`#33`) UI 는 `#214` 로 폐기됐다 — 진입은 접근 주소 고유 ID + encX·encY 로, 추적 키는 X 내부 필드로 대체됐다.
