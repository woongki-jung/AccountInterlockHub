# 방향 정의서 (PRD) — AccountInterlockHub

본 문서는 AccountInterlockHub 프로젝트의 방향(무엇을·왜)을 정의하는 directing 단계 산출물이다. 상세 요구사항·사양은 이후 spec 단계가 본 문서를 입력으로 작성한다. 추적 일감: `accountinterlockhub#23`(최초 방향), `accountinterlockhub#214`(핵심 연동 정책 변경 — 암호화 연동 플로우).

- 정보 구조(IA): [`ia/IA.md`](ia/IA.md)
- 개발사양: [`devspec/infra.md`](devspec/infra.md) · [`devspec/database.md`](devspec/database.md) · [`devspec/external-apis.md`](devspec/external-apis.md)

## 제품 정의 (한 줄)

서로 다른 서비스 사이의 **사용자 연동을, 관리자가 발송처별로 구성**하고, **발송처가 암호화해 전달한 값을 사용자 동의·본인확인(생년월일)으로 복원**해 수신처로 중개하는 플랫폼. 허브는 사용자 정보를 **지속 보관하지 않는다**(연동 실행 시점의 일시 복호화만 수행).

## 목적·배경

- 개인정보를 **지속 보관하지 않고** 운영되는 B2B 솔루션(이하 '서비스 A')의 **회원 식별 정보**를 다양한 외부 서비스와 연동해야 하는 요구가 반복적으로 발생한다.
- 매번 개별 개발로 대응하면 비효율이 크다. **연동 구성을 플랫폼화**해 반복 작업을 줄이고 새 연동을 빠르게 붙일 수 있게 한다.
- 전달 값의 기밀성을 높이기 위해, 입력을 **단일 암호화 JSON**으로 받고 **사용자 본인확인(생년월일)**을 복호화 요소로 결합하는 방식으로 핵심 연동 플로우를 개정한다(`#214`, 2026-07-11).

## 목표

- 서비스 A의 회원을 외부 서비스(이하 '서비스 B' 및 그 외)와 연동하는 과정을, **코드 개발 없이 관리자 화면의 구성만으로** 추가·운영할 수 있게 한다.
- 연동 중개 과정에서 **사용자 정보를 보관하지 않고**(연동 실행 시 복호화한 원문 포함 미저장), 처리 추적에 필요한 최소 상태만 한시적으로 보관한다.
- 전달 값을 **암호화 전송 + 사용자 본인확인(생년월일)**으로 보호해, 링크 전달 경로만으로 원문이 곧바로 노출되지 않게 한다(강도 한계·후속 보완은 §시스템 제약사항 참조).
- 새 연동 구성에 드는 시간을 낮추고, 연동 처리의 성공률을 안정적으로 유지한다.

## 서비스 이용 대상 (2026-07-11 재정의 — `#214`)

1. **발송처 (서비스 A 운용사 · 연동 요청 주체)** — 사용자 키를 외부와 연동하려는 주체. 전달 데이터(X)를 생성·이중 암호화하고, 허브가 발송처별로 발급한 접근 주소에 암호값을 붙여 사용자를 유도한다.
2. **수신처 (서비스 B 운용사 · 연동 대상)** — 허브가 전달한 사용자 키로 회원 여부를 판단하고 자체 서비스에 연결한다. 처리완료를 허브에 콜백으로 통지한다.
3. **사용자 (서비스 A의 고객)** — 발송처가 전달한 링크로 접근 페이지에 진입해, **생년월일을 입력**하고 동의 항목 확인 후 연동을 승인한다.
4. **연동 관리자 (허브 운영자)** — 허브 관리자 화면에서 **발송처별 접근 주소(고유 ID)**·수신처 전달 주소·동의 항목을 등록·운영한다.

## 제공 가치

- **구성만으로 연동**: 발송처별 접근 주소·수신처 전달 주소·동의 항목을 관리자 화면에서 설정하는 것으로 새 연동을 만든다.
- **암호화 전송 + 본인확인**: 발송처가 전달 데이터를 이중 암호화(encX/encY)해 전달하고, 사용자의 **생년월일 입력으로만 복원**된다. 허브는 값을 저장하지 않고 연동 실행 시점에만 복호화한다.
- **동의 기반 투명성**: 사용자가 어떤 항목에 동의하는지 화면으로 확인하고 스스로 승인/거부를 결정한다.
- **처리 추적**: 발송처가 연동 요청의 처리·결과 확인 상태와 수신처의 처리완료 여부를 조회할 수 있고, 수신처는 처리 완료를 콜백으로 통지한다.

## 성과 지표

- **서비스 사용량** — 연동 처리 요청 건수·활성 연동 구성(발송처 접근 주소) 수.
- **연동 성공률** — 전체 연동 요청 대비 정상 복호화·전달·완료 비율.

## 수행 범위

### 포함 (MVP)

1. **관리자 — 발송처 접근 주소 구성**: 발송처별 **접근 주소를 생성(고유 ID 부여)**하고, 그 주소에 **수신처(서비스 B) 전달 주소**와 **사용자 노출 동의 항목**을 설정한다. 접근한 주소(고유 ID)가 **요청처(발송처) 구분값**이 된다. *(기존 '전달 파라미터 + 사용자 키값 파라미터 exactly-one 지정'(`#33`)은 폐기·대체 — 입력이 단일 암호화 JSON 으로 바뀌고, 추적 키는 전달 데이터(X) 내부 필드가 대신한다.)*
2. **암호화 연동 규약 (발송처 수행·허브 정의)**: 전달 데이터 X(JSON) 생성 → `encX = AES256(X, 키=발송처키+생년월일yyMMdd)` → `encY = AES(그 키, 키=생년월일)`. 발송처가 허브 접근 주소에 encX·encY 를 파라미터로 붙여 URL 을 만들고 사용자를 유도한다. 허브는 이 규약을 정의·문서화하고 복호화로 대응한다(암호화 자체는 발송처 수행).
3. **사용자 — 접근·동의·승인 페이지**: 발송처 링크로 진입, **생년월일 입력**, 구성된 동의 항목 노출·동의 체크 후 연동 승인.
4. **연동 실행 (허브 복호화·전달)**: 승인 시 허브가 생년월일로 `encY→키→encX→X` 복호화 → X 를 접근 주소에 설정된 수신처(B) 주소로 전달 → 성공 시 사용자에게 **연동 완료 페이지** 노출.
5. **처리상태 확인 API**: 발송처가 **연동 추적 키**(X 내부 필드) 기준으로 처리·결과 확인 상태를 조회.
6. **연동 완료 확인·완료 콜백 API**: 수신처가 전달받은 추적 키 기준으로 완료를 허브에 통지(콜백)하고, 발송처가 그 키 기준으로 수신처의 처리완료 여부를 확인.
7. **상태 보관·정리**: 추적 키별 최소 상태 저장 + 일 단위 배치로 보관 기간 경과분 삭제.
8. **연동이력 저장**: 연동 추적 키 기준으로 연동 요청~완료 콜백까지의 이력을 저장하고, 6의 API 근거 데이터로 사용.

담당자 확정: 위 8개가 초기 구현 범위다. 보안 강화(링크 만료·1회성·발송처 서명 검증)는 이번 범위에서 제외하고 후속 보완한다(§미결·확인 필요).

### 제외 (허브 책임 밖)

- **사용자 정보·복호화 원문의 지속 보관** — 허브는 사용자 정보와 복호화한 전달 데이터(X) 원문을 저장하지 않는다(처리 추적용 최소 상태와 연동이력의 추적 키값은 예외).
- **암호화 키(발송처키)의 생성·배포·관리** — 발송처키의 생성·보관·발송처↔허브 공유 방식은 발송처·운영 정책 소관이다. 허브는 복호화 규약만 정의한다.
- **전달 데이터(X)의 업무적 유효성 판단** — 허브는 규약대로 복호화·중개하되 X 내용의 업무 신뢰성은 판단하지 않는다.

## 우선순위

구조적 의존 순서에 따른다.

1. **관리자(발송처 접근 주소 구성)** — 최우선. 여기서 설정된 접근 주소·수신처 주소·동의 항목이 있어야 사용자 페이지·연동 실행이 성립한다.
2. **사용자(접근·생년월일·동의·승인) + 연동 실행(복호화·전달)** — 관리자 구성 위에서 동작.
3. **처리상태·완료 확인·완료 콜백 API 및 보관 배치** — 연동 실행 이후의 추적·통지·정리.

## 시스템 제약사항

- **기술 스택**: 관리자·사용자 웹 모두 백엔드 NestJS(API) + 프런트엔드 ReactJS, 기본 언어 TypeScript. 상세 [`devspec/infra.md`](devspec/infra.md).
- **배포 형태**: ReactJS 정적 빌드를 NestJS가 정적 콘텐츠로 서빙하여, 운영 환경에서 **하나의 Azure App Service**로 배포한다.
- **관리자 접근 제어**: 관리자 경로는 NestJS 단에서 IP 기반 접속 제한을 적용한다.
- **데이터 원칙**: 허브는 사용자 정보·복호화 원문(X)을 **지속 보관하지 않는다**. 복호화는 **연동 실행 시점에 메모리 상에서만** 수행하고 전달 후 폐기한다. 저장은 처리 추적용 추적 키별 상태(처리 성공 여부·결과 확인 여부·처리일시·결과 확인일시)와 **연동 추적 키 기준 연동이력**으로 한정한다. 보관분은 일 배치로 삭제한다(처리상태는 결과 확인 완료 후 3개월, 연동이력은 동일 규칙 준용 기본안). 상세 [`devspec/database.md`](devspec/database.md).
- **암호화·복호화 규약**: 전달 데이터는 발송처가 이중 암호화(`encX`=AES256(X, 발송처키+생년월일), `encY`=AES(그 키, 생년월일))해 전달하고, 허브는 사용자 생년월일로 `encY→키→encX→X` 순 복호화한다. IV 는 키 앞자리를 필요 길이만큼 절단해 쓰고 부족분은 `_` 로 채운다. 블록 모드·키 32바이트 정규화·패딩 등 세부는 spec 에서 확정한다. 상세 [`devspec/external-apis.md`](devspec/external-apis.md).
- **보안 강도 한계(알려진 제약)**: `encX`·`encY` 가 URL 파라미터로 노출되고 생년월일(yyMMdd)은 경우의 수가 작아, 링크와 암호값을 확보한 자가 오프라인 전수대입으로 원문을 복원할 수 있다. 초기 범위는 이 수준을 수용하며, 링크 만료(TTL)·1회성 사용·발송처 서명 검증 등 강화는 **후속 보완 항목**으로 둔다(2026-07-11 담당자 '추후 보완' 지시 — `#214`).
- **데이터베이스**: PostgreSQL. 운영은 Azure의 PostgreSQL 서비스(Azure Database for PostgreSQL), 개발·로컬은 별도 PostgreSQL 서버.
- **신뢰 위임**: 발송처키의 관리와 전달 데이터의 업무적 유효성은 발송처(서비스 A)가 책임진다. 허브는 규약대로 복호화·중개한다.

## 프로그램 구성·기술스택·검증 방법

서비스를 이루는 프로그램과 각 기술스택·검증 방법(qa 케이스)이다. qa 단계가 이 표로 프로그램별 검증 케이스를 선택한다([`qa-execution.md`](../../ai/strategies/qa-execution.md) §케이스 선택).

| 프로그램 | 기술스택 | 검증 방법(qa 케이스) |
|---|---|---|
| 사용자 웹 (접근·생년월일·동의·승인·완료) | ReactJS(TS) + NestJS 정적 서빙 | 웹·SPA UI |
| 관리자 웹 (발송처 접근 주소 구성) | ReactJS(TS) + NestJS 정적 서빙 | 웹·SPA UI |
| 백엔드 API 서버 (복호화·전달·상태/콜백 API) | NestJS(TypeScript) | API·서버 |
| 상태 보관 배치 | NestJS 스케줄 작업 | API·서버(배치 동작 검증) |

- 사용자 웹·관리자 웹·API·배치는 **단일 App Service**로 배포되는 하나의 애플리케이션 안에 함께 구성된다(배포 단위는 하나, 논리 프로그램은 위 4종).

## 미결·확인 필요

- **관리자 인증 방식**: 관리자 접근 제어로 IP 기반 제한은 확정. IP 제한 외 로그인 인증(계정·비밀번호 등)을 둘지는 spec 단계에서 확정한다.
- **연동이력 보관 기간**: 기본안은 처리상태와 동일 규칙(결과 확인 완료 후 3개월) 준용. 사용량·성공률 통계 목적의 장기 보관 여부는 담당자 확정 대기(`#33`).
- **보안 강화(후속 보완)**: 링크 만료(TTL)·1회성 사용·발송처 서명 검증 등 — 초기 범위 제외, 후속 반영(`#214`).
- **암호화 규약 세부**: 블록 모드·키 32바이트 정규화·패딩·IV 규칙 검증·`encX` 전달 형식(GET 파라미터 길이 한계)은 spec 단계에서 확정한다.
- **실패 처리 UX**: 복호화 실패(잘못된 생년월일)·수신처 전달 실패 시 사용자 화면·재시도 방식은 spec 단계에서 확정한다.
