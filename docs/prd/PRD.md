# 방향 정의서 (PRD) — AccountInterlockHub

본 문서는 AccountInterlockHub 프로젝트의 방향(무엇을·왜)을 정의하는 directing 단계 산출물이다. 상세 요구사항·사양은 이후 spec 단계가 본 문서를 입력으로 작성한다. 추적 일감: `accountinterlockhub#23`.

- 정보 구조(IA): [`ia/IA.md`](ia/IA.md)
- 개발사양: [`devspec/infra.md`](devspec/infra.md) · [`devspec/database.md`](devspec/database.md) · [`devspec/external-apis.md`](devspec/external-apis.md)

## 제품 정의 (한 줄)

서로 다른 서비스 사이의 **사용자 연동을, 관리자가 건별로 구성**하고 **사용자 동의를 거쳐 중개**하는 플랫폼. 중개 과정에서 사용자 정보를 저장하지 않는 것을 원칙으로 한다.

## 목적·배경

- 개인정보를 직접 다루지 않고 운영되는 B2B 솔루션(이하 '서비스 A')의 **회원 식별 정보**를 다양한 외부 서비스와 연동해야 하는 요구가 반복적으로 발생한다.
- 매번 개별 개발로 대응하면 비효율이 크다. **연동 구성을 플랫폼화**해 반복 작업을 줄이고 새 연동을 빠르게 붙일 수 있게 한다.

## 목표

- 서비스 A의 회원을 외부 서비스(이하 '서비스 B' 및 그 외)와 연동하는 과정을, **코드 개발 없이 관리자 화면의 구성만으로** 추가·운영할 수 있게 한다.
- 연동 중개 과정에서 **사용자 정보를 보관하지 않고**, 처리 추적에 필요한 최소 상태만 한시적으로 보관한다.
- 새 연동 구성에 드는 시간을 낮추고, 연동 처리의 성공률을 안정적으로 유지한다.

## 서비스 이용 대상

1. **서비스 A (연동 요청 주체)** — 회원을 외부 서비스와 연동하려는 B2B 솔루션. 회원 고유 키를 허브로 전달한다.
2. **최종 사용자 (서비스 A의 회원)** — 연동 진행 시 허브의 동의 화면을 통해 동의/거부를 결정한다.
3. **서비스 B 및 기타 외부 서비스 (연동 대상)** — 허브가 전달한 사용자 키로 회원 여부를 판단하고 서비스를 제공한다.
4. **연동 관리자** — 허브 관리자 화면에서 연동 구성을 등록·운영한다.

## 제공 가치

- **구성만으로 연동**: 연동 대상·전달 파라미터·동의 항목·전달 주소를 관리자 화면에서 설정하는 것으로 새 연동을 만든다.
- **개인정보 부담 최소화**: 허브는 사용자 정보를 저장하지 않고, 값의 신뢰성 판단(암·복호화 등)은 전송 주체에 위임한다.
- **동의 기반 투명성**: 사용자가 어떤 항목에 동의하는지 화면으로 확인하고 스스로 결정한다.
- **처리 추적**: 서비스 A가 연동 요청의 처리·결과 확인 상태와 서비스 B의 처리완료 여부를 조회할 수 있고, 서비스 B는 처리 완료를 콜백으로 통지한다.

## 성과 지표

- **서비스 사용량** — 연동 처리 요청 건수·활성 연동 구성 수.
- **연동 성공률** — 전체 연동 요청 대비 정상 전달·완료 비율.

## 수행 범위

### 포함 (MVP)

1. **관리자 — 연동 구성 관리**: 서비스 A의 호출 주소·전달 파라미터, 사용자 노출 동의 항목, 완료 시 서비스 B로 전달할 주소값 설정. 전달 파라미터 중 **사용자 키값 파라미터를 정확히 1개 반드시 지정**해야 한다(필수·exactly-one — 2026-07-08 담당자 확정, `accountinterlockhub#33`. 지정이 0개이거나 2개 이상이면 저장을 차단).
2. **사용자 — 이용 동의 페이지**: 구성된 동의 항목 노출, 동의/거부 처리.
3. **연동 실행**: 서비스 A의 데이터(회원 키·파라미터) 수신 → 동의 완료 시 서비스 B로 요청 전달.
4. **처리상태 확인 API**: 서비스 A가 요청 키값 기준으로 처리·결과 확인 상태를 조회.
5. **상태 보관·정리**: 요청 키값별 최소 상태 저장 + 일 단위 배치로 보관 기간 경과분 삭제.
6. **연동 완료 확인·완료 콜백 API**: 서비스 B가 정보연계 처리를 완료하고 전달받은 키값 기준으로 완료를 허브에 통지(콜백)하는 API, 그리고 서비스 A가 그 키값 기준으로 서비스 B의 처리완료 여부를 확인하는 API.
7. **연동이력 저장**: 지정된 사용자 키값 파라미터 값을 기준으로 연동 요청~완료 콜백까지의 이력을 저장하고, 6의 API 제공 근거 데이터로 사용.

담당자 확인 결과 **1차에서 제외하는 기능은 없다**. 위 7개가 초기 구현 범위다(6·7은 2026-07-06 담당자 지시로 추가 — `accountinterlockhub#33`).

### 제외 (허브 책임 밖)

- 회원 키의 암·복호화·해시 등 **값 신뢰성 처리** — 전송 주체(서비스 A)가 전담. 허브는 값의 신뢰도를 판단하지 않는다(그대로 중개).
- **사용자 정보의 지속 보관** — 허브는 사용자 정보를 저장하지 않는다(처리 추적용 최소 상태와 연동이력의 지정 사용자 키값은 예외).

## 우선순위

구조적 의존 순서에 따른다.

1. **관리자(연동 구성)** — 최우선. 여기서 설정된 정보가 있어야 사용자 페이지·연동 실행이 성립한다.
2. **사용자(동의·연동 실행)** — 관리자 구성 위에서 동작.
3. **처리상태·완료 확인·완료 콜백 API 및 보관 배치** — 연동 실행 이후의 추적·통지·정리.

## 시스템 제약사항

- **기술 스택**: 관리자·사용자 웹 모두 백엔드 NestJS(API) + 프런트엔드 ReactJS, 기본 언어 TypeScript. 상세 [`devspec/infra.md`](devspec/infra.md).
- **배포 형태**: ReactJS 정적 빌드를 NestJS가 정적 콘텐츠로 서빙하여, 운영 환경에서 **하나의 Azure App Service**로 배포한다.
- **관리자 접근 제어**: 관리자 경로는 NestJS 단에서 IP 기반 접속 제한을 적용한다.
- **데이터 원칙**: 허브는 사용자 정보를 보관하지 않는다. 저장은 처리 추적용 요청 키값별 상태(처리 성공 여부·결과 확인 여부·처리일시·결과 확인일시)와, 연동 구성에서 지정한 **사용자 키값(불투명 값) 기준 연동이력**으로 한정한다 — 키값의 신뢰성 처리는 전송 주체 책임이며 허브는 개인정보 원문을 다루지 않는다. 보관분은 일 배치로 삭제한다(처리상태는 결과 확인 완료 후 3개월, 연동이력은 동일 규칙 준용 기본안). 상세 [`devspec/database.md`](devspec/database.md).
- **데이터베이스**: PostgreSQL. 운영은 Azure의 PostgreSQL 서비스(Azure Database for PostgreSQL), 개발·로컬은 별도 PostgreSQL 서버.
- **신뢰 위임**: 회원 키의 무결성·위변조 방지는 전송 주체(서비스 A)가 책임진다.

## 프로그램 구성·기술스택·검증 방법

서비스를 이루는 프로그램과 각 기술스택·검증 방법(qa 케이스)이다. qa 단계가 이 표로 프로그램별 검증 케이스를 선택한다([`qa-execution.md`](../../ai/strategies/qa-execution.md) §케이스 선택).

| 프로그램 | 기술스택 | 검증 방법(qa 케이스) |
|---|---|---|
| 사용자 웹 (동의·연동) | ReactJS(TS) + NestJS 정적 서빙 | 웹·SPA UI |
| 관리자 웹 (연동 구성) | ReactJS(TS) + NestJS 정적 서빙 | 웹·SPA UI |
| 백엔드 API 서버 | NestJS(TypeScript) | API·서버 |
| 상태 보관 배치 | NestJS 스케줄 작업 | API·서버(배치 동작 검증) |

- 사용자 웹·관리자 웹·API·배치는 **단일 App Service**로 배포되는 하나의 애플리케이션 안에 함께 구성된다(배포 단위는 하나, 논리 프로그램은 위 4종).

## 미결·확인 필요

- **관리자 인증 방식**: 관리자 접근 제어로 IP 기반 제한은 확정. IP 제한 외 로그인 인증(계정·비밀번호 등)을 둘지는 spec 단계에서 확정한다.
- **연동이력 보관 기간**: 기본안은 처리상태와 동일 규칙(결과 확인 완료 후 3개월) 준용. 사용량·성공률 통계 목적의 장기 보관 여부는 담당자 확정 대기(`accountinterlockhub#33`).
- **연동이력의 사용자 키값 저장 형태**: 서비스 A가 전달한 원문(불투명 문자열) 그대로 저장하는 전제 — 담당자 확인 대기(`accountinterlockhub#33`).
- 위 항목 외 방향 정의서 필수 항목은 담당자 합의로 충족됐다.
