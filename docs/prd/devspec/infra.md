# 개발사양 — 설치·배포·인프라

AccountInterlockHub의 인프라·배포 고정값이다. spec·build·qa가 단일 출처로 참조한다. 방향 근거는 [`../PRD.md`](../PRD.md) §시스템 제약사항.

## 애플리케이션 구성

- 하나의 애플리케이션 안에 **사용자 웹·관리자 웹·백엔드 API·상태 보관 배치**가 함께 구성된다(논리 프로그램 4종, 배포 단위 1개).
- 프런트엔드(ReactJS, TypeScript)는 정적 빌드 산출물로 만들어지고, 백엔드(NestJS, TypeScript)가 이 정적 산출물을 **정적 콘텐츠로 서빙**한다. 사용자·관리자 화면 모두 같은 방식으로 제공된다.

## 배포 형태

- 운영 환경은 **하나의 Azure App Service**로 배포한다. 별도의 프런트엔드 호스팅을 두지 않는다(NestJS가 API와 정적 화면을 함께 제공).
- 구체 App Service 이름·리소스 그룹 등 배포 대상 식별자는 build/배포 준비 시점에 확정하고 [`../../../CLAUDE.env.md`](../../../CLAUDE.env.md)에 변수로 추가한다.

## 환경 구분

1. **운영(prod)** — Azure App Service + Azure의 PostgreSQL 서비스(Azure Database for PostgreSQL).
2. **개발·로컬(dev/local)** — 별도로 구축한 PostgreSQL 서버를 사용한다(Azure 미사용). 애플리케이션은 로컬에서 기동한다.

- 환경별 접속 정보(연결 문자열 등 비밀 값)는 커밋하지 않으며, 비밀 관리 방식은 build 단계에서 확정한다.

## 접근 제어

- **관리자 경로**는 NestJS 단에서 **IP 기반 접속 제한**을 적용한다. 허용 IP 목록은 운영 구성 값으로 관리한다.
- 사용자 경로(동의·연동 실행)는 서비스 A의 진입 흐름을 통해 접근한다.

## 미결

- 관리자 IP 제한 외 로그인 인증 여부는 spec 단계에서 확정한다([`../PRD.md`](../PRD.md) §미결·확인 필요).
- 비밀 값(연결 문자열·허용 IP) 관리 수단, 로깅·모니터링 구성은 build 단계 상세.
