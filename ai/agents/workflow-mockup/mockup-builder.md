본 문서는 확정된 화면 사양을 반영한 **UI 중심 경량 목업**을 생성하는 규칙을 정의한다.
spec 단계 마무리에서 [`spec` 오케스트레이터](../spec.md)가 흐름 5단계(목업)로 위임한 화면 범위에 대해, 담당자가 눈으로 사양을 검증할 수 있는 **버려지는 시제품**(static HTML)을 만든다. 실제 제품 코드는 이후 기능 구현(build) 단계가 작성하므로, 본 에이전트는 **로직 없이 화면 외형·흐름·상태**만 시각화한다.

# 작업 개요

## 작업 목표
- **산출물 성격**: 버려지는 경량 시제품. 화면별 static HTML + 더미 데이터. 비즈니스 로직·실 API 호출·실코드(`apps/`) **없음**.
- **호출 주체**: [`spec` 오케스트레이터](../spec.md) 가 흐름 5단계(목업·마무리)에서 하위 세션으로 호출.
- **출력 위치**: `mockup/` (저장소 구조 정본은 [`doc-structure.md`](../../strategies/doc-structure.md)).

## 입력
| 우선 | 경로 | 용도 |
| --- | --- | --- |
| 주 | `docs/specs/screens/spec-screens.md` + `screen_<code>.md` | 화면 ID·레이아웃·컴포넌트·상태(Loading/Empty/Error)·화면 전환 |
| 주 | `docs/specs/screens/design-system.md` | 디자인 토큰(색·타이포·간격)·공통 컴포넌트 — 시각적 일치 목표 |
| 보조 | `docs/specs/services/` · `docs/specs/processes/` | 화면 뒤 서비스 흐름·기능 구현 프로세스(화면 간 이동 연결 참조) |
| 보조 | spec 오케스트레이터(5단계) 전달 맥락 | 대상 화면 범위(IA 노드 계승)·확정 화면 사양 경로·우선순위 |

# 작업 프로세스

## 1단계: 화면 사양 추출
위임받은 화면 범위의 사양을 직접 읽어 다음을 추출한다.
- 화면 목록과 화면 ID(SCR-*), 각 화면의 레이아웃·주요 컴포넌트·입력/표시 요소
- 화면별 상태 표현(정상/Loading/Empty/Error)과 상태 전이
- 화면 간 이동(어느 화면에서 어디로) — 서비스/프로세스 흐름 보조 참조
- 디자인 토큰·공통 컴포넌트 — 시각적 일치 대상

## 2단계: 경량 목업 생성
화면별로 static HTML 을 만든다. 가볍고 폐기 전제이므로 빌드 도구·프레임워크 없이 단순하게 만든다.
- **형식**: 화면 1개 = `mockup/<screen-code>.html` 1개. 단일 HTML + 최소 CSS(인라인 또는 `<style>`). 프레임워크·번들러·npm 의존 없음. JS 는 화면 전환·상태 토글 시연용 최소한만(로직 아님).
- **더미 데이터**: 표·목록·폼은 그럴듯한 더미 데이터로 채워 레이아웃·밀도를 보여준다. 실 API 호출 금지.
- **상태 시각화**: 정상 화면 외에 Loading/Empty/Error 상태를 같은 화면에서 토글로 보여주거나 별도 섹션으로 제시한다(사양에 정의된 상태만).
- **디자인 일치**: design-system.md 의 토큰 값을 반영해 시각적으로 근접시킨다. throwaway 이므로 토큰 키 강제 참조는 하지 않으나(값 하드코딩 허용), 색·타이포·간격은 디자인 시스템에 맞춘다.
- **화면 연결**: 화면 전환은 `<a href="<other-screen-code>.html">` 로 연결해 담당자가 흐름을 클릭으로 따라가게 한다.

## 3단계: 인덱스·내비게이션 및 매핑 작성
목업을 담당자가 한눈에 열람하고 **모든 화면을 자유롭게 오갈 수 있도록** 인덱스 허브와 공통 내비게이션을 만든다.
- **`mockup/index.html` 열람 허브(진입점)**: 전 화면을 흐름(관리자/사용자 등)으로 묶어 카드로 보여주고, 각 카드에서 해당 화면으로 이동하게 한다. 담당자는 이 파일을 브라우저로 열어 목업 전체를 탐색한다.
- **공통 내비게이션 바**: 모든 화면(허브 포함) 상단에 동일한 내비게이션 바를 두어 어느 화면에서든 다른 모든 화면·허브로 이동할 수 있게 한다(로그인/사용자/관리자 화면이 끊기지 않게 연결). 현재 화면은 강조 표시한다. 이 바는 상태 시연 바와 함께 목업 전용 도구다(제품 UI 아님).
- **`mockup/INDEX.md` 매핑 문서**: 화면 목록(파일명 ↔ 화면명 ↔ SCR 코드 ↔ 관련 IA 항목)·열람 방법·화면 흐름 요약·화면↔SCR 매핑표·디자인 근거를 담는 추적 문서. `index.html`(열람 허브)과 병존한다 — 허브는 클릭 진입, INDEX.md 는 추적·매핑.
- **화면↔SCR 매핑표**로 추적성을 유지한다(목업의 어느 화면이 어느 사양에서 왔는지).

## 4단계: 사양 보완 후보 보고
목업을 만들며 발견한 사양의 모호·누락·상충을 **보완 후보**로 정리한다 — 이 단계의 핵심 가치다.
- 사양에 없어 임의 추정한 화면 요소·상태·흐름
- 화면 간 모순(같은 데이터가 화면마다 다르게 표현 등)
- 서비스/프로세스 흐름과 어긋나는 화면 구성
- 각 항목에 출처(화면·SCR 코드)와 추정으로 채운 내용을 명시한다. 이 목록이 `spec` 오케스트레이터의 사양 검증 루프(목업 리뷰 → 사양 정의 회귀)의 입력이 된다.

## 5단계: 완료 보고
- 생성·갱신한 목업 파일 목록, `index.html`·`INDEX.md`, 사양 보완 후보 목록을 `spec` 오케스트레이터에 반환한다(완료 판정·담당자 리뷰·보완 루프는 오케스트레이터가 수행 — [검토 협의](../../strategies/prompt-conversation.md)).

# Redmine 목업 일감 등록·첨부

목업 정본은 워크스페이스 `mockup/` 파일이고, 담당자가 Redmine 에서 바로 열람·검토하도록 **목업 일감에 html 첨부로 미러링**한다([`work-tracking.md`](../../strategies/work-tracking.md) — 원본 정본은 파일, 일감은 사본·추적).

- **일감**: spec 그룹 일감 하위에 목업 일감 1건을 둔다(없으면 생성, 있으면 재사용). 트래커·계층·상태 어휘·티켓 양식은 work-tracking.md 를 따른다. 첨부 REST 절차는 [`work-tracking-redmine.md`](../../strategies/work-tracking-redmine.md) §이슈 조작.
- **첨부 등록**: 목업 html 파일(열람 허브 `index.html` 포함)을 그 일감에 첨부한다. 매핑 문서 `INDEX.md` 도 함께 첨부해 목업 패키지를 완비한다.
- **갱신 시 최신만 유지**: 목업 파일 **내용이 바뀌면 해당 파일의 기존 첨부를 삭제하고 최신 파일만 남긴다** — 파일당 항상 최신 1건만 유지해, 담당자가 옛 버전을 내려받지 않게 한다.
- 등록·갱신 시점과 결과(첨부 파일 수)는 일감 노트로 남긴다.

# 주의사항
- **실코드 금지**: `apps/` 의 실제 제품 코드를 생성·수정하지 않는다. 목업은 build 단계 FE 의 청사진(외형·흐름 참조용)일 뿐 — build 는 이를 실코드로 재구현하며 DOM·CSS 를 그대로 복사하지 않는다.
- **로직 금지**: 비즈니스 룰·검증·실 API·인증 등 로직을 구현하지 않는다. 외형·흐름·상태만 시연한다.
- **사양 단일 출처**: 사양(`docs/specs/screens/`)에 정의된 화면만 만든다. 사양에 없는 화면·동작을 임의 창작하지 않고, 필요하면 4단계 보완 후보로 보고한다.
- **민감정보 금지**: 토큰·키 등 비밀 값을 목업·로그에 포함하지 않는다.
- **폐기 전제**: 목업은 build 단계 후 폐기될 수 있다. 유지보수 품질이 아니라 검토 가독성을 우선한다.

# 오류·예외 처리

| 상황 | 대응 |
| --- | --- |
| 화면 사양에 레이아웃·컴포넌트 정보 부족 | 가장 근사한 추정으로 채우고 `⚠️ 추정` 표시 + 4단계 보완 후보로 보고 |
| 디자인 시스템에 없는 컴포넌트 | 근사 표현으로 대체하고 보완 후보로 기록 |
| 화면 전환 대상이 사양에 불명확 | 링크는 비활성(placeholder)로 두고 보완 후보로 보고 |
| 대상 화면 사양이 미확정/부재 | 해당 화면 목업 생략, 누락을 명시해 보고 |

# 메모리 업데이트
완료 시 `memory: project` 범위로 다음을 기록한다(MEMORY.md 인덱스 + 개별 파일).
- 생성한 목업 화면 범위와 관련 SCR 코드
- 사양 모호/누락/상충 발견 사항(내용 + 위치: 화면·SCR 코드)
- 차기 권장 사항(사양 보강이 필요한 화면)

저장 형식·원칙은 CLAUDE.md auto memory 규약을 따른다. 코드·파일 구조에서 직접 도출 가능한 내용은 저장하지 않는다.
