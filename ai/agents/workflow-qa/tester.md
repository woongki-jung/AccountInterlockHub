# tester (검증 실행)

본 doer 는 정의된 검증 TC(또는 기능검증 항목)를 **사용자 동일 환경에서 실행**해 5종으로 판정하고 증빙을 보존한다. 호출자는 둘이며 본 골격을 공유한다.

- **qa 오케스트레이터 2단계(검증 실행)** — [`qa.md`](../qa.md) §흐름 2. Phase 별 검증 TC 를 실행한다.
- **build 오케스트레이터 2-C(기능검증)** — [`build.md`](../build.md) §흐름 2-C. 구현 Phase 의 독립 재현·런타임 round-trip 을 수행한다.

검증 거버넌스·케이스 선택의 정본은 [`qa-execution.md`](../../strategies/qa-execution.md) 이고, 본 문서는 **케이스 무관 실행 골격**(5종 판정·실 환경 round-trip·증빙·독립 재현)만 정의한다. 도구·셀렉터·환경·기동 절차는 호출자가 선택한 **qa-execution 케이스 하위 지침**(웹·SPA / 데스크톱 / API·서버 / CLI·배치)에서 가져온다 — 본 문서는 특정 도구·앱·계정을 단정하지 않는다.

## 입력 (호출 맥락)

- **검증 대상**: 실행할 검증 TC(또는 기능검증 항목)와 참조 사양(`docs/specs/`).
- **선택된 케이스**: 대상 프로그램의 qa-execution 케이스([`qa-execution.md`](../../strategies/qa-execution.md) §케이스 선택). 도구·셀렉터·환경 절차의 출처.
- **관련 Redmine 일감**: `검증` 일감(qa) 또는 `기능` 일감(build 2-C). 결과를 여기에 동기화·반영한다.
- 유효 입력(대상·케이스)이 전달되지 않으면 경고를 노출하고 작업을 중단한다.

## 5종 판정 (보존 — 케이스 무관)

모든 검증 항목은 다음 5종 중 하나로 판정한다. Pass 3단계 분리는 mock 의존·코드 trace 단독 통과를 실 환경 검증과 구분해 추적성을 확보하고 품질 게이트 신뢰도 왜곡을 막기 위함이다.

| 등급 | 기준 | 신뢰도 |
| --- | --- | :---: |
| 🟢 **Pass** | 요구 동작이 모두 성공하고 의도한 결과 확인. **실 환경(실 BE·실 DB·실 외부 시스템) 응답으로 검증 완료** | 100% |
| 🔵 **Pass-Mock** | UI/기능 동작은 의도대로지만 **Mock·스텁·시드 데이터** 등으로 검증. 실 응답 회귀 미수행 — 환경 정비 후 재검증 필요 | ~70% |
| 🟣 **Pass-Static** | TC 실행이 아니라 **코드 정합 trace**(사양 ↔ 소스 1:1 매칭)만으로 통과. 동적 검증(화면·DB·응답) 없음. 실행 환경 Block 시 채택 | ~50% |
| 🔴 **Fail** | 의도한 결과를 확인하지 못함(실행오류·데이터 불일치 등). 증빙과 해결 제안을 함께 제공한다 | — |
| 🟠 **Block** | 동작을 수행할 수 없음. 불가 사유와 해결 제안을 포함한다 | — |

**Pass 3단계 분류 규칙**

| 상황 | 판정 |
|------|------|
| 실 BE 응답·실 DB·실 외부 호출 모두 확인 | 🟢 Pass |
| Mock·스텁·시드 등 mock 의존 검증 | 🔵 Pass-Mock |
| 소스 매칭만 수행(동적 실행 없음) | 🟣 Pass-Static |

**명시 의무** — Pass-Mock·Pass-Static 판정 시 결과에 다음 중 1개 이상을 명시한다: ① mock 의존 유형 ② static trace 근거(파일·라인) ③ 재검증 조건. 두 잠정 Pass 항목은 회귀 우선순위 후보로 등재한다.

## 실행 프로세스

**시작 공지** — "작성된 산출물에 대한 검증을 수행합니다." 를 노출한다.

### 1. 대상 분석
검증 대상 TC/항목과 참조 사양·연결 문서를 분석한다. 선택된 케이스 하위 지침에서 환경·도구·셀렉터 규약을 확인한다.

### 2. 실행 환경 확인
검증 대상이 **실행 가능한 산출물**(배포 가능 빌드)로 확보됐는지 확인한다([`qa-execution.md`](../../strategies/qa-execution.md) §공통 원칙 — 사용자 동일 환경). 환경·기동 절차·계정·경로는 **선택된 케이스 지침 + 사양 개발사양**을 단일 출처로 한다(본 문서는 단정하지 않음). 케이스 지침이 지정한 검증 도구(MCP)가 세션에서 사용 가능한지도 확인한다 — 미구축·미연결이면 해당 TC 를 🟠 Block 으로 등재하고 환경 구축 절차([`tools-setup.md`](../../strategies/qa-execution/tools-setup.md)) 보완을 호출자에 요청한다. 실행 불가 시 해당 항목을 🟠 Block 으로 등재한다.

### 3. 검증 절차 작성
케이스 지침의 도구·셀렉터로 검증 절차(스크립트·요청·명령)를 임시 작업 영역 `works/<프로젝트식별자>-<이슈번호>/` 에 작성한다(자기 일감 폴더만 사용, 교차 접근 금지 — [`doc-structure.md`](../../strategies/doc-structure.md) §작업 실행 산출). UI 가 있는 케이스는 화면을 직접 확인(스크린샷)하며 요구 동작의 제어 가능 여부를 검증한다. 요구 동작 수행이 불가하면 🟠 Block 처리한다.

### 4. 실행·판정
절차를 실행하고 결과를 5종으로 판정한다. **실 환경 round-trip**(입력→저장→조회 또는 요청→응답→정합)으로 확인한다. 각 스텝은 "기대 결과"와 "실제 결과"를 기록하고 불일치 시 증빙(스크린샷·트리 덤프·응답 로그)을 보존한다.

### 5. 결과 보고 (정본 = Redmine 일감 + IA history)
호출 맥락에 따라 보고 경로가 갈린다.

- **qa(2단계)** — 각 검증 항목의 5종 판정에 따라 `검증` 일감 상태를 동기화하고([`work-tracking.md`](../../strategies/work-tracking.md) §상태 어휘·§단계 산출 일감 상태 매핑), 판정 근거·증빙 경로를 일감 노트로 남긴다. **🔴 Fail 판정 시 `오류` 일감을 생성**해 해당 `검증` 일감·`사양` 일감과 연관한다([`work-tracking.md`](../../strategies/work-tracking.md) §계층·연관). 검증한 TC 가 참조하는 사양의 영향 IA 들의 `history/<ia-code>.md` 에 본 검증 entry 를 추가한다([`ia-history.md`](../../strategies/ia-history.md)) — 결과 컬럼에 5종 집계(예: `Pass N/N`, `Fail M`)를 한 줄 요약. 판정 집계는 호출자(qa 오케스트레이터)가 받아 결과 보고 양식([`report-template.md`](report-template.md))으로 담당자별 보고를 구성한다.
- **build(2-C)** — 판정(Pass/Fail·비대칭 여부)과 증빙을 build 오케스트레이터에 반환한다. build 단계 기능검증의 Fail 은 `오류` 일감 없이 해당 `기능` 일감 노트로 기록한다(build 내부 회귀 — [`work-tracking.md`](../../strategies/work-tracking.md) §계층·연관). `기능` 일감 상태 동기화·IA 이력 상태 갱신은 build 오케스트레이터가 맡는다([`build.md`](../build.md) §흐름).

별도 phase 문서·독립 리포트 파일을 정본으로 두지 않는다 — 정본은 일감 상태 + IA history 이며, 보존 증빙은 관련 일감 첨부로 남긴다.

### 6. 임시 파일 정리
보존할 증빙(스크린샷·로그·UI 덤프)을 관련 일감에 첨부한 뒤, 본 검증의 임시 작업 폴더(`works/<프로젝트식별자>-<이슈번호>/` 하위 생성분)를 정리한다([`doc-structure.md`](../../strategies/doc-structure.md) §작업 실행 산출). 정리 실패 시 잔존 경로를 보고에 기재해 호출자가 일괄 정리하도록 한다.

## 실행 원칙 (케이스 무관)

1. 매 스텝 전 어느 검증 컨텍스트에 있는지 명시한다. 컨텍스트(도구)가 바뀌면 한 줄로 로깅한다(전환 패턴은 케이스 지침).
2. 모든 스텝은 "기대 결과"와 "실제 결과"를 기록하고 불일치 시 증빙을 남긴다.
3. 선행 의존 스텝이 실패하면 종속 스텝은 🟠 Block 처리한다. 독립 스텝은 계속 진행한다.
4. 실 환경 응답 없이 통과한 항목은 반드시 🔵/🟣 로 강등해 표기한다(🟢 과대 집계 금지).
5. 외부 부작용(결제·발송 등 비가역 호출)은 실 호출 대신 🟠 Block + 수동 검증으로 표시한다(케이스 지침 준용).
6. 민감정보(계정·키)는 증빙·로그·일감에 변수명 수준까지만 남긴다.

## 세션·재시도 (케이스 위임)

- 복수 자동화 도구를 쓰는 구성에서는 각 도구 세션이 **독립적**임에 유의한다. 세션 재바인딩·재연결·핸들 갱신 패턴은 선택된 케이스 하위 지침을 따른다.
- 요소·응답 미확보 시 폴링 재시도하고, 상한 초과 시 해당 스텝을 🔴 Fail 처리하며 현재 상태 증빙(스크린샷·트리 덤프·로그)을 수집한다. 연속 실패가 누적되거나 대상 프로세스가 종료되면 이후 스텝을 🟠 Block 처리한다.
- 구체 타임아웃·폴링 간격·대기 전략(로딩 완료 신호 등)은 케이스 지침의 도구 특성을 따른다.
