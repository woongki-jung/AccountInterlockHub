# 품질검증 결과 보고 템플릿

본 문서는 [`qa.md`](../qa.md) 오케스트레이터 3단계(결과 평가·보고)가 생성하는 품질검증 결과 보고서의 작성 형식을 정의한다. 단계 흐름·담당자별 그룹화의 정본은 [`stages/qa.md`](../../strategies/stages/qa.md) §결과 보고이며, 본 문서는 그 양식 슬롯을 채운다. 판정 5종·검증 방법은 도구·앱을 단정하지 않고 [`tester.md`](tester.md)·[`qa-execution.md`](../../strategies/qa-execution.md) 를 준용한다(템플릿이므로 frontmatter 없음).

## 작성 원칙

- 모든 실패·Block·경고·후속 액션은 **주관 담당자(기획 / 디자인 / 개발 / QA)** 기준으로 그룹화하여 작성한다([`stages/qa.md`](../../strategies/stages/qa.md) §결과 보고 — 담당자별 그룹화).
- 실패 항목은 표 대신 **섹션 형식**으로 작성하며, 항목별로 재현 절차·기대/실제 동작·오류 로그·예상 원인·해결 방안을 모두 포함한다.
- 담당 매핑 가이드:
  - **기획**: TC 기대 동작이 사양과 불일치하거나 정의가 모호한 경우, 정책 결정 필요 항목
  - **디자인**: 레이아웃·간격·색상·아이콘·타이포·인터랙션 피드백·a11y 등 시각/UX 결함
  - **개발**: 기능 결함·예외 처리 미흡·API 오류·성능 저하·셀렉터(요소 식별자) 미비
  - **QA**: 환경 미비·외부 의존성 장애·검증 스크립트 결함·Flaky·셀렉터 우회

```markdown

**종합 평가**: ✅ 완료 / ⚠️ 조건부 통과 / ❌ 실패 — Pass N / Fail N / Block N
**대상 빌드**: <commit SHA / 빌드 식별자>
**실행 기간**: <YYYY-MM-DD HH:mm ~ YYYY-MM-DD HH:mm>
> 종합 판정 사유 한 줄 요약

## 품질 게이트 판정
- 높음 우선순위 TC 실 Pass율: N% (기준: 100%)
- 전체 TC 실 Pass율: N% (기준: ≥ 95%)
- 미해소 Fail: N건 (기준: 0)
- 게이트 통과 여부: ✅ / ⚠️ / ❌

> 게이트 기준 정본은 [stages/qa.md](../../strategies/stages/qa.md) §완성도 기준(프로젝트 조정값은 방향 정의서 위임). 실행 우선순위 Critical·High 는 게이트의 "높음" 축에 매핑된다.

## Phase별 실행 결과

|-------|-----------------|--------|------|--------|---------|---------|---------|------|
| 1 | <TC-ID> | <ia-code> | E2E | 자동 | 🟢 Pass | Ns | 개발 | - |

## 🔵🟣 잠정 Pass 항목 — 실 환경 재검증 대기

> Mock 의존(🔵) 또는 코드 trace 단독(🟣)으로 Pass 판정된 항목. 실 BE·DB·외부 시스템 응답 미검증 상태이므로 환경 정비 후 회귀 대상.
> 신뢰도: 🔵 Pass-Mock ~70% / 🟣 Pass-Static ~50%
> 잠정 Pass 는 누적될수록 실질 커버리지 착시를 만든다 — Block 해소·환경 정비 완료 시점에 **회귀 우선순위를 한 단계 상향**해 일괄 재검증한다.

| 분류 | 건수 | 합계 대비 비율 |
|------|----:|------------:|
| 🔵 Pass-Mock | N | N.N% |
| 🟣 Pass-Static | N | N.N% |
| **잠정 합계** | **N** | **N.N%** |

**상위 잠정 Pass 영역 (재검증 우선순위)**

| TC ID / 영역 | 판정 | Mock·Static 의존 유형 | 재검증 조건 |
|------------|:----:|---------------------|---------|
| `<TC-ID>` | 🔵 | <어댑터 mock fallback 유형> | 실 BE 응답 확보 후 재실행 |
| `<TC-ID>` | 🟣 | 코드 trace (동적 검증 미수행) | 산출물 풀 기동 + 동적 round-trip 검증 |

(N건 — 본 검증 범위의 잠정 Pass 항목을 우선순위 순으로 등재. 개별 항목은 `검증` 일감 노트·IA history 참조.)

## TC 실패·차단 스냅샷

> qa 오케스트레이터가 검증 종료 시 모든 🔴 Fail / 🟠 Block TC 를 1건 1행으로 담은 **실패·차단 스냅샷**을 1회 생성한다([`stages/qa.md`](../../strategies/stages/qa.md) §산출물). 개별 TC ID·기대 결과·실제 결과는 스냅샷을 정본으로 참조한다.

- **직전 스냅샷 대비 변화**:

| 변동 유형 | 건수 |
|---------|----:|
| 🆕 신규 Fail | N |
| 🆕 신규 Block | N |
| ✅ 해소 (직전 Fail/Block → 본 검증 부재 or Pass) | N |
| 🔁 결과 변경 (Fail ↔ Block) | N |

본 검증 합계: 🔴 Fail N건 / 🟠 Block N건.
  ```
