본 문서는 프로젝트가 사용자에 제공하는 모든 화면 구성 요소를 서술하기 위한 규칙을 정의한다.
아래 지침에 따라 제공된 요구사항 및 관련 문서를 기반으로 화면 구성과 디자인 요소에 대한 정의를 수행한다.

용어 약어 — SCR은 화면 정의서 코드, MDL은 데이터 모델, FN은 공통 기능 코드, PROC는 기능구현 프로세스 정의서, POL(=policy)은 정책 규칙, FE는 프론트엔드를 의미한다.

# 호출 주체·입력

`spec` 오케스트레이터([`../spec.md`](../spec.md))가 도메인 의존 순서에 따라 호출하는 doer 다(화면은 5순위). 오케스트레이터가 작업 맥락을 전달한다 — **작업 범위**(전체 또는 IA 노드), **관련 Redmine `사양` 일감**(있으면), **선행 도메인 산출물**(정책·서비스·데이터·기능 사양과 분석 대상 PRD·IA). 범위 밖 기존 사양은 입력 참조로만 쓰고 수정하지 않는다. 유효한 작업 맥락이 없으면 경고 후 중단한다.

# 작업 프로세스

## 1단계: 요구사항 분석

- 시작 시 전달받은 작업 맥락(작업 범위·분석 대상 문서)을 확인한다.
- 분석된 내용에서 다음 항목을 추출한다.
	- 서비스 시나리오에서 도출되는 화면 목록.
	- 사용자 역할별 접근 가능한 화면 범위.
	- 화면에서 표시·입력해야 하는 데이터 항목 (MDL 참조).
	- 화면에서 호출하는 API 엔드포인트 (FN 참조) 및 트리거하는 PROC 코드.
	- 정책에서 화면에 영향을 주는 제약사항 (접근 제어, 입력값 검증, 표시 제한 등).
	- 반복 사용되는 UI 패턴 (목록, 폼, 모달, 알림 등).

## 2단계: 디자인 시스템 정의

- 프로젝트 전반에 적용할 디자인 시스템을 정의한다.
	- **디자인 토큰**: 색상 팔레트, 타이포그래피, 간격(spacing), 그림자, 라운딩 등.
	- **공통 컴포넌트**: 버튼, 입력 필드, 테이블, 카드, 모달, 토스트, 네비게이션 등.
	- **레이아웃 시스템**: 그리드 구조, 반응형 브레이크포인트, 공통 레이아웃 패턴.
	- **상태 표현**: 로딩, 빈 상태, 에러 상태, 비활성 상태의 표현 규칙.
	- **접근성 기준**: 색상 대비, 키보드 탐색, 스크린리더 지원 수준.
- 정책 문서에서 UI에 영향을 주는 항목을 반영한다.
	- 개인정보 마스킹 표시 규칙 (SEC/DATA 정책).
	- 역할별 메뉴/버튼 노출 제어 (AUTH 정책).

## 3단계: 화면 구조 설계

- 서비스 시나리오를 기반으로 화면 흐름을 설계한다.
- 화면 코드 체계를 설계한다 (SCR-nnn 형태).

## 4단계: 화면 상세 설계

- 각 화면에 대해 다음을 정의한다.
	- 화면 레이아웃 구성 (영역 배치).
	- 사용하는 컴포넌트 목록 및 배치.
	- 표시 데이터 항목과 데이터 모델(MDL) 매핑.
	- 사용자 인터랙션 정의 (클릭, 입력, 스크롤 등) — **트리거 PROC 코드 명시**.
	- 호출하는 API(FN) 및 호출 시점.
	- 화면 상태 전이 (초기 로딩 → 데이터 표시 → 사용자 액션 → 결과 반영) — **PROC 단계와 1:1 매핑**.
	- 정책 기반 조건부 표시/숨김 규칙.
	- 입력 폼 유효성 규칙 — **FE 검증 의사코드로 변환 가능 수준** (정규식·길이·범위·필수).

## 5단계: 화면 정의 문서 생성

### 5-1: 화면 정의 목록 문서 생성

`docs/specs/screens/spec-screens.md` 파일에 전체 화면 정의서의 개요와 하위 문서 목록을 작성한다.

- 디자인 시스템 요약.
- 화면 코드 체계 (SCR-nnn).
- 화면 분류 및 목록.
- 화면 간 이동 경로 요약 (Navigation Flow).
- 사용자 역할별 화면 접근 권한 매트릭스.

### 5-2: 개별 화면 정의 문서 생성

각 화면별로 `docs/specs/screens/screen_<screen-code>.md` 파일을 생성한다.

```markdown
# [화면명] 화면 정의

## 개요
- **화면 목적**: 이 화면이 사용자에게 제공하는 기능 및 가치
- **관련 PRD 요구사항**: PRD 내 관련 섹션 및 원문 인용

---

## [SCR 코드] [화면명]

### 기본 정보
| 항목 | 내용 |
|------|------|
| 화면명 | 화면 이름 |
| 화면 경로 | URL 경로 (예: /dashboard, /users/:id) |
| 접근 권한 | 접근 가능한 사용자 역할 (AUTH 정책 참조) |
| 관련 서비스 | 이 화면이 구현하는 SVC 코드 |
| 트리거 PROC | 본 화면의 액션이 트리거하는 PROC 코드 목록 |
| 선행 화면 | 이 화면에 진입하기 위한 이전 SCR 코드 |
| 관련 IA 항목 | 본 사양이 속한 IA 노드의 `ia-code`(들). 횡단 사양은 `공통` (정본 [`../../strategies/ia.md`](../../strategies/ia.md) §참조 규약) |

### 레이아웃 구성
- 화면의 영역 배치 (헤더, 사이드바, 본문, 푸터 등).
- 각 영역에 배치되는 컴포넌트 목록.
- 반응형 대응 시 레이아웃 변화.

### 데이터 표시
| 표시 항목 | 데이터 모델(MDL) | 속성 | 표시 형식 | 마스킹 규칙 |
|-----------|-----------------|------|-----------|-------------|
| 사용자명 | MDL-xxx | name | 텍스트 | 마스킹 정책 인용 (POL <그룹>-NNN) |

### 사용자 인터랙션

| 인터랙션 | 대상 요소 | 동작 | 호출 API(FN) | **트리거 PROC** | 결과 |
|----------|-----------|------|-------------|----------------|------|
| 클릭 | 저장 버튼 | 폼 데이터 제출 | FN-xxx | PROC-yyy | 성공 토스트 + SCR-zzz 이동 |
| 페이지 mount | (자동) | 목록 조회 | FN-aaa | PROC-bbb | 테이블 렌더링 |

> 트리거 PROC 칸은 [PROC 정의서](../processes/spec-process.md) 의 PROC 코드와 1:1 정합해야 한다.
> 한 인터랙션은 1개의 PROC 를 트리거한다 (PROC 가 내부에서 다른 PROC 를 호출하는 것은 PROC 내부 호출 관계로 표현).

### 화면 상태 전이

| 상태 | 진입 조건 | PROC 단계 매핑 | 표시 내용 |
|------|----------|---------------|-----------|
| Initial | 페이지 mount | (트리거 전) | 스켈레톤 UI |
| Loading | API 응답 대기 중 | PROC-xxx F1 진입 직후 | 스피너 + 디세이블 |
| Loaded | 200 응답 수신 | PROC-xxx F2 응답 처리 | 데이터 렌더링 |
| Empty | 응답 0건 | PROC-xxx F2 응답 처리 후 분기 | 안내 + 생성 유도 |
| Error | 4xx/5xx 응답 | PROC-xxx F2 onError | 에러 메시지 + 재시도 |

> 화면 상태 전이는 PROC 의사코드의 FE 단계(F1·F2 등)와 1:1 매핑되어야 한다.

### 입력 폼 정의 (해당 시)

| 필드명 | 입력 유형 | 데이터 모델(MDL) | 필수 | 유효성 규칙 (FE 검증 의사코드) | 에러 메시지 |
|--------|-----------|-----------------|------|--------------------------------|-------------|
| 이메일 | text input | MDL-xxx.email | Y | `/^[\w.+-]+@[\w-]+\.[\w.-]+$/.test(value)` | "올바른 이메일을 입력해주세요" |
| 휴대폰 | text input | MDL-xxx.phone | Y | `/^010\d{8}$/.test(value.replace(/-/g, ""))` | "휴대폰 번호 형식이 올바르지 않습니다" |
| 식별코드 | text input | MDL-yyy.code | Y | `value.trim().length > 0 && value.length <= 20` | "식별코드를 입력해주세요" |

> 유효성 규칙은 PROC 의사코드의 FE 검증 단계에서 그대로 사용 가능한 표현이어야 한다.
> 정규식·범위·필수·길이 제약은 [POL 규칙](../policies/spec-policies.md) 인용 시 정책 코드를 함께 명시한다.

### 조건부 표시

| 대상 요소 | 표시 조건 | 관련 정책 |
|-----------|-----------|-----------|
| 삭제 버튼 | `user.role === 'ADMIN'` | AUTH-xxx |
| 마스킹 해제 토글 | `user.permissions.includes('UNMASK')` | SEC-xxx |

### 이동 경로 (Navigation)

| 이동 대상 | 트리거 | 조건 | 트리거 PROC |
|-----------|--------|------|-------------|
| SCR-xxx 목록 화면 | 목록 버튼 클릭 | 없음 | (네비게이션만, PROC 없음) |
| SCR-xxx 상세 화면 | 행 클릭 | 데이터 존재 시 | PROC-yyy (상세 조회 트리거) |

### 구현 가이드
- 개발자가 이 화면을 구현할 때 참고할 핵심 사항.
- 관련 기술 패턴 (특정 라이브러리·프레임워크 강제 금지).
```

> 변경이력은 본문에 인라인 표로 두지 않는다 — 영향 IA 노드의 이력([`../../strategies/ia-history.md`](../../strategies/ia-history.md))으로 위임한다.

`docs/specs/screens/design-system.md` 파일에 디자인 시스템 전체를 작성한다.

- 산출물을 Redmine `사양` 트래커 일감으로 미러·동기화하고([`../../strategies/work-tracking.md`](../../strategies/work-tracking.md) §단계별 미러링), 영향 IA 노드의 이력을 갱신한다([`../../strategies/ia-history.md`](../../strategies/ia-history.md)).
- 일감 상태는 `진행` 을 유지한다 — `해결` 전이는 교차검증 통과 후 spec 오케스트레이터 몫이다. 세부 진척은 일감 노트로 기록한다([`work-tracking.md`](../../strategies/work-tracking.md) §단계 산출 일감 상태 매핑).
- 작업 범위·도출 결과·보류 항목을 `spec` 오케스트레이터에 보고한다.

# 문서 작성 원칙

## 역할 준수
- 본 문서는 화면 구성과 디자인 시스템을 정의하는 것이 목적으로, 비즈니스 로직 상세나 DB 스키마는 포함하지 않는다.
- PROC 의사코드의 FE 단계 내용은 PROC 정의서로 위임하고, 본 문서는 화면 액션 → PROC 트리거 매핑까지만 책임진다.

## 명확성
- 모호한 표현 금지 — "적절한 간격을 둔다" 대신 "컴포넌트 간 간격 16px, 섹션 간 간격 32px".
- 입력 폼 유효성 규칙은 FE 검증 의사코드로 그대로 사용 가능한 수준으로 작성.

## 일관성
- 모든 화면에 동일한 문서 구조 적용.
- 화면 코드(SCR) 채번 일관성 유지.
- 디자인 시스템에 정의된 컴포넌트와 토큰을 일관되게 참조.

## 완결성
- 모든 화면에 레이아웃, 데이터 표시, 인터랙션, 상태 정의가 포함되어야 한다.
- 사용자 인터랙션 표에 트리거 PROC 칸이 채워져야 한다.

## 추적 가능성
- 각 화면이 PRD의 어떤 요구사항에서 비롯되었는지 기술.
- 화면과 연관된 SVC / MDL / FN / POL 코드 기술.
- 트리거 PROC 코드를 명시하여 PROC 의사코드와 양방향 추적 가능.

# PROC 와의 상호 참조 (필수)

화면의 사용자 인터랙션은 PROC 의 진입 트리거가 된다.
SCR 작성 시 다음을 반드시 고려한다.

- 한 인터랙션은 1개의 PROC 를 트리거한다.
- 화면 상태 전이는 PROC 의사코드의 FE 단계명(F1·F2 등)과 1:1 매핑되어야 한다.
- 입력 폼 유효성 규칙은 PROC 의사코드의 FE 검증 단계에서 그대로 사용 가능한 표현이어야 한다.
- 데이터 표시의 마스킹 규칙은 PROC 의사코드의 BE "응답 DTO 변환" 단계 또는 FE "응답→FE 모델 어댑터" 단계와 정합한다.

# 산출물 검증 체크리스트

산출물 작성 완료 후 다음을 확인한다 (6 카테고리는 [prd-to-process.md §산출물 검증 체크리스트](prd-to-process.md) 와 동일 형식).

## 적용 범위 검증
- ⬜ 서비스 시나리오의 모든 사용자 접점이 화면으로 정의되었는가?
- ⬜ 디자인 시스템(토큰, 컴포넌트, 레이아웃)이 정의되었는가?
- ⬜ 모든 PROC 의 진입 트리거가 SCR 인터랙션에 매핑되었는가?

## 정의 완결성 검증
- ⬜ 모든 화면에 레이아웃, 데이터 표시, 인터랙션, 상태가 정의되었는가?
- ⬜ 사용자 인터랙션 표에 트리거 PROC 칸이 모두 채워져 있는가?
- ⬜ 화면 상태 전이가 PROC 단계와 1:1 매핑 가능 수준으로 작성되었는가?
- ⬜ 입력 폼 유효성 규칙이 FE 검증 의사코드로 변환 가능한 수준인가? (정규식·범위·필수)
- ⬜ 표시 데이터가 MDL 코드 + 속성 매핑까지 작성되었는가?
- ⬜ 마스킹 규칙이 정책 코드 인용과 함께 작성되었는가?

## 실행 제약 검증
- ⬜ 접근 권한 정책(AUTH)이 화면별로 반영되었는가?
- ⬜ 조건부 표시 규칙이 표현식 또는 정책 코드 인용으로 작성되었는가?
- ⬜ 빈 상태·에러 상태의 표현이 디자인 시스템과 정합한가?

## 일관성 검증
- ⬜ 모든 화면에 동일한 문서 구조 적용?
- ⬜ 화면 코드 채번이 spec-screens.md 목록과 일치하는가?
- ⬜ 디자인 토큰·컴포넌트 인용이 design-system.md 와 일관되는가?

## 의존관계 검증
- ⬜ 화면 간 이동 경로(Navigation Flow)가 양방향으로 추적 가능한가?
- ⬜ 트리거 PROC 코드가 모두 실재하는가? (깨진 참조 0건 — 후행 도메인(PROC 등) 참조는 초회 작성 시 예약 채번 허용, 실재 확인은 교차검증(spec-reviewer) 시점)
- ⬜ 인용된 SVC / MDL / FN / POL 코드가 모두 실재하는가? (깨진 참조 0건)

## 역할 준수 검증
- ⬜ 비즈니스 로직 상세(BE 의사코드·SQL)가 포함되지 않고 PROC 정의서로 위임되었는가?
- ⬜ DB 스키마 상세는 ENT 정의서로 위임되었는가?
- ⬜ 특정 프레임워크·라이브러리를 강제하는 표현이 없는가?
- ⬜ 모호 항목과 해석 근거가 검토 협의([`../../strategies/prompt-conversation.md`](../../strategies/prompt-conversation.md) §검토 협의)로 처리·기록되었는가?

# 출력 형식

- 화면 정의 목록 파일: `docs/specs/screens/spec-screens.md`
- 개별 화면 정의 파일: `docs/specs/screens/screen_<screen-code>.md`
- 디자인 시스템 파일: `docs/specs/screens/design-system.md`
- 언어: 한국어
- 형식: Markdown
- 화면 코드 채번: `SCR-001`, `SCR-002`, ...

# 주의사항

- PRD, 서비스 기능 정의, 공통 기능 정의에 근거하지 않은 화면을 임의로 추가하지 않는다.
- 특정 구현 기술(프레임워크, 라이브러리)을 강제하지 않고 화면의 요구사항만 정의한다.
- 비즈니스 로직 상세는 공통 기능 정의서(FN) 및 PROC 정의서에, DB 스키마는 데이터 정의서(ENT)에 위임하고 본 문서에서는 참조만 한다.
- 디자인 시스템에 정의된 컴포넌트와 토큰을 우선 사용하고, 화면 고유 스타일이 필요한 경우 그 사유를 명시한다.
- 기존 작성 내용에서 변경이 필요한 경우 변경 지점·사유·영향범위·추가 검토 제안사항을 영향 IA 노드의 이력([`../../strategies/ia-history.md`](../../strategies/ia-history.md))에 기록한다.

# 메모리 업데이트

PRD 분석 후 다음 사항을 프로젝트 메모리에 기록한다:
- 도출된 화면 분류 체계 및 코드 목록.
- SCR 인터랙션 → PROC 트리거 매핑 관계.
- 디자인 시스템의 주요 결정 사항 (토큰, 컴포넌트 구성 등).
- 화면 간 주요 이동 경로 구조.
- 입력 폼 유효성 규칙 카탈로그 (정규식·범위·필수 분포).
- PRD에서 화면으로 변환 시 판단이 필요했던 항목.
- 정책 기반 조건부 표시 규칙에서 추가 논의가 필요한 사항.
