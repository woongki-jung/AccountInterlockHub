본 문서는 요구사항을 분석하고 사용자 정의 및 서비스 시나리오를 바탕으로 구현되어야 할 서비스 항목을 서술하기 위한 규칙을 정의한다.
아래 지침에 따라 제공된 요구사항 및 정책 문서를 바탕으로 사용자 정의, 서비스 구분, 제공 기능, 사용 시나리오를 서술한다.

용어 약어 — SVC는 서비스 시나리오 정의서 코드, PROC는 기능구현 프로세스 정의서 코드, MDL은 데이터 모델, POL(=policy)은 정책 규칙, BR은 분기 코드(Branch), EX는 예외 코드(Exception)를 의미한다.

# 호출 주체·입력

`spec` 오케스트레이터([`../spec.md`](../spec.md))가 도메인 의존 순서에 따라 호출하는 doer 다(서비스는 2순위 — 정책 산출물을 입력으로 받는다). 오케스트레이터가 작업 맥락을 전달한다 — **작업 범위**(전체 또는 IA 노드), **관련 Redmine `사양` 일감**(있으면), **선행 도메인 산출물**(분석 대상 PRD·IA·정책 정의). 범위 밖 기존 사양은 입력 참조로만 쓰고 수정하지 않는다. 유효한 작업 맥락이 없으면 경고 후 중단한다.

# 작업 프로세스

## 1단계: 요구사항 분석

- 시작 시 전달받은 작업 맥락(작업 범위·분석 대상 문서)을 확인한다.
  분석 대상 문서와 각 문서가 참조하는 연결 문서의 내용을 자세히 분석한다.
- 분석된 내용에서 다음 항목을 추출한다.
	- 사용자 유형 및 역할 구분 (서비스 이용 주체 식별).
	- 각 사용자가 달성하려는 핵심 목표 및 동기.
	- 서비스가 제공해야 할 주요 기능 목록.
	- 기능 간 선후관계 및 의존성 (어떤 기능이 다른 기능의 선행 조건인지).
	- 기능별 입력/출력 데이터 개요 (MDL 코드 매핑 후보).
	- 외부 시스템 연동 요구사항 (API, 알림, 결제 등).
	- 기능의 우선순위 또는 필수/선택 구분.
	- 정책 문서에서 기능에 영향을 주는 제약사항 (접근 제어, 비즈니스 규칙 등).

## 2단계: 사용자 정의
- 1단계에서 추출한 사용자 유형을 정리하고 다음을 정의한다.
	- 각 사용자 역할의 명칭 및 설명.
	- 역할별 주요 목적 및 사용 맥락.
	- 역할 간 관계 및 권한 차이.
	- 외부 시스템 사용자 (API 연동 등).
- 정책 문서(AUTH, BIZ 등)에서 역할 기반 제약사항을 확인하여 반영한다.

## 3단계: 서비스 시나리오 정의
- 사용자 역할별 핵심 시나리오를 도출한다.
	- 사용자가 목적을 달성하기 위한 주요 흐름 (Happy Path).
	- 흐름 중 분기가 발생하는 지점과 각 분기의 조건.
	- 오류 또는 예외 상황에서의 대응 흐름.
	- 시나리오 간 선후관계 및 의존성.
- 시나리오를 서비스 단위로 그룹화한다.
- 서비스 코드 체계를 설계한다 (SVC-nnn 형태).
- **PROC 분해 가능성 사전 검토**: 한 시나리오가 너무 길어지면(예: 8 단계 초과) PROC 단계와 1:N 매핑 시 PROC 가 12 단계를 초과할 위험이 있으므로 사전에 시나리오를 분해한다.

## 4단계: 서비스 기능 정의 문서 생성

### 4-1: 서비스 기능정의 목록 문서 생성

`docs/specs/services/spec-services.md` 파일에 전체 서비스 기능 정의서의 개요와 하위 문서 목록을 작성한다.
서비스 기능 정의서는 다음 내용을 포함한다.

- 사용자 정의 (2단계 결과).
- 서비스 시나리오 요약 (3단계 결과).
- 서비스 코드 체계 (SVC-nnn).
- 서비스 기능 분류 및 목록.
- 서비스 간 의존관계 요약.

### 4-2: 개별 서비스 기능 정의 문서 생성

각 서비스별로 `docs/specs/services/service_<service-code>.md` 파일을 생성한다.

```markdown
# [서비스 기능명] 기능 정의

## 개요
- **서비스 목적**: 이 서비스가 해결하려는 문제 또는 제공하려는 가치
- **관련 PRD 요구사항**: PRD 내 관련 섹션 및 원문 인용

---

## [SVC 코드] [기능명]

### 기본 정보
| 항목 | 내용 |
|------|------|
| 기능명 | 서비스 기능 이름 |
| 분류 | 서비스 기능 분류 |
| 주 사용자 | 이 기능을 사용하는 주요 역할 |
| 우선순위 | 높음 / 보통 / 낮음 |
| 선행 기능 | 이 기능 사용 전 완료되어야 하는 기능 코드 |
| 매핑 PROC | 본 시나리오를 구현하는 PROC 코드 (1:N) |
| 관련 IA 항목 | 본 사양이 속한 IA 노드의 `ia-code`(들). 횡단 사양은 `공통` (정본 [`../../strategies/ia.md`](../../strategies/ia.md) §참조 규약) |

### 서비스 기능

| 기능 번호 | 기능 내용 | 입력 (MDL 코드) | 출력 (MDL 코드) | 비고 |
|-----------|-----------|----------------|----------------|------|
| F-001 | 구체적인 기능 서술 | MDL-xxx (또는 원시 입력) | MDL-yyy | 추가 설명 |

> 입력/출력은 가능한 한 [데이터 모델(MDL)](../datas/spec-datas.md) 코드로 매핑한다.
> MDL 매핑이 어려운 원시 타입(string·int·boolean 등)은 그대로 표기하되, 향후 MDL 신설 후보로 표기.

### 사용 시나리오

#### Happy Path

| 단계 | 단계명 | 주체 | 행동/처리 | 입력 | 출력 | 매핑 PROC 단계 |
|------|--------|------|----------|------|------|---------------|
| 1 | 진입 | 사용자 | 화면 SCR-xxx 진입, 진입 조건 검증 | (URL/메뉴) | 화면 표시 | PROC-xxx F1 |
| 2 | 입력 | 사용자 | 입력 폼 작성 | (사용자 입력) | form state | PROC-xxx F1 |
| 3 | 제출 | 사용자/시스템 | 제출 → 검증 → 영속화 | form state | 응답 DTO | PROC-xxx F1·B1·B2·B3 |
| 4 | 결과 | 시스템 | 응답 표시 + 후속 흐름 | 응답 DTO | (UI 갱신) | PROC-xxx F2 |

> 매핑 PROC 단계 칸은 [PROC 정의서](../processes/spec-process.md) 의 의사코드 단계명(F1·B2 등)과 1:1 정합해야 한다.

#### 분기 흐름

| 코드 | 발생 조건 | 분기 처리 | 결과 |
|------|----------|----------|------|
| BR-001 | 사용자 입력 X 조건 충족 | 단계 N에서 단계 M으로 우회 | 분기 결과 상태 |

#### 예외 흐름

| 코드 | 발생 조건 | 처리 방식 | 사용자 응답 |
|------|----------|----------|-------------|
| EX-001 | 사전 데이터 누락 | 단계 중단, 에러 페이지 노출 | "데이터를 찾을 수 없습니다." |
| EX-002 | 외부 시스템 응답 실패 | 재시도 → 보상 트랜잭션 → 알림 | "잠시 후 다시 시도해주세요." |

### 사용자 정의
- 이 기능과 관련된 사용자 역할 및 각 역할의 허용 범위.

### 관련 서비스
- 연계되거나 영향을 주는 다른 SVC 코드 및 관계 설명.

### 연관 정책 (policy)

| 정책 코드 | 적용 단계 | 적용 위치 (validate/transform/mask/audit) | 위반 시 처리 |
|-----------|----------|------------------------------------------|--------------|
| <그룹>-NNN-RR | 단계 N | 적용 위치 | 위반 처리 (HTTP·EX 코드) |

### 예외 사항
- 기능이 정상 동작하지 않는 조건 및 대응 방법.

### 구현 가이드
- 개발자가 이 기능을 구현할 때 참고할 핵심 사항.
- 관련 기술 패턴 (특정 라이브러리·프레임워크 강제 금지).
```

> 변경이력은 본문에 인라인 표로 두지 않는다 — 영향 IA 노드의 이력([`../../strategies/ia-history.md`](../../strategies/ia-history.md))으로 위임한다.

## 5단계: 작업 완료 보고
- 산출물을 Redmine `사양` 트래커 일감으로 미러·동기화하고([`../../strategies/work-tracking.md`](../../strategies/work-tracking.md) §단계별 미러링), 영향 IA 노드의 이력을 갱신한다([`../../strategies/ia-history.md`](../../strategies/ia-history.md)).
- 일감 상태는 `진행` 을 유지한다 — `해결` 전이는 교차검증 통과 후 spec 오케스트레이터 몫이다. 세부 진척은 일감 노트로 기록한다([`work-tracking.md`](../../strategies/work-tracking.md) §단계 산출 일감 상태 매핑).
- 작업 범위·도출 결과·보류 항목을 `spec` 오케스트레이터에 보고한다.

# 문서 작성 원칙

## 역할 준수
- 본 문서는 기능을 정의하는 것이 목적으로, UI 구성은 포함하지 않는다.
  별도 정의될 화면정의서와 충돌하지 않도록 구현되어야 할 기능에 집중하여 서술한다.
- 의사코드·SQL 등 PROC 영역의 내용은 포함하지 않는다.

## 명확성
- 모호한 표현 금지 — "적절히", "충분히" 대신 수치나 조건으로 명시.
- 기능 정의는 개발자가 코드로 바로 변환할 수 있을 만큼 구체적으로 기술.
- Happy Path 의 각 단계는 PROC 단계와 매핑 가능한 단위로 분해한다.

## 일관성
- 모든 서비스 기능에 동일한 문서 구조 적용.
- 서비스 코드 채번 일관성 유지 (SVC-NNN).
- 상충되는 기능 정의가 없도록 교차 검토.
- 분기·예외 코드(BR-/EX-)는 PROC 의 BR-/EX- 코드와 동일 채번 사용 (PROC 가 SVC 시나리오를 구현하므로 같은 분기·예외에 같은 코드).

## 완결성
- 사용자 중심으로 서비스 시나리오에 따라 흐름이 이어질 수 있도록 어떤 사용 단계인지, 다음 단계는 어떤 경우의 수가 있는지 명시.
- Happy Path · 분기 · 예외 흐름 3개를 모두 작성한다.

## 추적 가능성
- 각 기능이 PRD의 어떤 요구사항에서 비롯되었는지 기술.
- 기능과 연관된 정책(policy) 기술.
- 매핑 PROC 코드 인용으로 PROC 의사코드와 1:N 추적 가능.

# 산출물 정의

## 사용자 정의
- 사용자 역할 목록 및 각 역할의 설명.
- 역할별 접근 가능한 서비스 범위.
- 역할 간 관계 (상위/하위·위임 등) — 표/텍스트로 기술(다이어그램 대신).

## 서비스 시나리오 정의
- 사용자 역할별 핵심 시나리오 목록.
- 시나리오별 Happy Path · 분기 · 예외 표.
- 시나리오 간 선후관계·의존성 — 표/텍스트로 기술(다이어그램 대신).

## 서비스 기능정의
- 서비스 코드별 기능 목록.
- 각 기능의 입력/출력 (MDL 코드 매핑).
- 기능 간 의존관계 및 연계 흐름.
- 정책(policy)과의 매핑 관계.
- 매핑 PROC 코드 (SVC ↔ PROC 1:N).

# PROC 와의 상호 참조 (필수)

본 문서의 SVC 시나리오는 PROC 의사코드의 입력이 된다.
SVC 작성 시 다음을 반드시 고려한다.

- 한 SVC 시나리오는 1개 이상의 PROC 로 구현된다 (1:N 관계).
- Happy Path 의 각 단계는 PROC 의 의사코드 단계명(F1·B2 등)과 1:1 매핑되어야 한다.
- SVC 의 BR-/EX- 코드는 PROC 의 분기 및 예외 흐름 표에서 동일 코드로 인용된다.
- SVC 가 정의하는 입력/출력 MDL 코드는 PROC 의 데이터 변환 흐름 표에서 입력/출력 형태로 직접 인용된다.

# 산출물 검증 체크리스트

산출물 작성 완료 후 다음을 확인한다 (6 카테고리는 [prd-to-process.md §산출물 검증 체크리스트](prd-to-process.md) 와 동일 형식).

## 적용 범위 검증
- ⬜ PRD의 모든 기능 요구사항이 하나 이상의 서비스 기능으로 정의되었는가?
- ⬜ 모든 사용자 역할이 식별되고 역할별 서비스 범위가 정의되었는가?
- ⬜ 모든 SVC 가 1개 이상의 PROC 코드와 매핑되었는가?

## 정의 완결성 검증
- ⬜ 각 서비스 기능의 입력/출력이 구체적으로 기술되었는가? (MDL 코드 매핑 우선)
- ⬜ Happy Path 표의 각 단계가 PROC 단계명과 매핑 가능한 단위인가?
- ⬜ 분기 흐름 표(BR-)가 작성되었는가?
- ⬜ 예외 흐름 표(EX-)가 작성되었는가?
- ⬜ 연관 정책 표에 정책 코드·적용 단계·적용 위치·위반 시 처리가 명시되었는가?

## 실행 제약 검증
- ⬜ Happy Path 가 8 단계 이내 또는 적절히 분해되었는가? (PROC 12 단계 한계 보호)
- ⬜ 외부 시스템 연동의 실패 처리(재시도·보상·알림)가 예외 흐름 표에 기술되었는가?

## 일관성 검증
- ⬜ 모든 서비스 기능에 동일한 문서 구조 적용?
- ⬜ 서비스 코드 채번이 spec-services.md 목록과 일치하는가?
- ⬜ BR-/EX- 코드가 PROC 의 동일 분기·예외에 동일 코드로 사용되었는가?

## 의존관계 검증
- ⬜ 기능 간 선후관계 및 의존성이 명시되었는가?
- ⬜ 매핑 PROC 코드가 모두 실재하는가? (깨진 참조 0건 — 후행 도메인(PROC 등) 참조는 초회 작성 시 예약 채번 허용, 실재 확인은 교차검증(spec-reviewer) 시점)
- ⬜ 인용된 MDL/policy 코드가 모두 실재하는가? (깨진 참조 0건)

## 역할 준수 검증
- ⬜ UI 구성·표시 형식 등 화면 정의서에 속하는 내용이 포함되지 않았는가?
- ⬜ 의사코드·SQL 등 PROC 정의서에 속하는 내용이 포함되지 않았는가?
- ⬜ 특정 프레임워크·라이브러리를 강제하는 표현이 없는가?
- ⬜ 모호 항목과 해석 근거가 검토 협의([`../../strategies/prompt-conversation.md`](../../strategies/prompt-conversation.md) §검토 협의)로 처리·기록되었는가?

# 출력 형식

- 서비스 정의 파일: `docs/specs/services/spec-services.md`
- 서비스 기능정의서: `docs/specs/services/service_<service-code>.md`
- 언어: 한국어
- 형식: Markdown
- 서비스 기능정의 코드 채번: `SVC-001`, `SVC-002`, ...

# 주의사항

- PRD에 명시되지 않은 기능을 임의로 추가하지 않는다.
- PRD 및 정책 정의를 준수하는 범위 내에서 필요 기능을 정의한다.
- 특정 구현 기술(프레임워크, 라이브러리)을 강제하지 않고 기능의 요구사항만 정의한다.
- 기존 작성 내용에서 변경이 필요한 경우 변경 지점·사유·영향범위·추가 검토 제안사항을 영향 IA 노드의 이력([`../../strategies/ia-history.md`](../../strategies/ia-history.md))에 기록한다.

# 메모리 업데이트

PRD 분석 후 다음 사항을 프로젝트 메모리에 기록한다:
- 도출된 사용자 역할 목록 및 주요 특성.
- 서비스 기능 분류 체계 및 코드 목록.
- SVC ↔ PROC 매핑 관계 요약 (한 시나리오 당 PROC 수 분포).
- 분기·예외 코드(BR-/EX-) 카탈로그 (SVC ↔ PROC 공유).
- 서비스 간 주요 의존 관계.
- PRD에서 기능으로 변환 시 판단이 필요했던 항목 (모호한 요구사항, 해석 근거 등).
- 정책(policy)과의 매핑에서 추가 논의가 필요한 사항.
