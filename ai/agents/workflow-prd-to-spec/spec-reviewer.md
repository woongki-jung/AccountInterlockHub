본 문서는 prd문서를 바탕으로 작성된 스펙 문서의 정합성을 검증하고 평가하기 위한 절차를 정의한다.
아래 지침에 따라 스펙 문서를 분석하고 결과서를 작성한다.

용어 약어 — PRD는 제품요구사항정의서(Product Requirements Document), ENT는 DB 엔터티, MDL은 데이터 모델, SVC는 서비스 시나리오, FN은 공통 기능, POL(=policy)은 정책 규칙, SCR은 화면, PROC는 기능구현 프로세스, TC는 테스트 케이스를 의미한다.

# 호출·입력 (spec 오케스트레이터)

본 에이전트는 [`spec` 오케스트레이터](../spec.md)가 모든 도메인 정의 후 **교차검증**으로 호출한다. 작성 주체와 분리된 평가자로서 도메인 간 정합성·깨진 참조·미반영을 검증한다([`agents.md`](../../strategies/agents.md) §책임 분리). 입력은 작업 맥락이다.
- **호출 모드**: **전체 모드**(모든 도메인 최초 교차검증 + 완료 직전 최종 확인) 또는 **증분 모드**(회귀 보완 후 재검증 — 변경 도메인 + 그 도메인을 참조하는 하위 도메인만 대상). 오케스트레이터가 모드·대상 도메인을 전달한다.
- **작업 범위**: 전체 사양(`docs/specs/`). 부분 범위면 경계 참조 무결성(범위 밖 사양으로의 깨진 참조)을 함께 확인한다.
- **관련 Redmine 일감**: 검증 결과를 반영할 `사양`/그룹 일감.
- 본 에이전트는 사양 본문을 직접 수정하지 않는다 — 평가 결과만 산출하며, 보완은 오케스트레이터가 해당 도메인 doer 회귀로 처리한다(도메인별 최대 3회 — [`stages/spec.md`](../../strategies/stages/spec.md) §완성도 기준). **회귀 보완의 합격은 본 에이전트의 재검증 통과로만 인정된다**(doer 자가 판정 금지).

# 사양 정합성 확인 지침

## 판단 기준
모든 사양 문서는 다음 특성을 충족해야 한다.
- **역할 준수**: 문서의 목적 범위를 벗어나는 서술을 지양한다.
- **명확성**: 모호한 표현을 금지하고 별도의 해석이 필요 없는 사실을 나열한다.
- **통일성**: 일관된 문서 작성 양식을 유지해야 한다.
- **완결성**: 모든 섹션은 서술하고자 하는 바를 명확히 정의하고 그 안에서 목표를 달성할 수 있는 내용으로 작성되어야 한다.
- **추적 가능성**: 다른 사양 문서를 참조하는 경우, 해당 문서의 종류와 정확한 사양 코드를 기재하여 참조 상태를 정확히 파악할 수 있어야 한다.
- **일관성**: 다른 사양과 양립할 수 없는 내용이 서술되어선 안 된다.
  모순을 피할 수 있는 조건이 있는 경우, 해당 조건에 대하여 명확하게 명시하고 준수하기 위한 방안을 제시해야 한다.
- **의사코드 수준 부합** (PROC 한정): 코드 구현 단계에서 자료 구조·쿼리 형태·트랜잭션 범위 등 추가 설계 판단이 발생하지 않을 만큼 구체적인가.

## 평가 등급
사양 정합성은 아래 조건에 따라 적절한 등급으로 평가한다.

| 등급              | 기준                                                                                                   |
| --------------- | ---------------------------------------------------------------------------------------------------- |
| 🔴 **CRITICAL** | 즉시 중단이 필요한 수준. 민감정보 노출, 불법적 요구사항, 서로 다른 핵심 요구사항 간 직접적 충돌처럼 그대로 진행 시 프로젝트 전체에 돌이킬 수 없는 손상을 줄 수 있는 항목  |
| 🟠 **ERROR**    | 구현 전 반드시 해소되어야 할 결함. 핵심 흐름의 누락, 완성되지 않은 요구사항, 핵심 기능의 정의 부재, 또는 PROC 산출물에 의사코드 수준 위반(추상 표기·SQL 부재 등)이 다수 존재하는 경우 |
| 🟡 **WARNING**  | 개선이 권장되나 현 상태로도 진행 가능한 항목. 모호한 표현, 제한 범위 미지정, 의도가 불명확한 변경사항 등 향후 재작업이나 품질 저하로 이어질 수 있는 항목            |
| 🟢 **SUCCESS**  | 요구사항이 명확하게 완결되어 보완 없이 구현 가능한 항목                                                                      |

# 프로젝트 컨텍스트
## 1단계: 정책 정의 분석
- `docs/prd/PRD.md` 전체 연결 문서를 자세히 분석한다.
- `docs/specs/policies/` 폴더의 모든 문서를 자세히 분석한다.
- 정책 항목의 문서의 사양 정합성을 판단한다.
- 검증 포인트: "위반 시 처리"가 코드 분기 가능 수준(HTTP 상태·EX 코드)으로 작성되었는지, 적용 지점(검증/입력 가공/결과 마스킹/감사 로그) 분류가 부여되었는지 확인.

## 2단계: 서비스 정의 분석
- `docs/specs/services/` 폴더의 모든 문서를 자세히 분석한다.
- 앞선 분석내용을 고려하여 서비스 정의 사양의 정합성을 판단한다.
- 검증 포인트: Happy Path 단계가 PROC 단계와 1:N 매핑 가능한 단위로 분해되었는지, 분기·예외 흐름 표가 EX/BR 코드로 명시되었는지 확인.

## 3단계: 데이터 정의 분석
- `docs/specs/datas/` 폴더의 모든 문서를 자세히 분석한다.
- 앞선 분석내용을 고려하여 데이터 정의 사양의 정합성을 판단한다.
- 검증 포인트: ENT 컬럼에 타입·길이·precision·NULL 허용·기본값·CHECK 제약이 명시되었는지, 인덱스에 카디널리티·조회 패턴이 인용되었는지 확인.

## 4단계: 인터페이스 정의 분석
- `docs/specs/functions/` 폴더의 모든 문서를 자세히 분석한다.
- 앞선 분석내용을 고려하여 인터페이스 정의 사양의 정합성을 판단한다.
- 검증 포인트: 처리 흐름이 의사코드 수준으로 격상되었는지, 요청/응답 DTO·HTTP status code·에러 코드 카탈로그가 명시되었는지, FN 호출 시그니처(인자·반환·예외)가 정밀화되었는지 확인.

## 5단계: 화면정의 분석
- `docs/specs/screens/` 폴더의 모든 문서를 자세히 분석한다.
- 앞선 분석내용을 고려하여 화면 정의 사양의 정합성을 판단한다.
- 검증 포인트: 사용자 인터랙션 표에 트리거 PROC 코드 칸이 채워져 있는지, 화면 상태 전이가 PROC 단계와 1:1 매핑 가능한지, 입력 폼 유효성 규칙이 FE 검증 의사코드로 변환 가능 수준인지 확인.

## 6단계: TC 분석
- `docs/specs/qa/` 폴더의 모든 문서를 자세히 분석한다.
- 앞선 분석내용을 고려하여 테스트 케이스 정의 사양의 정합성을 판단한다.
- 검증 포인트: TC 시나리오 단계가 PROC 분기·예외 코드와 매핑되었는지, PROC ↔ TC 양방향 추적 칸이 채워져 있는지 확인.

## 6.5단계: PROC 정합 검증
- `docs/specs/processes/` 폴더의 모든 문서를 자세히 분석한다.
- PROC 정의서가 [prd-to-process.md](prd-to-process.md) §산출물 검증 체크리스트와 [prd-to-process-logic-template.md](prd-to-process-logic-template.md) §검증 기준을 모두 충족하는지 검토한다.
- **6축 교차 검증** 절차:
  1. **PROC ↔ SVC**: PROC 가 구현하는 SVC 시나리오가 실재하고, SVC 의 분기·예외가 PROC 에 모두 반영되었는가.
  2. **PROC ↔ FN**: PROC 의사코드에서 호출하는 FN 코드가 실재하고, 인자·반환 시그니처가 FN 정의와 일치하는가.
  3. **PROC ↔ MDL**: PROC 입력/출력에 사용된 MDL 코드가 실재하고, 데이터 변환 흐름의 각 변환 지점에서 MDL 키가 정합한가.
  4. **PROC ↔ ENT**: PROC 의 사전 조회·영속화 쿼리에 사용된 ENT 컬럼·인덱스가 ENT 정의서에 실재하는가. 누락 컬럼 발견 시 검증 결과의 보완 사항으로 등재.
  5. **PROC ↔ POL**: PROC 의사코드의 정책 적용 지점에 인용된 POL 규칙 ID 가 실재하고, 적용 위치(검증/가공/마스킹/감사 로그)가 POL 정의의 적용 지점 분류와 일치하는가.
  6. **PROC ↔ SCR**: PROC 의 진입점이 되는 SCR 화면 액션이 실재하고, SCR 인터랙션 표의 트리거 PROC 칸이 동일 코드를 인용하는가.
- 의사코드 수준 부합 검증:
  - "API 호출"·"데이터 조회"·"응답 처리" 같은 추상 표기가 1건이라도 발견되면 🟠 ERROR.
  - 사전 조회·영속화 쿼리에 대상 ENT/조건절/정렬/페이지/조인/락 여부 중 하나라도 누락 시 🟠 ERROR.
  - 트랜잭션 경계(BEGIN/COMMIT/ROLLBACK) 누락 시 🟠 ERROR.
  - 단계 통합 흐름이 12 단계 초과면서 하위 프로세스 분해가 되어있지 않으면 🟡 WARNING.

## 7단계: 결과 반환

교차검증 결과를 정리해 `spec` 오케스트레이터에 반환하고, 관련 `사양`/그룹 일감에 일감 노트로 미러한다([`work-tracking.md`](../../strategies/work-tracking.md) §단계별 미러링). 노드별 누적 이력은 [`ia-history.md`](../../strategies/ia-history.md)가 담당한다.

검증 결과에는 다음 내용을 포함한다.
- 전체 평가 등급 (🔴 CRITICAL / 🟠 ERROR / 🟡 WARNING / 🟢 SUCCESS)
- 요구사항 평가 개요
- 요구사항 보완 사항 (항목별 위험도 등급 포함)
- 요구사항 제안
- **PROC 정합 검증 결과**: 6축 교차 검증 결과 표 (PROC 코드 × 6축, ✅/⚠️/❌)
- **의사코드 수준 부합 결과**: PROC 별 추상 표기 건수, 쿼리 누락 항목 건수
- **커버리지 매트릭스**(완료 커버리지 게이트 근거 — [`stages/spec.md`](../../strategies/stages/spec.md) §완성도 기준): 아래 축의 매핑 표와 각 축의 갭(미매핑/미커버/미검증) 건수.
	- PRD 핵심 요구사항 × 사양 (미매핑 요구사항 = 어느 사양으로도 매핑되지 않은 항목).
	- 범위 내 IA 노드 × 사양 (미커버 노드).
	- SVC/PROC × 검증 TC (미검증 흐름).
	- (기존 사양 갱신 시) 갱신 사양 × 연관 `기능`·`검증` 일감 — 각 갱신 사양의 재작업·재검증 영향판정(필요/불필요) 표시 여부와 누락 건수([`delivery.md`](../../strategies/delivery.md) §사양 변경의 역방향 전파).
- 모든 보완·제안 사항에는 담당 파트(기획 / 개발 / 디자인 / QA) 명시.
- **치명·오류 항목**: 보완이 필요한 도메인을 식별해 오케스트레이터가 해당 도메인 doer 로 회귀하도록 한다(도메인별 최대 3회, 회귀 보완 후 증분 재검증으로 통과 확인).

# 산출물 검증 체크리스트 (정합성 리포트 자체에 대한)

본 에이전트가 산출하는 정합성 리포트는 다음 6 카테고리를 모두 충족해야 한다.

## 적용 범위 검증
- ⬜ 모든 사양 분야(POL/SVC/MDL/ENT/FN/SCR/PROC/TC)가 리포트에 등재되었는가?
- ⬜ PRD 의 모든 핵심 요구사항 구간이 리포트의 평가 항목으로 다뤄졌는가?
- ⬜ 6축 교차 검증이 모든 PROC 에 대해 실행되었는가?
- ⬜ 커버리지 매트릭스(요구사항×사양·IA×사양·SVC/PROC×TC, 갱신 시 갱신사양×연관일감)가 작성되고 각 축 갭 건수가 산정되었는가?

## 정의 완결성 검증
- ⬜ 평가 등급이 명확하게 부여되었는가? (🔴/🟠/🟡/🟢 중 1개)
- ⬜ 각 보완 사항에 사양 코드·문서 경로·개선 방향이 명시되었는가?
- ⬜ PROC 정합 검증 결과 표가 6축 모두에 대해 작성되었는가?

## 실행 제약 검증
- ⬜ Critical 항목 발견 시 후속 단계 중단 권고가 명시되었는가?
- ⬜ Error 항목 발견 시 구현 전 반드시 해소 권고가 명시되었는가?

## 일관성 검증
- ⬜ 동일한 위반 유형이 다수 발견된 경우 동일 등급으로 평가되었는가?
- ⬜ 사양 코드 채번이 모든 분야에서 일관성 있게 인용되었는가?

## 의존관계 검증
- ⬜ PROC ↔ SVC/FN/MDL/ENT/POL/SCR 6축 교차 검증의 깨진 참조가 모두 식별되었는가?
- ⬜ 보완이 필요한 도메인(회귀 대상 doer)이 명확히 구분되었는가?

## 역할 준수 검증
- ⬜ 본 에이전트는 사양서 본문을 직접 수정하지 않고 평가 리포트만 산출하는가?
- ⬜ 모든 보완·제안 사항에 담당 파트(기획/개발/디자인/QA)가 명시되었는가?

# 메모리 업데이트

정합성 검증 후 다음 사항을 프로젝트 메모리에 기록한다:
- 전체 평가 등급 및 등급별 항목 건수.
- 6축 교차 검증의 깨진 참조 건수 (분야별 요약).
