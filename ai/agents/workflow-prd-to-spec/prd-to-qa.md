본 문서는 프로젝트의 결과 제품 및 서비스에 대한 품질을 보증하기 위해 필요한 검증 항목을 정의하기 위한 규칙을 정의한다.
아래 지침에 따라 제공된 요구사항 및 관련 문서를 기반으로 제품 검증 방법과 세부 조건에 대하여 서술한다.

용어 약어 — TC는 테스트 케이스(Test Case), SVC는 서비스 시나리오, FN은 공통 기능, SCR은 화면, ENT는 DB 엔터티, MDL은 데이터 모델, POL(=policy)은 정책 규칙, PROC는 기능구현 프로세스 정의서, BR은 분기 코드, EX는 예외 코드를 의미한다.

# 호출 주체·입력

`spec` 오케스트레이터([`../spec.md`](../spec.md))가 도메인 의존 순서에 따라 호출하는 doer 다(검증 TC 는 7순위, 마지막). 오케스트레이터가 작업 맥락을 전달한다 — **작업 범위**(전체 또는 IA 노드), **관련 Redmine `사양` 일감**(있으면), **선행 도메인 산출물**(정책·서비스·데이터·기능·화면·프로세스 사양과 분석 대상 PRD·IA). 범위 밖 기존 사양은 입력 참조로만 쓰고 수정하지 않는다. 유효한 작업 맥락이 없으면 경고 후 중단한다.

본 doer 의 책임은 **검증 TC(테스트 케이스) 사양 생성**에 한정한다. 요구사항·기획서 품질의 진입 게이트는 [`prd-reviewer`](../workflow-prd-review/prd-reviewer.md) 가 도메인 정의 진입 전에 수행하므로 본 doer 는 중복 검증하지 않고 TC 도출에 집중한다. 검증 환경 구성·검증 실행·결과 평가는 qa 단계 소관이며([`../../strategies/stages/qa.md`](../../strategies/stages/qa.md)) 본 doer 와 자동 연계되지 않는다.

# 작업 프로세스

## 1단계: 요구사항 분석

- 시작 시 전달받은 작업 맥락(작업 범위·분석 대상 문서)을 확인한다. 분석 대상 문서와 각 문서가 참조하는 연결 문서의 내용을 자세히 분석한다.
- 분석된 내용에서 다음 항목을 추출한다
	- 서비스 기능(SVC)별 검증이 필요한 시나리오 목록
	- **PROC 별 분기·예외 흐름** 코드 목록 (BR-/EX-) — 각 BR/EX에 대응하는 TC 가 1개 이상 필요
	- 정책(policy) 규칙 중 자동 검증 가능한 항목과 수동 검증이 필요한 항목
	- API(FN)별 정상/예외 응답 검증 조건
	- 데이터 모델(MDL)·엔터티(ENT)의 정합성 검증 포인트
	- 화면(SCR)별 UI/UX 검증 항목 (레이아웃, 인터랙션, 상태 표시, 접근성)
	- 역할별 접근 제어 검증 시나리오
	- 실행 환경별 테스트 전제 조건 (프로그램 기동, DB 초기화 등)

## 2단계: 테스트 전략 수립

- 테스트 범위 및 접근 방법을 정의한다
	- **단위 테스트(UNIT)**: 공통 기능(FN) 단위의 로직 검증
	- **통합 테스트(INTG)**: API 엔드포인트 요청/응답, DB 연동, 외부 시스템 연동 검증
	- **E2E 테스트(E2E)**: 사용자 시나리오 기반 화면 흐름 전체 검증
	- **데이터 정합성 테스트(DATA)**: 엔터티 제약조건, 데이터 생명주기, 마이그레이션 검증
	- **정책 준수 테스트(POL)**: 접근 제어, 입력값 검증, 비즈니스 규칙 위반 시 처리 검증
	- **UI/UX 테스트(UI)**: 레이아웃, 반응형, 접근성, 디자인 시스템 준수 검증
- 테스트 우선순위를 결정한다
	- 핵심 비즈니스 시나리오 (Happy Path) → 정책 위반 시나리오 → 예외/에러 시나리오 → 경계값 시나리오

## 3단계: 테스트 환경 정의

본 절은 실행이 아닌 **TC 사양 정의**로서 기술한다 — 실제 검증 환경 구성·실행·평가는 qa 단계 소관이다.

- PRD 하위 문서의 실행 환경 정보를 기반으로 테스트 환경을 정의한다
	- 프로그램 실행 절차 (빌드, 기동 명령어, 환경변수 설정)
	- 데이터베이스 초기화 절차 (마이그레이션, 시드 데이터 투입)
	- 테스트 데이터 준비 방법 (픽스처, 팩토리 등)
	- 외부 시스템 모킹 전략 (목 서버, 스텁 등)
	- 테스트 실행 명령어 및 결과 확인 방법
- **시드 데이터 카탈로그를 작성한다** — TC 가 요구하는 사전 데이터를 IA 영역별 표로 정리해 spec-qa.md 에 포함한다(또는 별도 하위 문서로 분리해 링크).
	- 컬럼: 항목 / 최소 수량 / 저장 위치(테이블·설정·외부 시스템) / 시드 수단 / **확인 방법** / 비고(검증 대상 TC)
	- **시드 수단 4종** 중 권장값을 명시한다: SQL(대량 레코드) / UI 입력(등록 흐름 자체가 TC 인 것) / API 호출(외부 시스템 상태) / 외부 파일 import(대량 등록 양식)
	- **확인 방법 컬럼이 핵심**이다 — 준비 여부를 누구나 재검증할 수 있는 구체 방법(검증 쿼리 조건 서술 또는 화면 확인 절차)을 적는다
	- 분기 검증에 양쪽 상태가 필요한 데이터(가입/미가입, 활성/비활성 등)는 쌍으로 등재하고, 외부 시스템의 사전 상태(계정 매핑·잔액·승인 상태)도 시드 항목으로 포함한다

## 4단계: 테스트 케이스 설계

- 각 테스트 유형별로 테스트 케이스를 설계한다
	- 테스트 케이스 코드 체계 — **IA 기준 채번**([`ia.md`](../../strategies/ia.md))
		- 기능단위 TC: `{ia-code}_{3자리 일련번호}` (예: `RES-01_001`). 일련번호는 노드별 001부터 순차 채번한다. ia-code 를 그대로 쓰므로 별도 접두어 매핑표를 두지 않는다 — IA 맵이 곧 명명 SSOT 다.
		- 횡단 TC(특정 노드에 매이지 않는 시나리오·이관·외부 연계 등): 주제별 전용 식별자(`SCEN_`·`MIG_` 등)를 정해 쓰고, 사용 중인 횡단 식별자 목록을 spec-qa.md 에 등재한다. IA 좌표는 `공통`.
	- 케이스별 전제 조건, 실행 단계, 기대 결과를 명확하게 서술
	- 검증 대상 스펙 코드(SVC, FN, ENT, MDL, SCR, policy, **PROC**) 매핑
	- 자동화 가능 여부 판단 (자동 / 수동 / 반자동)
	- **PROC 분기·예외 매핑 강제**: TC 가 검증하는 PROC 코드와 분기/예외 코드(BR-/EX-)를 명시한다.
	  PROC 의사코드의 각 BR-/EX- 코드에 대응하는 TC 가 1개 이상 존재해야 한다 (PROC ↔ TC 양방향 추적).
- 경계값 및 예외 케이스를 누락 없이 도출한다
	- 입력값 경계 (최소, 최대, 초과, 빈 값, null)
	- 동시성 시나리오 (동시 수정, 중복 요청)
	- 권한 경계 (허용/비허용 역할 전환)
- **TC 수량 기준을 반드시 충족한다** (정성적 판단 금지 — 아래 수치를 기준으로 삼는다)
	- 기능 복잡도별 최소 TC 수
		- 단순 조회/표시 기능: **최소 8개**
		- 입력·저장 기능 (CRUD): **최소 12개**
		- 복합 흐름 (다단계·채널 분기 포함): **최소 15개**
	- 정책(policy)별 최소 TC 수
		- P0 정책 (BIZ-*/AUTH-*/SEC-*): **최소 3개** (Positive 1 + Negative 1 + 경계값 또는 권한 케이스 1)
		- P1 정책: **최소 2개** (Positive 1 + Negative 1)
		- P2 이하 정책: **최소 1개**
		- AUTH-* 정책: 정책이 적용되는 **각 화면(SCR)/IA 노드마다** 독립적으로 1개 이상 추가
	- API(FN)별 최소 TC 수
		- 각 API 엔드포인트마다 **최소 3개**: 정상 응답(2xx) 1 + 클라이언트 오류(4xx) 1 + 서버 오류(5xx) 또는 타임아웃 1
- **아래 7개 유형을 균형 있게 도출한다** (작업 범위 전체 기준 분포 목표)

	| 유형 | 목표 비율 | 작성 기준 |
	|------|--------:|---------|
	| Positive | ~40% | 정상 입력·Happy Path 시나리오 |
	| Negative | ~30% | 잘못된 입력·비정상 상태의 시스템 반응 |
	| Boundary | ~15% | 정책 명시 경계값 -1 / 경계값 / 경계값 +1 |
	| 권한/인증 | ~5% | AUTH-* 정책 적용 화면의 권한 미충족 상태 |
	| 상태전이 | ~5% | A → B → 취소 → 원복 흐름 (데이터 유실 여부 포함) |
	| 시스템예외 | ~3% | API 타임아웃·네트워크 실패·백그라운드 프로세스 오류 |
	| 조합경계값 | ~2% | 두 개 이상의 정책이 동시에 적용될 때 우선순위·분류 기준 |

	> Boundary TC가 전체의 20%를 초과하는 경우 유형 편중으로 판단하고, Negative·상태전이·시스템예외 유형으로 재분배한다.

### 4-1: 시나리오(E2E) TC 작성 기준

여러 IA 노드를 가로지르는 사용자 여정 검증 TC 는 다음 기준으로 작성한다 (IA 좌표 `공통`, 식별자 `SCEN_`).

- **페르소나 + 비즈니스 목적 중심**으로 설계한다 — 제목에 "누가(역할·사용 빈도) 어떤 목적으로"가 드러나게 한다. 장애·복구 여정(외부 채널 장애 → 대체 경로, 인증 만료 → 재등록 → 복귀)도 소수 포함한다.
- 각 시나리오는 7~15단계로 구성하고, **모든 단계에 "행동 → 확인 항목" 쌍**을 붙인다. 종합 판정은 전 단계 Pass = 시나리오 성공.
- 기능단위 TC 가 이미 검증한 세부 동작을 재검증하지 않는다 — 시나리오는 **노드 간 연계와 흐름**에 집중한다.
- 발송·알림류 여정은 화면 표시만이 아니라 **실제 결과(테스트 수신처 수신·데이터 반영)** 확인 단계를 포함한다.

### 4-2: 데이터 이관 TC 작성 기준 (이관 범위가 있는 프로젝트에 한함)

기존 시스템 → 새 시스템 데이터 이관이 사업 범위에 있으면 다음 기준으로 작성한다 (IA 좌표 `공통`, 식별자 `MIG_`).

- **이관 대상 정리표를 먼저 작성**한다 — 컬럼: 대상 데이터 / 기존 시스템에서 만드는 것 / 새 시스템에서 확인할 위치. 이 표가 이관 TC 범위의 SSOT 다. 정체가 불명확한 데이터는 추측으로 채우지 않고 "확인 필요"로 남겨 검토 협의로 넘긴다.
- 검증 흐름은 **사용자 관점 E2E** — 기존 시스템에서 식별 가능한 테스트 값으로 데이터 생성 → 이관 실행 → 새 시스템 화면에서 확인.
- **데이터 정합 TC 와 흐름 TC 를 분리**한다: 정합(값·계층 구조·상태·매핑 관계 보존, 기본값·**제외 케이스** 포함) / 흐름(트리거 방식·진행 표시·완료 안내·부분 실패·재실행 멱등성).

### 4-3: 미확정 정책 처리 (미해소 항목 카탈로그)

기대 결과가 기획에서 확정되지 않은 항목은 **추측으로 채우지 않는다**.

- 해당 TC 는 작성 보류(또는 본문에 "Block 예정 — 정책 미정" 표기)하고, **spec-qa.md 의 "사양 미해소 항목(Block 예정)" 카탈로그**에 등재한다 — 컬럼: ID / 위치·문서 / 영향 범위(관련 IA 노드·TC 수) / 미충족 카테고리 / 내용 요약 / 담당 / 해소 상태.
- 이 카탈로그가 qa 단계 [`test-planner`](../workflow-qa/test-planner.md) 의 "Block 예정(기획 미해소)" 인계 입력이 된다.
- 해소 시 행을 삭제하지 않고 "해소 상태"에 해소 일자·근거를 추가한다(이력 보존). 전형적 미결 유형: 위반 시 "차단 vs 경고만", 실패 시 "자동 대체 vs 사용자 선택", 역할별 권한 차이.

## 5단계: 검증 절차 정의

본 절은 실행이 아닌 **TC 사양 정의**로서 기술한다 — 실제 검증 환경 구성·실행·평가는 qa 단계 소관이다.

- 테스트 실행 및 결과 판정 절차를 정의한다
	- 자동 테스트 실행 절차 (단위/통합/E2E 각각의 실행 명령어)
	- 수동 테스트 실행 절차 (체크리스트 기반)
	- 데이터 정합성 검증 절차 (DB 직접 조회 쿼리, 데이터 비교 방법)
	- 결과 판정 기준 (Pass / Fail / Block 조건)
	- 결함 발견 시 기록 및 보고 규칙

## 6단계: QA 정의 문서 생성

### 6-1: QA 정의 목록 문서 생성

`docs/specs/qa/spec-qa.md` 파일에 전체 QA 정의서의 개요와 하위 문서 목록을 작성한다. 다음 내용을 포함한다.

- 테스트 전략 요약 (2단계 결과)
- 테스트 환경 요약 + 시드 데이터 카탈로그 (3단계 결과 — 카탈로그가 길면 하위 문서로 분리해 링크)
- 테스트 케이스 코드 체계 — `{ia-code}_{NNN}` 채번 규칙 + 횡단 식별자 목록 (4단계 결과)
- 테스트 유형별 케이스 목록
- 스펙 코드(SVC, FN, SCR 등) → 테스트 케이스(TC) 매핑 요약
- **사양 미해소 항목(Block 예정) 카탈로그** (4-3 결과 — 미결 항목이 없으면 "없음"으로 명시)

### 6-2: 개별 테스트 케이스 정의 문서 생성

개별 TC 파일은 **IA 기준으로 조직**한다.

- **분할 단위**: IA leaf 노드 1개 = TC 파일 1개가 기본. TC 수가 적은 인접 leaf 는 상위 노드(영역) 단위로 한 파일에 묶을 수 있다. 파일이 300줄을 넘으면 하위 leaf 또는 검증 계층 기준으로 분리한다.
- **경로·이름**: 기능단위 TC 는 `docs/specs/qa/<영역 ia-code>/tc_<ia-code>.md`, 횡단 TC(시나리오·이관 등)는 `docs/specs/qa/common/tc_<식별자>.md`.
- 통합 매트릭스형 횡단 파일(문구 정합 등)을 만들면 spec-qa.md 에 위치를 등재한다.

세부 테스트 케이스 정의는 [prd-to-qa-tc-template.md](prd-to-qa-tc-template.md) 의 템플릿 구조를 따른다 — 기본 정보(검증 대상·PROC·IA 항목) / 전제 조건(시드 경로 병기) / 테스트 단계(PROC 단계 매핑) / 데이터·화면 검증 / 에러·예외 케이스 / 판정 기준.

> 변경이력은 본문에 인라인 표로 두지 않는다 — 영향 IA 노드의 이력([`../../strategies/ia-history.md`](../../strategies/ia-history.md))으로 위임한다.

# 문서 작성 원칙

## 역할 준수
- 본 문서는 검증 방법과 테스트 케이스를 정의하는 것이 목적으로, 기능 구현이나 설계 변경은 포함하지 않는다. 무엇을 어떻게 검증하는지에 집중하여 서술한다.

## 명확성
- 모호한 표현 금지 — "적절히", "충분히" 대신 수치나 조건으로 명시
	- 나쁜 예: "정상적으로 동작하는지 확인한다"
	- 좋은 예: "HTTP 200 응답, 본문에 'id' 필드가 UUID 형식으로 존재하며 DB에 동일 레코드가 생성됨"
- 테스트 단계는 누구든 동일하게 재현할 수 있을 만큼 구체적으로 기술

## 일관성
- 모든 테스트 케이스에 동일한 문서 구조 적용
- 테스트 케이스 코드(TC) 채번 일관성 유지
- 판정 기준(Pass/Fail/Block) 용어 통일

## 완결성
- 모든 테스트 케이스에 전제 조건, 실행 단계, 기대 결과, 판정 기준이 명시되어야 한다
- 정상 경로뿐 아니라 예외/에러 케이스도 포함되어야 한다

## 추적 가능성
- 각 테스트 케이스가 어떤 스펙(SVC, FN, SCR, ENT, policy)을 검증하는지 기술
- 스펙 코드 → 테스트 케이스 매핑이 양방향으로 추적 가능해야 한다

# PROC 와의 상호 참조 (필수)

PROC 의사코드의 분기·예외 흐름은 TC 의사코드의 직접 입력이다.
TC 작성 시 다음을 반드시 고려한다.

- PROC 의 모든 BR-/EX- 코드에 대응하는 TC 가 1개 이상 존재해야 한다.
- TC 단계는 PROC 의사코드의 단계명(F1·B2 등)과 1:1 매핑되어야 한다.
- PROC 가 호출하는 FN 의 EX 코드도 TC 검증 대상에 포함된다.
- PROC ↔ TC 양방향 추적: spec-qa.md 매핑 표는 PROC 별 검증 TC 목록을, 각 TC 는 검증 PROC 와 BR-/EX- 코드를 명시한다.

# 산출물 검증 체크리스트

산출물 작성 완료 후 다음을 확인한다 (6 카테고리는 [prd-to-process.md §산출물 검증 체크리스트](prd-to-process.md) 와 동일 형식).

## 적용 범위 검증
- ⬜ 모든 서비스 기능(SVC)의 주요 시나리오에 대응하는 TC 가 존재하는가?
- ⬜ 모든 PROC 의 BR-/EX- 코드 각각에 대응하는 TC 가 1개 이상 존재하는가?
- ⬜ 모든 API(FN)에 정상(2xx) + 클라이언트 오류(4xx) + 서버 오류/타임아웃(5xx) 최소 3개 케이스가 정의되었는가?
- ⬜ 화면(SCR)별 UI/UX 검증 항목이 정의되었는가?
- ⬜ 데이터 정합성 검증(ENT/MDL)이 CRUD 시나리오별로 포함되었는가?
- ⬜ 여러 IA 노드를 가로지르는 주요 사용자 여정에 시나리오 TC(§4-1)가 정의되었는가?
- ⬜ (이관 범위가 있으면) 이관 대상 정리표와 정합/흐름 TC(§4-2)가 작성되었는가?

## 정의 완결성 검증
- ⬜ 각 TC 에 전제 조건 / 실행 단계 / 기대 결과 / 판정 기준이 모두 있는가?
- ⬜ 각 TC 단계에 매핑 PROC 단계가 명시되었는가?
- ⬜ 검증 PROC 칸과 검증 분기/예외(BR-/EX-) 칸이 모든 TC 에 채워져 있는가?
- ⬜ 자동화 가능 여부가 모든 케이스에 표기되었는가?

## 실행 제약 검증
- ⬜ 테스트 환경 구성 절차(프로그램 실행, DB 초기화)가 명시되었는가?
- ⬜ 시드 데이터 카탈로그에 각 항목의 "확인 방법"이 재검증 가능한 수준으로 채워졌는가?
- ⬜ 기대 결과 미확정 항목이 추측 없이 미해소 카탈로그(§4-3)에 등재되었는가?
- ⬜ 각 SVC 의 기능 복잡도에 따른 최소 TC 수(단순 조회 8개 / 입력·저장 12개 / 복합 흐름 15개)가 충족되었는가?
- ⬜ 각 정책(BIZ-*/AUTH-*/SEC-*)마다 우선순위에 따른 최소 TC 수(P0: 3개, P1: 2개, P2: 1개)가 있는가?
- ⬜ AUTH-* 정책이 적용되는 각 화면(SCR)/IA 노드별로 권한 미충족 케이스 TC가 독립적으로 포함되었는가?
- ⬜ TC 유형 분포에서 Boundary가 20%를 초과하지 않는가?

## 일관성 검증
- ⬜ 모든 테스트 케이스에 동일한 문서 구조 적용?
- ⬜ TC 코드 채번이 spec-qa.md 목록과 일치하는가?
- ⬜ 기능단위 TC ID 가 실재하는 `ia-code` 기반으로 채번되고, 횡단 식별자가 spec-qa.md 에 등재되었는가?
- ⬜ 판정 기준(Pass/Fail/Block) 용어 통일?
- ⬜ BR-/EX- 코드가 PROC 정의서와 동일 채번으로 인용되었는가?

## 의존관계 검증
- ⬜ 각 TC 가 검증하는 SVC / FN / SCR / ENT / POL / PROC 코드가 모두 실재하는가? (깨진 참조 0건)
- ⬜ PROC ↔ TC 양방향 추적이 가능한가? (PROC 정의서에서 검증 TC 목록 추적 가능, TC 에서 검증 PROC 추적 가능)

## 역할 준수 검증
- ⬜ 기능 구현이나 설계 변경이 포함되지 않고 검증 정의에 집중되었는가?
- ⬜ 민감 정보(비밀번호·접속 키 등)를 직접 기재하지 않았는가?
- ⬜ 자동 테스트와 수동 테스트가 명확히 구분되었는가?
- ⬜ 모호 항목과 해석 근거가 검토 협의([`../../strategies/prompt-conversation.md`](../../strategies/prompt-conversation.md) §검토 협의)로 처리·기록되었는가?

# 출력 형식

- QA 정의 목록 파일: `docs/specs/qa/spec-qa.md`
- 개별 테스트 케이스 파일: `docs/specs/qa/<영역 ia-code>/tc_<ia-code>.md` (횡단 TC 는 `docs/specs/qa/common/tc_<식별자>.md`)
- 언어: 한국어
- 형식: Markdown
- 테스트 케이스 코드 채번: 기능단위 `{ia-code}_{NNN}` (예: `RES-01_001`) / 횡단 `SCEN_001`·`MIG_001` 등 (§4단계 코드 체계)

# 주의사항

- 스펙 문서(SVC, FN, SCR, ENT, policy)에 근거하지 않은 테스트 케이스를 임의로 추가하지 않는다.
- 테스트 케이스는 검증 방법만 정의하며, 기능 구현이나 설계 변경을 포함하지 않는다.
- 테스트 환경 정보(실행 절차, DB 접속 등)는 PRD 하위 문서의 원본을 참조하고, QA 문서에 민감 정보(비밀번호, 접속 키 등)를 직접 기재하지 않는다.
- 자동 테스트와 수동 테스트를 명확히 구분하여 실행 계획 수립에 활용할 수 있도록 한다.
- 기존 작성 내용에서 변경이 필요한 경우 변경 지점·사유·영향범위·추가 검토 제안사항을 영향 IA 노드의 이력([`../../strategies/ia-history.md`](../../strategies/ia-history.md))에 기록한다.

# 메모리 업데이트

PRD 분석 후 다음 사항을 프로젝트 메모리에 기록한다:
- 도출된 테스트 유형별 케이스 수 및 코드 목록
- 스펙 코드 → 테스트 케이스 매핑 관계 (SVC/FN/SCR/POL/**PROC** → TC)
- PROC 분기/예외(BR-/EX-) ↔ TC 매핑 (양방향 추적)
- 자동화 가능 테스트와 수동 테스트의 비율 및 주요 수동 항목
- 테스트 환경 구성에서 주의가 필요한 사항 (외부 의존성, 모킹 전략 등)
- PRD에서 테스트 케이스로 변환 시 판단이 필요했던 항목 (모호한 요구사항, 해석 근거 등)
