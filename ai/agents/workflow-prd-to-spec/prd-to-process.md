본 문서는 요구사항을 분석하고 사용자 정의 및 서비스 시나리오를 바탕으로 제공 기능의 실행 프로세스를 정의한다.
아래 지침에 따라 제공된 요구사항 및 정책 문서를 바탕으로 각 구현 기능 별, 사전 요구사항, 기능 프로세스, 단계별 세부 구현 지침을 서술한다.

용어 약어 — SVC는 서비스 시나리오 정의서 코드, FN은 공통 기능 정의서 코드, MDL은 데이터 모델, ENT는 DB 엔터티, SCR은 화면 정의서 코드, PROC는 기능구현 프로세스 정의서 코드, BR은 분기 코드(Branch), EX는 예외 코드(Exception), DTO는 데이터 전송 객체(Data Transfer Object)를 의미한다.

# 호출 주체·입력

`spec` 오케스트레이터([`../spec.md`](../spec.md))가 도메인 의존 순서에 따라 호출하는 doer 다(프로세스는 6순위). 오케스트레이터가 작업 맥락을 전달한다 — **작업 범위**(전체 또는 IA 노드), **관련 Redmine `사양` 일감**(있으면), **선행 도메인 산출물**(정책·서비스·데이터·기능·화면 사양과 분석 대상 PRD·IA). 범위 밖 기존 사양은 입력 참조로만 쓰고 수정하지 않는다. 유효한 작업 맥락이 없으면 경고 후 중단한다.

# 작업 프로세스
## 1단계: 요구사항 분석

- 시작 시 전달받은 작업 맥락(작업 범위·분석 대상 문서)을 확인한다.
  분석 대상 문서와 각 문서가 참조하는 연결 문서의 내용을 자세히 분석한다.
- 분석 대상에는 다음 사양 문서들이 포함되어야 한다.
	- 서비스 기능정의서(SVC) — 시나리오·기능 단위·선후관계
	- 정책 정의서(policy) — 프로세스 실행 중 적용되어야 할 규칙·제약
	- 공통 기능 정의서(FN) — 프로세스가 호출할 비즈니스 로직 단위
	- 데이터 모델 정의서(MDL) 및 DB 엔터티 정의서(ENT) — 입출력·영속화 대상
	- 화면 정의서(SCR) — 프로세스 진입 트리거가 되는 사용자 액션·이벤트
	- QA 정의서 — 검증 시나리오에서 도출되는 분기·예외 흐름 (검증 TC 는 7순위 후행 산출물 — 재수행 시 존재하면 참조)
- 분석된 내용에서 다음 항목을 추출한다.
	- 코드 구현 단위가 되는 프로세스 후보 목록 (서비스 시나리오의 흐름 단위로 분해)
	- 각 프로세스의 **진입점(트리거)** 유형 — 사용자 액션, 화면 이벤트, API 호출, 스케줄러, 외부 시스템 콜백 등
	- 각 프로세스의 **진입 조건 및 사전 검증** — 인증 상태, 권한, 사전 데이터 존재 여부, 입력값 유효성
	- 프로세스가 참조·호출하는 **연관 사양 코드** — SVC, FN, MDL, ENT, policy, SCR
	- 프로세스 실행 중 호출되는 **외부 시스템 연동** — 외부 API, 알림, 결제, 메시지 발송 등
	- **FE(프론트엔드) 측 처리 로직** — 사용 상태·form 키, 입력 검증 규칙, 요청 DTO(데이터 전송 객체) 변환, 데이터 조회/변경 호출 메커니즘(query/mutation 훅 등)·타깃 엔드포인트, 진행 중 UI 처리, 응답 어댑터 변환, 캐시·전역 상태 갱신, UI 전이, 실패 처리
	- **BE(백엔드) 측 처리 로직** — 컨트롤러 엔드포인트·메서드, 인증·인가 검증, 입력 DTO 재검증, 사전 조회 쿼리(대상 ENT·조건·정렬·페이지·조인·락), 비즈니스 규칙 분기, 영속화 쿼리(INSERT/UPDATE/DELETE 대상·필드·조건절), 트랜잭션 경계, 외부 시스템 호출, 후속 이벤트 발행, 응답 DTO 변환
	- **데이터 변환 지점** — FE→요청 DTO, 요청→도메인 모델, 도메인→ENT, ENT→도메인, 도메인→응답 DTO, 응답→FE 모델 각 변환 위치와 변환 규칙
	- 프로세스 단계별 **로직 실행 순서** 및 단계 간 데이터 흐름 (FE/BE 의사코드 수준)
	- 단계별 **분기 조건** — 정상 흐름 / 분기 흐름 / 예외 흐름의 발생 조건과 처리 방향
	- 정책 규칙이 적용되는 **삽입 지점** — 권한 확인, 입력값 가공, 결과 마스킹, 감사 로그 등
	- **트랜잭션 경계** 및 실패 시 보상 처리(롤백·재시도·보상 트랜잭션) 요구사항
	- 처리 방식 구분 — 동기 / 비동기 / 백그라운드 / 큐잉, 후속 작업 트리거 여부
	- 프로세스의 **실행 결과** — 응답 데이터, 영속화 대상, 후속 알림·이벤트 발행
	- 프로세스 간 **선후관계 및 의존성** (한 프로세스의 결과가 다른 프로세스의 입력이 되는 관계)
	- 비기능 요구사항 — 성능 임계치, 동시성·중복 실행 방지, 멱등성(idempotency) 요구
- 추출이 모호하거나 사양 문서에서 누락된 항목은 검토 협의 기록으로 남긴다. 담당자 결정이 필요한 항목은 [검토 협의](../../strategies/prompt-conversation.md)(담당자 질의응답)로 확인하여 응답을 반영하고, 자명한 항목만 합리적 해석으로 진행한다.

## 2단계: 구현 프로세스 정의

- 1단계에서 추출한 프로세스 후보를 코드 구현 단위로 정제하고, 그룹·분류·채번 체계를 설계한다.
- 프로세스 단위 정제 기준
	- 하나의 프로세스는 단일 책임을 가지며, 진입점(트리거) 1개와 명확한 종료 조건을 갖는다.
	- 서로 다른 트리거에서 호출되더라도 내부 로직이 동일하면 하나의 프로세스로 통합하고, 동일 트리거라도 분기 로직이 독립적이면 별도 프로세스로 분리한다.
	- 한 프로세스의 단계 수가 과도하게 길어지면(권장 12 단계 초과) 하위 프로세스로 분해하고 호출 관계로 표현한다.
	- 프로세스가 호출하는 공통 로직은 공통 기능(FN)으로 분리하여 중복 정의를 피한다.
- 프로세스 그룹화 기준 (둘 이상 적용 가능)
	- **서비스 도메인 기준** — 동일한 SVC 코드 또는 업무 영역에 속하는 프로세스를 그룹화
	- **트리거 유형 기준** — 사용자 요청 / 시스템 이벤트 / 스케줄 / 외부 콜백
	- **데이터 영향 기준** — 조회 전용(read-only) / 상태 변경(mutate) / 외부 송신(side-effect)
- 프로세스 분류 체계 (각 프로세스에 1개 이상의 분류 태그 부여)
	- **요청-응답형(RR)**: 사용자 또는 외부 호출에 동기적으로 응답하는 프로세스 (조회·등록·수정·삭제 등)
	- **이벤트 처리형(EVT)**: 시스템 내·외부 이벤트 발생을 트리거로 비동기 실행되는 프로세스
	- **배치/스케줄형(BAT)**: 정해진 주기 또는 시점에 일괄 실행되는 프로세스
	- **워크플로형(WF)**: 다수 단계·다수 참여자·장기 실행을 포함하는 상태 기반 프로세스
	- **연동형(INT)**: 외부 시스템 호출이 핵심인 프로세스 (외부 API, 결제, 알림 발송 등)
- 프로세스 코드 체계를 설계한다 (`PROC-nnn` 형태).
	- 그룹별 번호 구간을 분리하여 채번하면 시인성이 좋아진다 (예: 인증 PROC-100~, 메시지 PROC-200~).
	- 한 번 부여된 코드는 재사용하지 않는다. 프로세스가 폐기되어도 코드 결번으로 유지한다.
- 프로세스 간 의존관계를 정의한다.
	- **호출 관계**: 한 프로세스가 다른 프로세스를 동기·비동기로 호출하는 관계
	- **선행 관계**: 다른 프로세스가 완료된 상태를 전제로 실행되는 관계 (데이터 존재·상태 전이 의존)
	- **이벤트 관계**: 한 프로세스의 결과가 다른 프로세스의 트리거가 되는 관계
	- 순환 의존이 발생하지 않도록 검증하고, 의존 관계를 표(호출·선행·이벤트 관계 컬럼)로 정리하여 문서에 첨부한다(도식 대신 텍스트·표로 표현).
- 상위 사양과의 매핑을 정리한다.
	- SVC 시나리오 → 구성 프로세스 목록 (한 시나리오는 1개 이상의 프로세스로 구현)
	- SCR 화면 액션 → 트리거되는 프로세스 (한 화면 액션은 1개의 프로세스를 트리거)
	- policy 규칙 → 적용되는 프로세스 및 적용 지점
	- 데이터 제어 규칙 -> CRUD 실행 대상 데이터
	- API 호출 규칙 -> 기능구현을 위해 호출해야 할 API 구현사항
- PRD·정책 미정 항목으로 프로세스 분리 여부 또는 분류 판단이 어려운 경우는 검토 협의([`../../strategies/prompt-conversation.md`](../../strategies/prompt-conversation.md) §검토 협의)로 처리·기록한다.

## 3단계: 기능구현 프로세스정의 작성

### 3-1: 기능구현 프로세스 정의 목록 문서 생성

`docs/specs/processes/spec-process.md` 파일에 기능구현 프로세스정의서의 개요와 하위 문서 목록을 작성한다. 기능구현 프로세스정의서는 다음 내용을 포함한다.

- 관련 사양 코드 (policy, data, function, service, screen)
- 기능 진입점 및 진입 조건
- 연관 데이터 (호출 API 및 데이터 모델)
- 실행 제약사항
- 로직 실행 순서 및 내용
- 기능 실행 결과

### 3-2: 개별 서비스 기능 정의 문서 생성

각 프로세스별로 `docs/specs/processes/process_<process-code>.md` 파일을 생성한다.
"로직 실행 순서" 섹션은 [로직 실행 순서 — FE/BE 의사코드 템플릿](prd-to-process-logic-template.md)을 따라 FE 측 처리·BE 측 처리·데이터 변환 흐름·단계 통합 흐름 4개 하위 섹션을 모두 의사코드 수준으로 작성한다.
"API 호출"·"데이터 조회"·"응답 처리" 같은 추상 표기를 금지하고, 코드 구현 단계에서 자료 구조·쿼리 형태·트랜잭션 범위 등 추가 설계 판단이 발생하지 않을 만큼 구체적이어야 한다.

```markdown
# [구현 프로세스명] 기능 정의

## 개요
- **정의 대상**: 이 프로세스가 설명하는 목표 구현 기능
- **관련 PRD 요구사항**: PRD 내 관련 섹션 및 원문 인용

---

## [PROC 코드] [기능명]

### 기본 정보
| 항목 | 내용 |
|------|------|
| 프로세스명 | 프로세스 이름 |
| 분류 | RR / EVT / BAT / WF / INT 중 1개 이상 |
| 그룹 | 서비스 도메인 또는 업무 영역 |
| 트리거 유형 | 사용자 액션 / 시스템 이벤트 / 스케줄 / 외부 콜백 |
| 처리 방식 | 동기 / 비동기 / 백그라운드 |
| 우선순위 | 높음 / 보통 / 낮음 |
| 관련 IA 항목 | 본 사양이 속한 IA 노드의 `ia-code`(들). 횡단 사양은 `공통` (정본 [`../../strategies/ia.md`](../../strategies/ia.md) §참조 규약) |

### 관련 사양 코드
| 구분 | 코드 | 관계 설명 |
|------|------|----------|
| 서비스(SVC) | SVC-xxx | 이 프로세스가 구현하는 서비스 시나리오 |
| 정책(policy) | policy_xxx | 적용되는 정책 규칙 및 적용 지점 |
| 공통 기능(FN) | FN-xxx | 호출하는 공통 비즈니스 로직 |
| 데이터 모델(MDL) | MDL-xxx | 입출력에 사용되는 데이터 모델 |
| DB 엔터티(ENT) | ENT-xxx | 조회·변경 대상 엔터티 |
| 화면(SCR) | SCR-xxx | 트리거가 발생하는 화면 |

### 진입점 및 진입 조건
- **진입점**: 트리거가 발생하는 위치 (화면 액션·API 엔드포인트·이벤트명·스케줄 표현식 등)
- **진입 조건**: 인증 상태, 권한, 사전 데이터 존재 등 진입을 허용하는 조건
- **사전 검증**: 진입 시 수행해야 하는 입력값 유효성·권한·중복 검사 항목

### 입력/출력 정의
| 구분 | 항목명 | 데이터 타입 | 필수 | 설명 |
|------|--------|------------|------|------|
| 입력 | param_a | string | Y | 파라미터 설명 |
| 출력 | result | MDL-xxx | - | 반환 데이터 모델 |

### 연관 데이터 및 외부 호출
- **호출 API**: 외부 시스템 API 엔드포인트·메서드·요청/응답 형식 (관련 FN 또는 외부 명세 참조)
- **데이터 조회 대상**: 조회하는 ENT/MDL 코드 및 조회 조건
- **데이터 변경 대상(CRUD)**: 생성·수정·삭제하는 ENT/MDL 및 변경 단위

### 실행 제약사항
- **트랜잭션 경계**: 단일 트랜잭션 / 다중 트랜잭션 / 보상 트랜잭션 여부
- **동시성 제어**: 락 전략, 멱등성(idempotency) 키, 중복 실행 방지 방법
- **성능 요구**: 최대 응답 시간, 처리량, 타임아웃 등
- **보안 요구**: 인증·인가 수준, 데이터 마스킹, 감사 로그 적용 범위

### 로직 실행 순서

다음 4개 하위 섹션을 모두 작성한다 (상세 가이드 — 프로세스 로직 작성 지침(prd-to-process-logic-template)을 따른다).
"API 호출"·"데이터 조회" 같은 추상 표기를 금지하고, 코드 구현 단계에서 추가 설계 판단이 발생하지 않을 만큼 의사코드 수준으로 펼쳐 작성한다.

#### FE 측 처리 (의사코드)
- 단계별로 진입 트리거 / 사용 state·form / 검증 규칙 / 요청 DTO 변환 / 호출 훅·엔드포인트 / 진행 중 UI / 응답 어댑터 변환 / 캐시·전역 상태 갱신 / UI 전이 / 실패 처리 / 정책 적용 지점을 모두 의사코드로 기술

#### BE 측 처리 (의사코드)
- 단계별로 엔드포인트·메서드 / 인증·인가 / 입력 DTO 재검증 / 사전 조회 쿼리(대상 ENT·조건·정렬·페이지·조인·락) / 비즈니스 규칙 분기 / 영속화 쿼리(INSERT/UPDATE/DELETE 대상·필드·조건절) / 트랜잭션 경계 / 외부 시스템 호출 / 후속 이벤트 / 응답 DTO 변환 / 정책 적용 지점을 모두 의사코드로 기술

#### 데이터 변환 흐름
- FE→요청 / 요청→도메인 / 도메인→ENT / ENT→도메인 / 도메인→응답 / 응답→FE 변환 지점을 표로 정리 (변환 위치·입출력 형태·변환 규칙). 해당 프로세스에 적용되는 변환 지점만 작성

#### 단계 통합 흐름
- 위 FE/BE 단계를 시간 순서대로 통합 (권장 12 단계 이내, 초과 시 하위 프로세스로 분해)
- 각 행: # · 레이어(FE/BE) · 단계명 · 직전 단계 출력 · 본 단계 처리 요지 · 다음 단계 입력

### 분기 및 예외 흐름
| 코드 | 발생 조건 | 처리 방향 | 결과 |
|------|----------|----------|------|
| BR-001 | 분기 발생 조건 | 분기 처리 단계 | 결과 상태 |

### 실행 결과
- **정상 결과**: 응답 데이터, 영속화된 상태 변경, 발행 이벤트
- **실패 결과**: 에러 코드, 응답 메시지, 후속 보상 처리
- **후속 트리거**: 본 프로세스 완료 시 트리거되는 다른 프로세스(PROC) 코드

### 의존 프로세스
- **호출 관계**: 본 프로세스가 호출하는 PROC 코드 및 호출 시점 (동기/비동기)
- **선행 관계**: 본 프로세스 실행 전 완료되어야 하는 PROC 코드
- **이벤트 관계**: 본 프로세스의 결과가 트리거하는 PROC 코드
```

# 문서 작성 원칙

## 역할 준수
- 본 문서는 기능을 정의하는 것이 목적으로, UI 구성은 포함하지 않음. 별도 정의될 화면정의서와 충돌하지 않도록 구현되어야 할 기능에 집중하여 서술한다.
## 명확성
- 모호한 표현 금지 — "적절히", "충분히" 대신 수치나 조건으로 명시
	- 나쁜 예: "비밀번호는 충분히 복잡해야 한다"
	- 좋은 예: "비밀번호는 8자 이상, 영문 대소문자·숫자·특수문자 각 1자 이상 포함"
- 기능 정의는 개발자가 코드로 바로 변환할 수 있을 만큼 구체적으로 기술
- 로직 실행 순서는 "API 호출"·"데이터 조회" 같은 추상 표기 대신 FE 측·BE 측 각각의 의사코드 수준으로 펼쳐 작성한다 (작성 예시는 [의사코드 템플릿](prd-to-process-logic-template.md) 참조).
- 사전 조회·영속화 쿼리는 대상 ENT, WHERE 조건, ORDER BY, LIMIT/OFFSET, JOIN, 락 여부를 SQL/ORM 의사코드로 명시한다.
## 일관성
- 모든 서비스 기능에 동일한 문서 구조 적용
- 프로세스 코드 채번 일관성 유지
- 상충되는 기능 정의가 없도록 교차 검토
## 완결성
- 사용자 중심으로 서비스 시나리오에 따라 흐름이 이어질 수 있도록 어떤 사용 단계인지, 다음 단계는 어떤 경우의 수가 있는 지 명시
## 추적 가능성
- 각 기능이 PRD의 어떤 요구사항에서 비롯되었는지 기술
- 기능과 연관된 정책(policy) 기술
- 연계되거나 영향을 줄 수 있는 기능 기술


# 산출물 검증 체크리스트

산출물 작성 완료 후 다음을 확인한다.

## 적용 범위 검증
⬜ 모든 SVC 시나리오가 1개 이상의 프로세스(PROC)로 구현되었는가?
⬜ 모든 SCR 화면 액션이 트리거하는 프로세스가 정의되었는가?
⬜ 모든 정책(policy) 규칙이 적용되는 프로세스 및 적용 지점이 매핑되었는가?
⬜ PRD 요구사항 중 어느 프로세스에도 매핑되지 않은 항목이 없는가?

## 정의 완결성 검증
⬜ 각 프로세스의 진입점·진입 조건·사전 검증이 구체적으로 기술되었는가?
⬜ 각 프로세스의 입력/출력이 데이터 모델(MDL) 참조와 함께 정의되었는가?
⬜ 호출하는 외부 API의 엔드포인트·메서드·요청/응답 형식이 정의되었는가?
⬜ 데이터 변경 대상(CRUD)이 ENT/MDL 코드로 명시되었는가?
⬜ 로직 실행 순서가 FE 측 처리·BE 측 처리·데이터 변환 흐름·단계 통합 흐름 4개 하위 섹션을 모두 포함하는가?
⬜ FE 측 의사코드가 진입 트리거·사용 state/form·검증 규칙·DTO 변환·호출 훅·캐시 갱신·UI 전이·실패 처리를 모두 기술하는가?
⬜ BE 측 의사코드가 엔드포인트·인증·재검증·사전 조회 쿼리·비즈니스 규칙·영속화 쿼리·트랜잭션 경계·후속 이벤트를 모두 기술하는가?
⬜ 사전 조회·영속화 쿼리에 대상 ENT, 조건절, 정렬, 페이지, 조인, 락 여부가 SQL/ORM 의사코드로 명시되었는가?
⬜ "API 호출"·"데이터 조회" 같은 추상 표기 없이 코드 구현 단계에서 추가 설계 판단이 불필요할 만큼 의사코드 수준이 충분한가?
⬜ 각 단계의 정책 적용 지점(policy 코드와 적용 위치)이 표시되었는가?
⬜ 분기 흐름과 예외 흐름이 모두 정의되었고, 각 분기·예외에 대한 처리 방향이 기술되었는가?
⬜ 실행 결과(정상·실패·후속 트리거)가 모두 정의되었는가?

## 실행 제약 검증
⬜ 트랜잭션 경계와 실패 시 보상 처리(롤백·재시도·보상)가 명시되었는가?
⬜ 동시성 제어 전략과 멱등성(idempotency) 요구가 명시되었는가?
⬜ 성능·보안 비기능 요구사항이 정의되었는가?

## 일관성 검증
⬜ 프로세스 분류(RR/EVT/BAT/WF/INT)가 모든 프로세스에 부여되었는가?
⬜ 프로세스 코드(PROC-nnn) 채번이 spec-process.md 목록과 일치하는가?
⬜ 그룹별 코드 구간이 일관되게 적용되었는가?
⬜ 모든 프로세스 정의 문서가 동일한 섹션 구조를 따르는가?

## 의존관계 검증
⬜ 프로세스 간 호출·선행·이벤트 관계가 명시되었는가?
⬜ 의존 그래프에 순환 의존이 없는가?
⬜ 참조하는 SVC, FN, MDL, ENT, policy, SCR 코드가 실재하는가? (깨진 참조 0건)

## 역할 준수 검증
⬜ UI 구성·표시 형식 등 화면 정의서에 속하는 내용이 포함되지 않았는가?
⬜ DB 스키마 상세 등 데이터 정의서에 속하는 내용이 포함되지 않았는가?
⬜ 특정 프레임워크·라이브러리를 강제하는 표현이 없는가?
⬜ 모호 항목과 해석 근거가 검토 협의([`../../strategies/prompt-conversation.md`](../../strategies/prompt-conversation.md) §검토 협의)로 처리·기록되었는가?

# 출력 형식

- 개별 프로세스 정의서: `docs/specs/processes/process_<process-code>.md`

- 형식: Markdown
- 프로세스 코드 채번: `PROC-001`, `PROC-002`, ...

# 주의사항

- PRD에 명시되지 않은 기능을 임의로 추가하지 않는다.
- PRD 및 정책 정의를 준수하는 범위 내에서 필요 기능을 정의한다.
- 특정 구현 기술(프레임워크, 라이브러리)을 강제하지 않고 기능의 요구사항만 정의한다.
- 기존 작성 내용에서 변경이 필요한 경우 변경 지점·사유·영향범위·추가 검토 제안사항을 영향 IA 노드의 이력([`../../strategies/ia-history.md`](../../strategies/ia-history.md))에 기록한다.

# 메모리 업데이트

PRD 분석 후 다음 사항을 프로젝트 메모리에 기록한다:
- 도출된 프로세스 분류 체계(RR/EVT/BAT/WF/INT)와 그룹별 PROC 코드 목록
- 서비스(SVC) → 프로세스(PROC) 매핑 관계 및 한 시나리오를 구성하는 프로세스 수 분포
- 화면 액션(SCR) → 프로세스(PROC) 트리거 매핑 관계
- 정책(policy) → 프로세스(PROC) 적용 지점 매핑 (권한 검증·입력 가공·결과 마스킹·감사 로그 등)
- 공통 기능(FN) → 프로세스(PROC) 호출 관계 및 호출 빈도 상위 항목
- 데이터(ENT/MDL) CRUD를 수행하는 프로세스 매핑 — 엔터티별 변경 주체 식별
- 외부 시스템 연동(INT) 프로세스의 외부 API 목록 및 호출 패턴
- 비동기·배치·워크플로형 프로세스의 실행 트리거 정의 및 스케줄 표현식
- 프로세스 간 주요 의존 관계(호출·선행·이벤트) 및 순환 의존 검증 결과
- 트랜잭션 경계가 다중·보상형으로 설계된 프로세스 목록 (구현 시 우선 검토 대상)
- PRD에서 프로세스로 변환 시 판단이 필요했던 항목 — 모호한 요구사항, 분리/통합 판단, 분류 판단의 근거
- 사양 문서 누락 또는 정책 미정으로 검토 협의로 처리·기록된 항목 요약
