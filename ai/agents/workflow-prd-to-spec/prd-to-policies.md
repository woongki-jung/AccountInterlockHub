본 문서는 프로젝트 전체에 적용되어야 할 각종 정책을 도출하기 위한 지침을 작성하기 위한 규칙을 정의한다.
아래 지침에 따라 제공된 요구사항을 바탕으로 비즈니스 규칙, 접근 제어, 데이터 보존, 보안 정책 등 서비스 운영에 필요한 규칙 체계를 일관되게 정의한다.

용어 약어 — POL(=policy)은 정책 규칙, EX는 예외 코드(Exception code), HTTP status는 응답 상태 코드, PROC는 기능구현 프로세스 정의서 코드, FN은 공통 기능 코드를 의미한다.

# 호출 주체·입력

`spec` 오케스트레이터([`../spec.md`](../spec.md))가 도메인 의존 순서에 따라 호출하는 doer 다(정책은 1순위). 오케스트레이터가 작업 맥락을 전달한다 — **작업 범위**(전체 또는 IA 노드), **관련 Redmine `사양` 일감**(있으면), **선행 도메인 산출물**(분석 대상 PRD·directing 산출물·IA). 범위 밖 기존 사양은 입력 참조로만 쓰고 수정하지 않는다. 유효한 작업 맥락이 없으면 경고 후 중단한다.

# 작업 프로세스

## 1단계: PRD 분석
- 시작 시 전달받은 작업 맥락(작업 범위·분석 대상 문서)을 확인한다.
  분석 대상 문서와 각 문서가 참조하는 연결 문서의 내용을 자세히 분석한다.
- 분석된 내용에서 다음 항목들을 추출한다.
	- 사용자 역할 및 권한 체계 (인증/인가 정책)
	- 데이터 접근 제어 규칙
	- 비즈니스 규칙 및 제약 조건 (입력값 제한, 상태 전이 규칙 등)
	- 데이터 보존 및 삭제 정책 (보존 기간, 소프트 삭제 여부)
	- 개인정보 처리 정책 (수집 항목, 보존 기간, 제3자 제공 등)
	- 보안 정책 (비밀번호 규칙, 세션 만료, 암호화 대상 등)
	- 서비스 제한 정책 (요청 횟수 제한, 파일 크기 제한 등)
	- 알림 및 커뮤니케이션 정책

## 2단계: 정책 분류 및 구조 설계
- 정책을 논리적 그룹으로 분류한다.
	- **인증/인가 정책(AUTH)**: 로그인, 권한, 세션 관리
	- **데이터 정책(DATA)**: 보존, 삭제, 개인정보, 접근 제어
	- **비즈니스 규칙(BIZ)**: 도메인별 핵심 업무 규칙
	- **보안 정책(SEC)**: 암호화, 입력 검증, API 보안
	- **운영 정책(OPS)**: 제한 정책, 알림, 감사 로그
- 정책 간 의존성과 우선순위를 파악한다.
- 논리적 그룹에 따라 정책 코드 체계를 설계한다.

## 3단계: 정책 정의 문서 생성

### 3-1: 정책 목록 문서 생성
`docs/specs/policies/spec-policies.md` 파일에 전체 정책 정의서의 개요와 하위 문서 목록을 작성한다.
정책 정의서는 다음 내용을 포함한다.
논리적 그룹에 따라 그룹코드-nnn 형태로 정책 코드를 정의한다.
- 정책 설계 규칙 및 기본 방향
- 코드체계
- 우선순위 및 충돌 해결 원칙
- 정책 분류체계
- 정책 목록

### 3-2: 개별 정책 정의 문서 생성
각 정책 그룹별로 `docs/specs/policies/policy_<policy-code>.md` 파일을 생성한다.
세부 정책 정의는 다음 템플릿 구조를 따른다.

```markdown
# [정책명] 정책 정의

## 개요
- **정책 목적**: 이 정책이 해결하려는 문제 또는 보장하려는 품질
- **관련 PRD 요구사항**: PRD 내 관련 섹션 및 원문 인용

---

## [정책코드] [정책명]

### 기본 정보
| 항목 | 내용 |
|------|------|
| 정책명 | 정책 이름 |
| 분류 | AUTH / DATA / BIZ / SEC / OPS 중 1개 |
| 적용 범위 | 이 정책이 적용되는 대상 시스템·기능·데이터 |
| 우선순위 | 높음 / 보통 / 낮음 |
| 관련 정책 | 의존 또는 참조하는 다른 정책 코드 |
| 적용 PROC | 본 정책이 인용되는 PROC 코드 목록 (PROC 측 의사코드의 정책 적용 지점에서 참조) |
| 관련 IA 항목 | 본 사양이 속한 IA 노드의 `ia-code`(들). 횡단 사양은 `공통` (정본 [`../../strategies/ia.md`](../../strategies/ia.md) §참조 규약) |

### 정책 규칙

| 규칙 ID | 규칙 내용 | 적용 조건 | 적용 지점 | 위반 시 처리 (HTTP·EX 코드) |
|---------|-----------|-----------|-----------|----------------------------|
| <그룹>-NNN-RR | 측정 가능한 규칙 서술 (수치·조건·임계값) | 규칙이 적용되는 조건 | 검증 / 입력 가공 / 결과 마스킹 / 감사 로그 중 1개 이상 | 예: 401 EX-AUTH-001 / 422 EX-BIZ-005 / 토스트 메시지 / 무시 + 감사 로그 등 |

- **규칙 ID 채번 규칙**: `<그룹>-<순번 3자리>-<리비전 2자리>` 형태로 작성한다.
  예: `AUTH-001-01`(인증 첫 번째 규칙 v1), `BIZ-005-02`(비즈니스 5번 규칙 v2 개정).
  한 번 부여된 규칙 ID 는 재사용하지 않는다.
- **적용 지점 분류** (필수):
	- **검증(validate)**: 요청 진입 시 입력값·권한·상태 사전 검증.
	- **입력 가공(transform)**: 검증 통과 후 도메인 모델로 변환할 때 적용 (트림·정규화·기본값 보충).
	- **결과 마스킹(mask)**: 응답 DTO 생성 시 민감 정보 마스킹·필드 제거·표시값 변환.
	- **감사 로그(audit)**: 트랜잭션 종료 후 감사 로그에 기록.
- **위반 시 처리** (필수): 코드 분기 가능 수준으로 작성.
	- HTTP 응답 (상태 코드 + EX 코드 + 메시지) 형태가 기본.
	- 사용자에게 노출하지 않는 정책 위반은 "감사 로그 기록 + 무시" 같은 명시적 처리 방향 작성.

### 예외 사항
- 규칙을 적용하지 않는 예외 케이스 및 예외 처리 방법
- 예외 케이스에도 ID를 부여(`EXC-<그룹>-<순번>`)하여 PROC 의사코드에서 인용 가능하도록 한다.

### 적용 지점 매핑

| 규칙 ID | 적용 PROC | 적용 단계 (PROC 의사코드 단계명) | 적용 위치 (검증/가공/마스킹/감사) |
|---------|----------|-------------------------------|--------------------------------|
| <그룹>-NNN-RR | PROC-xxx | 예: B2 사전 조회 | 검증 |

> 본 표는 [prd-to-process](prd-to-process.md) §3-2 PROC 정의서의 "정책 적용 지점" 섹션과 1:1 정합해야 한다.

### 구현 가이드
- 정책 관련 사항을 구현할 때 참고할 핵심 항목
- 관련 기술 패턴 (특정 라이브러리·프레임워크 강제 금지)
```

> 변경이력은 본문에 인라인 표로 두지 않는다 — 영향 IA 노드의 이력([`../../strategies/ia-history.md`](../../strategies/ia-history.md))으로 위임한다.

## 4단계: 작업 완료 보고
- 산출물을 Redmine `사양` 트래커 일감으로 미러·동기화하고([`../../strategies/work-tracking.md`](../../strategies/work-tracking.md) §단계별 미러링), 영향 IA 노드의 이력을 갱신한다([`../../strategies/ia-history.md`](../../strategies/ia-history.md)).
- 일감 상태는 `진행` 을 유지한다 — `해결` 전이는 교차검증 통과 후 spec 오케스트레이터 몫이다. 세부 진척은 일감 노트로 기록한다([`work-tracking.md`](../../strategies/work-tracking.md) §단계 산출 일감 상태 매핑).
- 작업 범위·도출 결과·보류 항목을 `spec` 오케스트레이터에 보고한다.

# 문서 작성 원칙

## 명확성
- 모호한 표현 금지 — "적절히", "충분히" 대신 수치나 조건으로 명시.
	- 나쁜 예: "비밀번호는 충분히 복잡해야 한다".
	- 좋은 예: "비밀번호는 8자 이상, 영문 대소문자·숫자·특수문자 각 1자 이상 포함".
- 정책 규칙은 개발자가 코드로 바로 변환할 수 있을 만큼 구체적으로 기술.
- 위반 시 처리는 HTTP 상태·EX 코드·사용자 메시지 단위로 표현.

## 일관성
- 모든 정책에 동일한 문서 구조 적용.
- 정책 코드 채번 일관성 유지(`<그룹>-<순번>-<리비전>`).
- 상충되는 정책이 없도록 교차 검토.

## 완결성
- "어떤 경우에" 정책이 적용되는지와 "위반 시 어떻게 처리하는지"를 반드시 명시.
- 예외 사항도 명확하게 정의.
- 적용 지점 분류(validate/transform/mask/audit)가 모든 규칙에 부여되어야 한다.

## 추적 가능성
- 각 정책이 PRD의 어떤 요구사항에서 비롯되었는지 기술.
- 정책 간 상호 참조 명시.
- 적용 PROC 코드를 모두 인용하여 PROC 의사코드와 1:1 추적 가능하도록 한다.

# 정책 분류별 서술항목

## 인증/인가 정책 예시
- 로그인 방식 및 세션 관리 규칙.
- 역할 기반 접근 제어 (RBAC) 또는 권한 체계.
- 비밀번호 규칙, 계정 잠금 정책.

## 데이터 정책 예시
- 데이터 보존 기간 및 삭제 규칙.
- 개인정보 수집·이용·제공 범위.
- 소프트 삭제 vs 하드 삭제 적용 기준.
- 데이터 접근 권한 규칙.

## 비즈니스 규칙 예시
- 핵심 업무 흐름의 상태 전이 규칙.
- 입력값 제약 조건 (최대 길이, 허용 형식 등).
- 중복 방지 규칙, 고유성 제약.

## 보안 정책 예시
- 민감정보 암호화 대상 및 알고리즘 기준.
- API 인증 방식.
- 입력값 검증 및 주입 공격 방어 기준.

## 운영 정책 예시
- API 요청 횟수 제한 (Rate Limiting).
- 파일 업로드 크기 및 형식 제한.
- 감사 로그 기록 대상 및 보존 기간.

# PROC 와의 상호 참조 (필수)

PROC 의사코드의 정책 적용 지점에서 본 문서의 규칙 ID 가 인용된다.
정책 정의 작성 시 다음을 반드시 고려한다.

- 한 정책 규칙은 0~다수의 PROC 에서 인용될 수 있다.
- PROC 의사코드의 단계명(예: `B2 사전 조회`, `B3 영속화 트랜잭션`)을 인용할 때는 PROC 코드와 단계명을 함께 명시한다.
- 적용 위치(validate/transform/mask/audit)는 PROC 의사코드 단계와 정합해야 한다.
	- validate → BE 의 "입력 DTO 재검증" 또는 "사전 조회 후 비즈니스 규칙 분기" 단계.
	- transform → BE 의 "도메인 모델 생성" 또는 FE 의 "요청 DTO 변환" 단계.
	- mask → BE 의 "응답 DTO 변환" 단계.
	- audit → BE 의 "트랜잭션 커밋 후 감사 로그 기록" 단계.

# 산출물 검증 체크리스트

산출물 작성 완료 후 다음을 확인한다 (6 카테고리는 [prd-to-process.md §산출물 검증 체크리스트](prd-to-process.md) 와 동일 형식).

## 적용 범위 검증
- ⬜ PRD의 모든 비즈니스 규칙이 하나 이상의 정책으로 정의되었는가?
- ⬜ 인증/인가가 필요한 모든 기능에 접근 제어 정책이 정의되었는가?
- ⬜ 개인정보를 다루는 모든 처리 단계에 개인정보 처리 정책이 매핑되었는가?
- ⬜ 모든 PROC 가 인용하는 정책 규칙이 본 문서에 실재하는가?

## 정의 완결성 검증
- ⬜ 각 정책 규칙이 수치 또는 조건으로 명확하게 기술되었는가?
- ⬜ 위반 시 처리에 HTTP 상태 코드와 EX 코드가 명시되었는가?
- ⬜ 적용 지점 분류(validate/transform/mask/audit)가 모든 규칙에 부여되었는가?
- ⬜ 예외 사항이 명확하게 정의되었는가?
- ⬜ 적용 지점 매핑 표가 PROC 코드·단계명·적용 위치와 함께 작성되었는가?

## 실행 제약 검증
- ⬜ 각 위반 처리가 코드 분기 가능 수준으로 표현되었는가? (예: 401 EX-AUTH-001)
- ⬜ 감사 로그 정책의 보존 기간·기록 대상이 명시되었는가?

## 일관성 검증
- ⬜ 모든 정책에 동일한 문서 구조 적용?
- ⬜ 정책 코드 채번이 `<그룹>-<순번>-<리비전>` 패턴으로 일관 적용되었는가?
- ⬜ spec-policies.md 목록과 개별 정책 코드가 1:1 일치하는가?
- ⬜ 상충되는 정책이 없는가?

## 의존관계 검증
- ⬜ 정책 간 상호 참조가 명시되었는가?
- ⬜ 적용 PROC 가 모두 실재하는가? (깨진 참조 0건 — 후행 도메인(PROC 등) 참조는 초회 작성 시 예약 채번 허용, 실재 확인은 교차검증(spec-reviewer) 시점)

## 역할 준수 검증
- ⬜ UI 구성·표시 형식 등 화면 정의서에 속하는 내용이 포함되지 않았는가?
- ⬜ DB 스키마 상세 등 데이터 정의서에 속하는 내용이 포함되지 않았는가?
- ⬜ 특정 프레임워크·라이브러리를 강제하는 표현이 없는가?
- ⬜ 모호 항목과 해석 근거가 검토 협의([`../../strategies/prompt-conversation.md`](../../strategies/prompt-conversation.md) §검토 협의)로 처리·기록되었는가?

# 출력 형식
- 목록 파일: `docs/specs/policies/spec-policies.md`
- 개별 파일: `docs/specs/policies/policy_<policy-code>.md`
- 언어: 한국어
- 형식: Markdown
- 정책 코드 채번: `<그룹>-<순번 3자리>-<리비전 2자리>` (예: `AUTH-001-01`, `BIZ-005-02`)

# 주의사항
- PRD에 명시되지 않은 정책을 임의로 추가하지 않는다.
- 특정 구현 기술(프레임워크, 라이브러리)을 강제하지 않고 정책의 요구사항만 정의한다.
- 법적 요구사항(개인정보보호법 등)이 관련된 정책은 해당 법규를 명시한다.
- 기존 작성 내용에서 변경이 필요한 경우 변경 지점·사유·영향범위·추가 검토 제안사항을 영향 IA 노드의 이력([`../../strategies/ia-history.md`](../../strategies/ia-history.md))에 기록한다.

# 메모리 업데이트

PRD 분석 후 다음 사항을 프로젝트 메모리에 기록한다:
- 도출된 정책 분류 체계 및 코드 목록.
- 정책 간 주요 의존 관계.
- 적용 지점 분류별 정책 분포 (validate/transform/mask/audit 카운트).
- 정책 ↔ PROC 적용 지점 매핑 (어느 정책이 어느 PROC 의 어느 단계에서 인용되는지).
- 위반 시 처리 코드(EX-) 카탈로그.
- PRD에서 정책으로 변환 시 판단이 필요했던 항목 (모호한 요구사항, 해석 근거 등).
- 관련 법규 또는 컴플라이언스 요구사항 존재 여부.
