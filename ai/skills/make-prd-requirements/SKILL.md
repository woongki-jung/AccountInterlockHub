## 사용 상황

- 새 프로젝트를 시작하며 PRD(제품 요구사항 정의서) 가 비어 있을 때 (**신규 모드**)
- 기존 작성된 요구사항 문서의 갱신을 수행할 때 (**갱신 모드**) — 사용자 변화, 신규 기능 추가, 범위 변경, 우선순위 재조정, 결함 발견 등
- 담당자가 머릿속·메모·기존 자료에 흩어진 요구사항을 정리된 문서로 출력하고 싶을 때
- 후속 `prd-to-policies` / `prd-to-service` / `prd-to-screens` 등 사양 분석 에이전트가 사용할 입력 문서를 준비할 때

## 산출 문서

| 산출 문서 | 경로 | 역할 |
|---|---|---|
| PRD 루트 | `docs/prd/PRD.md` | 제품 한 줄 요약·배경·목표·성공 지표·범위 |
| 요구사항 루트 | `docs/prd/requirements/requirements.md` | 하위 4개 카테고리 문서를 묶는 인덱스 |
| 사용자 정의 | `docs/prd/requirements/users/user_<code>.md` | 페르소나·역할·핵심 시나리오·pain point |
| 정보 구조(IA) | `docs/prd/requirements/ia/` | 핵심 도메인 엔티티·관계·분류 체계 |
| 서비스 기능 | `docs/prd/requirements/services/service_<code>.md` | 사용자가 수행할 액션·시스템이 제공할 기능·우선순위 |
| 화면 정의 | `docs/prd/requirements/screens/screen_<code>.md` | 각 기능이 노출되는 화면·흐름·진입점 |

저장 경로 전체 트리는 [`ai/strategies/doc-structure.md`](../../strategies/doc-structure.md) 참조.

## 대화 원칙 (모든 단계 공통)

- **추측 금지** — 담당자가 명시하지 않은 내용은 반드시 질문. 단, 한 번에 묻는 질문 수는 최대 3개.
- 모호함 발견 시 → 옵션 2~3개 제시 + 권장안 1개 명시 → 담당자 선택.
- 각 산출 문서 작성 직후 담당자 검토 1회 → 수정 반영 → 다음 단계 진행 합의 확인.

## 실행 동작

### 1. 모드 식별 및 사전 준비

작업 시작 시 `docs/prd/PRD.md` 존재 여부로 모드 추정:
- 파일 없음 또는 골격만 있음 → **신규 모드** (§1.1)
- 파일 있음 + 요구사항 문서 1개 이상 작성됨 → **갱신 모드** (§1.2)

추정 결과를 담당자에게 한 줄로 보고하고 확인받은 뒤 분기.

#### 1.1 신규 모드 — 사전 자료 수집

담당자에게 다음을 묻고, 제공되는 자료가 있으면 우선 읽는다.

- 기존 기획 문서(PDF·docx·노션 등) 보유 여부
- 참고 디자인 시안(Figma URL·이미지) 보유 여부
- 경쟁/유사 서비스 참고할 것 보유 여부

이후 §2 부터 순차 실행.

**활용 외부 스킬**:
- PDF 자료 → [`ai/skills/opendataloader-pdf/SKILL.md`](../opendataloader-pdf/SKILL.md) (워크스페이스 등록 스킬) 로 마크다운 변환 후 분석.
- Figma URL 또는 디자인 시안 → `claude.ai Figma` MCP(Anthropic 공식: `mcp__claude_ai_Figma__get_design_context` / `get_screenshot` / `get_metadata`) 로 디자인 메타데이터·스크린샷 확보.
- Google Drive 공유 링크 → `Google Drive` MCP(Anthropic 공식) 로 직접 접근.

#### 1.2 갱신 모드 — 변경 범위 식별

1. 기존 문서 일괄 로딩: `docs/prd/PRD.md` + `docs/prd/requirements/requirements.md` + 하위 4개 카테고리 폴더 전수.
2. 담당자에게 갱신 트리거 질문 (왜 지금 바꾸는가):
   - 사용자 변화 / 신규 기능 / 범위 축소 / 우선순위 재조정 / 결함 발견 / 외부 환경 변화 중 어디에 해당하는가
   - 트리거를 발생시킨 구체적 사실 1~2줄 (예: "신규 페르소나 발견", "B2B 라인 추가 결정")
3. 변경 범위 매트릭스 작성 후 담당자 확인:

   | 변경 종류 | 영향 카테고리 | 변경 요약 |
   |---|---|---|
   | 수정 / 추가 / 폐기 | users / ia / services / screens / PRD | 1줄 요약 |

4. 매트릭스에서 영향 카테고리에 해당하는 §3~§6 단계만 선택 실행. 영향 없는 카테고리는 건너뛴다. 단, 어떤 카테고리가 바뀌면 다른 카테고리도 연쇄 영향이 있을 수 있으므로 §10 "갱신 모드 세부 지침" 의 연쇄 영향 규칙을 먼저 점검한다.

### 2. 프로젝트 목표 정의 (→ `PRD.md`)

**대화 흐름**:

1. "한 문장으로 어떤 제품인지" 요청.
2. 다음 3개 질문 (1번에 묶어 송신):
   - 누가 사용하는가 (대상 사용자 한 줄)
   - 어떤 문제를 해결하는가 (현재 사용자가 겪는 불편 한 줄)
   - 성공의 기준은 무엇인가 (3개월 / 6개월 시점 정량 지표 있으면)
3. 답변을 바탕으로 PRD.md 초안 작성:
   - 제목 / 한 줄 요약 / 배경 / 목표 / 성공 지표 / 범위(In/Out) / 비범위
4. 담당자에게 초안 제시 + "이 방향이 맞는지" 확인. 수정 요청 시 반영.

**수집 피드백**:
- 범위 In/Out 의 경계가 분명한지
- 성공 지표가 측정 가능한지 (불가능하면 측정 방법 함께 정의)
- "왜 지금" 에 대한 답이 있는지 (없으면 묻기)

### 3. 사용자 정의 (→ `users/user_<code>.md`)

**대화 흐름**:

1. PRD.md 의 대상 사용자를 1~3개 페르소나로 분해 제안. 담당자 승인 후 진행.
2. 각 페르소나별 다음 항목 질문:
   - 역할·소속 (예: 30대 직장인 / 의원 원장 / 학원 관리자)
   - 일상 흐름 중 본 제품을 쓰는 순간 (언제·어디서·무엇을 하다가)
   - 현재 그 순간에 겪는 가장 큰 불편 1~2개
   - 본 제품이 그 불편을 어떻게 줄이는지 (담당자 답변 → AI 가 한 문장으로 요약 후 확인)
3. 각 페르소나를 `user_<code>.md` 1개씩 저장. `<code>` 는 P1 / P2... 형태.

**수집 피드백**:
- 페르소나 간 차이가 명확한지 (겹치면 통합 제안)
- "pain point → 해결" 연결고리가 추상적이지 않은지

### 4. 정보 구조 정의 (→ `ia/`)

**대화 흐름**:

1. PRD.md + 사용자 정의를 바탕으로 핵심 엔티티 후보 5~10개 추출 후 담당자 제시.
   - 예: "고객 / 예약 / 매장 / 결제 / 리뷰" 같은 명사 단위.
2. 각 엔티티에 대해 질문:
   - 누가 만들고, 누가 보는가
   - 다른 엔티티와의 관계 (1:N / N:N / 소유 / 참조)
   - 분류 체계가 있다면 (예: 예약 상태 = 대기/확정/취소)
3. 엔티티 관계도를 mermaid 다이어그램으로 그리고 담당자 확인.
   - 복잡한 다이어그램이면 Figma MCP (`mcp__claude_ai_Figma__generate_diagram`) 로 FigJam 다이어그램 생성하여 공유 가능.
4. `ia/` 하위에 카테고리별 문서로 저장.

**수집 피드백**:
- 동일 개념을 다른 이름으로 부르는 곳이 있는지 (용어 통일)
- 누락된 엔티티가 있는지 (각 사용자 시나리오를 다시 한 번 훑으며 확인)

### 5. 서비스 기능 정의 (→ `services/service_<code>.md`)

**대화 흐름**:

1. 사용자 정의 + IA 를 바탕으로 사용자 액션 목록 추출. 담당자에게 표 형식으로 제시:

   | 사용자 | 액션 | 관련 엔티티 |
   |---|---|---|
   | P1 | 매장 검색 | 매장 |
   | P1 | 예약 생성 | 예약·매장 |

2. 각 액션을 서비스 기능으로 묶고, 다음 질문:
   - 이 기능이 없으면 어떤 일이 벌어지는가 (필수성 판단)
   - MoSCoW 분류 — Must / Should / Could / Won't (담당자가 모르면 AI 가 권장안 제시)
   - 기능 사용 빈도 추정 (자주 / 가끔 / 드물게)
3. 각 서비스 기능을 `service_<code>.md` 로 저장. `<code>` 는 S1 / S2... 형태. 문서 내용:
   - 기능명 / 목적 / 사용 페르소나 / 트리거 / 인수 기준 / 우선순위 / 관련 엔티티

**수집 피드백**:
- Must 가 전체의 70% 를 넘으면 → 범위 비현실적, 우선순위 재조정 권고
- 인수 기준이 모호한 항목은 "어떻게 되면 완료라고 부를지" 재질문

### 6. 화면 정의 (→ `screens/screen_<code>.md`)

**대화 흐름**:

1. 서비스 기능별로 노출 화면 후보를 표로 제시 후 담당자 합의.
2. 각 화면에 대해 질문:
   - 화면 진입점 (어떤 액션 후 들어오는가)
   - 화면이 보여야 하는 핵심 정보 3~5개
   - 화면에서 가능한 액션 1~3개
   - 후속 화면 (다음에 어디로 가는가)
3. 화면 흐름도를 mermaid `flowchart` 로 작성하여 담당자 확인.
4. 디자인 시안이 있는 경우:
   - 담당자가 Figma URL 제공 시 → `claude.ai Figma` MCP (`mcp__claude_ai_Figma__get_design_context` / `get_screenshot`) 로 시안 가져와 화면 정의서에 첨부.
   - 디자인 시안을 새로 만들 필요가 있으면 → 진행 방식을 담당자와 협의해 정한다 (본 스킬은 요구사항 수준의 화면 정의까지만 책임).
5. 각 화면을 `screen_<code>.md` 로 저장. `<code>` 는 SC1 / SC2... 형태.

**수집 피드백**:
- 동일 정보를 여러 화면에서 중복 표시하면 → 정보 우선순위 재정리 권고
- 화면 흐름이 막다른 길로 끝나는 곳 없는지 (귀환 경로 확인)

### 7. 요구사항 루트 작성 (→ `requirements.md`)

1~6 완료 후, 모든 하위 문서를 묶는 인덱스 작성:
- 사용자 / IA / 서비스 / 화면 각 카테고리의 문서 목록
- 카테고리별 한 줄 설명
- 변경 이력 (최초 작성일 + 담당자)

### 8. PRD 통합 갱신

`PRD.md` 의 "요구사항" 섹션에 `requirements.md` 링크 추가. PRD 자체는 §2 에서 작성한 골격을 유지하며, 요구사항 정의 과정에서 발견된 범위 변경·전제 변경만 PRD 본문에 반영.

### 9. 후속 단계 안내

작성 완료 후 담당자에게 다음 안내:

- 본 산출물은 spec 오케스트레이터 [`ai/agents/spec.md`](../../agents/spec.md) 의 입력으로 사용된다.
- 사양 정의(spec) 단계를 시작하려면 `spec` 오케스트레이터 실행을 요청하라.
- 갱신 모드인 경우 §10.4 의 다운스트림 영향 안내를 반드시 함께 전달한다.
- 사양 분석 단계의 작업을 계획적으로 분해하려면 [`ai/skills/writing-plans/SKILL.md`](../writing-plans/SKILL.md) (워크스페이스 등록 스킬) 활용 가능.

### 10. 갱신 모드 세부 지침

§1.2 에서 갱신 모드로 분기한 경우 §2~§9 실행 시 본 절 규칙을 함께 적용한다.

#### 10.1 변경 분류 (수정 / 추가 / 폐기)

각 변경 항목을 다음 셋 중 하나로 분류하여 처리.

| 분류 | 처리 |
|---|---|
| 수정 | 기존 코드(P1·S1·SC1 등) 유지. 본문 갱신 + 변경 로그 항목 추가 |
| 추가 | 다음 비어있는 코드 부여 (P3 → P4). 신규 문서 생성 |
| 폐기 | 본문에 `status: deprecated` 표기 + 폐기 사유·일자 기재. 파일 삭제 금지 (코드 충돌·이력 추적 보존) |

#### 10.2 코드(식별자) 유지 원칙

- 의미가 본질적으로 동일하면 코드 유지 (이름·세부사항 변경은 수정으로 처리).
- 의미가 본질적으로 바뀌면 기존 항목 폐기 + 새 코드 부여 (코드 재활용 금지).
- 판단 모호 시 담당자에게 "같은 개념의 변경인가, 다른 개념인가" 질문.

#### 10.3 연쇄 영향 규칙

한 카테고리의 변경은 다른 카테고리에 자동 영향을 줄 수 있다. §3~§6 진입 전 다음 표로 점검:

| 1차 변경 카테고리 | 점검할 후속 카테고리 | 점검 포인트 |
|---|---|---|
| users | services / screens | 폐기된 페르소나만 쓰던 기능·화면 존재? |
| ia | services / screens | 폐기·이름 변경된 엔티티를 참조하는 항목 존재? |
| services | screens / users / ia | 신규 기능이 새 화면·페르소나·엔티티를 요구하는지 |
| screens | services | 화면 변경으로 인해 가능 액션 / 인수 기준이 바뀌는지 |
| PRD (목표·범위) | 전체 카테고리 | 범위 축소 시 폐기 후보 점검 / 범위 확장 시 누락 점검 |

연쇄 영향이 식별되면 변경 범위 매트릭스(§1.2) 에 해당 카테고리를 추가하고 담당자에게 재확인.

#### 10.4 다운스트림 영향 평가

요구사항 변경은 이미 작성된 사양(`docs/specs/`) 에 영향을 줄 수 있다. §9 안내 직전에 점검.

1. 변경된 요구사항 항목별로 `docs/specs/` 하위에서 직간접 참조 검색 (grep / 파일 본문 확인).
2. 영향 받는 사양 문서 목록을 작성하여 담당자에게 보고:
   - 폐기된 요구사항을 입력으로 한 사양 → 사양 폐기 후보
   - 수정된 요구사항을 입력으로 한 사양 → 사양 재검토 후보
   - 추가된 요구사항 → 신규 사양 작성 후보
3. 담당자에게 후속 작업 안내:
   - 사양 정합성 검증 → `spec-reviewer` 에이전트 ([`ai/agents/workflow-prd-to-spec/spec-reviewer.md`](../../agents/workflow-prd-to-spec/spec-reviewer.md))
   - 전체 재실행 → spec 오케스트레이터 ([`ai/agents/spec.md`](../../agents/spec.md))

#### 10.5 변경 이력 기록 의무

- 갱신된 각 문서 하단의 "변경 로그" 섹션에 항목 추가: `<날짜> | <변경 종류> | <변경 요약> | <담당자>`.
- `requirements.md` 루트의 변경 이력에도 1줄 요약 추가.
- `PRD.md` 본문이 바뀌었다면 PRD 변경 로그에도 항목 추가.
- 갱신 사이클이 끝나면 매트릭스(§1.2) + 다운스트림 영향 목록(§10.4) 을 `requirements.md` 의 직전 갱신 이력 항목 아래에 첨부.
