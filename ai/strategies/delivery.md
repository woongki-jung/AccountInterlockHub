# 릴리스·배포·유지보수 전략

본 문서는 qa 배포 가능 판정 **이후**의 산출물 전달 과정 — 버전·릴리스, 배포 실행, 유지보수(핫픽스) — 를 정의하는 단일 출처다. 4단계 흐름은 [`base-workflow.md`](base-workflow.md), 작업 추적은 [`work-tracking.md`](work-tracking.md) 가 정본이다.

## 저장소 운용

- **저장소** — 워크스페이스 저장소 하나가 운영 문서(`ai/`)·산출 문서(`docs/`)·제품 코드(`apps/`)를 함께 관리한다([`doc-structure.md`](doc-structure.md)). 제품 코드를 별도 저장소로 분리해야 하면 담당자 협의로 정하고 그 위치를 개발사양(`docs/prd/devspec/`)에 기록한다.
- **브랜치·커밋·병합** — 운용 규칙은 [`git-flow.md`](git-flow.md) 가 단일 출처다: 단계 착수 1회 = 브랜치(`<단계>/<이슈번호>-<주제>`), Phase = commit(산출물+이력 같은 commit), 담당자 승인 시 main 병합(no-ff). main 은 승인 통과 기준선이다.

## 버전·릴리스

qa 배포 가능 판정([`stages/qa.md`](stages/qa.md) §다음 단계 이행 조건)과 담당자의 릴리스 지시로 시작한다.

1. **버전 채번** — SemVer(`MAJOR.MINOR.PATCH`) 를 기본으로 한다. 기능 추가 = MINOR, 결함 수정(핫픽스) = PATCH, 호환성 파괴 변경 = MAJOR.
2. **Redmine 버전 반영** — 프로젝트에 버전 객체를 생성하고 포함 일감의 배포버전(`fixed_version`)을 지정한다([`work-tracking-redmine.md`](work-tracking-redmine.md) §프로젝트 생성 표준 절차 5 — 로드맵에 잡힌다).
3. **릴리스 노트** — `docs/releases/<버전>.md` 1파일로 작성한다: 변경 요약(담당자 언어), 포함 일감 목록(이슈 번호), 배포 산출물 식별 정보(파일·경로·기준 commit).
4. **릴리스 태그** — main 의 기준 commit 에 `v<버전>` 태그를 부여한다([`git-flow.md`](git-flow.md) §릴리스 태그).

## 배포 실행

- 배포 산출물의 형식·배포 절차는 개발사양(`docs/prd/devspec/infra.md`)이 단일 출처다(본 문서는 방식을 단정하지 않는다).
- 배포 실행은 **담당자 요청으로 시작**한다 — 요청 없이 자동 트리거하지 않으며, 결과(성공·이상·롤백)를 릴리스 관련 (그룹) 일감 노트로 기록한다.
- 배포 후 결함이 확인되면 §유지보수·핫픽스 경로로 접수한다.

## 유지보수·핫픽스 경로

운영 중 결함이 4단계 흐름에 진입하는 규칙이다.

1. **접수** — 담당자가 작업세션 이슈로 결함을 보고하면 ai-pm 이 서브에이전트를 디스패치해 `오류` 일감으로 등록한다 — 영향 IA 를 카테고리로, 관련 사양·기능 일감을 연관으로([`work-tracking.md`](work-tracking.md)).
2. **영향 판단** — 결함 원인에 따라 진입 단계를 정한다.
	- **구현 결함** → build 를 부분 착수한다(spec 생략, 해당 사양과의 정합만 확인).
	- **사양 결함** → spec 을 부분 착수한 뒤 build 로 잇는다.
	- **긴급(서비스 중단급)** → 담당자 승인 하에 최소 수정으로 build 직행하고, 사후에 사양 정합을 맞춘다.
3. **수정·검증** — build 는 해당 `오류` 일감을 연관으로 진행하고, qa 는 **축약 범위**(해당 결함 TC + 회귀 우선순위 TC — [`stages/qa.md`](stages/qa.md) §결과 보고의 회귀 후보)로 수행한다.
4. **릴리스** — PATCH 버전으로 §버전·릴리스를 수행한다.

## 사양 변경의 역방향 전파

피드백 회귀([`base-workflow.md`](base-workflow.md) §전이 규칙)로 사양이 바뀌면, 사양 일감을 갱신하는 주체가 **연관된 기능·검증 일감을 점검**해 재작업·재검증 필요 여부를 각 일감 노트로 표시한다. 필요한 build·qa 부분 착수는 완료 보고에 포함해 ai-pm 이 후속 디스패치 대상으로 삼게 한다.
