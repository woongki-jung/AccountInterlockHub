# qa (품질검증) 단계 지침

본 문서는 [`base-workflow.md`](../base-workflow.md) §qa 의 상세 진행 지침이다. base-workflow §단계별 지침 공통 템플릿의 슬롯을 채우고, §개요의 단계 공통 흐름(계획 → 수행 → 검증 → 정리·보고)을 qa 에 매핑한다.

## 목적

산출물이 사양을 충족하는지 **사용자와 동일한 환경**에서 검증하고 배포 가능 여부를 평가한다. 검증 누락 없이 검증 TC 전체를 처리하는 것이 최우선 원칙이다.

## 착수 입력

- build 산출물 — 배포 산출물과 그 식별 정보(버전·경로·기준 commit — build 정리 보고가 인계).
- 검증 대상 사양·검증 TC(`docs/specs/` 및 검증 TC 사양).

## 핵심 활동

단계 공통 흐름에 따라 진행하되, 계획에 앞서 **검증 환경 구성**을 선행한다.

0. **환경 구성(선행)** — 워크스페이스 직접 기동이 아니라 **build 가 산출한 배포 산출물을 배포 절차대로 설치·기동한 것**을 검증 대상으로 삼는다(사용자 동일 환경 — 케이스별 세부는 [`qa-execution.md`](../qa-execution.md)). qa 가 산출물을 재빌드하지 않으며, 개발용(Debug) 구성 산출물은 쓰지 않는다. 설치·기동 → 동작 가능 상태 도달 → 검증 기준선(baseline) 확보까지 최소 1회 이상 시도한다.
1. **계획(TC 배정)** — 작업 범위(§작업 범위)를 확정하고 검증 TC 를 Phase 로 배정한다. 프로그램별 검증 케이스는 directing 프로그램 구성표에 따라 선택한다([`qa-execution.md`](../qa-execution.md) §케이스 선택). **TC 는 빠짐없이 처리**하며(1 TC = 1 Phase, Phase 수 상한 없음), 환경 한계로 수행 못 하는 TC 는 사유·해소조건과 함께 차기 이월로 명시한다(누락 금지).
2. **수행(검증 실행)** — Phase 별로 TC 를 실행해 5종으로 판정한다 — 🟢 Pass(실 환경) / 🔵 Pass-Mock / 🟣 Pass-Static / 🔴 Fail / 🟠 Block. 실 환경 round-trip 으로 확인하고 준비·실행·결과 증빙을 보존한다. 🔴 Fail 은 `오류` 일감을 생성해 해당 검증·사양 일감과 연관한다([`work-tracking.md`](../work-tracking.md) §계층·연관).
3. **검증(집계·게이트)** — 범위 내 전체 TC 가 판정 또는 이월로 빠짐없이 처리됐는지 대조하고(정합 식: 전체 TC 수 = 수행 + 명시적 이월), 품질 게이트(핵심 우선순위 TC 의 실 Pass율)를 평가한다.
4. **정리·보고** — 담당자별로 그룹화해 결과를 보고하고, 실패·차단 항목을 후속 처리 스냅샷으로 남긴 뒤 산출물을 정리한다.

### 결과 보고 — 담당자별 그룹화

모든 실패·차단·후속 액션은 주관 담당자(**기획 / 디자인 / 개발 / QA**) 기준으로 분류한다. 각 항목에 재현 절차·기대/실제 동작·증빙·예상 원인·해결 방안을 담는다. 잠정 Pass(Mock·Static)는 실 Pass 와 분리해 품질 게이트 신뢰도 왜곡을 막고, 회귀 우선순위 후보로 등재한다.

## 실행 주체

- **ai-pm** — 채널 대화에서 qa 작업을 식별해 서브에이전트를 디스패치하고 결과를 보고한다([`ai-pm.md`](../ai-pm.md) §디스패치 계약).
- **서브에이전트** — 환경 구성·계획·실행·정리를 수행하고, 진행·결과(판정·스냅샷 경로)를 관련 Redmine 일감에 반영한다.
- 모호하거나 비가역적 판단이 필요하면 가정하지 않고 담당 채널에 질의·응답을 참조해 진행한다(검토 협의 — [`prompt-conversation.md`](../prompt-conversation.md) §검토 협의). 백그라운드 서브에이전트의 질의는 ai-pm 릴레이로 처리한다([`ai-pm.md`](../ai-pm.md) §질의·승인 릴레이).
- 구체 에이전트 바인딩과 검증 환경·UI 자동화 도구 운용 세부는 품질 검증 수행 전략([`qa-execution.md`](../qa-execution.md))에 위임한다.

## 수행 방법

qa 는 ai-pm 이 **서브에이전트로 디스패치**해 수행한다.

1. **시작** — ai-pm 이 작업 맥락(검증 범위·대상 빌드·관련 일감)을 전달해 서브에이전트를 기동한다.
2. **수행·추적** — 계획 단계에서 각 검증 TC 를 Redmine **`검증` 트래커 일감**으로 등록하고, **각 TC 실행 즉시** 그 5종 판정에 따라 일감 상태를 동기화한다(실행 여부가 일감에 남도록 누락 금지) — **🟢 Pass→`완료`(닫힘)**, 🔵🟣 잠정 Pass→`해결`(재검증 사유·해소 조건 노트 필수), 🔴 Fail→`진행`, 🟠 Block→`보류`. **확정 판정(🟢)은 실행 주체가 직접 닫는다** — 추가 확인 여지가 없기 때문이며, `해결`은 확인이 더 남은 건에만 쓴다([`work-tracking.md`](../work-tracking.md) §단계 산출 일감 상태 매핑). 다만 **단계 이행·배포 가능 판정은 여전히 담당자 승인 사항**이다(§다음 단계 이행 조건) — 개별 TC 를 닫는 것이 단계 통과를 뜻하지 않는다. **최종 판정과 같은 갱신에서 제목 맨 앞에 상태 아이콘(🟢🔵🟣🔴🟠)을 붙인다** — 목록에서 판정을 즉시 훑기 위함이며, 재판정 시 기존 아이콘을 교체한다(덧붙이지 않는다 — [`work-tracking.md`](../work-tracking.md) §제목 상태 아이콘). 각 TC 일감에는 검증하는 사양의 `사양` 일감을 연관 일감으로 추가한다(§산출물).
3. **종료** — 완료 보고를 받은 ai-pm 이 품질 게이트 결과·후속 액션을 작업세션 이슈로 담당자에게 회신한다.

## 작업 범위

검증은 **전체** 또는 **부분(특정 섹션)** 으로 수행할 수 있다. 부분 수행 시 대상 외 TC 는 참조만 한다(공통 TC 는 범위와 무관하게 항상 포함). 부분 범위의 선별 단위는 IA 노드이며([`ia.md`](../ia.md)), 경로 규칙은 [doc-structure](../doc-structure.md)가 정의한다.

## 산출물

- **품질검증 결과 보고** — 담당자별 결과·품질 게이트·후속 액션.
- **실패·차단 스냅샷** — 후속 처리·회귀 추적용(1회 생성 후 수정 금지).
- **증빙·환경 기록** — 스크린샷·로그·환경 구성 결과.
- **Redmine 미러(공유용·추적)** — 계획 단계에서 각 검증 TC 를 `검증` 트래커 일감으로 등록하고, 검증 결과(5종 판정)에 따라 상태를 동기화한다(🟢 Pass→`완료`(닫힘) — 실행 주체가 직접 종결; 🔵🟣 잠정 Pass→`해결` + 재검증 사유). `검증` 일감은 검증 대상 일감(기능 등)의 하위 이슈로 두고, 검증하는 사양의 `사양` 일감을 **연관 일감**으로 추가한다. 트래커 구분·판정↔상태 매핑 정책 정본은 [work-tracking](../work-tracking.md).

검증 환경 표준·UI 자동화 도구·설치 경로는 품질 검증 수행 전략([`qa-execution.md`](../qa-execution.md)) 및 사양의 개발사양을 단일 출처로 한다.

## 완성도(완료) 기준

- **환경 baseline** — 0단계에서 배포 산출물의 동작 가능 상태·검증 기준선을 확보한다. baseline 미확보 시 검증을 수행하지 않고 **전체 Block 으로 보고**한다(수행 0·전건 이월 — §핵심 활동 0).
- 범위 내 모든 TC 가 5종 판정 또는 명시적 이월로 처리된다(누락 0).
- **구현↔검증 대조**(누락 0) — build 이 인계한 커버리지 매트릭스(구현 Phase)를 대조해 "구현됐으나 검증 TC 없는 Phase"(구현↔검증 공백)를 식별한다. 공백은 TC 를 임의 추가하지 않고 **spec 보완 제안**으로 등재한다(누락 0 = 공백 없음 또는 전건 보완 제안 등재).
- 품질 게이트(핵심 우선순위 TC 의 실 Pass율) 충족 여부 평가가 완료된다. **기본 기준**: 우선순위 높음 TC 실 Pass 100% · 전체 TC 실 Pass 95% 이상 · 미해소 Fail 0 — 프로젝트별 조정 값은 방향 정의서가 정의한다. 우선순위는 TC 사양의 우선순위(높음/보통/낮음)를 따른다.
- **재검증 루프** — Fail 은 qa 가 자체 수정하지 않는다. `오류` 일감 → 외부(build/spec) 수정 → qa 재검증(축약 범위: 해당 결함 TC + 회귀 우선순위 TC)의 교차단계 루프로 수렴한다. 회귀 카운터(동일 TC 재검증 최대 3회 — base-workflow §개요 3)를 넘어 수렴하지 않으면 담당자 판단을 구한다.
- 완료 판정은 base-workflow §전이 규칙 "완성도 게이트"를 정본으로 따른다.

## 다음 단계 이행 조건

- 품질 게이트를 충족하고 **담당자가 승인**하면 배포 가능으로 판정한다. 판정 이후의 릴리스·배포는 [`delivery.md`](../delivery.md) 를 따른다.
- **개별 `검증` 일감이 모두 닫혔다는 사실은 배포 가능 판정이 아니다** — 🟢 Pass TC 종결은 실행 주체의 정리 작업이고, 단계 승인은 품질 게이트 결과를 담당자가 받아 내리는 별도 판단이다(승인 대상은 단계 그룹 일감·작업세션 이슈 — [`work-tracking.md`](../work-tracking.md) §단계 산출 일감 상태 매핑).
- 사양·구현 결함이 확인되면 해당 선행 단계(spec/build)를 개별 프로세스로 다시 착수한다(base-workflow §전이 규칙 "피드백").
- qa 는 독립 단계다 — 자동으로 배포로 이어지지 않는다.
