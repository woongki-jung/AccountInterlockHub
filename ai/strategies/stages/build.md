# build (구현) 단계 지침

본 문서는 [`base-workflow.md`](../base-workflow.md) §build 의 상세 진행 지침이다. base-workflow §단계별 지침 공통 템플릿의 슬롯을 채우고, §개요의 단계 공통 흐름(계획 → 수행 → 검증 → 정리·보고)을 build 에 매핑한다.

## 목적

확정 사양을 실제 산출물(제품 프로그램·배포 구성)로 구현한다. FE(화면)·BE(서버 로직)를 모두 구현하되 **무게중심은 BE 비즈니스 로직·외부 연동**에 둔다. FE 는 승인된 목업을 청사진으로 삼아 설계 모호성을 줄인다.

## 착수 입력

- spec 산출물 — 확정 사양서(`docs/specs/`), 승인된 목업(FE 청사진).
- (있으면) 외부 API 사양·기존 코드베이스.

## 핵심 활동

단계 공통 흐름에 따라 진행한다.

1. **계획(구현 범위·Phase 편성)** — 작업 범위(§작업 범위)를 확정하고 구현 항목을 Phase 로 편성한다. **1 구현 항목 = 1 Phase 로 빠짐없이** 편성하고 Phase 수에 상한을 두지 않는다(범위가 과도하면 항목을 버리는 압축이 아니라 누락 없는 범위 분할로 나눠 실행). 확정 사양의 각 구현 대상 항목을 Phase 에 매핑해 **편성 누락(미커버)이 없도록** 하고, 이 매핑은 정리 단계 커버리지 매트릭스의 근거가 된다. BE 연동 결정 시 **외부 API 사양을 의무로 참조**해 "스키마 부재"를 본구현 회피 사유로 삼지 않는다.
2. **수행·검증(Phase별 구현)** — 각 Phase 를 **코드 작성 → 코드리뷰 → 기능검증**의 세 책임으로 처리한다(§책임 분리). 리뷰·검증에서 결함이 나오면 작성으로 되돌려 보완하고, 한 Phase 가 세 책임을 모두 통과한 뒤에만 다음 Phase 에 착수한다.
3. **배포 산출물·검증 게이트(런타임 동작)** — 모든 Phase 가 세 책임을 통과하면 배포 산출물을 빌드하고(형식은 프로그램 구성표·개발사양이 정의), **그 산출물(사용자 동일 환경)로 최종 런타임 round-trip 검증**(실제 실행·저장↔조회 정합)을 확인한다([`qa-execution.md`](../qa-execution.md) §적용 범위). 잠정 Pass(🔵 Mock·🟣 Static)는 게이트 통과 근거가 아니다 — 실 Pass 로 해소한다. 해소되지 않은 실패가 있으면 정리로 진입하지 않고 구현으로 회귀한다(수정 후 재빌드·재검증). 단 build 환경에서 실 Pass 가 원천적으로 불가능한 항목(외부 의존 미가용 등)은 무한 회귀 대신 사유·해소조건과 함께 **qa 이월 또는 담당자 보류**로 종결한다(§완성도 기준).
4. **정리·보고** — 결과·진척률·보류 항목과 **산출물 식별 정보(버전·경로·기준 commit)** 를 보고한다(qa 인계 입력).

### 책임 분리 (작성 ≠ 리뷰 ≠ 검증)

세 책임은 서로 다른 실행 주체가 맡는다. 작성 주체가 자기 작성물을 합격 판정·기능검증하지 않는다(독립 재현 원칙).

- **코드 작성** — 사양·계획대로 제품 코드를 작성한다. FE 는 승인된 목업을 레이아웃·흐름·상태의 청사진으로 참조하되 그대로 복사하지 않으며, 목업과 사양이 충돌하면 사양·디자인 시스템을 우선한다.
- **코드리뷰(판정)** — 계획·코딩 표준·사양 부합을 검토해 합격/보완을 판정한다.
- **기능검증** — 구현 기능을 실제 실행해 사양·인수 기준대로 동작하는지 독립 재현으로 확인한다. 작성자의 자가 검증은 완료 근거가 아니다.

## 실행 주체

- **ai-pm** — 채널 대화에서 build 작업을 식별해 서브에이전트를 디스패치하고 결과를 보고한다([`ai-pm.md`](../ai-pm.md) §디스패치 계약).
- **서브에이전트** — 계획·작성·리뷰·검증·정리의 책임을 분리해 수행하고, 진행·결과를 관련 Redmine 일감에 반영한다.
- 사양·계획에 근거가 없어 임의 판단이 필요하면 가정하지 않고 담당 채널에 질의·응답을 참조해 진행한다(검토 협의 — [`prompt-conversation.md`](../prompt-conversation.md) §검토 협의). 백그라운드 서브에이전트의 질의는 ai-pm 릴레이로 처리한다([`ai-pm.md`](../ai-pm.md) §질의·승인 릴레이).
- 구체 에이전트 바인딩은 [에이전트 전략](../agents.md)에서 정의한다.

## 수행 방법

build 는 ai-pm 이 **서브에이전트로 디스패치**해 수행한다.

1. **시작** — ai-pm 이 작업 맥락(범위·관련 일감·대상 사양)을 전달해 서브에이전트를 기동한다.
2. **수행·추적** — 계획 단계에서 각 Phase 를 Redmine **`기능` 트래커 일감**으로 등록하고, 구현 진척(작성→리뷰→검증→완료)에 따라 일감 상태를 동기화한다. **자체검증(세 책임) 통과 후 추가 확인이 필요 없는 Phase 는 `완료`(닫힘)까지 정리한다** — 잔여 확인 항목(잠정 Pass 🔵🟣 · 보류 · 사양 결함 제기 · 담당자 판단 대기)이 있는 Phase 만 `해결` 로 두고 그 항목을 노트로 명시한다([`work-tracking.md`](../work-tracking.md) §단계 산출 일감 상태 매핑). **정리 시 Phase 최종 상태를 제목 맨 앞 아이콘(🟢🔵🟣🔴🟠)으로 표기**한다(재판정 시 교체 — [`work-tracking.md`](../work-tracking.md) §제목 상태 아이콘). 각 Phase 일감에는 참조한 사양정의서의 `사양` 일감을 연관 일감으로 추가한다(§산출물).
3. **종료** — 완료 보고를 받은 ai-pm 이 결과·진척률과 qa 착수 승인 요청을 작업세션 이슈로 담당자에게 회신한다.

## 작업 범위

구현은 **전체** 또는 **부분(특정 섹션)** 으로 수행할 수 있다. 부분 수행 시 범위 밖 사양·코드는 입력 참조로만 쓰고 수정하지 않는다. 부분 범위의 선별 단위는 IA 노드이며([`ia.md`](../ia.md)), 경로 규칙은 [doc-structure](../doc-structure.md)가 정의한다.

## 산출물

- **제품 프로그램** — 대상 프로그램·빌드 엔트리·산출물 경로는 개발사양(`docs/prd/devspec/`)을 단일 출처로 한다(본 문서는 경로를 단정하지 않는다).
- **배포 산출물** — 프로그램별 배포 형식(설치 패키지·번들·이미지 등)은 개발사양이 정의한다. qa 착수 입력이므로 식별 정보(버전·경로·기준 commit)를 정리 보고에 포함한다.
- **단계 결과 보고** — Phase 편성·진행 기록 포함. 정본은 관련 Redmine 일감이다([doc-structure](../doc-structure.md) §작업 실행 산출).
- **Redmine 미러(공유용·추적)** — 계획 단계에서 각 Phase 를 `기능` 트래커 일감으로 등록하고, 진척에 따라 상태(신규→진행→완료)를 동기화한다. **추가 확인이 불필요한 Phase 는 `완료`(닫힘)로 종결**하고, 잔여 확인 항목이 있는 Phase 만 `해결` + 잔여 항목 노트로 남긴다. 각 Phase 일감은 참조한 사양정의서의 `사양` 일감을 **연관 일감**으로 추가해 사양↔구현을 잇는다. 트래커 구분·상태 매핑 정책 정본은 [work-tracking](../work-tracking.md).

## 완성도(완료) 기준

완료는 아래를 모두 충족할 때 성립한다. "모든 Phase 통과"(결함 0)와 "사양이 빠짐없이 Phase 로 편성됨"(누락 0)은 서로 다른 축이다.

- 범위 내 모든 Phase 가 코드리뷰 PASS + 기능검증 통과(런타임 검증 게이트 통과).
- **구현 커버리지**(누락 0) — 확정 사양의 모든 구현 대상 항목이 Phase 로 편성되어 **미편성(미커버) 0** 이다(계획 단계의 사양↔Phase 매핑을 정리 단계에서 대조). 갭은 담당자 명시 이월로만 허용한다. 정리 보고에 **사양×Phase×기능검증 커버리지 매트릭스**를 산출해 qa 인계 입력으로 넘긴다.
- **실 Pass 수렴** — 잠정 Pass(🔵 Mock·🟣 Static)는 게이트 통과 근거가 아니다. build 환경에서 실 Pass 가 원천적으로 불가능한 항목(외부 서비스 미가용 등)은 무한 회귀시키지 않고 사유·해소조건과 함께 **qa 이월(실 환경 재검증) 또는 담당자 판단 보류**로 종결한다(해당 Phase 는 `해결` + 이월 노트).
- 코드 빌드 성공 + 배포 산출물 빌드 성공(qa 착수 입력이므로 필수). 빌드 도구 부재 등으로 배포 산출물을 만들 수 없으면 보류 항목으로 명시하고 담당자와 협의한다.
- **Suggestion 종결** — 코드리뷰 Suggestion(수정 의무 없음)은 범위 내·저비용이면 자체보완, 그 외는 담당자·후속 이월로 분류해 완료 보고에 명시한다.
- 완료 판정은 base-workflow §전이 규칙 "완성도 게이트"를 정본으로 따른다.

## 다음 단계 이행 조건

- 완성도 기준을 충족하고 **담당자가 승인**하면, 빌드 산출물을 입력으로 **qa 단계를 수동 착수**할 수 있다.
- 검증 게이트·기능검증에서 사양 결함이 드러나면 spec 단계를 개별 프로세스로 다시 착수한다(base-workflow §전이 규칙 "피드백").
- build 는 독립 단계다 — 승인 없이 자동으로 qa 로 이어지지 않는다.
