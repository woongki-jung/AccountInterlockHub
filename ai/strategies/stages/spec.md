# spec (요구사항·스펙정의) 단계 지침

본 문서는 [`base-workflow.md`](../base-workflow.md) §spec 의 상세 진행 지침이다. base-workflow §단계별 지침 공통 템플릿의 슬롯을 채우고, §개요의 단계 공통 흐름(계획 → 수행 → 검증 → 정리·보고)을 spec 에 매핑한다.

## 목적

방향 정의서(PRD)를 구체적 요구사항·사양으로 옮긴다. 후행 build·qa 가 누락·왜곡 없이 진행되도록 정책·서비스·데이터·기능·화면·프로세스·검증 기준을 도메인별로 정의한다.

## 착수 입력

- directing 산출물 — 방향 정의서(`docs/prd/PRD.md`).
- (있으면) 기존 사양·선행 논의·외부 인터페이스 자료.

## 핵심 활동

단계 공통 흐름에 따라 진행한다.

1. **계획(범위·정합성 게이트)** — 작업 범위(§작업 범위)를 확정하고 **요구사항 정합성을 먼저 평가(진입 게이트)**한다. 방향 정의서에 충돌·공백·모호가 남아 있으면 사양 정의에 진입하지 않고 directing 으로 회귀한다.
2. **수행(도메인별 사양 정의)** — §도메인 의존 순서대로 도메인 하나씩 깊이 있게 정의한다. 상위(선행) 도메인 산출물을 입력으로 하위 도메인을 정의해 누락을 줄인다.
3. **검증(도메인 간 교차검증)** — 도메인별 정의가 끝나면 전체 사양의 도메인 간 정합성(깨진 참조·미반영)을 교차검증한다. 치명·오류가 남으면 해당 도메인으로 회귀해 보완하되, **보완의 합격 판정은 작성자(doer)가 아니라 교차검증(평가자)이 한다** — 회귀 보완 후 반드시 교차검증을 재실행해 통과해야 루프가 닫힌다. 재검증 범위는 **변경 도메인 + 그 도메인을 참조하는 하위 도메인**(증분)으로 하고, 회귀는 도메인별 최대 3회다(§완성도 기준의 회귀 카운터). 마무리 목업에서 사양이 바뀌면 교차검증으로 복귀해 재통과한 뒤에만 완료로 진행한다. **완료 직전에는 전체 모드 교차검증을 1회 수행**해 증분 재검증이 놓친 횡단 영향을 확인한다.
4. **정리·보고** — 확정 사양과 도메인별·교차검증 결과를 취합해 진척률·보류 항목과 함께 보고한다. 완료를 막지 않는 🟡 WARNING·요구사항 제안은 §완성도 기준의 처리 분류(자체보완/담당자 이월)로 종결한다.

### 도메인 의존 순서

사양은 다음 의존 순서로 정의한다. 각 도메인은 선행 도메인의 산출물을 입력으로 쓰므로 선행이 끝난 뒤 착수한다.

| 순서 | 도메인 | 정의 내용 |
|----|------|---------|
| 1 | 정책 | 업무 규칙·제약·권한 등 전 영역 공통 기준 |
| 2 | 서비스 | 사용자에게 제공하는 기능 단위·흐름 |
| 3 | 데이터 | 엔터티·속성·관계 |
| 4 | 기능·API | 서비스를 구현하는 기능·인터페이스 계약 |
| 5 | 화면 | 화면 구성·컴포넌트·상태·전환 |
| 6 | 프로세스 | 화면·기능을 잇는 처리 흐름 |
| 7 | 검증 TC | 사양을 검증하는 테스트 케이스 |

범위에 해당 도메인의 변경·신규가 없으면 그 도메인 정의는 생략한다.

### 정보 구조(IA) 활용

IA 는 directing 단계에서 서비스 제공 기능단위까지 대부분 작성된다([`ia.md`](../ia.md)). spec 은 이를 **작업 단위로 소비**하고, 누락·변경만 보완한다(정련이 spec 의 주 산출이 아니다).

- **사양 ↔ IA 소속**: 각 사양정의서는 자신이 속한 `ia-code`(들)를 "관련 IA 항목"으로 기록한다. 횡단 사양(공통 정책·공유 엔터티 등)은 `공통`. 참조 방향은 사양 → IA 한 방향이다([`ia.md`](../ia.md) §참조 규약).
- **보완만**: 사양 정의 중 IA 누락·오류가 드러나면 `docs/prd/ia/IA.md` 를 보완한다(IA 정본은 directing 산출 — 큰 구조 변경은 directing 보완 루프로). 세부 구현기능은 IA 노드로 만들지 않고 기능·API 도메인 사양으로 다룬다.

### 사양 검증(목업) 루프 — 마무리 활동

목업은 **범위에 화면(SCR) 도메인 신규·변경이 있을 때** 수행한다. 화면 변경이 없으면 목업을 생략하고 그 사유를 완료 보고에 명시한다(단, 담당자가 명시 요청하면 화면 무변경이라도 수행한다). 화면에 영향을 주는 비화면 변경(정책·기능 등)은 하위 화면 사양 재정의로 이어져 SCR 변경이 생기므로 자연히 목업 대상이 된다.

사양이 확정되면 마무리로 화면 사양을 반영한 **UI 중심 경량 목업**(버려지는 시제품)을 만들어 담당자가 눈으로 사양을 검증한다. 리뷰에서 드러난 사양의 모호·누락·오류는 사양 정의로 되돌려 보완한다(목업 = 사양 검증 도구). **목업 보완으로 사양이 바뀌면 교차검증으로 복귀해 재통과하는 것이 담당자 승인의 전제다** — 화면 변경이 도메인 정합성(트리거 PROC·FN 등)을 깨뜨린 채 완료되는 것을 막는다. 승인된 목업은 build 단계 FE 구현의 청사진(참조)으로 넘어가며, 목업 자체는 실코드로 재구현되고 폐기된다.

## 실행 주체

- **ai-pm** — 채널 대화에서 spec 작업을 식별해 서브에이전트를 디스패치하고 결과를 보고한다([`ai-pm.md`](../ai-pm.md) §디스패치 계약).
- **서브에이전트** — 사양 정의를 수행한다. 계획·실행·평가·정리의 책임을 분리하고(작성 주체가 자기 산출을 합격 판정하지 않음), 진행·결과를 관련 Redmine 일감에 반영한다.
- 모호하거나 비가역적 판단이 필요하면 가정하지 않고 담당 채널에 질의·응답을 참조해 진행한다(검토 협의 — [`prompt-conversation.md`](../prompt-conversation.md) §검토 협의). 백그라운드 서브에이전트의 질의·목업 승인은 ai-pm 릴레이로 처리한다([`ai-pm.md`](../ai-pm.md) §질의·승인 릴레이).
- 구체 에이전트 바인딩은 [에이전트 전략](../agents.md)에서 정의한다.

## 수행 방법

spec 은 ai-pm 이 **서브에이전트로 디스패치**해 수행한다(directing 과 달리 ai-pm 이 직접 작성하지 않는다).

1. **시작** — ai-pm 이 작업 맥락(요청 채널·범위·관련 일감)을 전달해 서브에이전트를 기동한다.
2. **수행·추적** — 서브에이전트가 도메인별 사양을 `docs/specs/` 에 작성하고, 같은 내용을 공유용으로 Redmine **`사양` 트래커 일감**에 미러링한다(§산출물). 작업 진행은 종류·범위에 따라 **여러 Redmine 일감**으로 등록·반영한다.
3. **종료** — 완료 보고를 받은 ai-pm 이 결과·진척률과 build 착수 승인 요청을 작업세션 이슈로 담당자에게 회신한다.

## 작업 범위

사양 정의는 **전체** 또는 **부분(특정 섹션)** 으로 수행할 수 있다. 부분 수행 시 대상 외 기존 사양은 입력 참조로만 쓰고 수정하지 않으며, 경계 참조 무결성(깨진 참조)을 교차검증에서 확인한다. 부분 범위의 선별 단위는 IA 노드이며([`ia.md`](../ia.md)), 경로 규칙은 [doc-structure](../doc-structure.md)가 정의한다.

## 산출물

- **사양서(원본 정본)** — 도메인별 정의서 묶음. 워크스페이스 `docs/specs/<도메인>/` 가 단일 출처(SoT)다. 위치 정본은 [doc-structure](../doc-structure.md).
- **정보 구조(IA) 보완** — 필요 시 `docs/prd/ia/IA.md` 누락·변경 반영(IA 정본은 directing 산출; 양식 [`ia.md`](../ia.md)).
- **Redmine 미러(공유용)** — 각 사양정의서를 Redmine **`사양` 트래커 일감**으로 미러링해 담당자가 간편히 열람·공유하게 한다. 원본 정본은 `docs/specs/` 이고 일감은 그 사본이다(원본 갱신 시 일감도 갱신). 도메인별 사양정의서 1건 = `사양` 일감 1건을 기본으로 한다.
- **목업** — 사양 검증용 경량 목업(throwaway). build FE 청사진.
- **작업 티켓 연결** — 작업 진행은 Redmine 일감으로 추적한다(종류·범위에 따라 여러 일감). 트래커 구분(그룹·오류·기능·사양·검증)·이슈↔단계 매핑 등 정책 정본은 [work-tracking](../work-tracking.md).

## 완성도(완료) 기준

완료는 아래 게이트를 **모두** 충족할 때 성립한다. 앞 세 개는 "결함 0"(써놓은 사양끼리 안 깨짐), 커버리지 게이트는 "누락 0"(써야 할 사양을 빠뜨리지 않음)으로 서로 다른 축이다.

- **진입 게이트**(요구사항 정합성)를 통과한다 — 치명 결함 0.
- **교차검증** — 도메인 간 정합성에서 치명·오류 0. 회귀 보완분은 평가자 재검증(증분)을 통과하고, 완료 직전 전체 교차검증 1회를 통과한다(§핵심 활동 3).
- **커버리지 게이트**(누락 0) — 아래 세 지표에 갭이 없다. 갭은 담당자 명시 이월로만 허용한다. 근거는 spec-reviewer 의 커버리지 매트릭스다.
	- PRD 핵심 요구사항 → 사양 **미매핑 0**.
	- 범위 내 IA 노드 → 사양 **미커버 0**.
	- SVC/PROC → 검증 TC **미검증 0**.
- **역방향 전파**(기존 사양 갱신 시) — 갱신된 각 사양에 대해 연관 `기능`·`검증` 일감의 재작업·재검증 영향판정이 누락 없이 표시되고(누락 0), 필요 착수분이 완료 보고에 포함된다([`delivery.md`](../delivery.md) §사양 변경의 역방향 전파). 신규 사양만이면 해당 없음.
- **목업 승인**(범위에 화면 변경이 있을 때) — 목업 리뷰에서 담당자 승인에 도달한다(보완 항목은 명시). 화면 변경이 없어 목업을 생략하면 사유를 명시한다(§사양 검증(목업) 루프).
- **이월 종결** — 완료를 막지 않는 🟡 WARNING·요구사항 제안이 각각 처리 분류로 종결된다: WARNING 은 범위 내·저비용이면 자체보완(doer 회귀), 그 외는 담당자 이월. 요구사항 제안은 spec 이 임의 채택하지 않고 담당자 판단 이월 목록으로 완료 보고에 포함한다.
- **회귀 카운터** — 도메인별 최대 3회, 진전(잔여 결함 감소) 없으면 3회 전이라도 보류로 전환한다(base-workflow §개요 3). 소진 시 보류 보고로 담당자 판단을 구한다.
- 완료 판정은 base-workflow §전이 규칙 "완성도 게이트"를 정본으로 따른다.

## 다음 단계 이행 조건

- 완성도 기준을 충족하고 **담당자가 승인**하면, 확정 사양을 입력으로 **build 단계를 수동 착수**할 수 있다.
- 진입 게이트에서 방향 결함이 드러나면 directing 단계를 개별 프로세스로 다시 착수한다(base-workflow §전이 규칙 "피드백").
- spec 은 독립 단계다 — 승인 없이 자동으로 build 로 이어지지 않는다.
