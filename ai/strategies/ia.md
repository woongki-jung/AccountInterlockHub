# 정보 구조(IA) 작성 전략

본 문서는 제품 **정보 구조(IA)** 산출물의 **표준 양식**을 정의한다. IA 는 작업 범위 선별·사양 소속·이력·티켓 카테고리의 공통 기준(척추)이 되는 제품 영역 트리다. 단계별 작성 범위는 directing·spec 지침이 정의하고, 본 문서는 **무엇을 어떤 형식으로 적는가**(모델·코드·위치·양식)를 담당한다.

## IA 모델

- IA 는 제품의 기능·정보 **영역을 계층 트리**로 나타낸다. 상위 노드(영역) → 하위 노드 → **leaf**.
- **서술 범위 = 서비스 제공 기능단위까지**. leaf 는 사용자에게 제공하는 기능 단위(예: 예약 생성·예약 변경)다. **세부 구현기능**(내부 로직·서브함수·구현 분해)은 IA 에 넣지 않는다 — 그건 spec 의 기능·API 도메인 소관이다.
- 각 노드는 고유 **`ia-code`** 로 식별한다. IA 는 제품을 주소화하는 단일 좌표계다 — 사양·이력·티켓이 이 코드로 IA 를 가리킨다.
- **작성 시점**: IA 의 **대부분은 directing 단계에서 PRD 작성과 함께** 서비스 제공 기능단위까지 작성된다. spec 은 이 IA 를 소비하고(사양↔IA 소속) 누락·변경만 보완한다([`stages/directing.md`](stages/directing.md)·[`stages/spec.md`](stages/spec.md)).

## 위치·파일

- IA 산출물은 `docs/prd/ia/` 에 둔다(directing 산출 PRD 와 같은 계열).
- IA 맵 1개 `docs/prd/ia/IA.md` 가 트리 전체를 담는다. 노드가 많아지면 영역별 하위 파일로 분할할 수 있다(분할·경로 규칙은 [`doc-structure.md`](doc-structure.md)).

## `ia-code` 규칙

- 형식: `<영역약어>-<하위>` 계층 표기(예: `RES`, `RES-01`, `RES-01-EDIT`). 상위→하위가 코드에 드러나게 한다.
- 코드는 발급 후 **불변**(이력·사양·티켓이 영구 참조). 영역을 폐지해도 코드를 재사용하지 않는다.
- 횡단(특정 노드에 매이지 않는 공통 정책·공유 엔터티 등)은 `공통` 으로 표기한다.

## 표준 양식 (IA 맵)

`docs/prd/ia/IA.md` 는 노드 트리를 표로 담는다. 각 노드 행에 다음을 적는다.

- `ia-code` / 노드명 / **깊이**(영역·중간·leaf) / **우선순위**(directing) / 한 줄 설명.
- leaf 노드는 spec·build·qa 의 작업 범위 단위가 됨을 깊이로 표시한다.

```markdown
# 정보 구조 (IA)

| ia-code | 노드 | 깊이 | 우선순위 | 설명 |
|---|---|---|---|---|
| RES | 예약 | 영역 | 높음 | 예약 관련 전체 영역 |
| RES-01 | 예약 생성 | leaf | 높음 | 신규 예약 등록 |
| RES-02 | 예약 변경 | leaf | 보통 | 예약 수정·취소 |
| 공통 | (횡단) | — | — | 영역 무관 공통 정책·공유 엔터티 |
```

## 참조 규약

- **사양 → IA**: 각 사양정의서는 자신이 속한 `ia-code`(들)를 "관련 IA 항목"으로 기록한다(횡단은 `공통`). 참조 방향은 **사양 → IA 한 방향**이며, IA 맵에 사양 역링크를 달지 않는다(상위 IA 가 종속 순서에서 우선).
- IA 는 작업 범위 선별·노드별 이력·티켓 카테고리의 기준이 된다. 각 소비 정책(work-tracking 의 카테고리·ia-history 의 노드 이력 등)의 상세는 해당 전략이 정본이다.
