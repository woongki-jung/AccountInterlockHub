# qa 케이스: 데스크톱 네이티브 UI

[`../qa-execution.md`](../qa-execution.md) 의 케이스 하위 지침. 네이티브 데스크톱 GUI 프로그램의 검증 방식을 정의한다. 공통 원칙(5종 판정·동일 환경·독립 재현·증빙)은 부모 문서를 준용한다.

## 적용 조건 (선택 트리거)

directing 프로그램 구성표의 검증 방법이 **데스크톱 네이티브 UI** 인 프로그램. 예: 데스크톱 GUI 앱(.NET·Win32·기타 네이티브 프레임워크). 웹뷰 내장 하이브리드면 [웹·SPA UI](web-ui.md) 케이스를 병행한다.

## 환경 구성

- 검증 대상은 **설치 산출물(배포 산출물 설치본)을 기동**한다(사용자 동일 환경). 워크스페이스 직접 빌드 산출물 기동은 쓰지 않는다 — build 의 Phase별 기능검증만 예외([`../qa-execution.md`](../qa-execution.md) §적용 범위).
- 설치 경로·실행 파일·기동 인자는 프로젝트 개발사양(`docs/prd/`)을 단일 출처로 한다.

## 검증 도구·절차

- **지정 도구: pywinauto-mcp 등 OS UI 자동화 MCP** — 접근성 트리·컨트롤 식별자 기반으로 실제 창·컨트롤을 제어한다. 백엔드는 대상 UI 프레임워크로 선택한다 — UIA 지원(WinForms·WPF·UWP 등)은 `uia`, 구형 Win32 계열은 `win32`. 판별 기준·설치·등록·연결 검증은 [tools-setup.md](tools-setup.md).
- 시작 시 UI 트리 baseline 덤프를 확보해 셀렉터 산출 기준선으로 쓴다.
- 런타임 round-trip(입력→저장→조회 정합)으로 기능을 확인한다.
- 지정 도구 미연결로 실행할 수 없는 TC 는 🟠 Block 으로 등재하고 환경 보완을 요청한다.
- 확정 백엔드·수집된 식별자(`AutomationId`·창 클래스명 등)·포트는 프로젝트 구성에 맞춰 본 절에 채운다(수집 도구·절차: [tools-setup.md](tools-setup.md) §식별자 수집).

## 판정·증빙

- 5종 판정(🟢🔵🟣🔴🟠) — 부모 §공통 원칙.
- 준비·실행·결과 스크린샷과 UI 트리 덤프를 보존한다.
