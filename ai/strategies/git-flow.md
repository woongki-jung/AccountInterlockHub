# git flow 전략

본 문서는 워크스페이스 저장소의 **브랜치·커밋·병합(MR)** 운용 규칙을 정의하는 단일 출처다. 워크플로우 단위에 git 단위를 대응시킨다 — **단계 착수(디스패치) 1회 = 브랜치, Phase·작업 항목 = commit, 담당자 승인 = main 병합**. 저장소 구성(단일 저장소·산출물 위치)은 [`delivery.md`](delivery.md) §저장소 운용, 단계 흐름·승인 게이트는 [`base-workflow.md`](base-workflow.md) 가 정본이다.

## 원칙

- **main = 승인 통과 기준선** — main 에는 완성도 게이트와 담당자 승인을 통과한 작업만 병합한다. 후행 단계 착수(예: qa 의 검증 대상)와 릴리스의 기준 commit 은 항상 main 에서 잡는다.
- **작업 구간 = 브랜치** — 단계 착수(디스패치) 1회가 브랜치 1개다. 브랜치가 살아 있는 동안이 그 착수분의 작업 구간이며, 잔존 브랜치 목록이 곧 진행 중·보류 작업 목록이다.
- **Phase = commit** — 브랜치 안에서 Phase·작업 항목 1건이 commit 1건이다.

## 브랜치 (단계 착수 단위)

- **생성** — 단계 오케스트레이터(directing 은 ai-pm)가 착수 시 최신 main 에서 분기한다. 질의·승인 릴레이 후 재디스패치(재개)면 새로 만들지 않고 기존 브랜치를 이어 쓴다.
- **명명** — `<단계>/<이슈번호>-<주제>`
	- `<단계>` = `directing` / `spec` / `build` / `qa`.
	- `<이슈번호>` = 디스패치 기준 Redmine 일감 번호 — 그룹 일감 우선, 없으면 단계 대표 일감([`work-tracking.md`](work-tracking.md) §티켓 ID).
	- `<주제>` = 영문 소문자 kebab-case 요약.
	- 예: `spec/140-order-domain` · `build/152-user-auth` · `qa/171-release-1-2`.
- **핫픽스** — 유지보수 경로([`delivery.md`](delivery.md) §유지보수·핫픽스 경로)는 build 부분 착수이므로 같은 규칙으로 `build/<오류일감번호>-<주제>` 를 쓴다(별도 접두 없음).
- **운영·메타 작업** — 단계 밖 작업(운영 문서·워크스페이스 개선)은 `docs/<주제>`(문서·정책) 또는 `chore/<주제>`(설정·정리)로 분기한다. 병합 규칙은 단계 브랜치와 같다(승인 주체는 담당자).

## 커밋 (Phase 단위)

- **단위** — 1 Phase(작업 항목) = 1 commit 을 기본으로 한다. 산출물과 이력 갱신([`ia-history.md`](ia-history.md)·[`doc-revision.md`](doc-revision.md))을 같은 commit 에 담는다. 리뷰·검증 회귀의 보완은 같은 브랜치에 추가 commit 으로 쌓는다(이력 재작성 금지).
- **수행 주체** — 승인된 단계 디스패치의 위임 범위로 단계 작업 주체(doer)가 수행한다.
- **메시지 양식** — 제목은 한국어 50자 내외, 접두는 브랜치와 같은 단계명(운영 작업은 `docs:`/`chore:`).

	```
	<단계>: <작업 항목 요약>

	<변경 요지 — 필요 시>

	관련 일감: <프로젝트식별자>#NNN
	```

	- 예: `build: 로그인 API 구현 (Phase 3/8)` · `spec: 주문 도메인 서비스 사양 작성` · `docs: 검증 게이트 수치 정본화`.
	- 관련 일감이 여럿이면 쉼표로 나열하고, 일감 없는 운영 작업은 그 줄을 생략한다. 일감 번호 표기는 [`work-tracking.md`](work-tracking.md) §티켓 ID.

## 병합 (MR 전략)

로컬 저장소이므로 PR 플랫폼 없이 아래 절차가 병합 요청(MR)을 대신한다.

1. **조건** — 단계 완성도 게이트 통과 + **담당자 승인**. 다음 단계 이행 승인([`base-workflow.md`](base-workflow.md) §단계 진행 모델)과 같은 시점이며, 승인 증적은 (그룹) 일감 노트다([`work-tracking.md`](work-tracking.md) §생애주기·갱신 책임).
2. **주체** — 승인을 접수한 주체(통상 ai-pm, 담당자 직접 세션이면 그 세션)가 수행한다. **단계 오케스트레이터는 자기 브랜치를 병합하지 않는다**(승인 게이트 우회 방지). 병합은 업무 수행이 아니라 승인의 저장소 반영(관리 동작)이다.
3. **방식** — `--no-ff` 로 병합 commit 을 남긴다(Phase commit 보존, 착수분 경계 가시화). 병합 commit 제목: `Merge: <요약> (<브랜치명>)`.
4. **충돌** — main 이 앞서 있으면 **브랜치 쪽에서 main 을 병합해 해소**하고, 영향 범위를 재검증한 뒤 main 에 병합한다.
5. **원격 반영(origin push)** — main 병합 직후 **항상 origin 으로 push** 한다(main 브랜치 + 병합에 수반된 릴리스 태그). 병합 결과를 로컬에만 두지 않는다(2026-07-09 담당자 지시 정착). push 실패는 조용히 넘기지 않고 담당자에게 알린다([`ai-pm.md`](ai-pm.md) §글로벌 운영 원칙 — 에러 보고).
6. **병합 후** — 브랜치를 삭제한다(원격에 push 됐으면 원격도). 잔존 브랜치에는 진행 중·보류 작업만 남긴다.

- **반려·이월** — 승인 반려면 브랜치를 유지한 채 재작업한다. 장기 보류·이월이면 관련 일감을 `보류` 로 두고 브랜치를 남기되, 일감 노트에 브랜치명을 기록한다.

## 릴리스 태그

릴리스 확정 시([`delivery.md`](delivery.md) §버전·릴리스) main 의 기준 commit 에 `v<MAJOR.MINOR.PATCH>` 태그를 부여한다 — 릴리스 노트의 "기준 commit" 과 같은 지점이다. 부여한 태그는 §병합 5(원격 반영)에 따라 origin 으로 함께 push 한다.
