# 업무 티켓 관리 전략

본 문서는 담당자 요청부터 완료까지 업무 1건의 **생애주기를 Redmine 이슈(정본)로 단계 횡단 추적**하는 정책을 정의한다. 접속·식별자·프로젝트 생성 등 운영 세부는 하위문서 [`work-tracking-redmine.md`](work-tracking-redmine.md) 가 담당한다.

- **정본 = Redmine 이슈** — 업무 단위마다 Redmine 프로젝트를 두고(요청·작업 = 이슈), 담당자는 Redmine 에서 진행을 직접 조회한다.
- **산출물 원본은 워크스페이스 파일** — 사양·코드·목업의 정본(SoT)은 `docs/`·`apps/`·`mockup/` 등 워크스페이스 파일이다. Redmine 일감은 그 작업을 **추적·공유**하고 산출물 내용을 **미러링**한다(§단계별 미러링).
- 세션 대화·디스패치는 휘발성이므로, 접수·진행의 영속 기록은 이슈가 담당한다.

## 프로젝트·이슈 구조

- **Redmine 프로젝트 = 업무 단위 1개** — 제품 프로젝트는 1:1 로 생성하고(절차 [`work-tracking-redmine.md`](work-tracking-redmine.md) §프로젝트 생성 표준 절차), 워크스페이스 운영·메타 업무는 운영 프로젝트(`ai-workgroup-ops`)에서 추적한다.
- **이슈 1건 = 작업 1건** — 한 요청이 종류·범위에 따라 여러 일감으로 나뉠 수 있다([`ai-pm.md`](ai-pm.md) §디스패치 계약). 단계가 넘어가도 같은 작업은 같은 이슈를 유지한다.
- **연관·계층으로 연결** — 일감 관계(연관)와 상위-하위 이슈로 사양↔구현↔검증을 잇는다(§일감 유형·계층).
- **이슈 범주 = IA 노드** — 이슈가 어느 IA 노드(서비스 제공 기능단위)에 속하는지 Redmine **카테고리**로 표시한다([`ia.md`](ia.md)). 작업 범위 선별·이력이 같은 IA 좌표로 정렬된다.

## 일감 유형 (트래커)

작업 종류를 트래커로 구분한다. 숫자 id·생성 방법은 [`work-tracking-redmine.md`](work-tracking-redmine.md) 가 관리한다(인스턴스를 재구축해도 정책은 유지).

| 트래커 | 용도 |
|---|---|
| **그룹** | 업무 묶음·부모(우산) 일감, 워크스페이스 ops·메타 |
| **오류** | 결함·장애 해소 |
| **기능** | 기능·구현 작업 (build Phase 미러) |
| **사양** | 사양정의서(`docs/specs/`)의 Redmine 미러 (spec 산출) |
| **검증** | 검증 TC — 검증 대상 일감의 **하위 이슈**로 등록 (qa) |

### 계층·연관

- **그룹 일감**을 업무 단위의 우산으로 두고, 그 아래 사양·기능·오류·검증 일감을 자식으로 묶을 수 있다.
- **검증 일감**은 검증 대상(기능 등) 일감의 하위 이슈로 등록한다.
- build·qa 일감은 참조한 **사양 일감**을 연관 일감(relations)으로 추가해 사양↔구현↔검증을 잇는다([`stages/build.md`](stages/build.md)·[`stages/qa.md`](stages/qa.md)).
- **qa 단계에서 🔴 Fail 판정된 TC 는 검증 실행 주체가 `오류` 일감을 생성**해 해당 검증·사양 일감과 연관한다 — 결함이 일감으로 남아 후속 처리·유지보수 경로([`delivery.md`](delivery.md))로 이어진다. build 단계 내부 회귀(Phase 루프)는 오류 일감 없이 해당 기능 일감 노트로 기록한다.

## 단계별 미러링

각 단계 산출물을 일감으로 미러링한다. 원본 정본은 워크스페이스 파일이고 일감은 그 사본·추적이다(원본 갱신 시 일감 갱신). 단계별 상세는 각 stages 문서가 정본.

| 단계 | 미러 단위 → 일감 | 상태 동기화 |
|---|---|---|
| spec | 도메인별 사양정의서 1건 → **사양** 일감 1건 | 작성·확정 |
| build | Phase 1건 → **기능** 일감 1건 | 진척(작성→리뷰→검증→완료) |
| qa | 검증 TC 1건 → **검증** 일감 1건 | 검증 결과(5종 판정) |

directing 작업 자체도 이슈로 추적한다([`stages/directing.md`](stages/directing.md)).

## 상태 어휘

| 상태 | 의미 | 진입 시점 |
|---|---|---|
| 신규 | 접수됨·미착수 | 이슈 생성 |
| 진행 | 작업 중 | 디스패치·착수 |
| 해결 | 평가·리뷰 대기 | 실행 완료 보고 후 |
| 완료 (닫힘) | 완료 | 평가 통과·담당자 확인 |
| 보류 | 외부 응답 대기 | 검토 협의 질의 등 — 응답 시 이전 상태 복귀 |
| 거절 (닫힘) | 취소·철회 | 담당자 결정(사유 노트) |

### 단계 산출 일감 상태 매핑

단계별 미러 일감의 상태 전이 기준. 세부 진척(예: build 의 작성→리뷰→검증 구간)은 상태 어휘를 늘리지 않고 **이슈 노트**로 기록한다.

| 일감 | 신규 | 진행 | 해결 | 완료 |
|---|---|---|---|---|
| 사양 (spec) | 등록·작성 착수 전 | 작성·보완 중 | 교차검증 통과 | 단계 완료·담당자 승인 |
| 기능 (build Phase) | Phase 편성 등록 | 작성~리뷰~검증 진행(구간은 노트) | 세 책임 모두 통과 | 단계 게이트 통과·정리 |
| 검증 (qa TC) | TC 등록 | 실행 중 | 🟢 Pass 또는 잠정 Pass(🔵🟣 — 잠정 사유 노트 필수) | 단계 평가·담당자 확인 |

- 🟢/🔵/🟣 **Pass** → `해결`(잠정 Pass 🔵🟣 는 재검증 사유 노트 필수). `완료`(닫힘)는 평가·담당자 확인 몫이다 — 특히 qa 검증 일감은 **담당자 승인(배포 가능 판정) 후에만** 완료하며, 검증 실행 주체(qa 오케스트레이터·tester)는 `해결`까지만 설정한다([`stages/qa.md`](stages/qa.md) §다음 단계 이행 조건).
- 🔴 **Fail** → `진행` 유지 + 판정 노트. qa 단계의 Fail 은 `오류` 일감을 생성해 연관한다(§계층·연관).
- 🟠 **Block·명시적 이월** → `보류` + 해소 조건 노트. 해소 시 `진행` 복귀.
- 리뷰·검증 결함으로 회귀하면 상태는 `진행` 을 유지하고 회귀 사유·시도 횟수를 노트로 남긴다(최대 3회 — [`base-workflow.md`](base-workflow.md) §개요 공통 흐름).

## 티켓 ID

- ID = Redmine 이슈 번호 `<프로젝트식별자>#NNN`(예: `ai-workgroup-ops#14`). 별도 채번 없음.
- 발급 후 불변. 대화·디스패치·산출물·이력 어디서든 이 번호로 참조한다.

## 표준 이슈 양식

- **제목**: 작업을 한 줄로 요약(한글 허용).
- **설명 첫 줄 메타 머리말**: 요청자·요청 출처·단계(spec/build/qa). 이어 배경·산출물 링크·검토 협의 기록.
- **진행 로그 = 이슈 노트**: 상태 전이·산출물 생성 시점마다 노트 1건(모든 행위를 적지 않음). 작성자·시각은 Redmine 이 자동 기록한다.

## 생애주기·갱신 책임

| 시점 | 주체 | Redmine 동작 |
|---|---|---|
| 요청 접수 | 담당자 | **작업세션 이슈 생성**(상태 `신규`) — 이슈(스레드) 생성은 담당자 몫([`ai-pm.md`](ai-pm.md) §Slack→Redmine 매핑) |
| 인식·접수 확인 | ai-pm | 작업세션 이슈에 접수 노트 + 상태 `진행`. **봇은 이슈를 생성하지 않는다**(댓글·상태만) |
| 디스패치 | ai-pm | 서브에이전트 기동(작업세션 이슈 번호·맥락 전달). 산출 일감(사양·기능·오류·검증) 생성은 서브에이전트 소관 |
| 진행·미러 | 서브에이전트 | 산출물 작성과 함께 해당 일감 등록·갱신(미러·노트), 상태 동기화 |
| 완료 보고 | 서브에이전트 | 완료 시 상태 `해결` |
| 평가·전달 | ai-pm | 통과 → `완료`(산출물·링크 노트), 보완 → `진행` 복귀. 결과를 작업세션 이슈로 담당자에 회신 |
| 보류 | 질의 주체 | `보류` + 검토 협의 링크 노트. 응답 수신 시 이전 상태 복귀 |
| 취소 | ai-pm(담당자 지시) | `거절` + 사유 노트 |

- 상태·담당자를 바꾼 주체가 같은 동작의 노트로 사유를 남긴다. 별도 보드 갱신은 불필요하다(§조회).
- 단계 이행·게이트의 **담당자 승인·반려는 ai-pm 이 해당 (그룹) 일감 노트로 기록**한다 — 작업세션 이슈 대화는 통로이고 증적 정본은 일감 노트다([`ai-pm.md`](ai-pm.md) §질의·승인 릴레이).

## 티켓 생성 기준

다음 중 하나면 생성한다 — 산출물 생성·변경이 따르는 요청 / 서브에이전트 디스패치가 발생하는 작업 / 후속 추적(진행·완료 확인)이 필요한 요청. 단답으로 끝나는 질문·조회·인사는 티켓 없이 즉답한다. 애매하면 만들지 않고, 후속 작업이 실제 발생하는 시점에 만든다.

## 조회

별도 INDEX 파일을 두지 않는다 — Redmine 이슈 목록·필터가 보드 역할을 한다. 상태·담당자·카테고리·트래커로 걸러 본다. 자주 보는 뷰(진행 중·대기·완료)는 Redmine **저장 쿼리**로 만들어 담당자와 공유한다([`work-tracking-redmine.md`](work-tracking-redmine.md) §저장 쿼리).
