# IA 기반 이력 관리 전략

본 문서는 제품 작업의 누적 이력을 **IA 노드별 시계열**로 관리하는 정책을 정의한다. 어느 단계(spec/build/qa)의 작업이든 어느 IA 노드에 영향을 줬는지 한 곳에서 시간 순으로 조회·감사할 수 있게 한다. IA 모델·코드는 [`ia.md`](ia.md), 진행 중 업무의 현재 상태는 [`work-tracking.md`](work-tracking.md)(Redmine 일감)가 담당하고, 본 문서는 **완료된 작업의 IA별 시계열 기록**을 담당한다.

## 위치·파일 구조

루트 `history/` — 산출물(`docs/`·`apps/`)과 분리된 횡단 이력 인덱스.

- `history/README.md` — 양식·갱신 정책 요약(본 문서 링크).
- `history/<ia-code>.md` — IA 노드별 이력(1 파일 = 1 IA 노드, leaf 단위 기본).
- `history/common.md` — 횡단(`공통`) 사양 이력.
- `history/bak-<YYYY>.md` — 연도별 백업(§연도별 백업).

- **파일명 = `ia-code`** 그대로([`ia.md`](ia.md) §`ia-code` 규칙). 예: `RES-01.md`.
- **1 IA 노드 = 1 파일**(leaf 단위). 상위 노드는 하위 leaf 의 합집합이라 별도 파일을 두지 않는다.
- 신규 IA 의 첫 작업 시 해당 파일을 생성한다.

## Entry 양식

각 파일 본문은 단일 표 — **신규는 맨 위(최신순)**.

```markdown
# IA: <ia-code> — <노드 명칭>

> 본 IA 노드와 연관된 작업의 시계열 이력(최신순). 정책: ai/strategies/ia-history.md

| 일시 (KST) | 단계 | 산출물·결과 | 관련 일감 | 상태 |
|---|---|---|---|---|
| 2026-06-29 14:32 | spec | `service_xyz.md` 신규 | `proj#42` | ✅ |
| 2026-06-28 17:25 | qa | TC-001~005 → 5/5 Pass | `proj#57` | ✅ |
```

- **일시 (KST)**: `YYYY-MM-DD HH:MM` — 산출물 commit 직전 시각.
- **단계**: `spec`/`build`/`qa`. 횡단 사양이 본 IA 에 영향을 줬으면 `<단계> ⓒ`(§공통·횡단).
- **산출물·결과**: 산출물 경로 + 결과 한 줄 요약(검증은 집계 포함).
- **관련 일감**: 해당 작업의 Redmine 일감 번호([`work-tracking.md`](work-tracking.md)).
- **상태**: `✅` 완료 / `🚧` 진행 / `⏸️` 보류 / `❌` 실패·보완 / `ℹ️` 공통 참조.

## 갱신 절차

**산출물 commit 직전**에 작업 서브에이전트가 수행한다.

1. **영향 IA 수집** — 본 작업에서 생성·갱신한 산출물의 "관련 IA 항목"([`ia.md`](ia.md) §참조 규약)을 모은다. `공통` 은 §공통·횡단으로 처리.
2. **각 IA 파일 갱신** — 파일이 없으면 양식으로 생성, 표 맨 위에 entry 1행 삽입(상태 `🚧`, 공통 참조면 `ℹ️`). 회전이 필요하면 먼저 §연도별 백업.
3. **commit·일감 동기** — history 갱신을 산출물과 같은 commit 에 포함하고, 추적 중인 Redmine 일감에 진행 노트 1건을 남긴다([`work-tracking.md`](work-tracking.md) §생애주기·갱신 책임).
4. **상태 사후 갱신** — 평가가 확정되면 평가 주체(ai-pm·평가 서브에이전트)가 해당 row 의 상태만 갱신한다(새 row 를 만들지 않음).

## 공통·횡단 사양 처리

`공통`(IA 노드에 매이지 않는 횡단 사양)의 작업은:

1. **`history/common.md` 가 단일 소스** — entry 본문은 여기에만 두고, 영향 IA 컬럼에 영향받는 IA 노드를 나열한다.
2. **영향 IA 각 파일에 역링크 entry** — 단계 컬럼 `<단계> ⓒ`, 산출물 컬럼 `(공통 반영) <산출물> — common.md`, 상태 `ℹ️`. 본 IA 이력만 봐도 공통 변경 영향을 인지하게 한다.

복수 IA 를 명시한(공통 아님) 사양은 각 IA 파일에 정식 entry 를 중복 등록한다.

## 파일 길이·연도별 백업

`history/*.md` 는 append-only 라 [`document-master-guide.md`](document-master-guide.md) 300줄 제한에서 **제외**된다. 대신 연도 회전으로 활성 파일 크기를 관리한다.

- 활성 파일에는 **현재 연도 + 직전 연도** entry 만 둔다.
- entry 연도가 `현재연도 - 2` 이하면 활성 파일에서 분리해 `history/bak-<YYYY>.md`(연도별 1개, 모든 IA + common 을 IA 섹션으로 묶음)로 이전한다.
- 회전은 활성 파일에 entry 추가 직전 검사·수행한다. 활성 파일 맨 아래 row 연도가 `현재연도 - 1` 이상이면 회전 불필요(검사 종료).
- 특정 IA 전체 이력 = 활성 파일 + 각 `bak-<YYYY>.md` 의 그 IA 섹션 합집합.

## 책임 분담·백필

| 주체 | 책임 |
|---|---|
| 작업 서브에이전트 | 산출물 commit 직전 §갱신 절차(상태 `🚧`/`ℹ️`) |
| 평가 주체(ai-pm·평가 서브에이전트) | 평가 후 해당 row 의 상태만 갱신(✅/❌/⏸️) |

- 본 정책 도입 이전 작업은 백필하지 않는다(forward-only).
- 진행 중 업무의 현재 상태는 [`work-tracking.md`](work-tracking.md)(Redmine 일감)가 담당한다 — 본 이력은 완료 시계열·감사용이다.
