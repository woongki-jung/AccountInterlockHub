# 업무 수행 기반 전략

본 문서는 워크스페이스에서 수행되는 모든 업무를 **4단계**로 구분하고 각 단계의 진행 방식을 정의하는 최상위 흐름 문서다. [`CLAUDE.md`](../../CLAUDE.md) §워크스페이스 운용 전략이 본 문서를 참조한다.

## 개요

프로젝트 진행은 다음 4단계로 나뉜다.

1. **directing (방향설정)** — 프로젝트의 목적과 목표, 서비스 이용 대상과 제공 가치, 성과 지표 등 프로젝트의 방향을 정의한다.
2. **spec (요구사항·스펙정의)** — 프로젝트 방향을 구체적 요구사항·사양서로 옮긴다.
3. **build (구현)** — 사양을 실제 산출물로 만든다.
4. **qa (품질검증)** — 산출물이 사양을 충족하는지 검증한다.

이 4단계는 성격이 둘로 나뉜다 — **directing 은 담당자 참여로 방향을 준비하는 단계**이고, **spec·build·qa 는 그 방향을 제품으로 실현하는 하나의 제품개발 루프**다. directing 이 사람 판단을 앞당겨 결정해 두어, 루프가 사람 개입 없이 끝까지 돌 수 있도록 입력(방향 정의서·IA·프로그램 구성표·개발사양·환경 값)을 준비한다. 루프 내부의 피드백(spec↔build↔qa)과, 루프가 방향 결함을 발견해 directing 으로 되돌리는 피드백은 성격이 다르다(§전이 규칙 "피드백").

각 단계는 공통적으로 다음의 흐름을 따른다.

1. 작업 목표(범위) 및 계획 수립
2. 계획 수행
3. 수행 결과 검증 — 검증 결과가 충분한 수준에 도달하지 못하면 해당 부분을 회귀해 재진행한다. **회귀 시도는 회귀가 발생하는 작업 단위별로 독립 집계**한다(spec=도메인·build=Phase·qa=TC — 단 qa 의 '회귀'는 자체 수정이 아니라 외부(build/spec) 수정 후 재검증이다), **단위당 최대 3회**. 시도 횟수와 직전 대비 잔여 결함 수는 해당 작업 단위의 Redmine 일감 노트로 기록해 세션이 바뀌어도 유지한다. **회귀해도 잔여 결함이 줄지 않으면(진전 없음) 3회를 다 쓰기 전이라도 즉시 보류로 전환**해 담당자 판단을 구한다(무의미한 재시도 소진 방지). 회귀 보완의 합격은 작성 주체가 아니라 분리된 평가 주체의 재검증 통과로 인정한다(자가 판정 금지). **directing 은 인간 협의 단계라 기계적 회귀 상한을 적용하지 않는다** — 요구사항 세분화·루프 진입 준비도 확보가 수렴하거나 담당자가 충분하다고 판단할 때까지 반복한다
4. 정리 및 보고 — 전체 프로젝트 수행 범위를 기준으로 해당 단계의 프로젝트 수준 진척률을 산정한다. 3에서 반복 실행으로 충분한 결과를 얻지 못한 경우 이에 대한 보고와 함께 실행 실패로 보고한다

## 단계 진행 모델 (전이 규칙)

- **제품 루프 오케스트레이션** — spec·build·qa 는 **제품 루프 오케스트레이터**([`../agents/product-loop.md`](../agents/product-loop.md))가 하나의 루프로 제어한다. ai-pm 은 요청을 **내용으로 판별**해 directing(직접) 또는 제품 루프 오케스트레이터(디스패치) 둘로 분기한다. 루프는 어느 단계에서든 진입할 수 있다(단일 창구).
- **하이브리드 전이** — 루프 내부 단계 전이(spec→build, build→qa)는 핸드오프 검증(커버리지 인계) 통과 시 **자동 진행**하고, **체크포인트**(qa 배포 판정·main 병합)에서만 담당자 승인에서 멈춘다. main = 승인 기준선은 유지된다 — 승인 지점이 체크포인트로 모인다([`git-flow.md`](git-flow.md) §제품 루프 브랜치).
- **완성도 게이트** — 각 단계는 자체 완료 기준으로 "충분한 완성도"를 확인한다. 확인 주체·방법은 각 단계별 지침에서 정의한다.
- **피드백** — 후행 단계에서 선행 단계의 결함이 발견되면 선행 단계를 다시 착수한다. **루프 내부 결함**(qa Fail→build, build 사양결함→spec)은 오케스트레이터가 자동 회귀 편성하고(담당자 보고), **방향 결함**은 루프를 벗어나 directing 으로 에스컬레이션한다(담당자 판단).
- **독립 실행** — 요청은 특정 단계·범위를 지목해 착수할 수 있고, 한 작업이 반드시 4단계를 모두 통과함을 전제하지 않는다. 단일 단계 요청도 제품 루프 오케스트레이터가 해당 진입점으로 받아 범위만큼만 전진한다(단일 창구).

## 단계별 지침 공통 템플릿

각 단계의 상세 진행 지침은 다음 슬롯을 채워 작성한다(각 `stages/` 문서가 이 슬롯을 채운다).

- **목적** — 이 단계가 달성하는 것.
- **착수 입력** — 착수에 필요한 선행 산출물·요청.
- **핵심 활동** — 단계 내 주요 작업.
- **실행 주체** — 담당 역할(담당자·에이전트). 구체 에이전트 바인딩은 에이전트 전략에서 정의한다.
- **산출물** — 생성물의 위치·형식.
- **완성도(완료) 기준** — 게이트 통과 조건. 누가 어떻게 확인하는지 포함.
- **단계 보고** — 정리·보고 산출(완료 시점의 진척률·보류/미결·성공/실패).
- **다음 단계 이행 조건** — 어떤 승인을 거쳐 어느 단계로 넘어가는지.

## 단계별 지침

각 단계의 상세 진행 지침은 `stages/` 하위 단계별 문서에 둔다. 아래는 요약이며, 상세가 작성된 단계는 링크를 따른다.

### directing (방향설정)

프로젝트의 방향을 정의하고(무엇을·왜) 제품 루프가 소비할 입력을 준비한다. 담당자와 ai-pm 이 직접 협의(퍼실리테이션)하며, 요구사항 세분화·루프 진입 준비도 평가는 평가 doer(prd-reviewer)를 반복 호출해 수행한다. 산출물은 **방향 정의서**(목적·목표·이용 대상·제공 가치·성과 지표·범위·우선순위)와 그에 정합하는 IA·프로그램 구성표·개발사양·환경 값이다. 요구사항을 더 세분화할 여지가 없을 때(수렴)까지 다듬어 루프에 넘긴다.

→ 상세: [`stages/directing.md`](stages/directing.md)

### spec (요구사항·스펙정의)

방향 정의서를 구체적 요구사항·사양으로 옮긴다. 요구사항 정합성 게이트 → 도메인별(정책·서비스·데이터·기능·화면·프로세스·검증 TC) 의존 순서 정의 → 도메인 간 교차검증으로 진행하고, 마무리로 목업 검증 루프를 거친다. 산출물은 도메인별 사양서 묶음이다. ai-pm 이 서브에이전트로 디스패치해 수행한다.

→ 상세: [`stages/spec.md`](stages/spec.md)

### build (구현)

확정 사양을 제품으로 구현한다. 구현 항목을 Phase 로 빠짐없이 편성하고 각 Phase 를 코드 작성 → 코드리뷰 → 기능검증의 분리된 책임으로 처리한 뒤, 런타임 동작 검증 게이트를 거쳐 배포 산출물(프로그램 구성표가 정한 케이스별 형식 — 설치 패키지·번들·이미지 등)을 빌드한다. 무게중심은 BE 로직이며 FE 는 승인된 목업을 청사진으로 삼는다. ai-pm 이 서브에이전트로 디스패치해 수행한다.

→ 상세: [`stages/build.md`](stages/build.md)

### qa (품질검증)

산출물이 사양을 충족하는지 사용자 동일 환경(배포 산출물 설치·기동본)에서 검증한다. 검증 TC 를 빠짐없이 Phase 로 배정·실행해 5종으로 판정하고, 담당자별(기획·디자인·개발·QA)로 그룹화한 결과·품질 게이트로 배포 가능 여부를 평가한다. ai-pm 이 서브에이전트로 디스패치해 수행한다.

→ 상세: [`stages/qa.md`](stages/qa.md)

배포 가능 판정 이후의 릴리스·배포·유지보수(핫픽스)는 [`delivery.md`](delivery.md) 가 정의한다.
