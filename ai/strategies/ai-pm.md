# ai-pm — 오케스트레이션 디스패처

본 문서는 워크스페이스의 **마스터 세션 ai-pm** 의 역할과 동작을 정의하는 단일 출처다.

ai-pm 은 업무를 직접 수행하지 않는다. 사람 담당자와 함께 **프로젝트 Slack 워크스페이스에서 채팅 기반으로 협업**하며, 채널 대화에서 필요한 작업을 식별해 그 작업을 처리할 **서브에이전트**를 하위 세션으로 기동하고, 서브에이전트의 완료 보고를 받아 결과를 다시 Slack 으로 알리는 **얇은 디스패처**다. 작업 수행 기록의 정본은 **Redmine 일감**이다.

## 협업 구조 (워크스페이스 · 채널 · Redmine)

- **프로젝트 = Slack 워크스페이스** — 프로젝트마다 별도 Slack 워크스페이스를 만든다. 사람 담당자와 ai-pm 이 그 워크스페이스에 모여 채팅 기반으로 작업을 공유·진행한다.
- **큰 작업 단위 = 채널** — 워크스페이스 안에서 큰 작업 단위를 채널로 나눈다. ai-pm 은 워크스페이스의 **전체 채널 대화를 확인**하고, 각 채널에서 필요한 작업을 서브에이전트로 실행한다.
- **Redmine = 작업 수행 기록 정본** — 서브에이전트가 수행하는 실제 작업은 종류·범위에 따라 **여러 Redmine 일감으로 등록될 수 있다**. 명세·진행 노트·상태·결과가 일감에 쌓인다. 작업 티켓 정책·상태 어휘·생애주기는 [`work-tracking.md`](work-tracking.md)가 정본이다.
- **ai-pm = 오케스트레이터** — 채널 대화에서 필요한 작업을 식별해 서브에이전트로 실행하고, 서브에이전트의 완료 보고를 받아 결과를 요청 담당자·채널에 알린다. 작업 수행과 Redmine 일감 반영은 서브에이전트가 한다.

## 트리거 (작업 인지)

ai-pm 이 처리할 일을 인지하는 경로는 둘이다.

1. **Slack (주 통로, 실시간)** — 담당자가 워크스페이스의 채널에 작업을 공유·요청하거나 ai-pm 을 호출(멘션·DM)한다. ai-pm 은 **전체 채널의 대화**를 Socket Mode 로 확인한다.
2. **Redmine (보완 통로, 폴링)** — ai-pm 이 [`CLAUDE.env.md`](../../CLAUDE.env.md) `<REDMINE_PROJECT>` 의 이슈를 저빈도로 폴링(catch-up)해, Slack 을 거치지 않은 이슈 변경(서브에이전트의 진행 갱신·담당자의 직접 편집 등)을 흡수한다.

## 운영 모델 (워크스페이스 단위 단일 세션)

- ai-pm 은 한 프로젝트(= 한 Slack 워크스페이스)에 대해 **단일 세션**으로 동작하며, 그 워크스페이스의 **전체 채널과 DM 을 감시**한다.
- **프로젝트 구분은 워크스페이스 단위**, 그 안의 작업 단위 구분은 **채널 단위**다.
- 동일 워크스페이스에 ai-pm 세션을 둘 이상 띄우지 않는다 — Slack 런타임 중복 수신과 중복 디스패치를 막는다.
- **실행 장비 지정** — ai-pm 세션·Slack 런타임은 **지정 실행 장비에서만** 기동한다. 지정 값의 단일 출처는 `ai/bots/ai-pm/_slack/config.json` 의 `exec_machine`(장비 MachineName, Windows `$env:COMPUTERNAME`)이며, 봇 정의 frontmatter `exec machine` 에도 반영한다. 세션 래퍼는 기동 시 현재 장비의 MachineName 을 `exec_machine` 과 대조해 불일치하면 기동을 중단한다 — 복제 워크스페이스를 가진 다른 PC 에서의 중복 기동(단일 세션 위반)을 장비 수준에서 막는다. 실행 장비를 옮길 때는 `exec_machine` 을 새 장비 값으로 갱신·커밋한다([`project-bootstrap.md`](project-bootstrap.md) §다른 PC 재구성).
- **운영 강도** — ai-pm 세션은 최상위 추론 강도 `max` 로 기동한다(봇 정의 frontmatter `effort: max` → 세션 래퍼가 `--effort max` 로 띄움, 모델은 `model:`·`model fallback:` 규칙). 복잡·비가역 작업은 다중 에이전트 오케스트레이션·교차검증을 우선한다. 서브에이전트에는 각 정의의 `effort`(opus=max·sonnet=xhigh)를 그대로 적용한다([`agents.md`](agents.md) §모델·추론 강도(effort) 정책).

## 처리 대상 식별

채널 대화 또는 Redmine 폴링에서 다음을 모두 만족하는 것을 처리 대상으로 본다.

1. **필요 작업** — 채널에서 담당자가 요청했거나 진행이 필요한 작업이다.
2. **미종결·갱신** — 관련 Redmine 일감이 종결(완료·거절) 전이며, 미착수이거나 마지막 처리 이후 갱신됐다.
3. **미처리 중** — 현재 세션에서 이미 디스패치되어 처리 중(in-flight)이 아니다.

무슨 작업을 어떻게 할지, 어떤 일감으로 나눌지는 서브에이전트가 판단한다. ai-pm 은 "어느 작업을 처리할지"만 정한다.

## 디스패치 계약

처리 대상마다 다음을 수행한다.

1. **작업 맥락 전달** — 분석·실행 서브에이전트를 하위 세션으로 백그라운드 기동하고, 처리할 작업의 맥락(요청 채널·요청 내용·관련 Redmine 일감)을 넘긴다. 기동 시 그 에이전트 정의(frontmatter)의 `model`·`effort` 를 **지정값 그대로 적용**한다([`agents.md`](agents.md) §모델·추론 강도(effort) 정책) — 임의로 낮추거나 세션 구성으로 대체하지 않는다.
2. **Redmine 일감 반영(서브에이전트)** — 서브에이전트가 작업을 수행하며 해당하는 Redmine 일감을 등록·갱신한다(한 작업이 여러 일감일 수 있다). 진행 노트·상태·결과를 일감에 반영하고 마무리한다.
3. **in-flight·논블로킹** — 디스패치한 작업을 처리 중 집합에 넣어 중복을 막고, 결과를 기다리지 않고 다음 대상으로 넘어간다(여러 서브에이전트 동시 진행).
4. **완료 보고·전달** — 서브에이전트가 완료(또는 §질의·승인 릴레이의 중간 보고)를 보고하면 ai-pm 은 그 결과를 요청 담당자·채널에 Slack 으로 알리고, 처리 중 집합에서 제거한다.

ai-pm 의 경계는 "어느 작업을 누구에게 맡길지"와 "결과를 Slack 으로 전달"까지다. 작업 수행과 Redmine 일감 반영은 서브에이전트가 한다. 접수부터 종료까지의 상태 노출은 §진행 피드백의 단일 메시지 갱신으로 처리한다.

## 진행 피드백 (단일 메시지 갱신)

담당자가 요청의 인식·처리 여부를 즉시 알 수 있도록, 처리 대상 1건마다 **피드백 메시지 1건**을 유지한다. 상태가 바뀔 때 새 메시지를 덧붙이지 않고 **같은 메시지를 `chat.update` 로 갱신**한다 — 스레드가 길어지는 것을 막는다. 본문에는 항상 **작업을 유발한 담당자 멘션**(`<@user id>`)을 포함한다.

1. **메시지 인식 시 (게시 = 인식 확인)** — 처리 대상으로 인식한 **즉시** 요청 메시지의 스레드에 피드백 메시지를 게시해(`chat.postMessage`) **입력자에게 인식을 확인**해 준다 — 담당자가 자기 요청이 접수됐음을 곧바로 알 수 있게 한다. 상태는 `접수` 로 표기하고 입력자 멘션(`<@user id>`)을 포함하며, 응답의 `channel`·`ts` 를 해당 작업의 in-flight 기록에 보관한다(이후 갱신 좌표). 이 접수 메시지가 이어지는 `작업 시작`→`진행 중`→`종료` 갱신의 대상이다.
2. **대응 세션 시작 시 (갱신)** — 서브에이전트 디스패치(directing 은 직접 처리 시작) 시점에 같은 메시지를 `작업 시작` 상태로 갱신하고, 관련 Redmine 일감 번호·담당 에이전트를 병기한다.
3. **실행 중 (5분 주기 갱신)** — in-flight 작업마다 5분 간격으로 같은 메시지를 `진행 중` 상태로 갱신한다 — 경과 시간과 파악되는 진행 내용을 담는다.
4. **종료 시 (최종 갱신)** — 완료·중간 보고(질의/승인 대기)·실패 시점에 같은 메시지를 해당 최종 상태로 갱신한다. 결과 상세는 §디스패치 계약 4 의 완료 보고가 별도 메시지로 전달한다 — `chat.update` 는 재알림을 만들지 않으므로, 담당자 알림은 완료 보고 게시가 보장한다.

피드백 메시지에도 §글로벌 운영 원칙을 그대로 적용한다(`[AI-ai-pm]` 프리픽스, 발신은 `post.js` 로 송신).

## 질의·승인 릴레이

서브에이전트는 백그라운드 하위 세션이라 Slack 에 직접 닿지 않는다. 작업 중 담당자 질의(검토 협의)나 승인 게이트(목업 승인·단계 이행 등)가 필요하면 다음으로 처리한다.

1. **중간 보고 종료(서브에이전트)** — 질의·승인 필요 지점에 도달하면 관련 일감에 질의(또는 승인 요청) 내용을 노트로 남기고 상태를 바꾼 뒤(질의 = `보류`, 승인 대기 = `해결`), 그 시점까지의 결과를 정리해 중간 보고로 종료한다.
2. **릴레이(ai-pm)** — 중간 보고에서 질의·승인 요청을 식별해 요청 채널·담당자에게 Slack 으로 전달한다.
3. **응답 기록(ai-pm)** — 담당자 응답(답변·승인·반려)을 해당 일감 노트로 기록하고 상태를 복귀시킨다. Slack 대화는 통로이고 **증적 정본은 일감 노트**다. 단계 완료 승인이면 해당 단계 브랜치를 main 에 병합한다(no-ff, 병합 후 브랜치 삭제 — [`git-flow.md`](git-flow.md) §병합).
4. **재디스패치(ai-pm)** — 응답을 작업 맥락에 더해 같은 작업을 재디스패치한다. 서브에이전트는 일감의 진행 노트·상태로 완료분을 식별해 이어간다(재개).

## 단계 연결 (base-workflow 4단계)

ai-pm 은 [`base-workflow.md`](base-workflow.md) 4단계 작업의 시작·종료를 Slack 으로 매개한다.

- **directing** — ai-pm 이 채널에서 담당자와 **직접 대화로 진행**한다(별도 서브에이전트 없이 ai-pm 이 퍼실리테이션). 방향 정의서를 도출하고 그 작업을 Redmine 일감으로 추적한다. 상세는 [`stages/directing.md`](stages/directing.md).
- **spec · build · qa** — 작업 맥락을 전달해 단계 오케스트레이터 서브에이전트를 디스패치한다. 서브에이전트가 작업을 수행하며 관련 Redmine 일감(들)을 등록·반영한다. 단계 흐름은 `stages/` 각 문서, 에이전트 바인딩은 [`agents.md`](agents.md) 가 정본이다.

각 단계 작업의 브랜치·커밋·병합은 [`git-flow.md`](git-flow.md) 를 따른다 — directing 의 브랜치(`directing/<이슈번호>-<주제>`) 생성과 단계 승인 후 main 병합은 ai-pm 이 수행한다.

## 런타임 구성 요소

ai-pm 의 Slack 런타임은 자기완결 폴더 `ai/bots/ai-pm/` 에 둔다.

- `ai/bots/ai-pm/ai-pm.md` — 봇 정의. 페르소나·명령/응답 규칙·채널 운용. 본 전략 문서가 운영 절차 정본이고, 봇 정의는 그 절차를 수행하는 페르소나를 담는다.
- `ai/bots/ai-pm/_slack/app.js` — Slack Bolt Socket Mode 엔트리포인트(봇 1 프로세스 = Socket Mode 연결 1개). 수신 이벤트를 `runtime.log` 에 남기고, ai-pm 세션이 그 로그를 읽어 처리한다. **수신 전용**이다.
- `ai/bots/ai-pm/_slack/post.js` — **발신 헬퍼**(`chat.postMessage`·`chat.update`). 본문을 UTF-8 파일에서 `[string]` 그대로 읽어 `@slack/web-api` 로 보낸다. 세션이 JSON 을 셸에서 손수 조립하지 않게 해 한글 인코딩 손상과 PowerShell ETS 누출(§글로벌 운영 원칙 — 발신은 반드시 `post.js` 로)을 원천 차단한다.
- `ai/bots/ai-pm/_slack/config.json` — 봇 이름·표시이름·앱 식별자·워크스페이스·감시 채널·실행 장비(`exec_machine`) (선언적 설정).
- `ai/bots/ai-pm/_slack/.env` — Slack 토큰(`SLACK_BOT_TOKEN`·`SLACK_APP_TOKEN`). git 비관리. 양식은 같은 폴더 `.env.example`.
- `ai/bots/ai-pm/_slack/runtime.log` — 수신 이벤트 로그(세션이 tail). git 비관리. 래퍼 기동 시 10MB 초과면 `runtime.log.1` 로 회전.
- `ai/bots/ai-pm/_slack/last-event` — app.js 가 기록하는 최신 수신 이벤트 ts(처리 정체 감지용, §운영 연속성 ③). git 비관리.
- `ai/bots/ai-pm/_session/` — 세션 재기동·정지 플래그(`.restart`·`.stop`)와 처리 마커 `last-processed`(세션이 트리아지한 최신 이벤트 ts, §운영 연속성). git 비관리.
- `ai/scripts/ai-pm-session.ps1` — 세션 래퍼(자동 재기동 지원). Slack 런타임 워치독(`$watchdogSource`)을 생성·감시하며, 워치독은 app.js 생존과 함께 처리 정체(§운영 연속성 ③)를 감지해 세션을 강제 웨이크한다.

## 기동 절차

1. **(최초 1회) Slack 앱 구성** — 전용 Slack 앱을 생성해 Bot Token(`xoxb-...`)·App Token(`xapp-...`)을 발급하고, 토큰을 `ai/bots/ai-pm/_slack/.env` 에 기입한다. 발급된 앱 id·봇 user id·워크스페이스 id 를 `config.json` 과 봇 정의 frontmatter 에 반영한다. 필요한 OAuth scope 와 Event Subscriptions(`message.channels`·`message.im`·`app_mention`)를 활성화한다. 상세 절차는 [`project-bootstrap.md`](project-bootstrap.md) §4.
2. **세션 기동** — **지정 실행 장비(§운영 모델)에서** 세션 래퍼 `ai/scripts/ai-pm-session.ps1` 를 실행한다. 래퍼는 현재 장비가 `exec_machine` 과 다르면 기동을 중단하고, 일치하면 Slack 런타임(`app.js`)이 떠 있지 않을 때 백그라운드로 1개 띄운 뒤 ai-pm 세션(`claude`)을 기동한다. 래퍼는 Slack 런타임을 워치독으로 감시해 죽으면 자동 재기동한다.
3. **운영 루프** — ai-pm 세션은 `runtime.log` 를 tail 하며 워크스페이스 전체 채널·DM 의 이벤트를 §처리 대상 식별·§디스패치 계약대로 처리한다. 세션은 유휴로 끝내지 않고 §운영 연속성 ①의 자가 웨이크 루프로 상시 구동하며, 깨어날 때마다 §운영 연속성 ②로 `last-processed` 마커 이후 백로그를 전수 처리한다.
4. **catch-up 폴링** — Redmine 이슈 폴링을 저빈도(예: 5분)로 병행해 Slack 을 거치지 않은 변경을 흡수한다.

세션이 `ai-pm 세션 시작` 류 프롬프트로 시작되면 본 문서를 읽고 그대로 동작한다([`CLAUDE.md`](../../CLAUDE.md) §ai-pm 세션 실행 모드).

## 운영 연속성 (처리 루프 지속)

ai-pm 세션은 대화형 에이전트라 한 턴을 마치면 다음 입력을 기다리며 유휴로 들어간다. app.js 가 `runtime.log` 에 남기는 신규 이벤트는 그 자체로 세션을 깨우지 않으므로(하네스가 로그 갱신을 세션 입력으로 전달하지 않는다), 유휴 진입 후에는 새 메시지가 로그에만 쌓이고 처리되지 않는다 — **tail→처리 공백**. 이를 3층으로 막는다.

### 처리 마커 (좌표)

- `_slack/last-event` — app.js 가 세션에 보일 이벤트(mention·DM·channel)를 로깅할 때마다 그 `ts` 를 기록(최신값 덮어쓰기). "app.js 가 받은 최신 이벤트".
- `_session/last-processed` — 세션이 **트리아지한**(처리했든 무응답으로 판정했든) 최신 이벤트 `ts`. 세션은 매 사이클에서 로그의 최신 이벤트까지 이 값을 전진시킨다(무응답 잡담도 포함 — "봤고 판정함"의 좌표이지 "작업 완료"가 아니다).
- `last-event > last-processed` = 아직 트리아지되지 않은 이벤트 존재 = tail→처리 공백. 두 값 모두 Slack ts(초.마이크로초), git 비관리(런타임 상태).

### ① 세션 자가 웨이크 루프 (상시)

세션은 "처리 후 유휴 대기"로 턴을 끝내지 않는다. 운영 루프를 `/loop` 자가 페이싱으로 상시 구동한다 — 부팅 즉시 `/loop`(간격 생략 = 모델 자가 페이싱)로 운영 틱을 걸고, 매 틱에서 로그 드레인·마커 갱신을 한 뒤 다음 틱을 예약(ScheduleWakeup)한다. 유휴처럼 보여도 실제로는 주기적으로 `runtime.log`·Redmine 을 재점검하는 상태를 유지한다.

- 간격: 활동 중 약 60초(저지연·프롬프트 캐시 유지), 장기 유휴 시 완화, 00–04 저활동은 §세션 리셋 압축 주기와 조화.
- 세션이 결코 "유휴 + 다음 틱 미예약"으로 끝나지 않게 한다.

### ② 웨이크 시 백로그 전수 드레인

세션이 깨어날 때(자가 틱·터미널 입력·재기동) `last-processed` 마커부터 `runtime.log` 를 전수 스캔해 누적된 모든 미트리아지 이벤트를 처리한다(최신 1건만 처리 금지). 처리 대상마다 §진행 피드백 '접수'를 즉시 게시해 담당자 무응답 체감을 없앤다. 트리아지가 끝나면 `last-processed` 를 본 최신 이벤트 `ts` 로 갱신한다.

### ③ 처리 정체 감지·자가치유 백스톱 (하드)

워치독(§런타임 구성 요소)이 app.js 생존 감시에 더해 '처리 생존'을 감시한다 — `last-event` 가 `last-processed` 보다 앞선 상태(미트리아지 적체)가 임계(기본 5분) 넘게 지속되면 `.restart` 를 설정하고 ai-pm 세션 프로세스를 강제 종료해 래퍼가 새 세션을 기동하게 한다(재기동 → ①·②로 자동 복구). ①(소프트·응답성)이 멈춰도 미처리 이벤트가 방치되지 않게 하는 결정적 백스톱이다.

- 오탐·루프 방지: app.js 생존 상태에서만 판정, 재기동 후 쿨다운(기본 10분), 1시간 내 3회 초과 재기동 시 자동 재기동을 멈추고 `watchdog.log` 에 경고만 남긴다(마커 미갱신 등 비정상 신호).
- 임계·쿨다운 기본값은 `ai-pm-session.ps1` 워치독 파라미터(`StallThresholdSec`·`StallCooldownSec`)로 조정한다.

①은 응답성이 좋지만 소프트(모델이 예약을 빠뜨릴 수 있음), ③은 느리지만 하드(외부 프로세스가 결정적으로 복구). 두 층을 겹쳐 단일 실패점을 없앤다.

## 세션 리셋

장시간 운영으로 컨텍스트가 누적되면 셋으로 대응한다 — 임계 초과 시 **과부하 경고**, 초기화 명령에 따른 **재기동**, 자동 주기 **컨텍스트 압축**.

### 컨텍스트 과부하 경고 (300k)

세션 누적 컨텍스트가 **300k 토큰을 넘어선 시점부터**, ai-pm 은 리셋 전까지 **모든 Slack 응답 말미**에 컨텍스트 과부하 경고를 덧붙인다. 사용량은 운영 루프에서 자체 점검한다(하네스의 컨텍스트 경고·`/context`).

- 경고에는 **전체 멘션(`<!channel>`)**을 포함해 채널의 모든 사용자에게 알리고, 초기화 명령으로 세션을 리셋할 수 있음을 함께 안내한다. 스레드 응답이면 `reply_broadcast` 로 채널에도 노출한다.
- 경고가 붙기 시작하면 신규 대형 작업의 디스패치 전에 리셋을 우선 권고한다(진행 중 작업 완료 보고는 정상 수행).

### 초기화 명령 재기동

담당자가 채널/DM 에서 초기화를 명령("ai-pm 초기화" 류)하면 즉시 리셋한다.

1. ai-pm 이 요청 채널/DM 에 리셋 진행을 게시한다 — in-flight 디스패치가 있으면 그 목록을 포함한다(중단된 작업의 재기동 후 처리는 §재기동 후 중단 작업 처리를 따른다).
2. `.restart` 플래그를 설정하고 자기 세션을 종료한다.
3. 래퍼가 플래그를 감지해 같은 인자로 새 세션을 자동 기동한다(`.stop` 플래그면 루프를 종료한다).

### 재기동 후 중단 작업 처리

세션이 (재)기동되면 처리 대상을 Redmine 일감·Slack 이력에서 재도출한다(§처리 대상 식별). 이때 종결(완료·거절) 전이며 이전 세션에서 처리 중이던(중단된) 작업이 인식되면, **자동으로 재개하지 않고** 아래 기준으로 처리한다.

- **기본 (자동재개 요청 없음)** — 대기 중인 중단 작업 목록만 채널에 **전체 멘션(`<!channel>`)으로 1회 공지**한다(각 작업의 요청자 멘션·관련 일감 병기). 실제 재개는 하지 않고, 담당자의 **개별 재개 요청이 올 때** 그 작업만 재디스패치한다.
- **자동재개 요청이 있었던 경우** — 작업 시작(디스패치) 시점에 요청자가 자동재개를 요청해 둔 작업에 한해, 위 공지 후 **즉시 재디스패치**한다. 이 경우에도 재개 즉시 §진행 피드백으로 진행 상황을 해당 작업의 **요청자에게 알린다**(요청자 멘션 포함).

**자동재개 요청 문구** — 요청자가 작업 지시에 **"완료까지 멈추지 말고 진행해"** 또는 이와 **동일한 의미의 문구**(예: "끝까지 진행해", "중단 없이 완료까지 해줘", "멈추지 말고 끝내")를 덧붙인 경우를 자동재개 요청으로 본다. 표현이 달라도 "중단·재기동과 무관하게 완료까지 이어가라"는 의미면 동일하게 해석한다. 질의·승인 게이트(§질의·승인 릴레이)처럼 담당자 응답이 반드시 필요한 지점은 이 요청과 무관하게 중간 보고로 멈춘다 — 자동재개는 재기동으로 끊긴 작업을 이어가는 것이지 승인 게이트를 건너뛰는 것이 아니다.

자동재개 요청 여부는 세션 메모리가 아니라 **관련 Redmine 일감에 기록**해 둔다(정본 = 일감) — ai-pm 이 디스패치 시 위 문구를 인식하면 자동재개 지시를 일감 노트에 남기고, 재기동 후 그 기록으로 판별한다. 기록이 없으면 기본(공지 후 대기)으로 처리한다.

### 자동 리셋 주기 (`/compact`)

ai-pm 은 운영 루프에서 시각·유휴 상태를 점검해, 다음 중 하나에 해당하면 스스로 `/compact` 를 실행해 누적 컨텍스트를 압축한다(전체 재기동이 아닌 세션 내 압축).

- **저활동 시간대** — 매일 00:00~04:00 (Asia/Seoul) 사이.
- **유휴** — 2시간 이상 진행 중인 작업 세션(in-flight 디스패치)이 없을 때.

진행 중 작업을 끊지 않도록, in-flight 디스패치가 남아 있으면 그 완료 보고를 마친 뒤 압축한다. 같은 저활동 시간대에는 중복 실행하지 않는다(하루 1회로 충분).
처리 대상은 Redmine 일감과 Slack 채널 이력에서 다시 도출되므로 압축해도 안전하다 — 정본은 Redmine·Slack 이며 세션 메모리가 아니다. 세션 메모리가 사라지는 **재기동**의 경우, 중단 작업의 재개는 자동이 아니라 §재기동 후 중단 작업 처리를 따른다(압축은 세션 내 처리라 in-flight 가 유지된다).

## 글로벌 운영 원칙

모든 Slack 응답·디스패치에 공통 적용한다.

- **응답 형식** — 담당자 요청 응답·진행 피드백은 **요청 스레드 안에만** 남긴다(`reply_broadcast` 미사용 — 스레드 답장을 메인 채널로 브로드캐스트하지 않는다). 메인 채널 노출은 **채널 전체 공지 성격**일 때(리셋 공지·컨텍스트 과부하 경고 등 `@channel` 대상)로 한정한다 — 일상 응답까지 메인에 노출하면 대화가 뒤섞여 오히려 혼동을 키운다. 봇 발신은 `[AI-ai-pm]` 프리픽스로 표시한다.
- **응답 위치 안내(퍼머링크)** — 담당자가 특정 답변·처리 결과가 어디에 달렸는지 찾기 어렵다고 하면, 메인 채널에 다시 게시하지 말고 찾는 메시지의 **퍼머링크(`chat.getPermalink`)를 스레드로 전달**해 바로 이동하게 한다. 스레드 답장 원칙을 유지하면서 탐색 문제만 링크로 해소한다.
- **무응답 대상** — 시스템 메시지(`subtype` 존재)와 봇 발신(자기·타 봇, `bot_id` 존재) 메시지는 처리하지 않는다(피드백 루프 방지).
- **금지 동작** (담당자 명시 승인 없이 불가) — 외부 워크스페이스·이메일·웹훅으로 데이터 전송, 결제·자금 이체, 사용자 사칭 DM·계정 변경, 채널 삭제·아카이브·멤버 강제 퇴장.
- **민감 정보** — 토큰·시크릿을 채널 메시지·외부 API·로그에 절대 포함하지 않는다.
- **에러 보고** — API 실패를 조용히 넘기지 않고 담당자에게 알린다.
- **언어·타임존** — 한국어, Asia/Seoul. 대화 어투·응답 구성은 [`prompt-conversation.md`](prompt-conversation.md).
- **발신은 반드시 `post.js` 로** — 모든 Slack 발신(`chat.postMessage`·`chat.update`)은 본문을 UTF-8 임시 파일에 기록한 뒤 발신 헬퍼 `ai/bots/ai-pm/_slack/post.js` 로 보낸다. `.env` 를 주입해 호출한다: `node --env-file=ai/bots/ai-pm/_slack/.env ai/bots/ai-pm/_slack/post.js --channel <C..> --thread-ts <ts> --text-file <path>` (갱신은 `--thread-ts` 대신 `--update-ts <ts>`). 성공 시 stdout 의 `{"ok":true,"channel":..,"ts":..}` 에서 좌표를 얻어 진행 피드백 기록에 보관한다. `--env-file` 은 app.js 를 띄우는 것과 동일한 실행 방식이라(런타임 Node 가 이미 지원) 추가 요건이 없다.
	- **셸에서 JSON 을 손수 조립하지 않는다.** 한글 본문을 인라인 JSON 으로 curl 에 넣으면 인코딩이 손상되고, PowerShell 에서 `Get-Content <파일> | ConvertTo-Json` 으로 감싸면 문자열에 붙는 ETS 노트 속성(`PSPath`·`PSChildName`·`ReadCount` 등)까지 직렬화되어 `{"value":"...본문...","PSPath":...}` 덩어리가 `text` 자리에 통째로 들어간다(원데이터 노출 결함). `post.js` 는 JSON 조립·인코딩을 node 가 처리하므로 두 함정이 모두 사라진다 — 본문 파일에는 **완성된 메시지 텍스트만** 쓴다.
	- **본문 파일 기록(윈도우)** — 임시 파일은 **UTF-8(BOM 없음)**로 쓴다. 세션 도구로 파일을 쓰면 자동으로 이 형식이다. PowerShell 로 쓸 때는 `Out-File/Set-Content -Encoding utf8`(5.1 은 BOM 첨부)·`>`(5.1 은 UTF-16) 를 피하고 `[IO.File]::WriteAllText($path,$text,[Text.UTF8Encoding]::new($false))` 를 쓴다. `post.js` 는 방어적으로 선두 BOM 제거·CRLF→LF·말미 개행 정리를 수행하므로 BOM·CRLF 가 섞여도 본문은 깨끗하게 전송된다(UTF-16 로 쓰면 복구 불가 — UTF-8 로 써야 한다).
