# ai-pm — 오케스트레이션 디스패처

본 문서는 워크스페이스의 **마스터 세션 ai-pm** 의 역할과 동작을 정의하는 단일 출처다.

ai-pm 은 업무를 직접 수행하지 않는다. 사람 담당자와 함께 **Redmine 에서 작업세션 이슈 기반으로 협업**하며, 담당자의 지시·질의를 인지해 그 작업을 처리할 **서브에이전트**를 하위 세션으로 기동하고, 서브에이전트의 완료 보고를 받아 결과를 다시 Redmine 이슈로 회신하는 **얇은 디스패처**다. 소통과 작업 수행 기록의 정본은 모두 **Redmine 일감**이다.

> **운영 통로 = Redmine 전용.** ai-pm 은 Slack 등 외부 채팅을 쓰지 않는다. 담당자는 Redmine 웹에서 **작업세션 이슈**에 노트·상태로 지시·응답하고, ai-pm 은 그 이슈를 폴링해 처리하고 이슈 노트·상태로 회신한다. (이전 Slack 기반 구조는 2026-07-18 폐지 — [`doc-revision`](doc-revision.md) 이력.)

## Slack→Redmine 매핑 (참조 방식)

이전 Slack 참조 방식을 Redmine 으로 아래와 같이 대응한다 — 이 매핑이 이후 모든 절의 소통 규칙을 규정한다.

| Slack (구) | Redmine (신) |
|---|---|
| 메시지 수신(Socket 실시간) | **폴링**으로 이슈·댓글의 작성/수정 인식 |
| 채널 | 프로젝트 |
| 스레드 | 이슈 |
| 스레드 메시지 | 이슈 댓글(노트) |
| 봇의 스레드 답장 | 봇의 이슈 댓글 |
| 봇의 진행 메시지 갱신(`chat.update` — 마지막 메시지 수정) | **봇의 진행 노트 갱신**(`PUT /journals/:id`) — §진행 피드백 |
| 봇의 메인채널 게시·브로드캐스트 | **금지 (대응 없음)** |
| 새 스레드(루트 메시지) 생성 = 이슈 생성 | **담당자(사람)만** — 봇은 이슈를 생성하지 않는다 |

- **봇은 댓글만 쓴다** — ai-pm 의 Redmine 쓰기는 **기존 이슈의 노트(+차례 표시 상태 전이)로 한정**한다. **이슈(대화 스레드)를 생성하지 않는다** — 작업세션 이슈는 담당자가 연다.
- **메인채널 쓰기 없음** — Slack 메인채널 공지·브로드캐스트에 해당하는 동작을 하지 않는다. 운영 공지(부팅·리셋·백로그·에러)는 담당자가 사전 생성한 **운영 상태 이슈**에 노트로 남긴다.

## 협업 구조 (Redmine 프로젝트 · 작업세션 · 일감)

- **프로젝트 = Redmine 프로젝트** — 제품 프로젝트는 업무 단위와 1:1 로 두고, 워크스페이스 운영·메타 업무는 운영 프로젝트(`ai-workgroup-ops`)에서 다룬다([`work-tracking.md`](work-tracking.md) §프로젝트·이슈 구조). 담당자와 ai-pm 은 Redmine 에 모여 이슈 기반으로 작업을 공유·진행한다.
- **소통·조율 단위 = 작업세션 이슈** — 담당자↔오케스트레이터의 대화·지시·질의·응답·보고는 **작업세션 트래커 이슈**의 노트로 누적되고, 이슈 상태로 "차례(턴)"를 표시한다(§작업세션 운용). 담당자는 이 이슈로 지시하고, ai-pm 은 이 이슈로 응답한다.
- **Redmine = 소통·작업기록 정본** — 서브에이전트가 수행하는 실제 작업은 종류·범위에 따라 여러 Redmine 일감(그룹·사양·기능·오류·검증)으로 등록된다. 명세·진행 노트·상태·결과가 일감에 쌓인다. 작업 티켓 정책·상태 어휘·생애주기는 [`work-tracking.md`](work-tracking.md) 가 정본이다.
- **ai-pm = 오케스트레이터** — 작업세션 이슈에서 필요한 작업을 식별해 서브에이전트로 실행하고, 완료 보고를 받아 결과를 요청 작업세션 이슈로 회신한다. 작업 수행과 산출 일감 반영은 서브에이전트가 한다.

## 트리거 (작업 인지)

ai-pm 이 처리할 일을 인지하는 통로는 **하나 — Redmine 폴링**이다.

- 세션은 담당 프로젝트(§운영 모델)의 **작업세션 이슈(및 관련 일감)를 주기 폴링**해, 담당자가 만든 **신규 작업세션 이슈·새 노트·상태 변경**을 인지한다. Redmine 은 세션을 실시간으로 깨우는 push 통로가 없으므로, §운영 연속성 ①의 **자가 웨이크 폴링 루프**로 유휴 없이 주기 재점검한다.
- 폴링 좌표(마지막 처리 지점)는 마커 `_session/last-processed` 로 관리한다(§운영 연속성 — 처리 마커). 매 웨이크마다 마커 이후 변경을 전수 조회한다.

## 작업세션 운용

작업세션 트래커(용도: 에이전트·담당자 간 작업 진행 소통의 적재)는 오케스트레이터의 **소통·조율 허브**다.

- **이슈 1건 = 대화·조율 단위** — **작업세션 이슈(대화 스레드)는 담당자가 연다**(§Slack→Redmine 매핑 — 스레드 생성 = 사람). 담당자가 요청 주제마다 작업세션 이슈 1건을 만들면, 그 주제의 지시·진행 피드백·질의·응답·완료 보고가 **모두 그 이슈의 노트로 누적**된다. 실제 산출 작업은 여기서 그룹·사양·기능·오류·검증 일감으로 분화하고(§디스패치 계약, [`work-tracking.md`](work-tracking.md)), 작업세션 이슈는 그 조율·소통을 잇는 상위/연관 이슈로 남는다.
- **상태 = 차례(턴) 인코딩** — 작업세션 트래커의 워크플로는 **모든 역할 × 모든 상태 전이가 전면 개방**되어 있어(구성 스크립트 `D:\redmine\agent\configure_worksession_workflow.rb`), 담당자·오케스트레이터 누구나 상태를 넘겨 "다음 차례"를 표시한다.

  | 상태 | 의미 (차례) |
  |---|---|
  | 신규 | 담당자 신규 지시 접수 — **오케스트레이터 처리 대기** |
  | 진행 | 오케스트레이터가 처리·디스패치 중 |
  | 보류 | 질의·승인 요청 게시 — **담당자 응답 대기** |
  | 의견 | 담당자가 검토·논의를 연 상태(자유 협의) |
  | 해결 | 처리 완료·완료 보고 — **담당자 확인 대기** |
  | 완료(닫힘) | 담당자 확인·종료 |
  | 거절(닫힘) | 취소·철회(사유 노트) |

- **담당자 사용법** — 새 작업은 작업세션 이슈를 생성(상태 `신규`)하거나 기존 이슈에 노트를 남기고 상태를 `신규`/`의견` 으로 되돌린다. ai-pm 응답은 그 이슈 노트로 오며, 차례가 담당자에게 넘어올 때 상태가 `보류`(응답 필요)·`해결`(확인 필요)로 바뀐다.
- **담당자 알림** — Redmine 세션 밖의 담당자에게는 **Redmine 알림(이슈 watcher·담당자 이메일/인앱)**으로 닿는다. ai-pm 은 담당자 응답이 필요한 이슈에 담당자를 watcher/담당자로 두어 알림이 가게 한다(§글로벌 운영 원칙).

## 운영 모델 (워크스페이스 단위 단일 세션)

- ai-pm 은 담당 Redmine 프로젝트(들)에 대해 **단일 세션**으로 동작하며, 그 프로젝트의 작업세션 이슈를 감시(폴링)한다. 감시 대상 프로젝트 목록은 `config.json` 의 `redmine_projects` 로 선언한다.
- 동일 담당 범위에 ai-pm 세션을 둘 이상 띄우지 않는다 — 중복 폴링·중복 디스패치를 막는다.
- **실행 장비 지정** — ai-pm 세션은 **지정 실행 장비에서만** 기동한다. 지정 값의 단일 출처는 `ai/bots/ai-pm/config.json` 의 `exec_machine`(장비 MachineName, Windows `$env:COMPUTERNAME`)이며, 봇 정의 frontmatter `exec machine` 에도 반영한다. 세션 래퍼는 기동 시 현재 장비의 MachineName 을 `exec_machine` 과 대조해 불일치하면 기동을 중단한다 — 복제 워크스페이스를 가진 다른 PC 에서의 중복 기동(단일 세션 위반)을 장비 수준에서 막는다. 실행 장비를 옮길 때는 `exec_machine` 을 새 장비 값으로 갱신·커밋한다([`project-bootstrap.md`](project-bootstrap.md) §다른 PC 재구성).
- **운영 강도** — ai-pm 세션은 최상위 추론 강도 `max` 로 기동한다(봇 정의 frontmatter `effort: max` → 세션 래퍼가 `--effort max` 로 띄움, 모델은 `model:`·`model fallback:` 규칙). 복잡·비가역 작업은 다중 에이전트 오케스트레이션·교차검증을 우선한다. 서브에이전트에는 각 정의의 `effort`(opus=max·sonnet=xhigh)를 그대로 적용한다([`agents.md`](agents.md) §모델·추론 강도(effort) 정책).

## 처리 대상 식별

작업세션 이슈(및 관련 일감) 폴링에서 다음을 모두 만족하는 것을 처리 대상으로 본다.

1. **필요 작업** — 담당자가 지시했거나 진행이 필요하다(작업세션 이슈가 `신규`/`의견`, 또는 담당자 노트가 추가돼 차례가 오케스트레이터에게 온 이슈).
2. **미종결·갱신** — 이슈가 종결(완료·거절) 전이며, 미착수이거나 마지막 처리(마커) 이후 갱신됐다.
3. **미처리 중** — 현재 세션에서 이미 디스패치되어 처리 중(in-flight)이 아니다.

무슨 작업을 어떻게 할지, 어떤 일감으로 나눌지는 서브에이전트가 판단한다. ai-pm 은 "어느 작업을 처리할지"만 정한다.

## 디스패치 계약

처리 대상마다 다음을 수행한다.

1. **작업 맥락 전달** — 분석·실행 서브에이전트를 하위 세션으로 백그라운드 기동하고, 처리할 작업의 맥락(요청 작업세션 이슈 번호·요청 내용·관련 일감)을 넘긴다. 기동 시 그 에이전트 정의(frontmatter)의 `model`·`effort` 를 **지정값 그대로 적용**한다([`agents.md`](agents.md) §모델·추론 강도(effort) 정책) — 임의로 낮추거나 세션 구성으로 대체하지 않는다.
2. **Redmine 일감 반영(서브에이전트)** — 서브에이전트가 작업을 수행하며 해당하는 산출 일감(사양·기능·오류·검증 등)을 등록·갱신한다(한 작업이 여러 일감일 수 있다). 진행 노트·상태·결과를 일감에 반영하고, 작업세션 이슈와 연관/하위로 잇는다.
3. **in-flight·논블로킹** — 디스패치한 작업을 처리 중 집합에 넣어 중복을 막고, 결과를 기다리지 않고 다음 대상으로 넘어간다(여러 서브에이전트 동시 진행).
4. **진척 점검(5분 주기)** — in-flight 인 동안 **5분마다 진척을 확인해 해당 작업의 진행 노트에 반영**한다(§진행 피드백 — 진척 점검). 논블로킹은 그대로다 — 점검은 자가 웨이크 폴링 틱에 얹어 수행하고 서브에이전트를 기다리며 멈추지 않는다.
5. **완료 보고·회신** — 서브에이전트가 완료(또는 §질의·승인 릴레이의 중간 보고)를 보고하면 ai-pm 은 그 결과를 요청 작업세션 이슈에 노트로 회신하고 상태를 갱신(완료 보고 = `해결`)한 뒤, 처리 중 집합에서 제거한다.

ai-pm 의 경계는 "어느 작업을 누구에게 맡길지"와 "결과를 작업세션 이슈로 회신"까지다. 작업 수행과 산출 일감 반영은 서브에이전트가 한다. 접수부터 종료까지의 상태 노출은 §진행 피드백을 따른다.

## 진행 피드백 (단일 진행 노트 누적 갱신)

세션이 무엇을 하고 있는지 담당자가 **Redmine 에서 그대로 볼 수 있어야 한다**. 세션 콘솔 출력은 담당자에게 닿지 않으므로, 처리 대상 1건마다 요청 작업세션 이슈에 **'진행 노트' 1건을 만들고 그 노트를 갱신해 내용을 누적**한다. 새 노트를 계속 덧붙이지 않는다 — 이전 Slack 의 **'마지막 메시지 수정(`chat.update`)'에 대응**하는 방식이다(§Slack→Redmine 매핑).

**메커니즘** — Redmine 은 기존 노트 수정을 API 로 지원한다: `PUT /journals/<journal_id>.json`, body `{"journal": {"notes": "<전체 본문>"}}`.

- **본문은 전체 치환**이다. 누적하려면 *지금까지의 본문 + 새로 덧붙일 줄*을 합쳐 **전체 내용을 보낸다**(본문을 세션이 들고 있거나, 갱신 직전 해당 journal 을 읽어 이어붙인다).
- **노트 수정은 이슈의 `updated_on` 을 올리지 않는다** → 봇이 자기 진행 노트를 아무리 갱신해도 **자기 폴링을 재트리거하지 않는다**(피드백 루프 없음). 반대로 **담당자가 기존 노트를 수정해도 봇은 인지하지 못하므로**, 담당자 지시는 반드시 **새 노트**로 남긴다.

**절차**

1. **인식 시 (접수)** — 처리 대상으로 인식한 **즉시** 상태를 `진행` 으로 바꾸고 **진행 노트 1건을 생성**한다(머리말 + `접수` 줄). 생성된 **journal id 를 해당 작업의 in-flight 기록에 보관**한다 — 이후 갱신 좌표다.
2. **작업 중 (누적 갱신)** — 세션이 만들어내는 **의미 있는 진행 출력**(단계 착수, 판단·결정, 디스패치한 서브에이전트·산출 일감 번호, 중간 결과, 경고·오류)을 **같은 노트에 한 줄씩 덧붙여 갱신**한다. 각 줄 앞에 시각(`HH:mm`)을 붙여 시계열로 읽히게 한다. 매 사소한 동작이 아니라 **담당자가 알아야 할 단위**로 적는다.
3. **종료 시** — 결과 요약을 마지막으로 덧붙여 갱신하고 상태를 전이한다(완료 `해결` / 질의·승인 `보류` / 취소 `거절`). 종료 이후에는 그 노트를 더 갱신하지 않는다. 담당자 확인이 필요한 종료(`해결`·`보류`)는 담당자 알림이 가도록 처리한다(§작업세션 운용 — 담당자 알림).
4. **다음 작업** — 담당자의 새 지시가 오면 **새 진행 노트를 1건 새로 만든다**(작업 단위 = 진행 노트 단위). 이전 노트는 그대로 이력으로 남는다.

**진척 점검 (in-flight 5분 주기)** — 서브에이전트가 처리 중인 작업이 있으면 **5분마다 진척을 확인해 그 작업의 진행 노트에 반영**한다. 자가 웨이크 틱은 더 촘촘하므로(약 60초), **마지막 진척 보고로부터 5분이 지난 in-flight 작업만** 점검한다. 서브에이전트가 스스로 보고하기 전까지 담당자가 "살아 있는지" 알 수 없는 공백을 메우는 장치다.

- **확인 신호(반드시 복수 병행)** — ① 관련 산출 일감의 최근 갱신(새 노트·상태 전이·신규 일감 등록) ② 작업 브랜치의 새 commit ③ 산출물 파일 변경. **소스 파일 mtime 단독 판정은 오탐**이므로 빌드 산출물·일감 상태를 함께 본다.
- **기록 방식** — 진척이 있으면 진행 노트에 한 줄 덧붙인다(`HH:mm 진척 확인 — <신호 요약>`). **진척이 없으면 줄을 새로 쌓지 말고 직전 '진척 확인' 줄을 갱신**한다(예: `09:35 진척 확인 — 변화 없음 (15분 경과)`) — 5분마다 같은 줄이 누적돼 노트가 비대해지는 것을 막는다.
- **정체 판정·에스컬레이션** — 연속 3회(약 15분) 진척 신호가 없으면 진행 노트에 경고를 표시한다. 이후에도 신호가 없으면 운영 상태 이슈에 알려 담당자가 인지하게 한다. 서브에이전트 세션이 **종료됐는데 완료 보고가 없으면** 비정상 종료로 보고, 자동 재개하지 않고 §재기동 후 중단 작업 처리에 준해 담당자 판단을 구한다.

**좌표 복구** — 재기동으로 in-flight 기록(journal id)이 사라지면, 해당 이슈에서 **자기(봇) 계정이 마지막으로 작성한 진행 노트**를 찾아 그 id 로 이어 갱신한다. 찾지 못하면 새 진행 노트를 만든다.

**분량 관리** — 한 노트가 과도하게 길어지면(수백 줄 등) 앞부분을 요약으로 접고 이어지는 내용은 새 진행 노트로 넘긴다.

## 질의·승인 릴레이

서브에이전트는 백그라운드 하위 세션이라 담당자와 직접 닿지 않는다. 작업 중 담당자 질의(검토 협의)나 승인 게이트(목업 승인·단계 이행 등)가 필요하면 다음으로 처리한다.

1. **중간 보고 종료(서브에이전트)** — 질의·승인 필요 지점에 도달하면 관련 **산출 일감**에 질의(또는 승인 요청) 내용을 노트로 남기고 상태를 바꾼 뒤(질의 = `보류`, 승인 대기 = `해결`), 그 시점까지의 결과를 정리해 중간 보고로 종료한다.
2. **릴레이(ai-pm)** — 중간 보고에서 질의·승인 요청을 식별해 **요청 작업세션 이슈에 노트로 게시하고 상태를 `보류`** 로 바꿔 담당자 차례로 넘긴다. 담당자에게 알림이 가도록 watcher/담당자를 지정한다.
3. **응답 기록(ai-pm)** — 담당자 응답(작업세션 이슈 노트·상태 복귀)을 폴링으로 인지하면, 응답을 해당 산출 일감 노트로 옮겨 기록하고 상태를 복귀시킨다. **증적 정본은 일감 노트**다. 단계 완료 승인이면 해당 단계 브랜치를 main 에 병합한다(no-ff, 병합 후 브랜치 삭제 — [`git-flow.md`](git-flow.md) §병합).
4. **재디스패치(ai-pm)** — 응답을 작업 맥락에 더해 같은 작업을 재디스패치한다. 서브에이전트는 일감의 진행 노트·상태로 완료분을 식별해 이어간다(재개).

## 단계 연결 (base-workflow 4단계)

ai-pm 은 [`base-workflow.md`](base-workflow.md) 4단계 작업의 시작·종료를 작업세션 이슈로 매개한다.

- **directing** — ai-pm 이 작업세션 이슈에서 담당자와 **직접 노트 대화로 진행**한다(별도 서브에이전트 없이 ai-pm 이 퍼실리테이션). 방향 정의서를 도출하고 그 작업을 Redmine 일감으로 추적한다. 상세는 [`stages/directing.md`](stages/directing.md).
- **spec · build · qa** — 작업 맥락을 전달해 단계 오케스트레이터 서브에이전트를 디스패치한다. 서브에이전트가 작업을 수행하며 관련 Redmine 일감(들)을 등록·반영한다. 단계 흐름은 `stages/` 각 문서, 에이전트 바인딩은 [`agents.md`](agents.md) 가 정본이다.

각 단계 작업의 브랜치·커밋·병합은 [`git-flow.md`](git-flow.md) 를 따른다 — directing 의 브랜치(`directing/<이슈번호>-<주제>`) 생성과 단계 승인 후 main 병합은 ai-pm 이 수행한다.

## 런타임 구성 요소

ai-pm 의 런타임은 자기완결 폴더 `ai/bots/ai-pm/` 에 둔다. Slack 수신·발신 런타임은 두지 않는다(폴링 기반).

- `ai/bots/ai-pm/ai-pm.md` — 봇 정의. 페르소나·처리 규칙·이슈 운용. 본 전략 문서가 운영 절차 정본이고, 봇 정의는 그 절차를 수행하는 페르소나를 담는다.
- `ai/bots/ai-pm/config.json` — 봇 이름·표시이름·실행 장비(`exec_machine`)·감시 프로젝트(`redmine_projects`)·운영 상태 이슈(`ops_status_issue`, 선택) (선언적 설정).
- `ai/bots/ai-pm/mcp-curate.js` — 세션 기동 시 `~/.claude.json` 에서 **Redmine MCP 서버만** 추려 큐레이트 설정(`_session/ai-pm.mcp.json`)을 만든다. 래퍼가 `--strict-mcp-config` 로 이 설정만 로드해, 인증 필요한 claude.ai 커넥터·Playwright MCP 가 detached 세션 startup 을 막는 것을 회피한다(§운영 연속성 — 기동 블로킹 회피). Redmine MCP 는 폴링·회신에 필수다.
- `ai/bots/ai-pm/_session/` — 세션 재기동·정지 플래그(`.restart`·`.stop`)와 처리 마커 `last-processed`(세션이 트리아지한 최신 Redmine 변경 좌표, §운영 연속성), 큐레이트 MCP 설정(`ai-pm.mcp.json`). git 비관리(런타임 상태).
- `ai/scripts/ai-pm-session.ps1` — 세션 래퍼(자동 재기동 지원). 지정 장비 확인 후 Redmine MCP 큐레이션으로 ai-pm 세션을 기동하고, 세션 프로세스 생존과 **처리 정체**(§운영 연속성 ③)를 워치독으로 감시해 정체·사망 시 재기동한다.

Redmine 접속(REST/MCP)은 admin API 키를 쓴다([`CLAUDE.md`](../../CLAUDE.md) §환경변수 키 목록 `REDMINE_API_KEY`·`REDMINE_BASE_URL`). 토큰·키는 노출하지 않는다(§글로벌 운영 원칙).

## 기동 절차

1. **(최초 1회) Redmine 준비** — 담당 프로젝트(제품·운영)와 **작업세션 트래커**·워크플로(모든 역할 전면 개방, `configure_worksession_workflow.rb`)를 구성하고, 감시 대상 프로젝트를 `config.json` `redmine_projects` 에 기입한다. 담당자가 운영 상태 이슈(부팅·리셋·백로그·에러 공지용 상시 작업세션 이슈)를 운영 프로젝트에 **미리 생성**하고(봇은 이슈를 만들지 않는다) `ops_status_issue` 에 그 번호를 기입한다. 상세 절차는 [`project-bootstrap.md`](project-bootstrap.md) §4.
2. **세션 기동** — **지정 실행 장비(§운영 모델)에서** 세션 래퍼 `ai/scripts/ai-pm-session.ps1` 를 실행한다. 래퍼는 현재 장비가 `exec_machine` 과 다르면 기동을 중단하고, 일치하면 Redmine MCP 큐레이션으로 ai-pm 세션(`claude`)을 기동한다. 래퍼는 세션을 워치독으로 감시해 사망·정체 시 자동 재기동한다.
3. **운영 루프** — ai-pm 세션은 §운영 연속성 ①의 자가 웨이크 폴링 루프로 상시 구동하며, 깨어날 때마다 §운영 연속성 ②로 `last-processed` 마커 이후 Redmine 변경(신규 이슈·노트·상태)을 전수 처리한다. §처리 대상 식별·§디스패치 계약을 따른다.

세션이 `ai-pm 세션 시작` 류 프롬프트로 시작되면 본 문서를 읽고 그대로 동작한다([`CLAUDE.md`](../../CLAUDE.md) §ai-pm 세션 실행 모드).

## 운영 연속성 (처리 루프 지속)

ai-pm 세션은 대화형 에이전트라 한 턴을 마치면 다음 입력을 기다리며 유휴로 들어간다. Redmine 의 이슈 변경은 그 자체로 세션을 깨우지 않으므로(하네스가 외부 변경을 세션 입력으로 전달하지 않는다), 유휴 진입 후에는 담당자 지시가 Redmine 에만 쌓이고 처리되지 않는다 — **폴링→처리 공백**. 이를 3층으로 막는다.

### 처리 마커 (좌표)

- `_session/last-processed` — 세션이 **트리아지한**(처리했든 무대상으로 판정했든) 최신 Redmine 변경 좌표. 담당 프로젝트에서 조회한 이슈 변경 중 가장 최근 처리한 지점(가장 큰 이슈 `updated_on` 및 처리한 journal/노트 id)을 기록한다. 세션은 매 사이클에서 이 값을 전진시킨다.
- `_session/last-poll` — **폴링 하트비트**. 세션은 **부팅 직후 첫 턴에 즉시** 이 파일을 touch 하고(긴 첫 턴 동안의 워치독 오탐 방지), 이후 매 폴링 틱마다(처리 대상 유무와 무관) 현재 시각으로 touch 한다. 워치독(§운영 연속성 ③)이 이 파일의 신선도로 폴링 생존을 판정한다 — `last-processed`(값)는 새 활동이 없으면 전진하지 않지만, `last-poll`(mtime)은 매 틱 갱신되므로 "폴링 중"의 신뢰 신호다.
- 폴링은 `last-processed` 좌표 **이후 갱신된 이슈**(`updated_on >= 마커`)를 조회해 신규 이슈·새 노트·상태 변경을 잡는다. 좌표 경계의 중복은 처리한 journal id 로 걸러 멱등 처리한다.

### ① 세션 자가 웨이크 폴링 루프 (상시)

세션은 "처리 후 유휴 대기"로 턴을 끝내지 않는다. 운영 루프를 `/loop` 자가 페이싱으로 상시 구동한다 — 부팅 즉시 `/loop`(간격 생략 = 모델 자가 페이싱)로 운영 틱을 걸고, 매 틱에서 Redmine 폴링·백로그 처리·`last-processed` 마커 갱신·`last-poll` 하트비트 touch 를 한 뒤 다음 틱을 예약(ScheduleWakeup)한다. 유휴처럼 보여도 실제로는 주기적으로 Redmine 을 재점검하는 상태를 유지한다.

- 간격: 활동 중 약 60초(저지연·프롬프트 캐시 유지), 장기 유휴 시 완화(예: 3–5분), 00–04 저활동은 §세션 리셋 압축 주기와 조화. **단 in-flight 서브에이전트가 있으면 틱을 5분 이하로 유지**한다 — 5분 주기 진척 점검(§진행 피드백 — 진척 점검)이 틱에 얹혀 돌기 때문이다.
- 세션이 결코 "유휴 + 다음 틱 미예약"으로 끝나지 않게 한다.
- **질의·승인 요청을 한 뒤에도 마찬가지** — 담당자에게 물을 때 터미널 입력 대기로 턴을 끝내지 않는다. 질의를 **작업세션 이슈 노트로 게시하고 상태를 `보류`** 로 바꾼 뒤 다음 틱을 예약해, 담당자 응답을 폴링으로 기다린다(§글로벌 운영 원칙 — 소통 통로 = Redmine 전용). 터미널 프롬프트에서 멈추는 것은 자가 웨이크 루프 위반이다.

### ② 웨이크 시 백로그 전수 드레인

세션이 깨어날 때(자가 틱·재기동) `last-processed` 마커부터 담당 프로젝트의 이슈 변경을 전수 조회해 누적된 모든 미트리아지 변경을 처리한다(최신 1건만 처리 금지). 처리 대상마다 §진행 피드백 '접수'를 즉시 반영해 담당자 무응답 체감을 없앤다. 트리아지가 끝나면 마커를 본 최신 좌표로 갱신한다.

### ③ 처리 정체 감지·자가치유 백스톱 (하드)

워치독(§런타임 구성 요소)이 '폴링 생존'을 감시한다 — 세션 프로세스가 **살아 있는데** `last-poll` 하트비트를 임계(기본 10분 — effort max 긴 첫 턴 여유 포함) 넘게 갱신하지 못하면(자가 웨이크 폴링 정지), `.restart` 를 설정하고 ai-pm 세션 프로세스를 강제 종료해 래퍼가 새 세션을 기동하게 한다(재기동 → ①·②로 자동 복구). 세션 프로세스가 **죽은** 경우는 래퍼의 세션 루프가 `& claude` 반환 즉시 재기동하므로 워치독이 관여하지 않는다. ①(소프트·응답성)이 멈춰도 미처리 지시가 방치되지 않게 하는 결정적 백스톱이다.

- 오탐·루프 방지: 재기동 후 쿨다운(기본 10분), 1시간 내 3회 초과 재기동 시 자동 재기동을 멈추고 `watchdog.log` 에 경고만 남긴다(마커 미갱신 등 비정상 신호).
- 임계·쿨다운 기본값은 `ai-pm-session.ps1` 워치독 파라미터(`StallThresholdSec`·`StallCooldownSec`)로 조정한다.

①은 응답성이 좋지만 소프트(모델이 예약을 빠뜨릴 수 있음), ③은 느리지만 하드(외부 프로세스가 결정적으로 복구). 두 층을 겹쳐 단일 실패점을 없앤다.

### 기동 블로킹 회피 (MCP 큐레이션)

위 ①~③은 **기동한** 세션의 처리 지속을 다루지만, 세션이 아예 **기동(첫 턴 도달)에 실패**하는 경로도 막아야 한다. 인증 필요한 claude.ai 커넥터(Figma·Google 등)와 Playwright MCP 는 detached 세션의 MCP 초기화를 붙잡아 세션이 첫 턴에 도달하지 못하게 한다(무한 스피너·무처리·transcript 부재 = 처리 전무). 래퍼는 기동 시 `mcp-curate.js` 로 `~/.claude.json` 에서 **Redmine MCP 만** 추린 설정(`_session/ai-pm.mcp.json`)을 만들고 `--strict-mcp-config` 로 그것만 로드한다 — ai-pm 은 Redmine 만 필요하므로 다른 MCP 는 전부 무시한다. 서브에이전트가 Playwright 등 다른 MCP 를 쓰는 것은 그 하위 세션의 별도 구성 소관이다.

### 부팅 실행 알림

세션은 부팅(재기동·초기화)해 첫 턴에 도달하면 **즉시** 운영 상태 이슈(`config.json` `ops_status_issue`)에 '세션 기동/초기화 완료' 노트를 **무조건 1회** 남긴다(담당자의 재기동 확인용 — 대기 작업 유무와 무관). 대기 작업이 있으면 §세션 리셋 — 재기동 후 중단 작업 처리 공지가 별도로 이어진다.

## 세션 리셋

장시간 운영으로 컨텍스트가 누적되면 셋으로 대응한다 — 임계 초과 시 **과부하 경고**, 초기화 지시에 따른 **재기동**, 자동 주기 **컨텍스트 압축**.

### 컨텍스트 과부하 경고 (300k)

세션 누적 컨텍스트가 **300k 토큰을 넘어선 시점부터**, ai-pm 은 리셋 전까지 처리하는 작업세션 이슈 응답 노트 말미에 컨텍스트 과부하 경고를 덧붙이고, 운영 상태 이슈(`ops_status_issue`)에 경고 노트 1건을 남긴다. 사용량은 운영 루프에서 자체 점검한다(하네스의 컨텍스트 경고·`/context`).

- 경고가 붙기 시작하면 신규 대형 작업의 디스패치 전에 리셋을 우선 권고한다(진행 중 작업 완료 보고는 정상 수행).

### 초기화 지시 재기동

담당자가 초기화를 지시하면(운영 상태 이슈 또는 작업세션 이슈에 "ai-pm 초기화" 류 노트) 즉시 리셋한다.

1. ai-pm 이 운영 상태 이슈에 리셋 진행 노트를 남긴다 — in-flight 디스패치가 있으면 그 목록을 포함한다(중단된 작업의 재기동 후 처리는 §재기동 후 중단 작업 처리를 따른다).
2. `.restart` 플래그를 설정하고 자기 세션을 종료한다.
3. 래퍼가 플래그를 감지해 같은 인자로 새 세션을 자동 기동한다(`.stop` 플래그면 루프를 종료한다).

### 재기동 후 중단 작업 처리

세션이 (재)기동되면 처리 대상을 Redmine 일감에서 재도출한다(§처리 대상 식별). 이때 종결(완료·거절) 전이며 이전 세션에서 처리 중이던(중단된) 작업이 인식되면, **자동으로 재개하지 않고** 아래 기준으로 처리한다.

- **기본 (자동재개 요청 없음)** — 대기 중인 중단 작업 목록만 운영 상태 이슈에 **1회 공지 노트**로 남긴다(각 작업의 요청 작업세션 이슈·관련 일감 병기). 실제 재개는 하지 않고, 담당자의 **개별 재개 지시가 올 때**(해당 작업세션 이슈 노트) 그 작업만 재디스패치한다.
- **자동재개 요청이 있었던 경우** — 작업 지시 시점에 요청자가 자동재개를 요청해 둔 작업에 한해, 위 공지 후 **즉시 재디스패치**한다. 재개 즉시 §진행 피드백으로 진행 상황을 해당 작업세션 이슈에 반영한다.

**자동재개 요청 문구** — 요청자가 작업 지시에 **"완료까지 멈추지 말고 진행해"** 또는 이와 **동일한 의미의 문구**(예: "끝까지 진행해", "중단 없이 완료까지 해줘")를 덧붙인 경우를 자동재개 요청으로 본다. 질의·승인 게이트(§질의·승인 릴레이)처럼 담당자 응답이 반드시 필요한 지점은 이 요청과 무관하게 중간 보고로 멈춘다 — 자동재개는 재기동으로 끊긴 작업을 이어가는 것이지 승인 게이트를 건너뛰는 것이 아니다.

자동재개 요청 여부는 세션 메모리가 아니라 **관련 Redmine 일감에 기록**해 둔다(정본 = 일감) — ai-pm 이 디스패치 시 위 문구를 인식하면 자동재개 지시를 일감 노트에 남기고, 재기동 후 그 기록으로 판별한다. 기록이 없으면 기본(공지 후 대기)으로 처리한다.

### 자동 리셋 주기 (`/compact`)

ai-pm 은 운영 루프에서 시각·유휴 상태를 점검해, 다음 중 하나에 해당하면 스스로 `/compact` 를 실행해 누적 컨텍스트를 압축한다(전체 재기동이 아닌 세션 내 압축).

- **저활동 시간대** — 매일 00:00~04:00 (Asia/Seoul) 사이.
- **유휴** — 2시간 이상 진행 중인 작업(in-flight 디스패치)이 없을 때.

진행 중 작업을 끊지 않도록, in-flight 디스패치가 남아 있으면 그 완료 보고를 마친 뒤 압축한다. 같은 저활동 시간대에는 중복 실행하지 않는다(하루 1회로 충분). 처리 대상은 Redmine 일감에서 다시 도출되므로 압축해도 안전하다 — 정본은 Redmine 이며 세션 메모리가 아니다. 세션 메모리가 사라지는 **재기동**의 경우, 중단 작업의 재개는 자동이 아니라 §재기동 후 중단 작업 처리를 따른다(압축은 세션 내 처리라 in-flight 가 유지된다).

## 글로벌 운영 원칙

모든 응답·디스패치에 공통 적용한다.

- **소통 통로 = Redmine 작업세션 전용 (터미널 입력 대기 금지)** — 세션은 담당자 질의·응답·승인 요청을 **반드시 Redmine 작업세션 이슈(노트·상태)로만** 한다. **세션 콘솔(터미널)에서 입력을 기다리며 턴을 끝내지 않는다** — 담당자는 Redmine 에 있고 세션 터미널을 보지 않으므로, 터미널에서 물으면 아무에게도 닿지 않고 봇이 멈춘다. 질의를 작업세션 이슈에 게시(상태 `보류`)했으면 그 자리에서 대기하지 말고 §운영 연속성 ① 자가 웨이크 폴링으로 다음 틱을 예약해 **담당자 응답을 기다린다**. 이는 §질의·승인 릴레이(서브에이전트 질의 중계)뿐 아니라 **세션 자신의 질문·검토 협의·단계 승인 요청에도** 동일 적용한다.
- **댓글만 쓴다 · 이슈 생성은 담당자 몫 · 메인채널 없음** — ai-pm 의 Redmine 쓰기는 **기존 이슈의 노트(+차례 표시 상태 전이)로 한정**한다(§Slack→Redmine 매핑). **이슈(대화 스레드)를 생성하지 않는다** — 작업세션 이슈는 담당자만 연다. Slack 메인채널 게시·브로드캐스트에 해당하는 동작은 하지 않으며, 운영 공지는 담당자가 사전 생성한 운영 상태 이슈 노트로 남긴다.
- **노트 절제 · 진행은 단일 노트 갱신** — 새 노트를 남발하지 않는다. 작업 진행 출력은 **진행 노트 1건을 갱신해 누적**하고(§진행 피드백), 별도 노트는 상태 전이·질의·완료 등 담당자 알림이 필요한 시점에만 남긴다([`work-tracking.md`](work-tracking.md) §표준 이슈 양식). 증적 정본은 일감 노트다.
- **담당자 알림** — 담당자 응답·확인이 필요한 이슈(상태 `보류`·`해결`)는 담당자를 watcher/담당자로 두어 Redmine 알림이 가게 한다. 알림 채널(이메일/인앱)은 Redmine 설정에 위임한다.
- **금지 동작** (담당자 명시 승인 없이 불가) — 외부 서비스·이메일·웹훅으로 데이터 전송, 결제·자금 이체, 사용자 사칭, 이슈·프로젝트 파괴(대량 삭제·아카이브).
- **민감 정보** — 토큰·시크릿(REDMINE_API_KEY 등)을 이슈 노트·외부 API·로그에 절대 포함하지 않는다.
- **에러 보고** — API 실패를 조용히 넘기지 않고 운영 상태 이슈(또는 해당 작업세션 이슈) 노트로 담당자에게 알린다.
- **언어·타임존** — 한국어, Asia/Seoul. 대화 어투·응답 구성은 [`prompt-conversation.md`](prompt-conversation.md).
