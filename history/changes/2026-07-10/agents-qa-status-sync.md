- **왜**: qa 실행 후 대상 `검증` 일감의 상태가 갱신되지 않아 정상 실행 여부를 확인하기 어려웠다. 정책(work-tracking §매핑 표)은 이미 `해결=Pass`·`완료=담당자 확인` 이었으나, 실행 지침이 "5종 판정에 따라 동기화" 로 **간접 참조**만 해 실제 갱신이 누락됐다. build 처럼 완료 항목에 결과·상태를 반영하되, qa 최종 완료는 사람이 하도록 실행 주체를 `해결`까지로 못박는다(담당자 지시 2026-07-10).
- **영향**: qa 실행 주체(오케스트레이터·tester)의 상태 전이가 `신규→진행→해결`(실행 완결)까지로 명확해지고, `완료`(닫힘)는 담당자 승인 게이트로 분리. build(`기능` 일감)와의 대칭성 확보. 정책 단일 출처는 work-tracking §단계 산출 일감 상태 매핑.
