# strategies-ai-pm-two-tier-watcher (2026-07-19)

> ai-pm 을 **상시 LLM 세션 → 2계층(코드 워처 + 작업 시 세션)** 으로 전환. 컨텍스트 누적을 '영구'에서 '작업 1사이클'로 제한. 본문 정본은 `ai-pm.md` §운영 모델(2계층).

- **무엇**:
  - [`ai-pm.md`](../../../ai/strategies/ai-pm.md) §운영 모델 — **§2계층 구조** 신설. 1계층 워처(래퍼·PowerShell)가 Redmine 을 코드 폴링해 **작업 유무만 판정**(토큰 0), 2계층 세션(claude·effort max)은 **작업이 있을 때만 기동**. **세션 수명 = in-flight 가 있는 동안**이며, 비면 종료한다. 세션 정상 종료는 실패가 아니라 작업 사이클 완료다.
  - [`ai-pm.md`](../../../ai/strategies/ai-pm.md) §트리거 — 폴링 주체를 세션에서 **워처**로 이관. 워처 판정 규칙 3종(① inflight 비어있지 않음 ② 작업세션 상태 `신규`·`의견` ③ 워터마크 초과 + **작성자 ≠ 봇** 노트).
  - [`ai-pm.md`](../../../ai/strategies/ai-pm.md) §운영 연속성 — **처리 마커 → §세션 상태(`_session/state.json`)** 로 재작성. `watermark_journal_id`(세션이 사이클 종료 시 전진 의무) + `inflight[]`(**선기록 원칙** — 디스패치 직전 기록해 비정상 종료 시 인계). **종료 전 불변식** 신설(작업세션 이슈를 `진행` 으로 남기지 않는다). 구 `last-processed` 자유서술 마커 폐지.
  - [`ai-pm.md`](../../../ai/strategies/ai-pm.md) §운영 연속성 ① — '상시 자가 웨이크 루프' → **'in-flight 가 있는 동안'** 으로 한정. 백로그·in-flight 가 비면 **틱을 예약하지 않고 종료**. 질의·승인 후에도 그것이 마지막 in-flight 면 종료(담당자 응답은 워처가 감지).
  - [`ai-pm.md`](../../../ai/strategies/ai-pm.md) §런타임 구성 요소·§기동 절차 — 래퍼를 '세션 래퍼 겸 1계층 워처'로 재정의, 기동 절차를 워처 기동 → 작업 사이클 반복으로 재작성. Redmine 접속 불가 시 **상시 세션 모드로 폴백**(구 동작 보존).
  - [`ai-pm.md`](../../../ai/strategies/ai-pm.md) §부팅 실행 알림 — 사이클마다 부팅 노트를 남기면 운영 상태 이슈가 잡음으로 덮이므로, **래퍼 기동 시 첫 세션과 비정상 종료 복구 기동에만** 남기도록 한정.
  - [`ai-pm.md`](../../../ai/strategies/ai-pm.md) §세션 리셋 — 컨텍스트 장치들이 상시 대응책이 아니라 **한 사이클이 비정상적으로 길어질 때의 안전장치**임을 명시.
  - [`ai/scripts/ai-pm-session.ps1`](../../../ai/scripts/ai-pm-session.ps1) — 워처 구현. `Read-AiPmState`·`Get-Watermark`·`Get-InflightCount`·`Test-AiPmWork` 추가, 메인 루프를 **워처 대기 → 감지 시 세션 기동 → 종료 시 워처 복귀**로 전환. 워터마크 미전진 세션이 3회 연속이면 5분 백오프(무한 재기동 방지). 기존 워치독·모델/effort 추출·MCP 큐레이션·`.restart`/`.stop` 은 그대로.
  - [`ai/bots/ai-pm/config.json`](../../../ai/bots/ai-pm/config.json) — 워처 설정 `worksession_tracker_id`(7)·`watch_interval_sec`(60)·`post_run_cooldown_sec`(15) 추가.
  - `ai/bots/ai-pm/_session/state.json` — 신설(git 비관리). 스키마 `ai-pm/state@1`, 기존 워터마크 2662 승계.
  - [`ai/bots/ai-pm/ai-pm.md`](../../../ai/bots/ai-pm/ai-pm.md) §처리 절차 — step 0(기동 = 작업 있음이므로 즉시 처리 착수, 루프는 in-flight 생긴 뒤), step 1(state.json 승계), step 5(디스패치 선기록), step 9(사이클 종료 = 워터마크 전진 + inflight 정리 + **비면 종료**).

- **왜**: 담당자 지적 — **지속적인 Redmine 목록 조회로 컨텍스트가 무한 누적**된다. 원인은 하루 ~1440 틱 중 실제 작업이 있는 틱이 극소수인데 '변화 없음' 확인에 매번 `effort: max` LLM 턴을 쓴 것. 그 판정(워터마크 초과·작성자·트래커)은 **LLM 판단이 불필요한 순수 코드 조건**이라 워처로 내렸다. 담당자가 제시한 두 안(서브에이전트 분기 / 별도 스케줄러) 중, 전자는 부모 세션이 영구 생존해 누적이 남고 후자는 매 틱 콜드스타트 비용 + **서브에이전트가 세션의 자식이라 세션 종료 시 진행 중 작업이 유실**되는 문제가 있어, 둘의 장점을 합친 2계층을 채택했다(담당자 승인).

- **영향**: ai-pm 런타임 전반. 유휴 시 LLM 세션이 아예 뜨지 않으므로 유휴 토큰 비용이 0 이 된다. 감수 사항 — ① directing 다중 턴 대화가 세션 경계를 넘으면 맥락이 Redmine 노트로만 복원된다(정책상 "정본은 Redmine"에 부합) ② 작업 사이클마다 지침 로드 비용 발생(작업이 있을 때만이라 총량은 감소) ③ 응답 지연은 워처 주기(60초)로 종전과 동일.
  - **사전 검증(2026-07-19)**: 래퍼에서 워처 함수를 실제 로드해 라이브 Redmine 대상으로 3경로 확인 — 담당자 새 노트 감지 ✔(#407 journal 2073) / in-flight 존재 시 강제 기동 ✔ / **봇 자기 노트는 미감지 ✔**(자기 트리거 방지). 현재 상태에서는 '작업 없음' 으로 판정돼, 새 모델이면 세션이 떠 있지 않을 상황임을 확인했다.
  - **활성화**: 래퍼 자체를 교체해야 하므로 **워처·워치독·세션을 모두 내리고 새 래퍼로 재기동**해야 반영된다(세션 재기동만으로는 부족).

- **관련 일감**: 없음 — 워크플로(운영 문서·런타임) 개선이라 Redmine 미등록.
