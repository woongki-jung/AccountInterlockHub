# strategies-build-qa-issue-close (2026-07-19)

  - [`work-tracking.md`](../../../ai/strategies/work-tracking.md) §단계 산출 일감 상태 매핑 — 매핑 표(기능·검증 row) 갱신 + **원칙 문장 신설**("추가 확인이 필요 없는 건은 실행 주체가 `완료`까지 정리한다. `해결`은 '확인이 더 남았다'는 뜻으로만 쓴다"). 규칙: **🟢 Pass→`완료`(닫힘)** / 🔵🟣 잠정 Pass→`해결`(재검증 사유·해소 조건 필수) / 🔴 Fail→`진행` / 🟠 Block→`보류`, **기능(build Phase)** 은 세 책임 통과 + 잔여 확인 항목(잠정 Pass·보류·사양 결함 제기·담당자 판단 대기) 없음 → `완료`. **개별 일감 종결 ≠ 단계 승인** 문단과 **닫을 때 지킬 것**(하위 먼저·GET 실측) 추가.

  - [`work-tracking-redmine.md`](../../../ai/strategies/work-tracking-redmine.md) §도구 함정 — 종결 동작이 늘어남에 따라 실측 함정 2건 명문화: **`update_issue` 가 부분 실패를 `ok:true` 로 삼킴**(상태 전이는 `GET` 의 `status`·`closed_on` 으로 검증), **열린 하위가 있으면 부모 close 거부**(하위 먼저).

- **왜**: 담당자 지적 — 산출 일감이 최종적으로 `해결` 에만 머물러, **판정이 확정된 건과 확인이 남은 건이 같은 상태에 뒤섞였다.** `해결` 더미에서 담당자가 실제로 봐야 할 건을 골라내야 했고, 완료 현황이 Redmine 에서 한눈에 잡히지 않았다. `해결` 을 "확인 대기" 신호로 되돌리고, 확정 건은 실행 주체가 닫아 담당자 시야에서 치운다.

- **영향**: build·qa 실행 전반(오케스트레이터·tester). 기존 "완료(닫힘)는 담당자 확인 전용" 규정을 대체한다 — 단, **단계 이행·배포 가능 판정은 종전대로 담당자 승인 사항**이며(승인 대상 = 단계 그룹 일감·작업세션 이슈), 개별 TC·Phase 종결이 단계 통과를 뜻하지 않음을 3개 문서에 교차 명시했다. spec 산출(`사양` 일감)의 상태 정책은 이번 개정 범위 밖(교차검증 통과 → `해결`, 담당자 승인 → `완료` 유지). 2026-07-10 [`agents-qa-status-sync`](../2026-07-10/agents-qa-status-sync.md) 의 "실행 주체는 `해결`까지" 규정을 본 개정이 갱신한다.

- **관련 일감**: 없음 — 워크플로(운영 문서) 개선이라 Redmine 미등록.
